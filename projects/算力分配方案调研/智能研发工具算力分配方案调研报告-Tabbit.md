## 1. 调研背景与目标

### 1.1 背景

作为银行软件开发中心智能研发的牵头人，当前面临以下核心痛点：

- **算力资源不足**：私有化部署的智能研发工具算力有限，难以满足全部用户需求
- **用户需求量大**：研发团队对智能编码工具的需求持续增长，使用频率和强度不断提升
- **分配机制缺失**：缺乏科学合理的算力分配方案，导致资源使用效率不高

### 1.2 调研目标

本次调研旨在：

1. 分析国内外主流智能编码工具（含类龙虾 Claw 工具）的收费模式与算力分配机制
2. 重点关注 Token 经济模型和免费策略的可借鉴经验
3. 设计适合行内私有化部署场景的算力分配方案
4. 对比各方案优缺点，形成推荐方案

### 1.3 调研范围

- **国际工具**：GitHub Copilot、Cursor、Codeium（Windsurf）等
- **国内工具**：通义灵码、阿里云百炼、腾讯云混元、Kimi、DeepSeek 等
- **开源工具**：OpenClaw（龙虾）及其派生产品
- **核心关注**：Token 计费模式、免费策略、配额管理机制

## 3. Token 经济与免费策略分析

### 3.1 Token 计费模式演进三阶段

#### ① 明码标价的 API（最初阶段）

- 最透明的计费方式，如实时跳动的电表
- 价格门槛阻碍大规模普及，用户存在"算力焦虑"

#### ② Coding Plan 订阅制（妥协产物）

- 通过"每 5 小时 1200 次请求"等模糊规则包装成包月服务
- 局限性：不透明，1 次提问可能触发几十次模型调用
- 结果：用户体验断裂，高强度使用时突然降智或限流

#### ③ Token Plan（当前主流）

- 小米、腾讯于 2026 年 4 月推出
- 像手机流量包一样明确告知周期内可用 Token 数量
- 优势：兼顾算力紧缺现状和商业利益的唯一解法

### 3.2 Token 价格趋势

**国际模型**（每百万 Token）：

- 轻量级模型：0.20−0.20-0.20−0.75（输入），1.25−1.25-1.25−4.50（输出）
- 通用模型：1.75−1.75-1.75−3.00（输入），14−14-14−15（输出）
- 强力模型：5.00（输入），5.00（输入），5.00（输入），25-$30（输出）

**国内模型**：

- DeepSeek：百万 Token 几块钱（价格屠夫）
- 其他厂商：跟随 Token Plan 趋势，价格逐步市场化

### 3.3 免费策略参考（可借鉴经验）

#### 3.3.1 每日固定 Credit/Token

- Cursor Hobby 计划：免费使用基础功能
- 通义灵码个人基础版：50 轮次/月编码智能体对话
- GitHub Copilot 个人计划：包含不同额度 AI Credits 津贴

#### 3.3.2 任务获取 Token 机制

- 部分平台通过完成新手任务、邀请好友、参与活动等方式赠送 Token
- Get AI Perks 等平台聚合 220+ AI 工具初创公司优惠，总价值$7.6M+ 积分

#### 3.3.3 限时免费体验

- 通义灵码个人专业版限时免费（至 2026 年 5 月 20 日）
- 腾讯云 LightVela 免费提供 1 个月云端 Hermes Agent 服务

### 3.4 可借鉴的核心策略

基于行业调研，以下策略对行内算力分配方案设计具有参考价值：

| 策略         | 核心机制                            | 可借鉴点                   |
| ------------ | ----------------------------------- | -------------------------- |
| **分层配额** | 按用户类型/使用强度划分不同配额层级 | 平衡公平性与效率           |
| **任务激励** | 完成特定任务获取额外 Token          | 提升用户参与度和工具使用率 |
| **弹性调整** | 根据实时负载动态调整分配优先级      | 避免资源浪费               |
| **额度流转** | 允许闲置额度在团队内部流转          | 提升整体资源利用率         |

## 4. 算力分配方案设计

基于行业调研成果，结合行内私有化部署场景，设计以下 4 套算力分配方案：

### 4.1 方案一：基础配额制

**核心机制**：

- 为所有用户分配固定基础算力 token 额度
- 月度清零，不可累积
- 规则简单，统一管理

**示例配置**：

code

复制

```text
所有用户：500 token/月
月度清零，不可结转
超额后暂停使用至下月
```

**优点**：

- 规则简单易理解，用户接受度高
- 资源分配公平透明，无争议
- 管理成本低，易于实施

**缺点**：

- 无法匹配差异化需求
- 轻度用户资源浪费
- 重度用户额度不足，影响核心业务

**适用场景**：工具推广初期，用户使用习惯未形成

------

### 4.2 方案二：阶梯 Token 池制

**核心机制**：

- 按用户角色/团队规模划分 3-5 个算力池
- 不同池分配不同额度
- 部门级成本核算

**示例配置**：

code

复制

```text
普通研发人员：300 token/月
核心研发人员：800 token/月
技术专家/架构师：1500 token/月
团队负责人：2000 token/月（含团队调配权）
```

**优点**：

- 匹配差异化使用需求
- 核心团队获得充足资源保障
- 便于部门级成本核算和管理

**缺点**：

- 可能引发资源分配争议
- 灵活性不足，难以应对突发需求
- 角色界定可能存在模糊地带

**适用场景**：团队架构稳定，使用强度差异明显

------

### 4.3 方案三：贡献兑换制

**核心机制**：

- 基础配额 + 任务兑换 token
- 通过完成代码提交、评审、工具优化等任务获取额外算力
- 资源向高贡献者倾斜

**示例配置**：

code

复制

```text
基础配额：300 token/月（所有用户）


任务兑换规则：
- 代码提交：5 token/次
- 代码评审：10 token/次
- 工具优化建议：50 token/条（采纳后）
- 分享使用经验：30 token/次
- 帮助文档贡献：20 token/篇
- Bug 反馈（有效）：15 token/条


月度兑换上限：1000 token
```

**优点**：

- 激励用户深度使用工具
- 提升研发产出质量和工具使用率
- 资源向高贡献者倾斜，形成正向循环

**缺点**：

- 规则复杂度高，用户理解成本高
- 需建立任务量化体系和审核机制
- 初期推广成本高，需持续运营

**适用场景**：工具成熟阶段，需要提升用户粘性和活跃度

------

### 4.4 方案四：动态优先级制

**核心机制**：

- 基础配额 + 实时算力调度
- 根据用户当前任务紧急度、历史贡献度动态调整算力优先级
- 算力资源利用率最大化

**示例配置**：

code

复制

```text
基础配额：400 token/月（所有用户）


动态调度规则：
- 紧急任务（P0）：算力优先级提升 3 倍，需审批
- 重要任务（P1）：算力优先级提升 2 倍
- 普通任务（P2）：标准优先级
- 闲置 token 流转：每月限额 200 token，部门内部流转
- 月末回收：剩余 token 自动回收纳入公共算力池
```

**优点**：

- 算力资源利用率最大化
- 紧急任务获得优先保障
- 自适应业务需求变化

**缺点**：

- 技术实现复杂度高
- 需实时监控系统支持
- 用户感知度低，可能产生不公平感

**适用场景**：研发任务波动大，紧急需求较多，有技术能力支撑

## 6. 实施建议

### 6.1 分阶段实施计划

**第一阶段：试点验证（1 个月）**

- 选取 2-3 个典型团队进行方案试点
- 收集用户反馈，验证配额参数合理性
- 调整优化任务兑换规则

**第二阶段：全面推广（2-3 个月）**

- 基于试点结果优化方案参数
- 制定详细实施指南和培训材料
- 全中心范围内推广实施

**第三阶段：持续优化（长期）**

- 建立月度复盘机制
- 根据使用情况动态调整配额
- 持续优化任务兑换规则

### 6.2 配套机制建设

**1. 算力使用监控看板**

- 实时跟踪各层级 token 使用情况
- 部门/团队/个人多维度统计
- 异常使用预警（如单日超额、异常高频等）

**2. 用户反馈渠道**

- 建立线上反馈入口
- 定期收集用户意见和建议
- 每季度调整配额参数和兑换规则

**3. 培训与宣导**

- 编制算力分配方案使用指南
- 组织专题培训和答疑
