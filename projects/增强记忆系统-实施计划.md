## 三、四大核心机制

| 机制 | 触发条件 | 核心功能 |
|------|---------|---------|
| **/remember** | 每会话结束自动 | 从对话提炼关键事实 → 写入 `atomic/[type]/` |
| **/dream** | 每3天 heartbeat | 整合、去重、时间标准化、控容 MEMORY.md |
| **/reflect** | 每次失败/纠正 | 写 `memory/reflections/` 反思记录 |
| **/recall** | 每会话开始 | 加载 MEMORY.md + 最新 atomic 条目注入上下文 |

> 注：OpenClaw 为聊天式架构，/remember /dream 等为机制名称而非 slash command。

## 四、记忆格式规范

### 4.1 原子记忆（atomic/）frontmatter 模板

```markdown
---
type: feedback
date: 2026-04-12
importance: 8
topic: 前瞻研究报告流程
---

# 用户确认：报告框架需先确认再动笔

- 内容: 研究报告应先出框架，用户确认后再撰写
- 来源: user
- 场景: 报告制作
- **Why**: 用户强调流程要先确认框架
- **How**: 报告类任务，先输出框架，等用户点头
```

### 4.2 四类记忆定义

| 类型 | 何时保存 | 格式重点 |
|------|---------|---------|
| **user** | 用户角色/偏好/知识缺口时 | 画像描述型 |
| **feedback** | 用户纠正或确认一个非显而易见的方法 | **规则 → Why → How to apply** |
| **project** | 进行中项目的目标/动机/截止 | **事实 → Why → How to apply** |
| **reference** | 外部系统指针（链接/账号/路径） | 链接型 |

### 4.3 反思记录（reflections/）模板

```markdown
# 反思记录 - 2026-04-12

- 任务: [当前任务描述]
- 错误: [具体问题]
- 原因: [为什么会错]
- 修正策略: [下次怎么做]
- 避免方法: [如何预防同类错误]
```

## 五、Dream 机制详解（每3天执行）

### 5.1 触发条件

- heartbeat 检测到距上次 dream > 3天
- 或用户手动触发

### 5.2 执行步骤

1. **扫描**：遍历 `memory/atomic/` 所有子目录，记录文件列表
2. **分类**：按 type 分组，识别同一主题的多个文件
3. **去重**：同一主题多个文件 → 合并到主文件，注释旧版本
4. **时间标准化**："昨天"、"上周" → "2026-04-11"、"2026-04-07"
5. **矛盾处理**：保留最新版本，注释旧版本（不删除）
6. **控容**：MEMORY.md 超过200行 → 压缩过时代条
7. **日志**：记录本次合并了哪几个文件

### 5.3 日记录处理

- 超过30天的日记录文件 → 标记为归档（移入 `memory/archive/`）
- 归档文件仍可被 grep 检索，但不出现在主索引

## 九、与现有系统的关系

### 9.1 与 self-improving skill 的关系

self-improving skill 提供：
- corrections.md 纠正记录
- domains/ 领域知识
- projects/ 项目知识
- heartbeat-state.md 状态追踪

本系统补充：
- atomic/ 四类型结构化记忆（更规范）
- reflections/ 反思记录（更深入）
- Dream 整合机制（self-improving 无此功能）
- 每日报告（主动推送）

两者互补，不冲突。

### 9.2 与现有日记录的关系

- 日记录 `memory/YYYY-MM-DD.md` 作为**原始素材池**保留
- 有价值的记忆提炼到 `atomic/` 对应类型
- 日记录定期归档，不长期占用索引空间
