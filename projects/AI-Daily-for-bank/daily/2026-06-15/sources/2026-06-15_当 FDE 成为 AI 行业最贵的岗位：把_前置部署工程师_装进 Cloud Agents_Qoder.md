# 当 FDE 成为 AI 行业最贵的岗位：把"前置部署工程师"装进 Cloud Agents

> 原文链接：https://mp.weixin.qq.com/s/BCpqg1DAYcp9HeJPWXn9Ww
> 公众号：Qoder

5 月 4 日，两家公司同一天讲了同一件事

2025 年 5 月 4 日，在硅谷的同一个工作日里，Anthropic 和 OpenAI 几乎前后脚地公布了各自的"十亿美元级"企业 AI 合资计划。两家公司在公关稿件里分别用了不同的措辞，但被记者们反复追问之后，他们承认：自己正在大规模招募的核心岗位是同一个——

Forward-Deployed Engineer，前置部署工程师，简称 FDE。

很多研发同学第一眼看到 FDE 这三个字母时，会下意识把它读成 Full-stack Developer Experience。这是一个善意的误读，但和它要做的事情没有任何关系。FDE 不是一个新的 IDE 概念，也不是一个开发者关系岗位。

它是一个在 ToB AI 时代被重新发明出来的、决定模型能否真正"上岗"的角色。

如果你只看一句标语，那么这一波 FDE 浪潮的精神底色其实非常清晰：

"AI 是劳动力，不是软件。"

这是 Rolling AI 合伙人阿甘和刘开在一档播客对谈中反复强调的判断。也正是从这一句开始，业界一些原本沉默的角色：售前、交付、解决方案架构师，突然意识到自己头顶上多了一个新的标签，而这个标签的市场议价权，已经被 OpenAI、Anthropic、Palantir 拉到了之前难以想象的高度。

FDE 的前世今生：从 Palantir 的"驻场工程师"到 AI 时代的"翻译官"

要理解今天的 FDE 为什么这么贵，必须把镜头往前推十几年。

FDE 这个角色，最早是 Palantir 发明的。

在 Palantir 内部，它的全称叫 Forward Deployed Software Engineer (FDSE)，"前置部署"的"前"，指的就是物理意义上的前——他们不在总部写需求文档，而是直接拎着电脑飞到客户的现场，坐在国防部、银行、警察局、医院的办公区，和客户的业务专家一起把数据接通、把系统打通、把流程跑通。

Palantir 的产品本身极其复杂——Foundry、Gotham 这种数据平台不可能用一份"用户手册"交付。能让 Palantir 在情报、军工这种对结果极端敏感的领域立足的，并不是它的算法多厉害，而是它的 FDE 能把一个抽象的数据平台，

翻译成一个具体的、可以让作战指挥官在 30 秒内做出决策的应用。

换句话说，Palantir 的 FDE 不是售前，也不是交付——他们是

长在客户一线、对最终业务结果负责的工程师。

这个模式后来被 Silicon Valley Product Group 反复研究，被 Pragmatic Engineer Newsletter 反复写过，也被无数 ToB 公司模仿过、失败过。

时间快进到 2025。OpenAI 和 Anthropic 几乎在同一时间扣动了同一个扳机。区别只是：他们要"前置部署"的，不再是 Foundry，而是 GPT-4o 和 Claude。模型本身已经足够强了——强到可以通过律师考试、医师考试、写出能跑的代码。但当一家世界 500 强真的把 API key 拿回内网之后，他们发现：

模型不知道自己公司的术语；

模型不知道哪个表是事实表、哪个是临时表；

模型给出的回答没有人为它的错误负责；

模型今天能解决的问题，下周因为业务流程变了又得重做一遍。

最后一公里依然是最难的那一公里。

而这一公里，恰好就是 FDE 的主场。

模型不是生产力，中间差的是 Context、Orchestration 和 Accountability

如果你问一个一线 FDE，他在客户那里到底每天在干什么，他会告诉你：80% 的时间不是在写 Prompt，也不是在 fine-tune 模型，而是在做下面三件事——

第一件事：Context（上下文）。

模型本身并不天然认识"我们公司"。它不知道"中后台"在你们这里指的是供应链系统而不是管理后台，它不知道"老王那个项目"指的是哪个项目，它也不知道你们的"红线财年"截止在每年 3 月 31 日。FDE 的第一项工作，就是把这些散落在 Confluence、钉钉群、口头知识里的业务上下文，

重新组织成一份模型能消费的语料

——它可能是一组检索文档，可能是一份 system prompt，也可能是一组结构化的 schema 定义。

第二件事：Orchestration（编排）。

单次模型调用是没有商业价值的。一个真正能交付业务结果的 AI 系统，往往要做这样一连串动作：先去 CRM 拉取客户档案，再去 ERP 查最近三个月订单，再调内部风控接口跑一次评分，最后让模型基于这一串数据生成一份给客服的话术建议。FDE 的第二项工作，就是

把模型嵌进真实业务流程的链路里

——他需要懂 OAuth、懂消息队列、懂幂等、懂回滚，懂在第三步失败的时候应该如何降级，而不是把链路断在客户脸上。

第三件事：Accountability（对结果负责）。

这是最容易被忽略、却最关键的一环。一个 SaaS 工程师交付的是"系统可用率 99.9%"，但一个 FDE 交付的是"客服响应时长降低 30%"、"风控误判率下降 5pt"、"投资经理写研报的时间从 4 小时缩到 40 分钟"。

他对的不是 SLA，是 KPI。

这意味着他不能在系统跑通之后就拍拍屁股走人——他要做 A/B、要做归因、要在客户业务变化时持续迭代，直到结果真正长出来。

把这三件事放在一起看，你就会理解为什么 OpenAI 和 Anthropic 愿意为一个 FDE 开出比普通工程师高 50%–100% 的薪资。

模型只是一台发动机，而 FDE 是把发动机塞进汽车、再把汽车开上你家门口的那个人。

旧分工松动，新组织在出现

FDE 的崛起背后，其实是一次组织结构层面的重排。

在传统的 ToB 软件公司里，能力链路被拆得很整齐：销售签合同，售前画蓝图，交付实施部署，支持兜底答疑，研发负责产品演进。每个环节都用合同和工单链起来，每一环都对自己负责的那一格 SLA 负责。这种分工在过去 30 年里跑得很稳，因为软件本身是确定性的——交付意味着可执行的二进制和一份签字盖章的验收清单。

但 AI 不是这样。

AI 的价值是

概率性的、迭代性的、和业务深度耦合的。

这意味着：

售前画的蓝图很可能在三周后就过时；

交付部署的"系统"如果不持续调教，效果就会飘；

客户的"上线"不是终点，而恰恰是第一次真正的对齐。

旧分工在松动。那条原本横向切割的能力链路，正在被一个新的、纵向贯通的角色重新串起来——这就是 FDE。他既要能写代码，又要能听懂业务；既要懂模型的脾气，又要扛得住客户的 KPI。这是一个

反劳动分工的角色。

它存在的合法性，恰恰来自于"这件事没法分给三个人做"。

而当一件事没法分给三个人做的时候，紧接着的问题就是：它能不能不分给"人"做？

人做 FDE 太贵了，能不能让 Agent 去做？

这才是真正有意思的地方。

人做 FDE 的天花板是肉眼可见的。一个 FDE 一年能服务的客户数量很有限，他要飞客户现场、要对齐业务、要持续迭代，

他的时间是稀缺资源里最稀缺的那种。

OpenAI 在企业服务侧的扩张速度受限于 FDE 的招聘速度，Anthropic 也是。Palantir 在过去十几年里之所以始终是一家"看起来不大"的公司，本质原因之一就是 FDE 这个角色无法规模化。

但如果我们逐一拆解 FDE 在客户那里干的事，会发现这件事在 2025 年之后，

第一次有了被产品化的可能性：

"持续上下文" → Memory 与长期记忆机制；

"可编排的能力" → Tools、Skills、MCP；

"对结果负责" → 异步执行 + 主动汇报 + 结果回写；

"在客户系统里干活" → 隔离的运行环境 + 完整的工具链；

"今天巡检明天复盘" → 定时调度 + 事件驱动；

"越用越懂业务" → 跨会话的项目记忆与知识沉淀。

这其实就是一个 AI Agent 系统的产品 Spec。也就是说，

FDE 的终极形态，未必是更多的人，而是把 FDE 的能力封装成一台"可编程的劳动力"，让每一个团队、每一个项目，都能拥有自己的 AI FDE。

这正是 Cloud Agents 今天在做的事情。

Cloud Agents：把 FDE 的工作模式产品化

如果我们把 FDE 在客户现场所做的事拆解成几个机制，然后逐一去对应 Cloud Agents 的产品能力，会发现这两件事之间的对应关系几乎是一对一的。

Sessions：FDE 的"工作日"。

一个 Cloud Agent 的 Session，本质上是一段长时间运行的、异步的、可以中途回看的工作。它不是"调用一次模型拿到一个回答"，而是"派一个 Agent 进入云端沙箱、自己去把这件事做完"。这正是 FDE 在现场的工作方式——他不是在等指令，他是在解决一个明确的问题；他可能从早上忙到下午，中间会查表、会试错、会回头改方案，最终在收工时把交付物丢回来。Session 把这个工作日抽象成一个一等公民的对象，每一次 tool call、每一次思考都可以被回看、被审计、被复用。

Schedule Deployments：FDE 的"驻场班表"。

真正的 FDE 不是"被动响应"的——他会主动去看昨天的报表，会在每周一早上提醒客户哪几个指标偏离了预期。Cloud Agents 的 Schedule（cron 调度）能力把这种主动性产品化：你可以让 Agent 在每个工作日早上 9 点跑一次系统巡检，发现异常时自动写一份报告并把结论发到运维群；你可以让它每周五汇总本周代码 review 中发现的高频问题，自动整理成知识库的一篇文章。

它不再等你打开一个对话框，它在帮你提前打开。

Memory + Dreaming：FDE 的"老员工记性"。

为什么客户那么舍不得让 FDE 离开？因为他们已经知道太多东西了——知道哪些工单是历史包袱，知道哪些数据列不能动，知道某位高管最讨厌什么样的报告格式。这种知识用一份 README 是写不全的，它必须通过长时间的协作慢慢沉淀。Cloud Agents 的 Memory 机制让 Agent 在跨会话之间持续积累项目上下文，而 Dreaming（离线整理记忆）则让这些记忆从"流水账"变成"结构化知识"——它会主动去问自己："这个项目里我见过的最关键的三件事是什么？我下次该如何更快地处理类似情况？" 这是 AI 角色第一次具备"老员工感"。

Tools / Skills：FDE 的"工具腰带"。

FDE 在客户现场不会只用 IDE。他要会写 SQL，要懂 Kubernetes，要能直接 curl 客户的内部 API，必要时甚至要手动连一次堡垒机。Cloud Agents 把这些能力做成了可编程的工具与技能：默认工具集（Bash、文件、Web、代码执行）覆盖大部分通用动作；Custom Tools 支持把任意一个内部 API 变成 Agent 的"新技能"；Skills 则把一类反复发生的工作流（如"写一份发版 changelog"、"跑一次 SWE-bench 评估"）封装成可复用的程序化经验。

Agent 不再是一个赤手空拳的语言模型，它是一个穿着工具腰带的工程师。

Environments：FDE 的"驻场办公室"。

Agent 跑在哪？这件事比想象中重要。如果它跑在你的本机，你不敢让它执行 rm -rf；如果它跑在共享环境里，你不敢让它读写敏感数据。Cloud Agents 的 Environment 抽象提供了一个隔离的、带完整工具链和依赖的运行空间——它对客户可见、可重置、可挂载代码仓库（GitHub Repository、私有 GitLab），意味着 Agent 在干活时既有"现场感"，又有"边界感"。

Files / Model Selection / Effort：FDE 的"性价比开关"。

有时候你只想让 Agent 把一份文档读一下做个摘要，有时候你想让它做一次跨整个仓库的代码重构。前者用一个轻量模型加最小思考深度足矣，后者必须给到顶配模型 + 高 Effort。Cloud Agents 把模型选择和思考深度（Effort 档位）做成显式的旋钮，让你能在每一次任务前对"质量–成本–时间"做精确权衡。这是一种工程师视角的、对 token 和时延都有明确感知的产品语言。

把上面这些机制拼到一起，

你会发现 Cloud Agents 不是"另一个聊天机器人"，它在尝试还原的，是 FDE 这个岗位在客户现场每天发生的那些事。

四个真实场景：让 Cloud Agent 当一次"驻场员工"

抽象的能力清单很难感受。我把它放进四个具体的场景里。

场景一：Agent 作为"驻场运维"。

你有一个混合云架构，每天凌晨需要有人去检查数据库慢查询、检查队列积压、检查证书是否快过期。过去这是 SRE 的值班工作；现在你可以配一个 Cloud Agent，挂上数据库只读凭据和 Prometheus 工具，再配一条 cron schedule —— 每天 02:00 自动跑一次巡检，发现异常时调用钉钉机器人推送到值班群，把当日健康度报表存进 Files。SRE 第二天上班时，看到的不是一堆告警，而是

一份已经被分类、已经被解读的报告。

场景二：Agent 作为"驻场 Code Reviewer"。

团队里总有几个仓库是没人来得及看的——脚本工具、internal SDK、一些遗留的 monorepo 子模块。你可以挂载这些 GitHub Repository 给一个 Cloud Agent，配置它在每个新 PR 创建时自动 review，关注几条高优先级规则（线程安全、敏感字段、依赖更新）。它不会替代人类 reviewer，但它会

让人类 reviewer 不再被低价值的"接缝问题"反复消耗注意力。

更重要的是，每一次 review 的判断都会被它写进 Memory，下一次再看到类似 PR 时，它知道这是一种已经讨论过的模式。

场景三：Agent 作为"流程胶水"。

跨系统流程是 ToB 世界里最浪费人力的地方——客户提了一张工单，先要在 CRM 里建客户档案，再去财务系统申请白名单，再去 IAM 里开账号，最后要在 Wiki 里更新一份对接文档。这套流程没有任何系统能一站式完成，但用 Custom Tools，你可以把这五个内部 API 注入给一个 Cloud Agent，让它当那块"胶水"。Session 跑完，所有动作有记录、有回滚、有可审计的事件流。

场景四：Agent 的"越用越懂你"。

让一个新员工真正"懂业务"，往往需要半年。但一个 Cloud Agent 在你的项目里跑了三个月之后，它的 Memory 里已经积累了：你们的代码风格偏好、你们对 changelog 的口径要求（"产品 sense，不要逐条 commit 流水账"）、你们在做哪几个高优功能、哪些业务术语需要严格区分。它已经像一个

老员工

了——这种"老员工感"，正是 FDE 在客户那里最稀缺、也最值钱的资产。

写给做技术决策的同学

这一波 FDE 热潮，从结果上看是一个招聘市场的浮动。但它真正在告诉我们的事情，其实是一个更长期的产业判断：

ToB AI 的瓶颈，从来都不在模型本身，而在模型与业务之间的那条"驻场链路"。

当 OpenAI 和 Anthropic 都在用十亿美元级的合资公司去补这条链路的时候，作为技术决策者，你大概率不会指望靠招同样规模的 FDE 团队去补——这不现实，也跑不赢。

真正可行的路径，是把 FDE 的工作模式抽象成一种产品能力，让每一个项目、每一个团队都能"持有"自己的 Agent。

Memory 解决"懂业务"，Tools/Skills 解决"会干活"，Schedules 解决"主动盯"，Sessions/Environments 解决"独立交付"，Effort/Files 解决"成本可控"。

这正是 Cloud Agents 想要回答的问题，不只是"能不能跑一个 Agent"，而是"能不能像养一个 FDE 一样，养一个 Agent"。

模型会继续变强，这毫无悬念。但

让模型从"展示能力"走向"交付结果"的那一公里，会属于把 FDE 工作模式产品化的那一类系统。

FDE 的终极形态，不是会议室里更多的工程师，而是每个团队都拥有的、属于自己的 AI 同事。
