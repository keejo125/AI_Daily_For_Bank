# 灵码 Skills：团队最佳实践，一键共享给所有人

> 原文链接：https://mp.weixin.qq.com/s/6IiarHGaKNwVXACAXYRnzA
> 公众号：灵码Lingma

你有没有遇到过这样的场景：团队里写了一套完善的代码审查规范，新人入职第一天就要翻十几页 Wiki，结果还是在 PR 里踩同样的坑。或者你花了两个下午摸透了一个复杂的日志分析流程，下次同事遇到同样的问题，你又得口口相传一遍。

团队里那些最值钱的东西——代码规范、调试技巧、部署流程——往往散落在文档角落、聊天记录和个人脑袋里。它们不缺文字记录，缺的是一种能在工作现场即时生效的载体。

灵码 Skills 就是这个载体。

一句话理解 Skills

Skills 是灵码的技能扩展机制。你可以把它理解为给 AI 编程助手写的"岗位手册"：一旦加载，灵码就会在合适的场景自动调用这份知识，按照你定义的流程和标准来工作。

不需要写代码，不需要搭框架。一个

SKILL.md

文件，就是一个完整的技能。

01

两个核心能力

智能触发，无需记忆指令

Skills 不是传统意义上的命令行插件——你不需要记住每个技能的名字和参数。当你在对话中描述需求时，灵码会根据 Skill 的描述自动判断是否需要调用。

举个例子：你创建了一个

log-analyzer

技能，当你对灵码说"帮我看看这个日志哪里有问题"，它会自动识别意图并激活这个技能，按照你预设的分析步骤——错误分级、调用链追踪、性能热点标注——逐步执行。

当然，如果你就是想精确触发某个技能，也可以在对话框输入

/skill-name

手动调用。自动和手动两条路，随你选。

项目级 + 用户级，两层作用域

Skills 支持两个层级的部署：

用户级

（

~/.lingma/skills/{skill-name}/SKILL.md

）：跟着你的账号走，无论打开哪个项目都能用。适合你个人沉淀的通用工作流——比如你习惯的 Git 提交规范、你偏好的 API 文档格式。

项目级

（

.lingma/skills/{skill-name}/SKILL.md

）：跟着代码仓库走，所有克隆这个项目的人自动获得同一套技能。适合团队约定——代码审查规则、部署检查清单、接口设计规范。

这意味着什么？把

.lingma/skills/

目录提交到 Git 仓库，你的团队技能就随代码一起版本管理、一起分发、一起演进。新人 clone 代码的那一刻，就继承了整个团队的最佳实践。不需要额外的 onboarding 文档，不需要"你去问一下XX"。

02

创建你的第一个 Skill

有两种方式上手。

方式一：交互式创建（推荐新手）

在灵码对话框里输入：

/create-skill 将接口返回的错误码转换为用户友好的提示文案

create-skill

是灵码内置的技能创建助手，它会通过对话引导你逐步完善技能定义——描述、触发条件、执行步骤、输出格式。最终生成一个规范的

SKILL.md

文件，存放在用户级目录下。

如果你不确定 SKILL.md 该怎么写，这是最省心的起步方式。

方式二：手动创建

如果你已经清楚想要什么，直接在对应路径创建文件即可：

# 用户级（个人技能）

mkdir

-p ~/.lingma/skills/code-reviewer

touch

~/.lingma/skills/code-reviewer/SKILL.md

# 项目级（团队技能）

mkdir

-p .lingma/skills/deploy-checker

touch

.lingma/skills/deploy-checker/SKILL.md

SKILL.md

的核心结构很简洁：技能名称、触发描述、执行步骤。写完后重启 IDE，在对话框输入

/

就能看到已加载的技能列表。

一个重要的优先级规则：

同名技能，项目级覆盖用户级

。这意味着团队可以在项目里"定制化"某个通用技能的行为，而不影响成员在其他项目中的使用。

03

四个高频应用场景

场景一：标准化代码审查

创建一个

code-reviewer

技能，写入你们团队的审查要点：命名规范、异常处理策略、并发安全检查、性能红线。当有人说"帮我审查这段代码"，灵码就会按这套标准逐项过一遍，输出结构化的审查意见。

不再依赖 reviewer 的个人经验差异，也不再遗漏那些"大家都知道但每次都忘"的检查项。

场景二：日志诊断 SOP

运维或后端团队可以创建

log-analyzer

技能，把日志分析的 SOP 封装进去：先看错误等级分布，再追踪异常调用链，最后定位性能瓶颈。团队里每个人遇到日志分析需求时，拿到的都是同一套诊断思路。

场景三：API 文档即时生成

约定一套

api-doc-generator

技能，定义你们的文档规范（字段说明格式、示例请求的风格、错误码的分类方式），之后对着任何一段接口代码说"生成文档"，产出的格式都是统一的。

场景四：部署前检查清单

把上线前那张人人都见过但经常有人漏掉的 checklist 做成

deploy-checker

技能：环境变量是否配齐、数据库迁移是否执行、依赖版本是否锁定、回滚方案是否就绪。上线前跑一遍，心里踏实。

几条实用建议

技能描述决定触发精度。

SKILL.md 的描述字段越精确，灵码判断"该不该用这个技能"的准确率越高。与其写"用于代码相关任务"，不如写"当用户要求审查 Java/Go 代码的并发安全性时使用"。

从小技能开始迭代。

不要一上来就写一个覆盖全流程的巨型技能。先把一个具体痛点做成技能（比如"检查 SQL 注入风险"），跑通了再扩展。

用户级技能做个人武器库。

那些不适合强推给全团队、但你自己用着特别顺手的工作流——个性化的 commit message 风格、你习惯的测试用例生成策略——放在用户级，跨项目都能用。

04

开始使用

打开你的 IDE（VS Code、JetBrains 系列或灵码自研 IDE 均支持），确保已安装灵码插件。在对话框输入

/create-skill

，描述你想要自动化的第一个工作流程。

或者直接动手：在你的项目根目录创建

.lingma/skills/

目录，写一个 SKILL.md，提交到仓库。下次有同事问你"这个项目有什么规范要注意"，你可以告诉他：都在代码里了，灵码会提醒你的。
