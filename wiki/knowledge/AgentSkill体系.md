# Agent Skill 体系

> 整合Skill宇宙、女娲Skill、同事Skill等Skill构建方法论

---

## 核心共识

### 1. Skill = 能力封装单元
Skill是将AI能力固化为可复用模块的核心机制：[1]

| 维度 | 说明 |
|------|------|
| 定义 | 可复用的工作流、知识、经验的封装 |
| 价值 | 不需要每次从零开始推理 |
| 特点 | 跨项目、跨场景、可持续迭代 |

### 2. Skill构建三层体系
| 层级 | 内容 | 说明 |
|------|------|------|
| 方法论 | 构建原则、设计模式 | 如何设计好的Skill |
| 技术实现 | SKILL.md格式、参数定义 | 如何让AI理解Skill |
| 生态运营 | 分享、发现、评价机制 | 如何让Skill流通 |

### 3. Skill vs Prompt 的本质区别
| 维度 | Prompt | Skill |
|------|--------|-------|
| 作用域 | 单次对话 | 跨会话复用 |
| 可进化 | 否 | 是（可迭代更新） |
| 携带上下文 | 无 | 可携带关联文件、示例 |
| 触发方式 | 手动输入 | 可按路径/关键词自动激活 |

## 观点分歧

### 关于Skill是否需要标准化
| 观点 | 来源 | 立场 |
|------|------|------|
| 开放技能标准（agentskills.io） | [1] Skill Universe | 标准化派 |
| 各自定义、百花齐放 | [2][3] 女娲Skill | 灵活派 |

> **整合理解**：标准化有利于生态流通，个性化有利于场景适配。两者不矛盾，可以标准化框架+个性化内容。

## 核心Skill类型

### 1. Persona Skill（人格技能）
| 类型 | 用途 | 特点 |
|------|------|------|
| 同事.skill | 模拟特定同事的工作方式 | 保留个人风格 |
| 前任.skill | 保留离职员工的经验 | 知识不流失 |
| 自己.skill | 提炼个人最佳实践 | 自进化 |

**构建方法**：
1. 收集该人格的工作样本、文档、决策记录
2. 提炼核心方法论、偏好、禁忌
3. 封装为Skill格式

### 2. Workflow Skill（流程技能）
| 类型 | 用途 | 示例 |
|------|------|------|
| 代码审查.skill | 标准化审查流程 | 步骤+检查清单 |
| 架构评审.skill | 评审设计方案 | 维度+打分表 |
| 发布检查.skill | 规范发布流程 | 检查项+审批流 |

### 3. Knowledge Skill（知识技能）
将特定领域的知识封装为可查询的Skill：
- 领域规则（银行合规、监管要求）
- 技术栈文档（内部框架、中间件）
- API规范（接口定义、使用示例）

## 自进化Skill：达尔文.skill

> 核心灵感：Karpathy 的 autoresearch（7万⭐）—— 把"棘轮机制"从模型训练迁移到 Skill 优化

### 核心机制
借鉴 Karpathy 的 autoresearch：让 AI 自己跑实验、自己评估结果、只保留有改进的修改。autoresearch 优化训练代码（loss 下降=保留），达尔文优化 SKILL.md（分数上升=保留）。

### 与 autoresearch 的关键区别
| 维度 | autoresearch | 达尔文.skill |
|------|-------------|-------------|
| 优化对象 | 训练代码 | SKILL.md |
| 评估标准 | loss 数值（全自主） | 8维度人工可判断 |
| 判断方式 | 机器自主 | 机器初筛 + 人在回路终审 |

**关键设计：Human in the Loop** — Skill 的好坏有时需要人判断，所以达尔文加了人在回路：每次优化后系统暂停，展示 diff、分数变化、测试对比，人确认后才继续。

### 五条原则（踩坑总结）

| # | 原则 | 来源 |
|---|------|------|
| 1 | **单一可编辑资产** — 每次只改一个 SKILL.md | 同时改多个skill无法判断哪个生效 |
| 2 | **双重评估** — 结构评分 + 实测评分 | 结构满分但跑出来很差的情况 |
| 3 | **棘轮机制** — 分数只能升不能降，改差了就 git revert | 允许实验，失败不伤害 |
| 4 | **独立评分** — 修改agent不能是评分agent | 审计独立性（改的人不能自己给自己打分） |
| 5 | **人在回路** — 机器初筛，人做终审 | 目前还是人比机器更懂"好不好" |

### 8维度评分体系（100分制）

**结构维度 60分**：

| 维度 | 分值 | 考察点 |
|------|------|--------|
| Frontmatter 规范 | 8分 | 格式是否标准 |
| 工作流清晰度 | 15分 | 步骤是否完整 |
| 异常处理 | 10分 | 有没有边界情况 |
| 用户确认点 | 7分 | 关键决策前是否暂停 |
| 指令可执行性 | 15分 | 指令是否具体到可直接执行 |
| 文件路径存在性 | 5分 | 引用的文件路径是否真实存在 |

**效果维度 40分**：

| 维度 | 分值 | 考察点 |
|------|------|--------|
| 整体架构合理性 | 15分 | 结构设计是否合理 |
| 实测输出质量 | **25分** | 拿真实 prompt 跑一遍看效果 |

> 实测权重最高（25分）：一个格式完美但跑起来没用的 skill，不如一个写得粗糙但特别管用的 skill。

### 优化流程（5阶段）

```
Phase 0-1: 初始化 + 基线评估（为每个skill设计测试prompt，跑出起点分数）
Phase 2  : 核心循环（找最低分维度 → 针对性修改 → 独立agent评分 → 涨了留/降了回滚，每skill最多3轮）
Phase 3  : 汇总Before/After分数表
```

棘轮示例：`72分 → 78分（留）→ 75分（回滚到78）→ 84分（留）→ 最终78→84，净提升6分`

### 棘轮的美学
> 你可以放心做实验，失败不会伤害你。只有成功会被保留。

科学是棘轮、民主制度是棘轮、git 历史是棘轮——达尔文.skill 把这个古老的结构用在了 Skill 优化这个新地方。

## Skill UI：双形态的关键补全

> 来源：Qoder 官方《Agent Skills 终于有 UI 了》| [5]

### 问题：CLI 是 Agent 的母语，但人不是 Agent

Skill 一直是 Agent 原生的（SKILL.md + scripts/），但缺少 GUI 层：
- **输入端**：Agent 需要人做决策时，一条一条线性地问，效率低（选个设计方向要5轮对话）
- **输出端**：Agent 展示结果时，文本列数字不如一张图表直观

### 方案：Skill 双形态 = CLI（Agent 用）+ GUI（人用）

| 层 | 对象 | 形式 | 说明 |
|----|------|------|------|
| CLI 层 | Agent | SKILL.md + scripts/ | 文本驱动、自描述、可组合 |
| GUI 层 | 人 | Widget UI | 配置面板做决策 + Dashboard 看结果 |

### 两种 UI 模式

| 模式 | 原理 | 适用场景 | 特点 |
|------|------|----------|------|
| 动态生成 | Agent 根据运行时上下文实时生成 HTML/CSS/JS | 数据驱动（性能图表、Diff预览、数据对比） | 灵活，消耗 token |
| 预定义模板 | Skill 作者预写 HTML 放 assets/，通过 `window.__WIDGET_DATA__` 传入上下文 | 交互固定（配置表单、审批流程） | 零 token、毫秒加载、体验一致 |

同一个 Skill 可混合使用两种模式。

### 目录结构示例
```
my-frontend-design-skill/
├── SKILL.md
├── scripts/
│   └── generate.sh
└── assets/
    └── design-direction.html   ← 预定义模板
```

### 安全架构
| 机制 | 说明 |
|------|------|
| 沙箱隔离 | `sandbox="allow-scripts allow-forms"` 的 iframe，不加 `allow-same-origin` |
| 双阶段消毒 | 预览阶段剥离所有 `<script>`；沙箱阶段保留交互，仅移除 iframe/object |
| 第三方库白名单 | 仅放行 cdnjs.cloudflare.com、cdn.jsdelivr.net、unpkg.com、esm.sh |
| 受控接口 | 仅 sendToAgent()、openFile()、openUrl()、copyToClipboard() 四个 |

### 实战：`/create-skill-ui`
在 Qoder Quest 模式下，三轮对话即可为 Skill 加 UI：
1. 描述需求 → 6秒生成初版面板
2. 要求视觉增强（如调性预览图）→ Agent 调用图像生成
3. 加自定义功能（如品牌色拾色器）→ 持久化到 assets/

### 意义
> Skill 从指令集升级为**有知识、有脚本、有界面的完整微应用**。输入端配置面板解决"怎么告诉 Agent 你要什么"，输出端 Dashboard 解决"怎么看懂 Agent 给你的东西"。人和 Agent 的协作闭环才算完整。

## Skill 训练心法：从"能用"到"可靠"

> 来源：solonnliu《让AI变成Super员工的秘密：高效训练Skills》| [6]
> 实战背景：打磨 web-testing Skill（全站自动化测试 → 自动修复 → 自动验证闭环）

### 核心命题

> Skill 的作用，不是让 AI 更聪明，而是让 AI 在你的业务里**更可靠**。

大模型拥有通用能力，但缺乏岗位能力。通用能力解决"能不能做"，岗位能力解决"什么算完成、哪些步骤不能跳、哪些错误已发生过、交付物长什么样"。Skill 本质上是**业务操作系统**——模型提供通用智力，Skill 提供岗位 SOP。

### 四个翻车教训

| # | 坑 | 现象 | 根因 | 修正 |
|---|-----|------|------|------|
| 1 | AI 不会"穷举" | 漏掉藏在 Tab→表格→蓝色链接后的深层页面入口 | AI 靠主观判断决定"是否重要"，看见但没真正点进去 | 不能写"请仔细检查"，要写触发条件+必做动作+结束门槛（如"页面必须滚动到底、Tab切换后必须重新扫描"）|
| 2 | AI 只生成"最像成果的" | 要求三个文件（sitemap.md、test-report.md、test-report.html），AI 只生成了 HTML | HTML 最显眼最复杂，吸走模型注意力和上下文预算 | 顺序写死 + 每生成一个立刻检查存在性 + 阻断式完整性检查 |
| 3 | AI 选"最省事"方案 | 用 `python3 -c` 拼超长脚本，命令行长度上限被打爆 | 模型不具备当前执行环境的工程常识 | Skill 也是"工程生存指南"：禁止超长单行命令、禁止 base64 内嵌、优先写文件再执行 |
| 4 | 最危险的是"看起来生成了" | 报告结构看似完整，但逐页面详细模块被悄悄压缩 | 上下文紧张时 AI 进入"节约模式"，压缩不显眼的结构 | 补结构门禁：checklist 逐项检查，上一项没过不能进下一项 |

### 四条训练方法

| # | 方法 | 要点 |
|---|------|------|
| 1 | 先真跑3-5次，再写 Skill | 没有真实翻车写不出真的 Skill；重点看它怎么错（漏了什么、跳了哪步、哪些地方看起来做了其实没做）|
| 2 | 写 SOP，不写要求 | ❌ "请仔细检查页面" → ✅ "当页面存在 Tab/展开行/子表格时，切换后必须重新枚举链接" |
| 3 | 配 checklist + 门禁 | checklist 告诉 AI 检查什么，门禁告诉 AI 没检查过不能往下走；自觉是加分项，门禁是底线 |
| 4 | 让 AI 参与复盘并自动改 Skill | 不只是手改 Prompt，是让 AI 分析根因后直接修改 Skill，循环迭代直到满意 |

### checklist 示例（web-testing）

```
- [ ] 页面模块数 = sitemap 页面数
- [ ] 每页都有截图
- [ ] 每页都有 UI/UX 审查
- [ ] 每页都有功能测试结果表
- [ ] 每页都有本页问题汇总
- [ ] HTML 中 page-card 数量 = 页面数
- [ ] 问题汇总表存在
- [ ] 修复计划表存在
```

### 与达尔文.skill 的互补

| 维度 | 达尔文.skill | Skill训练心法 |
|------|-------------|--------------|
| 优化方式 | 自动（棘轮+8维度评分） | 人工（真实翻车→复盘→写规则）|
| 适用阶段 | Skill 已有初版，需迭代优化 | Skill 从0到1，需先跑出问题 |
| 核心机制 | 机器初筛+人在回路终审 | 真实任务→发现坑→写SOP+门禁 |
| 关系 | **先用心法建初版，再用达尔文自动优化** | |

> 金句：**训练 Skill 不是在抬高 AI 的上限，而是在抬高 AI 的下限。**

## 构建指南

### Skill设计原则
1. **单一职责**：一个Skill做一件事
2. **清晰边界**：明确输入、输出、适用场景
3. **可观测**：记录使用效果，持续优化
4. **可组合**：Skill之间可以组合使用

### SKILL.md基本结构
```markdown
---
name: skill-name
description: 一句话描述
trigger: 触发条件
---

# Skill名称

## 概述
一句话描述这个Skill做什么

## 触发条件
什么情况下应该使用这个Skill

## 执行步骤
1. 步骤一
2. 步骤二

## 注意事项
- 注意点1
- 注意点2

## 示例
\`\`\`示例输入
...
\`\`\`
```

## Skill设计模式：5种核心架构

> 来源：微信公众号（链接存档于 memory/）

### 5种核心模式

| 模式 | 适用场景 | 代表Skill |
|------|----------|----------|
| **线性流程** | 部署/安装/迁移等有明确步骤的操作 | vercel-deploy |
| **决策树 + 按需加载** | 大型平台选型、产品导航 | cloudflare-deploy |
| **循环迭代** | "做→验证→改进"反复执行 | test-driven-development |
| **接力棒循环** | 跨Session持久化，长期项目 | stitch-loop |
| **多阶段 + 检查点** | 跨越多天/多周，有Go/No-Go决策 | discovery-process |
| **特殊：思维框架** | 深度分析而非快速执行 | audit-context-building |

### 知识3层架构

| 层级 | Token预算 | 内容 |
|------|-----------|------|
| Frontmatter | ~100 | name + description |
| SKILL.md正文 | 2K-5K | 核心指令、决策树、流程步骤 |
| references/ | 1K-3K/个 | 详细文档，按需加载 |

> 总上下文占用建议 <10K tokens（主文件 + 1-2个参考文档）

### 模式选择决策树

```
Skill需要做什么？
├─ 执行有明确步骤的操作 → 模式1：线性流程
├─ 在大量选项中选方向 → 模式2：决策树+按需加载
├─ 单次会话反复做→验证→改进 → 模式3：循环迭代
├─ 跨多个session持续推进 → 模式4：接力棒循环
├─ 跨越多天/多周有阶段决策 → 模式5：多阶段+检查点
└─ 深度分析而非快速执行 → 特殊：思维框架
```


### Anthropic 开放规范 + 5种工具视角设计模式

> 来源：阿里云开发者《Agent Skill规范、构建与设计模式》| [10]
> 补充视角：Anthropic/Google ADK 的工具视角，与"5种核心架构"互补

#### Anthropic Skill 规范标准

| 字段 | 必填 | 说明 |
|------|------|------|
| name | ✅ | 唯一标识，1-64字符，小写字母+连字符，与文件夹名一致 |
| description | ✅ | 触发关键词，1-1024字符，决定何时激活 |
| license | ❌ | 许可证 |
| compatibility | ❌ | 环境兼容性 |
| metadata | ❌ | 扩展元数据 |
| allowed-tools | ❌ | 预授权工具列表（实验性）|

**description 是触发的关键**：写好 description 比写好指令主体更重要——Anthropic 将 description 作为 Skill 识别的核心依据。

#### 三层渐进式加载机制

| 层级 | 内容 | Token预算 |
|------|------|-----------|
| SKILL.md（必需） | Frontmatter + 核心指令 | 2K-5K |
| scripts/（按需） | 可执行脚本 | 按需加载 |
| references/（按需） | 参考文档 | 1K-3K/个 |

> 解决 MCP 上下文膨胀问题——不是一股脑加载，而是按触发条件渐进式注入。

#### Anthropic/Google ADK 5种设计模式

| 模式 | 核心逻辑 | 适用场景 | 关键约束 |
|------|----------|----------|----------|
| **Tool Wrapper** | 封装专家知识为工具集 | 特定技术栈（BigQuery/GKE/Firebase） | 输出格式一致性 |
| **Generator** | 批量生成结构化内容 | 代码/文档/测试批量生成 | 质量门槛控制 |
| **Reviewer** | AI审查AI的产出 | PR审查/安全扫描/代码风格检查 | 必须输出分级报告 |
| **Inversion** | Agent先采访用户，再执行 | 需求不明确的新项目 | 所有阶段完成前不得开始构建 |
| **Pipeline** | 带检查点的严格多步流水线 | 复杂多阶段任务（文档生成/代码生成） | 检查点前不得跳步 |

#### 模式组合推荐

| 组合 | 说明 | 典型场景 |
|------|------|----------|
| Pipeline + Reviewer | 管道最后一步加自动审查 | 文档生成后自动质量检查 |
| Generator + Inversion | 先收集信息再填充模板 | 需用户输入的结构化文档生成 |
| Pipeline + Tool Wrapper | 管道某些步骤加载专家知识 | 多步骤代码生成 |
| Inversion + Pipeline | 先完成需求收集再进入执行流水线 | 复杂项目全流程 |

> 与"5种核心架构"的关系：两者互补——"核心架构"偏**工作流拆分**，ADK模式偏**Agent交互方式**。编码用"循环迭代"+Generator，交付用"多阶段+检查点"+Pipeline，运维用"思维框架"+Reviewer。

## 金融岗位智能体：一岗一助手

> 来源：金融AI coding系列4 | [9]
> 核心命题：万能助手解决个人效率，一岗一助手解决金融级生产流程

### 万能助手的三重局限

| 局限 | 说明 |
|------|------|
| **责任模糊** | 跨越多个岗位，难以追溯"谁确认的、谁采纳的、谁放行的" |
| **上下文混乱** | 产品经理关心业务目标，架构师关心系统边界，测试关心覆盖——一个助手无法承载 |
| **效果无法量化** | 什么都能答一点，什么都不够深入，难以评估真实价值 |

### 一岗一助手五维价值

| 维度 | 说明 |
|------|------|
| **责任边界** | 需求是谁确认的，代码是谁采纳的，故障是谁处置的——边界清楚，责任才清楚 |
| **专业上下文** | 每个岗位有专属知识资产（业务规则/架构文档/测试资产/运维知识） |
| **工作流衔接** | 不是孤立回答，而是流程节点上的执行者和连接器 |
| **权限安全** | 需求智能体不能随意调生产日志，运维智能体不能绕过授权执行 |
| **效果评估** | 岗位智能体可以被量化（代码采纳率/缺陷发现率/故障定位准确率） |

### 全球共识：Agent治理先行

| 机构 | 关键实践 |
|------|----------|
| **DBS** | Agentic control plane，对所有agent监督，强调human oversight + audit trail + fallback机制 |
| **Citi** | 4万开发者用Devin，但**不允许agent部署代码**，只产artifacts，人工review后才放行 |
| **Morgan Stanley** | AI coding assistants成为标准工作流后，瓶颈转向代码审查/测试/安全/验证/部署 |
| **Bank of America** | Human oversight + transparency + accountability，效率提升20%+ |

> 金融行业共识：Agent必须进入工作流，也必须被治理。不是不敢用，而是必须把Agent放进权限边界、审核机制和治理框架里。

### 招行DevAgent样本

| 维度 | 数据 |
|------|------|
| 月均任务 | **4.8万+开发任务** |
| 模式 | ReAct（感知-规划-执行-反馈-进化） |
| 能力 | 现场感知+企业知识检索+任务级开发+跨文件生成 |
| 关键词 | 不是"通用问答"，而是"懂工程、懂企业知识的现场智能体" |

### 工商银行样本

六大岗位全覆盖：**需求→设计→编码→测试→交付→运维**

核心命题：不是"AI能不能写代码"，而是"AI能不能在金融级软件工程体系中稳定工作"。真正难的是让模型进入秩序，让生成进入流程，让结果进入审计，让责任有人承接。

### 六岗位智能体详解

#### 需求智能体：从"写需求"走向"定义意图"
- 把自然语言/会议讨论/业务说明转化为结构化需求资产
- 辅助生成原型，让业务/产品/设计/开发围绕"看得见的东西"讨论
- 产品经理从"翻译"变成"导演"——让业务/AI/设计/开发/测试在同一镜头对齐

#### 设计智能体：从"补文档"走向"沉淀系统边界"
- 理解应用架构/接口文档/表结构/历史代码/公共方法/已有设计
- 设计文档同时给人看、给AI看、给测试看、给交付看
- 架构师从"救火队长"变成"轨道设计师"——轨道铺得越清楚，AI越不容易脱轨

#### 编码智能体：从"敲代码"走向"带AI干活"
- 理解任务→读取上下文→遵守规约→调用工具→生成代码→生成单测→执行自检→自动修复
- 开发人员从"亲自下地干活"变成"带一组AI工人的工长"
- 活可以让AI干，但图纸对不对、工序对不对、质量过不过关，最后人得接得住

#### 测试智能体：从"后面接锅"走向"前面设防"
- 基于需求/设计/代码生成测试案例，覆盖正常/异常/边界/权限/数据口径
- 辅助构造测试数据，生成测试脚本，分析失败原因
- 测试从"最后一道安检"变成"全流程雷达"——起飞前/飞行中/降落前都持续发现风险

#### 交付智能体：从"临门救火"走向"提前控险"
- 资源供给阶段：识别代码库/分支/环境/流水线/发布单元
- 持续集成阶段：监控构建失败/门禁异常/部署异常，分析原因推荐修复
- 版本交付阶段：生成交付报告，识别代码增量/配置变更/门禁异常/环境差异/潜在风险
- 投产前：给出风险提示和处置建议
- 交付从"临门一脚"变成"塔台调度"——天气/跑道/路线/机组状态都要提前看清

#### 运维智能体：从"告警疲劳"走向"故障推理"
- 读取指标/日志/链路/告警/变更记录/历史故障案例
- 形成诊断计划，边查边调整，在多个可能原因中做排除
- 运维从"消防队"变成"城市神经系统"——不仅要救火，还要提前感知哪里升温/哪里堵塞/哪里可能出事

### 上下游协同：资产流转闭环

```
需求智能体输出 → 设计智能体读取 → 编码智能体执行 → 测试智能体验证 → 交付智能体检查 → 运维智能体反馈
                                          ↑                                            ↓
                                    缺陷反馈 ←─────────────────────────────────────────── 故障沉淀
```

一岗一助手不是把每个岗位做成小烟囱，而是把每个岗位变成一段标准化轨道，最后连成一条能跑起来的智能研发生产线。

### 开发中心治理：管人也要管Agent

未来需要一张**智能体组织图**：
- 每个智能体负责什么岗位
- 能调用什么工具
- 能访问什么数据
- 输出什么资产
- 由谁审核
- 进入哪个流程

> 没有这张图，AI越多越乱。有了这张图，AI才能从个人工具变成组织能力。

### 与本体系的关联

| 维度 | 关联 |
|------|------|
| **达尔文.skill** | 岗位智能体是静态分工，达尔文是动态优化——岗位Skill可被达尔文自动迭代 |
| **Skill训练心法** | 岗位智能体的上下文/规约/流程，本质是**岗位级Skill**——每个岗位需要一个专属Skill |
| **5种设计模式** | 岗位智能体可按岗位特点选择：编码用"循环迭代"，交付用"多阶段+检查点"，运维用"思维框架" |
| **Skill UI双形态** | 岗位智能体的配置面板/结果Dashboard，正是Skill UI的应用场景 |

> 金句：**一岗一助手，不是让岗位消失，而是让每个岗位都站到更高的位置。**

## 谷歌官方 Agent Skills 仓库发布

> 来源：InfoQ | [8]
> 链接：https://mp.weixin.qq.com/s/FUh3VjFdfvUY-Ypgke11rg

### 背景：MCP 的上下文膨胀问题

MCP 服务器为模型提供实时信息，但大规模使用时不加区分地加载上下文，导致推理质量下降 + Token 成本飙升。有团队发现智能体每次调用加载 1.5 万 tokens 指令，几乎不给实际内容留空间。

### Google 官方 Skill 仓库

- **地址**：github.com/google/skills
- **规模**：13 项技能，覆盖 GCP 核心服务（BigQuery、Cloud Run、GKE、Firebase、Gemini API 等）
- **3 大架构支柱**：安全性 / 可靠性 / 成本优化
- **3 项流程指南**：GCP 入门 / 身份验证 / 网络可观测性
- **特点**：经 Gemini 实测验证、兼容 MCP 标准、Google 负责维护更新、直接用 GCP 凭证

### Addy Osmani 开源 Agent Skills

- **作者**：Addy Osmani（Google Cloud AI 总监，Gemini 工程负责人）
- **规模**：2.4 万⭐，19 项工程技能 + 7 条命令
- **完整生命周期**：Define → Plan → Build → Verify → Review → Ship
- **本质**：通用工程纪律框架——约束 AI 按规范干活

### 两库对比

| 维度 | Osmani 开源库 | Google 官方库 |
|------|-------------|-------------|
| 解决问题 | 如何正确地构建 | 构建什么 + 如何操作 |
| 本质 | 通用工程纪律框架 | 特定技术栈操作知识 |
| 适用 | 所有开发场景 | GCP 云服务场景 |
| 共存 | ✅ 同一智能体环境可同时使用 | |

### Skill 的技术定位

> 比 Prompt 更持久 → 比 Fine-tuning 更轻 → 比 RAG 更主动 → 比 Tools 更丰富

这与本体系中「Skill vs Prompt 的本质区别」和「Skill构建三层体系」的观点一致——Skill 的核心价值是**可持久、可进化、可组合**。

### 对我们 Skill 体系的启示

Google 官方入场验证了 Skill 路线的正确性。Osmani 的工程纪律框架（Define→Ship）与我们「达尔文.skill」的自进化思路互补：
- **Osmani**：构建阶段的规范约束（如何正确地写 Skill）
- **达尔文**：运行阶段的自动优化（如何让 Skill 越用越好）
- **我们的实践**：先用心法建初版（训练心法），再用达尔文自动优化
