# Agent Skill 体系

> 整合Skill宇宙、女娲Skill、同事Skill等Skill构建方法论

---

## 核心共识

### 1. Skill = 能力封装单元
Skill是将AI能力固化为可复用模块的核心机制：[1]

| 维度 | 说明 |
|------|------|
| 定义 | 可复用的工作流、知识、经验的封装 |
| 价值 | 不需要每次从零开始推理 |
| 特点 | 跨项目、跨场景、可持续迭代 |

### 2. Skill构建三层体系
| 层级 | 内容 | 说明 |
|------|------|------|
| 方法论 | 构建原则、设计模式 | 如何设计好的Skill |
| 技术实现 | SKILL.md格式、参数定义 | 如何让AI理解Skill |
| 生态运营 | 分享、发现、评价机制 | 如何让Skill流通 |

### 3. Skill vs Prompt 的本质区别
| 维度 | Prompt | Skill |
|------|--------|-------|
| 作用域 | 单次对话 | 跨会话复用 |
| 可进化 | 否 | 是（可迭代更新） |
| 携带上下文 | 无 | 可携带关联文件、示例 |
| 触发方式 | 手动输入 | 可按路径/关键词自动激活 |

## 观点分歧

### 关于Skill是否需要标准化
| 观点 | 来源 | 立场 |
|------|------|------|
| 开放技能标准（agentskills.io） | [1] Skill Universe | 标准化派 |
| 各自定义、百花齐放 | [2][3] 女娲Skill | 灵活派 |

> **整合理解**：标准化有利于生态流通，个性化有利于场景适配。两者不矛盾，可以标准化框架+个性化内容。

## 核心Skill类型

### 1. Persona Skill（人格技能）
| 类型 | 用途 | 特点 |
|------|------|------|
| 同事.skill | 模拟特定同事的工作方式 | 保留个人风格 |
| 前任.skill | 保留离职员工的经验 | 知识不流失 |
| 自己.skill | 提炼个人最佳实践 | 自进化 |

**构建方法**：
1. 收集该人格的工作样本、文档、决策记录
2. 提炼核心方法论、偏好、禁忌
3. 封装为Skill格式

### 2. Workflow Skill（流程技能）
| 类型 | 用途 | 示例 |
|------|------|------|
| 代码审查.skill | 标准化审查流程 | 步骤+检查清单 |
| 架构评审.skill | 评审设计方案 | 维度+打分表 |
| 发布检查.skill | 规范发布流程 | 检查项+审批流 |

### 3. Knowledge Skill（知识技能）
将特定领域的知识封装为可查询的Skill：
- 领域规则（银行合规、监管要求）
- 技术栈文档（内部框架、中间件）
- API规范（接口定义、使用示例）

## 自进化Skill：达尔文.skill

> 核心灵感：Karpathy 的 autoresearch（7万⭐）—— 把"棘轮机制"从模型训练迁移到 Skill 优化

### 核心机制
借鉴 Karpathy 的 autoresearch：让 AI 自己跑实验、自己评估结果、只保留有改进的修改。autoresearch 优化训练代码（loss 下降=保留），达尔文优化 SKILL.md（分数上升=保留）。

### 与 autoresearch 的关键区别
| 维度 | autoresearch | 达尔文.skill |
|------|-------------|-------------|
| 优化对象 | 训练代码 | SKILL.md |
| 评估标准 | loss 数值（全自主） | 8维度人工可判断 |
| 判断方式 | 机器自主 | 机器初筛 + 人在回路终审 |

**关键设计：Human in the Loop** — Skill 的好坏有时需要人判断，所以达尔文加了人在回路：每次优化后系统暂停，展示 diff、分数变化、测试对比，人确认后才继续。

### 五条原则（踩坑总结）

| # | 原则 | 来源 |
|---|------|------|
| 1 | **单一可编辑资产** — 每次只改一个 SKILL.md | 同时改多个skill无法判断哪个生效 |
| 2 | **双重评估** — 结构评分 + 实测评分 | 结构满分但跑出来很差的情况 |
| 3 | **棘轮机制** — 分数只能升不能降，改差了就 git revert | 允许实验，失败不伤害 |
| 4 | **独立评分** — 修改agent不能是评分agent | 审计独立性（改的人不能自己给自己打分） |
| 5 | **人在回路** — 机器初筛，人做终审 | 目前还是人比机器更懂"好不好" |

### 8维度评分体系（100分制）

**结构维度 60分**：

| 维度 | 分值 | 考察点 |
|------|------|--------|
| Frontmatter 规范 | 8分 | 格式是否标准 |
| 工作流清晰度 | 15分 | 步骤是否完整 |
| 异常处理 | 10分 | 有没有边界情况 |
| 用户确认点 | 7分 | 关键决策前是否暂停 |
| 指令可执行性 | 15分 | 指令是否具体到可直接执行 |
| 文件路径存在性 | 5分 | 引用的文件路径是否真实存在 |

**效果维度 40分**：

| 维度 | 分值 | 考察点 |
|------|------|--------|
| 整体架构合理性 | 15分 | 结构设计是否合理 |
| 实测输出质量 | **25分** | 拿真实 prompt 跑一遍看效果 |

> 实测权重最高（25分）：一个格式完美但跑起来没用的 skill，不如一个写得粗糙但特别管用的 skill。

### 优化流程（5阶段）

```
Phase 0-1: 初始化 + 基线评估（为每个skill设计测试prompt，跑出起点分数）
Phase 2  : 核心循环（找最低分维度 → 针对性修改 → 独立agent评分 → 涨了留/降了回滚，每skill最多3轮）
Phase 3  : 汇总Before/After分数表
```

棘轮示例：`72分 → 78分（留）→ 75分（回滚到78）→ 84分（留）→ 最终78→84，净提升6分`

### 棘轮的美学
> 你可以放心做实验，失败不会伤害你。只有成功会被保留。

科学是棘轮、民主制度是棘轮、git 历史是棘轮——达尔文.skill 把这个古老的结构用在了 Skill 优化这个新地方。

## Skill UI：双形态的关键补全

> 来源：Qoder 官方《Agent Skills 终于有 UI 了》| [5]

### 问题：CLI 是 Agent 的母语，但人不是 Agent

Skill 一直是 Agent 原生的（SKILL.md + scripts/），但缺少 GUI 层：
- **输入端**：Agent 需要人做决策时，一条一条线性地问，效率低（选个设计方向要5轮对话）
- **输出端**：Agent 展示结果时，文本列数字不如一张图表直观

### 方案：Skill 双形态 = CLI（Agent 用）+ GUI（人用）

| 层 | 对象 | 形式 | 说明 |
|----|------|------|------|
| CLI 层 | Agent | SKILL.md + scripts/ | 文本驱动、自描述、可组合 |
| GUI 层 | 人 | Widget UI | 配置面板做决策 + Dashboard 看结果 |

### 两种 UI 模式

| 模式 | 原理 | 适用场景 | 特点 |
|------|------|----------|------|
| 动态生成 | Agent 根据运行时上下文实时生成 HTML/CSS/JS | 数据驱动（性能图表、Diff预览、数据对比） | 灵活，消耗 token |
| 预定义模板 | Skill 作者预写 HTML 放 assets/，通过 `window.__WIDGET_DATA__` 传入上下文 | 交互固定（配置表单、审批流程） | 零 token、毫秒加载、体验一致 |

同一个 Skill 可混合使用两种模式。

### 目录结构示例
```
my-frontend-design-skill/
├── SKILL.md
├── scripts/
│   └── generate.sh
└── assets/
    └── design-direction.html   ← 预定义模板
```

### 安全架构
| 机制 | 说明 |
|------|------|
| 沙箱隔离 | `sandbox="allow-scripts allow-forms"` 的 iframe，不加 `allow-same-origin` |
| 双阶段消毒 | 预览阶段剥离所有 `<script>`；沙箱阶段保留交互，仅移除 iframe/object |
| 第三方库白名单 | 仅放行 cdnjs.cloudflare.com、cdn.jsdelivr.net、unpkg.com、esm.sh |
| 受控接口 | 仅 sendToAgent()、openFile()、openUrl()、copyToClipboard() 四个 |

### 实战：`/create-skill-ui`
在 Qoder Quest 模式下，三轮对话即可为 Skill 加 UI：
1. 描述需求 → 6秒生成初版面板
2. 要求视觉增强（如调性预览图）→ Agent 调用图像生成
3. 加自定义功能（如品牌色拾色器）→ 持久化到 assets/

### 意义
> Skill 从指令集升级为**有知识、有脚本、有界面的完整微应用**。输入端配置面板解决"怎么告诉 Agent 你要什么"，输出端 Dashboard 解决"怎么看懂 Agent 给你的东西"。人和 Agent 的协作闭环才算完整。

## Skill 训练心法：从"能用"到"可靠"

> 来源：solonnliu《让AI变成Super员工的秘密：高效训练Skills》| [6]
> 实战背景：打磨 web-testing Skill（全站自动化测试 → 自动修复 → 自动验证闭环）

### 核心命题

> Skill 的作用，不是让 AI 更聪明，而是让 AI 在你的业务里**更可靠**。

大模型拥有通用能力，但缺乏岗位能力。通用能力解决"能不能做"，岗位能力解决"什么算完成、哪些步骤不能跳、哪些错误已发生过、交付物长什么样"。Skill 本质上是**业务操作系统**——模型提供通用智力，Skill 提供岗位 SOP。

### 四个翻车教训

| # | 坑 | 现象 | 根因 | 修正 |
|---|-----|------|------|------|
| 1 | AI 不会"穷举" | 漏掉藏在 Tab→表格→蓝色链接后的深层页面入口 | AI 靠主观判断决定"是否重要"，看见但没真正点进去 | 不能写"请仔细检查"，要写触发条件+必做动作+结束门槛（如"页面必须滚动到底、Tab切换后必须重新扫描"）|
| 2 | AI 只生成"最像成果的" | 要求三个文件（sitemap.md、test-report.md、test-report.html），AI 只生成了 HTML | HTML 最显眼最复杂，吸走模型注意力和上下文预算 | 顺序写死 + 每生成一个立刻检查存在性 + 阻断式完整性检查 |
| 3 | AI 选"最省事"方案 | 用 `python3 -c` 拼超长脚本，命令行长度上限被打爆 | 模型不具备当前执行环境的工程常识 | Skill 也是"工程生存指南"：禁止超长单行命令、禁止 base64 内嵌、优先写文件再执行 |
| 4 | 最危险的是"看起来生成了" | 报告结构看似完整，但逐页面详细模块被悄悄压缩 | 上下文紧张时 AI 进入"节约模式"，压缩不显眼的结构 | 补结构门禁：checklist 逐项检查，上一项没过不能进下一项 |

### 四条训练方法

| # | 方法 | 要点 |
|---|------|------|
| 1 | 先真跑3-5次，再写 Skill | 没有真实翻车写不出真的 Skill；重点看它怎么错（漏了什么、跳了哪步、哪些地方看起来做了其实没做）|
| 2 | 写 SOP，不写要求 | ❌ "请仔细检查页面" → ✅ "当页面存在 Tab/展开行/子表格时，切换后必须重新枚举链接" |
| 3 | 配 checklist + 门禁 | checklist 告诉 AI 检查什么，门禁告诉 AI 没检查过不能往下走；自觉是加分项，门禁是底线 |
| 4 | 让 AI 参与复盘并自动改 Skill | 不只是手改 Prompt，是让 AI 分析根因后直接修改 Skill，循环迭代直到满意 |

### checklist 示例（web-testing）

```
- [ ] 页面模块数 = sitemap 页面数
- [ ] 每页都有截图
- [ ] 每页都有 UI/UX 审查
- [ ] 每页都有功能测试结果表
- [ ] 每页都有本页问题汇总
- [ ] HTML 中 page-card 数量 = 页面数
- [ ] 问题汇总表存在
- [ ] 修复计划表存在
```

### 与达尔文.skill 的互补

| 维度 | 达尔文.skill | Skill训练心法 |
|------|-------------|--------------|
| 优化方式 | 自动（棘轮+8维度评分） | 人工（真实翻车→复盘→写规则）|
| 适用阶段 | Skill 已有初版，需迭代优化 | Skill 从0到1，需先跑出问题 |
| 核心机制 | 机器初筛+人在回路终审 | 真实任务→发现坑→写SOP+门禁 |
| 关系 | **先用心法建初版，再用达尔文自动优化** | |

> 金句：**训练 Skill 不是在抬高 AI 的上限，而是在抬高 AI 的下限。**

## 构建指南

### Skill设计原则
1. **单一职责**：一个Skill做一件事
2. **清晰边界**：明确输入、输出、适用场景
3. **可观测**：记录使用效果，持续优化
4. **可组合**：Skill之间可以组合使用

### SKILL.md基本结构
```markdown
---
name: skill-name
description: 一句话描述
trigger: 触发条件
---

# Skill名称

## 概述
一句话描述这个Skill做什么

## 触发条件
什么情况下应该使用这个Skill

## 执行步骤
1. 步骤一
2. 步骤二

## 注意事项
- 注意点1
- 注意点2

## 示例
\`\`\`示例输入
...
\`\`\`
```

## Skill设计模式：5种核心架构

> 来源：微信公众号（链接存档于 memory/）

### 5种核心模式

| 模式 | 适用场景 | 代表Skill |
|------|----------|----------|
| **线性流程** | 部署/安装/迁移等有明确步骤的操作 | vercel-deploy |
| **决策树 + 按需加载** | 大型平台选型、产品导航 | cloudflare-deploy |
| **循环迭代** | "做→验证→改进"反复执行 | test-driven-development |
| **接力棒循环** | 跨Session持久化，长期项目 | stitch-loop |
| **多阶段 + 检查点** | 跨越多天/多周，有Go/No-Go决策 | discovery-process |
| **特殊：思维框架** | 深度分析而非快速执行 | audit-context-building |

### 知识3层架构

| 层级 | Token预算 | 内容 |
|------|-----------|------|
| Frontmatter | ~100 | name + description |
| SKILL.md正文 | 2K-5K | 核心指令、决策树、流程步骤 |
| references/ | 1K-3K/个 | 详细文档，按需加载 |

> 总上下文占用建议 <10K tokens（主文件 + 1-2个参考文档）

### 模式选择决策树

```
Skill需要做什么？
├─ 执行有明确步骤的操作 → 模式1：线性流程
├─ 在大量选项中选方向 → 模式2：决策树+按需加载
├─ 单次会话反复做→验证→改进 → 模式3：循环迭代
├─ 跨多个session持续推进 → 模式4：接力棒循环
├─ 跨越多天/多周有阶段决策 → 模式5：多阶段+检查点
└─ 深度分析而非快速执行 → 特殊：思维框架
```
