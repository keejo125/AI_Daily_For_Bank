# Agent Skill 体系

> 整合Skill宇宙、女娲Skill、同事Skill等Skill构建方法论

---

## 核心共识

### 1. Skill = 能力封装单元
Skill是将AI能力固化为可复用模块的核心机制：[1]

| 维度 | 说明 |
|------|------|
| 定义 | 可复用的工作流、知识、经验的封装 |
| 价值 | 不需要每次从零开始推理 |
| 特点 | 跨项目、跨场景、可持续迭代 |

### 2. Skill构建三层体系
| 层级 | 内容 | 说明 |
|------|------|------|
| 方法论 | 构建原则、设计模式 | 如何设计好的Skill |
| 技术实现 | SKILL.md格式、参数定义 | 如何让AI理解Skill |
| 生态运营 | 分享、发现、评价机制 | 如何让Skill流通 |

### 3. Skill vs Prompt 的本质区别
| 维度 | Prompt | Skill |
|------|--------|-------|
| 作用域 | 单次对话 | 跨会话复用 |
| 可进化 | 否 | 是（可迭代更新）|
| 携带上下文 | 无 | 可携带关联文件、示例 |
| 触发方式 | 手动输入 | 可按路径/关键词自动激活 |

## 观点分歧

### 关于Skill是否需要标准化
| 观点 | 来源 | 立场 |
|------|------|------|
| 开放技能标准（agentskills.io） | [1] Skill Universe | 标准化派 |
| 各自定义、百花齐放 | [2][3] 女娲Skill | 灵活派 |

> **整合理解**：标准化有利于生态流通，个性化有利于场景适配。两者不矛盾，可以标准化框架+个性化内容。

## 核心Skill类型

### 1. Persona Skill（人格技能）
| 类型 | 用途 | 特点 |
|------|------|------|
| 同事.skill | 模拟特定同事的工作方式 | 保留个人风格 |
| 前任.skill | 保留离职员工的经验 | 知识不流失 |
| 自己.skill | 提炼个人最佳实践 | 自进化 |

**构建方法**：
1. 收集该人格的工作样本、文档、决策记录
2. 提炼核心方法论、偏好、禁忌
3. 封装为Skill格式

### 2. Workflow Skill（流程技能）
| 类型 | 用途 | 示例 |
|------|------|------|
| 代码审查.skill | 标准化审查流程 | 步骤+检查清单 |
| 架构评审.skill | 评审设计方案 | 维度+打分表 |
| 发布检查.skill | 规范发布流程 | 检查项+审批流 |

### 3. Knowledge Skill（知识技能）
将特定领域的知识封装为可查询的Skill：
- 领域规则（银行合规、监管要求）
- 技术栈文档（内部框架、中间件）
- API规范（接口定义、使用示例）

## 自进化Skill：达尔文.skill

> 核心灵感：Karpathy 的 autoresearch（7万⭐）—— 把"棘轮机制"从模型训练迁移到 Skill 优化

### 核心机制
借鉴 Karpathy 的 autoresearch：让 AI 自己跑实验、自己评估结果、只保留有改进的修改。autoresearch 优化训练代码（loss 下降=保留），达尔文优化 SKILL.md（分数上升=保留）。

### 与 autoresearch 的关键区别
| 维度 | autoresearch | 达尔文.skill |
|------|-------------|-------------|
| 优化对象 | 训练代码 | SKILL.md |
| 评估标准 | loss 数值（全自主）| 8维度人工可判断 |
| 判断方式 | 机器自主 | 机器初筛 + 人在回路终审 |

**关键设计：Human in the Loop** — Skill 的好坏有时需要人判断，所以达尔文加了人在回路：每次优化后系统暂停，展示 diff、分数变化、测试对比，人确认后才继续。

### 五条原则（踩坑总结）

| # | 原则 | 来源 |
|---|------|------|
| 1 | **单一可编辑资产** — 每次只改一个 SKILL.md | 同时改多个skill无法判断哪个生效 |
| 2 | **双重评估** — 结构评分 + 实测评分 | 结构满分但跑出来很差的情况 |
| 3 | **棘轮机制** — 分数只能升不能降，改差了就 git revert | 允许实验，失败不伤害 |
| 4 | **独立评分** — 修改agent不能是评分agent | 审计独立性（改的人不能自己给自己打分）|
| 5 | **人在回路** — 机器初筛，人做终审 | 目前还是人比机器更懂"好不好" |

### 8维度评分体系（100分制）

**结构维度 60分**：

| 维度 | 分值 | 考察点 |
|------|------|--------|
| Frontmatter 规范 | 8分 | 格式是否标准 |
| 工作流清晰度 | 15分 | 步骤是否完整 |
| 异常处理 | 10分 | 有没有边界情况 |
| 用户确认点 | 7分 | 关键决策前是否暂停 |
| 指令可执行性 | 15分 | 指令是否具体到可直接执行 |
| 文件路径存在性 | 5分 | 引用的文件路径是否真实存在 |

**效果维度 40分**：

| 维度 | 分值 | 考察点 |
|------|------|--------|
| 整体架构合理性 | 15分 | 结构设计是否合理 |
| 实测输出质量 | **25分** | 拿真实 prompt 跑一遍看效果 |

> 实测权重最高（25分）：一个格式完美但跑起来没用的 skill，不如一个写得粗糙但特别管用的 skill。

### 优化流程（5阶段）

```
Phase 0-1: 初始化 + 基线评估（为每个skill设计测试prompt，跑出起点分数）
Phase 2  : 核心循环（找最低分维度 → 针对性修改 → 独立agent评分 → 涨了留/降了回滚，每skill最多3轮）
Phase 3  : 汇总Before/After分数表
```

棘轮示例：`72分 → 78分（留）→ 75分（回滚到78）→ 84分（留）→ 最终78→84，净提升6分`

### 棘轮的美学
> 你可以放心做实验，失败不会伤害你。只有成功会被保留。

科学是棘轮、民主制度是棘轮、git 历史是棘轮——达尔文.skill 把这个古老的结构用在了 Skill 优化这个新地方。

## Skill UI：双形态的关键补全

> 来源：Qoder 官方《Agent Skills 终于有 UI 了》| [5]

### 问题：CLI 是 Agent 的母语，但人不是 Agent

Skill 一直是 Agent 原生的（SKILL.md + scripts/），但缺少 GUI 层：
- **输入端**：Agent 需要人做决策时，一条一条线性地问，效率低（选个设计方向要5轮对话）
- **输出端**：Agent 展示结果时，文本列数字不如一张图表直观

### 方案：Skill 双形态 = CLI（Agent 用）+ GUI（人用）

| 层 | 对象 | 形式 | 说明 |
|----|------|------|------|
| CLI 层 | Agent | SKILL.md + scripts/ | 文本驱动、自描述、可组合 |
| GUI 层 | 人 | Widget UI | 配置面板做决策 + Dashboard 看结果 |

### 两种 UI 模式

| 模式 | 原理 | 适用场景 | 特点 |
|------|------|----------|------|
| 动态生成 | Agent 根据运行时上下文实时生成 HTML/CSS/JS | 数据驱动（性能图表、Diff预览、数据对比） | 灵活，消耗 token |
| 预定义模板 | Skill 作者预写 HTML 放 assets/，通过 `window.__WIDGET_DATA__` 传入上下文 | 交互固定（配置表单、审批流程）| 零 token、毫秒加载、体验一致 |

同一个 Skill 可混合使用两种模式。

### 目录结构示例
```
my-frontend-design-skill/
├── SKILL.md
├── scripts/
│   └── generate.sh
└── assets/
    └── design-direction.html   ← 预定义模板
```

### 安全架构
| 机制 | 说明 |
|------|------|
| 沙箱隔离 | `sandbox="allow-scripts allow-forms"` 的 iframe，不加 `allow-same-origin` |
| 双阶段消毒 | 预览阶段剥离所有 `<script>`；沙箱阶段保留交互，仅移除 iframe/object |
| 第三方库白名单 | 仅放行 cdnjs.cloudflare.com、cdn.jsdelivr.net、unpkg.com、esm.sh |
| 受控接口 | 仅 sendToAgent()、openFile()、openUrl()、copyToClipboard() 四个 |

### 实战：`/create-skill-ui`
在 Qoder Quest 模式下，三轮对话即可为 Skill 加 UI：
1. 描述需求 → 6秒生成初版面板
2. 要求视觉增强（如调性预览图）→ Agent 调用图像生成
3. 加自定义功能（如品牌色拾色器）→ 持久化到 assets/

### 意义
> Skill 从指令集升级为**有知识、有脚本、有界面的完整微应用**。输入端配置面板解决"怎么告诉 Agent 你要什么"，输出端 Dashboard 解决"怎么看懂 Agent 给你的东西"。人和 Agent 的协作闭环才算完整。

## Skill 训练心法：从"能用"到"可靠"

> 来源：solonnliu《让AI变成Super员工的秘密：高效训练Skills》| [6]
> 实战背景：打磨 web-testing Skill（全站自动化测试 → 自动修复 → 自动验证闭环）

### 核心命题

> Skill 的作用，不是让 AI 更聪明，而是让 AI 在你的业务里**更可靠**。

大模型拥有通用能力，但缺乏岗位能力。通用能力解决"能不能做"，岗位能力解决"什么算完成、哪些步骤不能跳、哪些错误已发生过、交付物长什么样"。Skill 本质上是**业务操作系统**——模型提供通用智力，Skill 提供岗位 SOP。

### 四个翻车教训

| # | 坑 | 现象 | 根因 | 修正 |
|---|-----|------|------|------|
| 1 | AI 不会"穷举" | 漏掉藏在 Tab→表格→蓝色链接后的深层页面入口 | AI 靠主观判断决定"是否重要"，看见但没真正点进去 | 不能写"请仔细检查"，要写触发条件+必做动作+结束门槛（如"页面必须滚动到底、Tab切换后必须重新扫描"）|
| 2 | AI 只生成"最像成果的" | 要求三个文件（sitemap.md、test-report.md、test-report.html），AI 只生成了 HTML | HTML 最显眼最复杂，吸走模型注意力和上下文预算 | 顺序写死 + 每生成一个立刻检查存在性 + 阻断式完整性检查 |
| 3 | AI 选"最省事"方案 | 用 `python3 -c` 拼超长脚本，命令行长度上限被打爆 | 模型不具备当前执行环境的工程常识 | Skill 也是"工程生存指南"：禁止超长单行命令、禁止 base64 内嵌、优先写文件再执行 |
| 4 | 最危险的是"看起来生成了" | 报告结构看似完整，但逐页面详细模块被悄悄压缩 | 上下文紧张时 AI 进入"节约模式"，压缩不显眼的结构 | 补结构门禁：checklist 逐项检查，上一项没过不能进下一项 |

### 四条训练方法

| # | 方法 | 要点 |
|---|------|------|
| 1 | 先真跑3-5次，再写 Skill | 没有真实翻车写不出真的 Skill；重点看它怎么错（漏了什么、跳了哪步、哪些地方看起来做了其实没做）|
| 2 | 写 SOP，不写要求 | ❌ "请仔细检查页面" → ✅ "当页面存在 Tab/展开行/子表格时，切换后必须重新枚举链接" |
| 3 | 配 checklist + 门禁 | checklist 告诉 AI 检查什么，门禁告诉 AI 没检查过不能往下走；自觉是加分项，门禁是底线 |
| 4 | 让 AI 参与复盘并自动改 Skill | 不只是手改 Prompt，是让 AI 分析根因后直接修改 Skill，循环迭代直到满意 |

### checklist 示例（web-testing）

```
- [ ] 页面模块数 = sitemap 页面数
- [ ] 每页都有截图
- [ ] 每页都有 UI/UX 审查
- [ ] 每页都有功能测试结果表
- [ ] 每页都有本页问题汇总
- [ ] HTML 中 page-card 数量 = 页面数
- [ ] 问题汇总表存在
- [ ] 修复计划表存在
```

### 与达尔文.skill 的互补

| 维度 | 达尔文.skill | Skill训练心法 |
|------|-------------|--------------|
| 优化方式 | 自动（棘轮+8维度评分）| 人工（真实翻车→复盘→写规则）|
| 适用阶段 | Skill 已有初版，需迭代优化 | Skill 从0到1，需先跑出问题 |
| 核心机制 | 机器初筛+人在回路终审 | 真实任务→发现坑→写SOP+门禁 |
| 关系 | **先用心法建初版，再用达尔文自动优化** | |

> 金句：**训练 Skill 不是在抬高 AI 的上限，而是在抬高 AI 的下限。**

## 构建指南

### Skill设计原则
1. **单一职责**：一个Skill做一件事
2. **清晰边界**：明确输入、输出、适用场景
3. **可观测**：记录使用效果，持续优化
4. **可组合**：Skill之间可以组合使用

### SKILL.md基本结构
```markdown
---
name: skill-name
description: 一句话描述
trigger: 触发条件
---

# Skill名称

## 概述
一句话描述这个Skill做什么

## 触发条件
什么情况下应该使用这个Skill

## 执行步骤
1. 步骤一
2. 步骤二

## 注意事项
- 注意点1
- 注意点2

## 示例
\`\`\`示例输入
...
\`\`\`
```

## Skill设计模式：5种核心架构

> 来源：微信公众号（链接存档于 memory/）

### 5种核心模式

| 模式 | 适用场景 | 代表Skill |
|------|----------|----------|
| **线性流程** | 部署/安装/迁移等有明确步骤的操作 | vercel-deploy |
| **决策树 + 按需加载** | 大型平台选型、产品导航 | cloudflare-deploy |
| **循环迭代** | "做→验证→改进"反复执行 | test-driven-development |
| **接力棒循环** | 跨Session持久化，长期项目 | stitch-loop |
| **多阶段 + 检查点** | 跨越多天/多周，有Go/No-Go决策 | discovery-process |
| **特殊：思维框架** | 深度分析而非快速执行 | audit-context-building |

### 知识3层架构

| 层级 | Token预算 | 内容 |
|------|-----------|------|
| Frontmatter | ~100 | name + description |
| SKILL.md正文 | 2K-5K | 核心指令、决策树、流程步骤 |
| references/ | 1K-3K/个 | 详细文档，按需加载 |

> 总上下文占用建议 <10K tokens（主文件 + 1-2个参考文档）

### 模式选择决策树

```
Skill需要做什么？
├─ 执行有明确步骤的操作 → 模式1：线性流程
├─ 在大量选项中选方向 → 模式2：决策树+按需加载
├─ 单次会话反复做→验证→改进 → 模式3：循环迭代
├─ 跨多个session持续推进 → 模式4：接力棒循环
├─ 跨越多天/多周有阶段决策 → 模式5：多阶段+检查点
└─ 深度分析而非快速执行 → 特殊：思维框架
```


### Anthropic 开放规范 + 5种工具视角设计模式

> 来源：阿里云开发者《Agent Skill规范、构建与设计模式》| [10]
> 补充视角：Anthropic/Google ADK 的工具视角，与"5种核心架构"互补

#### Anthropic Skill 规范标准

| 字段 | 必填 | 说明 |
|------|------|------|
| name | ✅ | 唯一标识，1-64字符，小写字母+连字符，与文件夹名一致 |
| description | ✅ | 触发关键词，1-1024字符，决定何时激活 |
| license | ❌ | 许可证 |
| compatibility | ❌ | 环境兼容性 |
| metadata | ❌ | 扩展元数据 |
| allowed-tools | ❌ | 预授权工具列表（实验性）|

**description 是触发的关键**：写好 description 比写好指令主体更重要——Anthropic 将 description 作为 Skill 识别的核心依据。

#### 三层渐进式加载机制

| 层级 | 内容 | Token预算 |
|------|------|-----------|
| SKILL.md（必需） | Frontmatter + 核心指令 | 2K-5K |
| scripts/（按需）| 可执行脚本 | 按需加载 |
| references/（按需）| 参考文档 | 1K-3K/个 |

> 解决 MCP 上下文膨胀问题——不是一股脑加载，而是按触发条件渐进式注入。

#### Anthropic/Google ADK 5种设计模式

| 模式 | 核心逻辑 | 适用场景 | 关键约束 |
|------|----------|----------|----------|
| **Tool Wrapper** | 封装专家知识为工具集 | 特定技术栈（BigQuery/GKE/Firebase）| 输出格式一致性 |
| **Generator** | 批量生成结构化内容 | 代码/文档/测试批量生成 | 质量门槛控制 |
| **Reviewer** | AI审查AI的产出 | PR审查/安全扫描/代码风格检查 | 必须输出分级报告 |
| **Inversion** | Agent先采访用户，再执行 | 需求不明确的新项目 | 所有阶段完成前不得开始构建 |
| **Pipeline** | 带检查点的严格多步流水线 | 复杂多阶段任务（文档生成/代码生成）| 检查点前不得跳步 |

#### 模式组合推荐

| 组合 | 说明 | 典型场景 |
|------|------|----------|
| Pipeline + Reviewer | 管道最后一步加自动审查 | 文档生成后自动质量检查 |
| Generator + Inversion | 先收集信息再填充模板 | 需用户输入的结构化文档生成 |
| Pipeline + Tool Wrapper | 管道某些步骤加载专家知识 | 多步骤代码生成 |
| Inversion + Pipeline | 先完成需求收集再进入执行流水线 | 复杂项目全流程 |

> 与"5种核心架构"的关系：两者互补——"核心架构"偏**工作流拆分**，ADK模式偏**Agent交互方式**。编码用"循环迭代"+Generator，交付用"多阶段+检查点"+Pipeline，运维用"思维框架"+Reviewer。

## 多Agent协作模式：Subagent vs Agent Team

> 来源：七牛开发者《从 Subagent 到 Agent Team》| [12]
> 核心命题：一个Agent不够用时，如何组织多Agent系统

### 为什么需要多Agent

Agent越强，单Agent承载的任务越重。开发场景中任务天然是碎片的——读代码→查接口→写实现→补测试→看日志→做Review，所有信息堆进同一个上下文，Agent容易丢主线。

多Agent系统形成了两种主流协作方式：

### Subagent：任务委派机制

Subagent = 主Agent派出去的专项助手。主Agent负责整体目标，Subagent只做一件边界清楚的事：

- Subagent A → 读认证相关代码
- Subagent B → 整理测试失败原因
- Subagent C → 安全检查
- Subagent D → Review改动影响

**核心价值**：保持主Agent上下文干净。测试日志几千行、代码搜索几十个文件——这些全塞进主对话会严重干扰判断。交给Subagent，主Agent只收摘要结果。

**适用场景**：查代码、跑测试、读文档、做Review、做检索——边界清楚、结果明确、可独立完成。

### Agent Team：团队协作机制

Agent Team = 多个Agent围绕同一目标协作，按角色分工：规划→实现→测试→Review→总结。

**核心价值**：不只是"并行"，更是**多视角校验**。以代码Review为例：
- 安全Agent：缺少权限校验
- 性能Agent：每次查库接口会变慢
- 测试Agent：应补缓存失效场景
- 维护性Agent：逻辑最好抽成独立函数

**适用场景**：复杂Bug排查、跨模块开发、方案设计、大型重构——需要多角色讨论方向、验证方案、同步信息。

### 选型标准

| | Subagent | Agent Team |
|---|---|---|
| 本质 | 专项助手 | 小项目组 |
| 判断标准 | 边界清楚，只需返回结果 | 需多角色协作、互相验证 |
| 典型任务 | 查代码/跑测试/读文档 | 方案设计/复杂Bug/跨模块开发 |

### 业界实践

| 工具/框架 | 实现方式 |
|---|---|
| LangChain | Supervisor模式，Subagent当工具调用 |
| Claude Code | Subagent处理专项任务，Agent Team协调成员 |
| OpenAI Codex | 并行启动专项agent做安全/质量/Bug检查 |
| AutoGen | Group Chat模式，多Agent共享消息线索协作 |

### 风险提醒

- **Subagent 拆太碎** → 结论零散，主Agent需判断哪些有用、哪些冲突
- **Agent Team 协作混乱** → 谁分配任务？谁判断优先级？冲突时谁做决策？改动重叠怎么办？

> 多Agent价值不在于让更多人说话，而在于让合适的Agent，在合适的位置，完成合适的任务。

## 金融岗位智能体：一岗一助手

> 来源：金融AI coding系列4 | [9]
> 核心命题：万能助手解决个人效率，一岗一助手解决金融级生产流程

### 万能助手的三重局限

| 局限 | 说明 |
|------|------|
| **责任模糊** | 跨越多个岗位，难以追溯"谁确认的、谁采纳的、谁放行的" |
| **上下文混乱** | 产品经理关心业务目标，架构师关心系统边界，测试关心覆盖——一个助手无法承载 |
| **效果无法量化** | 什么都能答一点，什么都不够深入，难以评估真实价值 |

### 一岗一助手五维价值

| 维度 | 说明 |
|------|------|
| **责任边界** | 需求是谁确认的，代码是谁采纳的，故障是谁处置的——边界清楚，责任才清楚 |
| **专业上下文** | 每个岗位有专属知识资产（业务规则/架构文档/测试资产/运维知识）|
| **工作流衔接** | 不是孤立回答，而是流程节点上的执行者和连接器 |
| **权限安全** | 需求智能体不能随意调生产日志，运维智能体不能绕过授权执行 |
| **效果评估** | 岗位智能体可以被量化（代码采纳率/缺陷发现率/故障定位准确率）|

### 全球共识：Agent治理先行

| 机构 | 关键实践 |
|------|----------|
| **DBS** | Agentic control plane，对所有agent监督，强调human oversight + audit trail + fallback机制 |
| **Citi** | 4万开发者用Devin，但**不允许agent部署代码**，只产artifacts，人工review后才放行 |
| **Morgan Stanley** | AI coding assistants成为标准工作流后，瓶颈转向代码审查/测试/安全/验证/部署 |
| **Bank of America** | Human oversight + transparency + accountability，效率提升20%+ |

> 金融行业共识：Agent必须进入工作流，也必须被治理。不是不敢用，而是必须把Agent放进权限边界、审核机制和治理框架里。

### 招行DevAgent样本

| 维度 | 数据 |
|------|------|
| 月均任务 | **4.8万+开发任务**|
| 模式 | ReAct（感知-规划-执行-反馈-进化）|
| 能力 | 现场感知+企业知识检索+任务级开发+跨文件生成 |
| 关键词 | 不是"通用问答"，而是"懂工程、懂企业知识的现场智能体" |

### 工商银行样本

六大岗位全覆盖：**需求→设计→编码→测试→交付→运维**

核心命题：不是"AI能不能写代码"，而是"AI能不能在金融级软件工程体系中稳定工作"。真正难的是让模型进入秩序，让生成进入流程，让结果进入审计，让责任有人承接。

### 六岗位智能体详解

#### 需求智能体：从"写需求"走向"定义意图"
- 把自然语言/会议讨论/业务说明转化为结构化需求资产
- 辅助生成原型，让业务/产品/设计/开发围绕"看得见的东西"讨论
- 产品经理从"翻译"变成"导演"——让业务/AI/设计/开发/测试在同一镜头对齐

#### 设计智能体：从"补文档"走向"沉淀系统边界"
- 理解应用架构/接口文档/表结构/历史代码/公共方法/已有设计
- 设计文档同时给人看、给AI看、给测试看、给交付看
- 架构师从"救火队长"变成"轨道设计师"——轨道铺得越清楚，AI越不容易脱轨

#### 编码智能体：从"敲代码"走向"带AI干活"
- 理解任务→读取上下文→遵守规约→调用工具→生成代码→生成单测→执行自检→自动修复
- 开发人员从"亲自下地干活"变成"带一组AI工人的工长"
- 活可以让AI干，但图纸对不对、工序对不对、质量过不过关，最后人得接得住

#### 测试智能体：从"后面接锅"走向"前面设防"
- 基于需求/设计/代码生成测试案例，覆盖正常/异常/边界/权限/数据口径
- 辅助构造测试数据，生成测试脚本，分析失败原因
- 测试从"最后一道安检"变成"全流程雷达"——起飞前/飞行中/降落前都持续发现风险

#### 交付智能体：从"临门救火"走向"提前控险"
- 资源供给阶段：识别代码库/分支/环境/流水线/发布单元
- 持续集成阶段：监控构建失败/门禁异常/部署异常，分析原因推荐修复
- 版本交付阶段：生成交付报告，识别代码增量/配置变更/门禁异常/环境差异/潜在风险
- 投产前：给出风险提示和处置建议
- 交付从"临门一脚"变成"塔台调度"——天气/跑道/路线/机组状态都要提前看清

#### 运维智能体：从"告警疲劳"走向"故障推理"
- 读取指标/日志/链路/告警/变更记录/历史故障案例
- 形成诊断计划，边查边调整，在多个可能原因中做排除
- 运维从"消防队"变成"城市神经系统"——不仅要救火，还要提前感知哪里升温/哪里堵塞/哪里可能出事

### 上下游协同：资产流转闭环

```
需求智能体输出 → 设计智能体读取 → 编码智能体执行 → 测试智能体验证 → 交付智能体检查 → 运维智能体反馈
                                          ↑                                            ↓
                                    缺陷反馈 ←─────────────────────────────────────────── 故障沉淀
```

一岗一助手不是把每个岗位做成小烟囱，而是把每个岗位变成一段标准化轨道，最后连成一条能跑起来的智能研发生产线。

### 开发中心治理：管人也要管Agent

未来需要一张**智能体组织图**：
- 每个智能体负责什么岗位
- 能调用什么工具
- 能访问什么数据
- 输出什么资产
- 由谁审核
- 进入哪个流程

> 没有这张图，AI越多越乱。有了这张图，AI才能从个人工具变成组织能力。

### 与本体系的关联

| 维度 | 关联 |
|------|------|
| **达尔文.skill** | 岗位智能体是静态分工，达尔文是动态优化——岗位Skill可被达尔文自动迭代 |
| **Skill训练心法** | 岗位智能体的上下文/规约/流程，本质是**岗位级Skill**——每个岗位需要一个专属Skill |
| **5种设计模式** | 岗位智能体可按岗位特点选择：编码用"循环迭代"，交付用"多阶段+检查点"，运维用"思维框架" |
| **Skill UI双形态** | 岗位智能体的配置面板/结果Dashboard，正是Skill UI的应用场景 |

> 金句：**一岗一助手，不是让岗位消失，而是让每个岗位都站到更高的位置。**

## Skill工程化设计：CLI执行层+Workflow机制

> 来源：腾讯技术工程 | [11]
> 核心命题：把 Agent 当"算法"用——确定性程序接管流程，AI 只做决策

### 核心哲学：把 Agent 当"算法"用

> 把 Agent 当成一个算法来用：给它输入，它给你指定格式的输出。中间的推理过程你不关心，但结果是确定的、可预期的。

LLM 是概率模型，不是函数。每次调用都像在概率空间里掷骰子——你能控制它大致落在哪片区域，但不能控制精确的落点。

两个真实的工程痛点：
- **Token 是钱，试错是浪费**：Agent 在模糊需求前反复揣摩、多轮尝试、走了一半发现方向不对再重来——每一步都在烧 Token
- **路径不确定，每次执行像开盲盒**：同一个提示词，换个模型、换个上下文顺序，输出可能完全不同

**解决方案**：不改变河的本性，但给它修好渠。让 Agent 保留"理解人话、做判断、组织表达"的能力，但把所有确定性的事——流程顺序、数据格式、API 调用、状态管理——全部从它脑子里拿出来，交给一个它触碰不到的确定性程序。

### CLI 作为执行层：消除不确定性

Agent 在精确操作上频繁翻车：拼 HTTP 请求时漏了 header、写 YAML 时缩进错了、调接口时把 `userId` 写成了 `user_id`。这些是能力类型不匹配，不是模型不够聪明。

**决策：引入 CLI，让 Agent 只做"大脑"**：

| | Agent（大脑） | CLI（手脚） |
|--|--|--|
| 做什么 | 理解意图、收集参数、组织回复 | 调 API、写文件、管状态 |
| 沟通方式 | 只输出 JSON 参数 | 只返回 JSON 结果 |
| 不确定性 | 有（被限定在决策范围内） | 无（相同输入 = 相同输出） |

### 三层工具管理：让 AI 从不读文件

当工具数量从 5 个扩展到 50 个时，现有方案都会遇到瓶颈：全量写进 SKILL.md 导致上下文爆炸、拆成多个文件让 AI 需要判断读哪个、MCP schema 不可定制。核心问题是 Agent 仍然需要做"我该读什么"的决策。

**解决方案：CLI 强制接管，Agent 不读文件**：

| 层级 | 内容 | Agent 何时可见 |
|------|------|:--:|
| 索引层 | 工具名 + 一句话描述 | 始终可见 |
| 元数据层 | 字段名、类型、是否必填、枚举值 | 用户请求某工具时按需加载 |
| 规则层 | IGNORE/NOTE/ENUM 注解 | 融合在元数据输出中，AI 不直接看到 |

Agent 从未"主动读取"过任何文件。所有信息都是 CLI"喂"给它的——什么时候给什么，由 CLI 决定，不由 Agent 决定。

### discover 热更新：工具规则自动生成

`discover` 命令不仅同步后端工具列表，还自动生成使用规则：

```bash
# CLI discover 自动生成的 .rules 文件示例
IGNORE: id                ← 系统自动分配，别让 Agent 传
IGNORE: insert_time       ← 时间戳，后端自动写
NOTE: host 必须是已加白的有效域名
ENUM: auth_mode: 0-预鉴权, 1-SDK接入, 2-openapi接入, 3-未接入
```

**意义**：后端加了一个新工具，CLI 自动识别出哪些字段是系统字段、哪些有枚举值。连规则都是自动生成的，而不是人写的。

## Workflow 机制：把散落的工具串成可靠的流水线

> Workflow 要解决的三个核心难题：流程顺序、步间通信、中断恢复

### 文件系统即工作流

Workflow 不写在 Skill 代码里，它就是文件系统上的一组 Markdown 文件。新增一个工作流的全部成本：在 `workflows/` 目录下新建一个文件夹 ← 完。

```
workflows/create-permission-set/
├── WorkFlow.md              ← 只有名字 + 一句话描述
└── references/
    ├── 01-collect-project-info.md
    ├── 02-create-project.md
    ├── 03-collect-role-info.md
    ├── 04-create-role.md
    └── 05-summary.md
```

**收益**：业务人员也能改（不用写代码）、版本管理天然支持（每步变更都有 git diff）、排序即逻辑（重排步骤 = 重命名文件前缀）。

### 步进式披露：AI 永远只看到当前一步

把 5 个步骤一次性写进提示词时，AI 做完前两步就开始"即兴创作"。设计：流程不再是提示词里的一段文字，而是文件系统里的一组步骤文件。Agent 通过 CLI 与流程交互，只有两个命令：`--start` 启动流程返回第 1 步内容；`--advance` 提交当前步数据返回下一步要做什么。

**AI 从来不需要"自由"。它需要的是"当前该做什么"的极度清晰。**

### Gate（门禁）：把开放题变成填空题

每个步骤文件头部声明 gate schema——精确定义本步需要提交什么、什么类型、是否必填：

```yaml
gate:
  schema:
    project_name: { type: string, required: true, desc: "项目名称" }
    host: { type: string, required: true, desc: "服务域名" }
```

Agent 调 `--advance --gate-data '{...}'` 时，CLI 做三件事：required 字段没给？拒绝推进；schema 没声明的字段？直接丢弃；全部满足？存储数据，推进下一步。

**约束越强，AI 反而越自由——当"完成标准"是确定性的 schema 时，Agent 的认知负担从"判断什么算完成"降级为"把这几个格子填好"。**

### 状态持久化：记忆不在 AI 脑子里

CLI 把整个流程的运行状态写到磁盘上的 JSON 文件（`.state`）。任何时候、任何会话、任何模型——调一个 `--current` 就能从断点续上。

### 自举：造工具的工具

workflow-creator Skill 让 AI 自己生成 Workflow——不只能运行，还能自己长出新的能力：

```
设计（Agent + workflow-creator）→ 构建（CLI 写文件）→ 执行（运行时 CLI 的工作流引擎）→ 用户提出新需求 → 回到"设计"
```

### 对金融机构的意义

| 痛点 | 解决思路 |
|------|----------|
| AI 在精确操作上不可靠 | CLI 接管所有精确操作（HTTP/字段名/格式）|
| Skill 能力扩展 = 修改代码 | Workflow 是文件系统，新增能力 = 新建文件夹 |
| 工具多了上下文爆炸 | 三层工具管理，Agent 只看索引表 |
| 多步骤流程 AI 跳步骤 | Gate 把开放题变成填空题 |
| IDE 崩了流程中断 | state 文件持久化，任何时候 --current 续上 |

## 第三块拼图：MetaSkill组织（花叔实践）

基于OpenSquilla项目的MetaSkill实践，与女娲（造技能）和达尔文（优化技能）形成完整闭环。[3]

### 三块拼图

| 拼图 | 工具 | 作用 |
|------|------|------|
| **造** | 女娲 | 把思维方式蒸馏成可跑Skill |
| **优化** | 达尔文 | Skill自己进化，分数低的自动迭代到高的 |
| **组织** | MetaSkill | 把多个技能组织成真能跑的流水线 |

### MetaSkill本质

**传统工作流编排**：提前把流程写死，Agent照着画好的线跑（我给你画好路线图）

**MetaSkill**：不写流程，只把规则和能用的技能告诉模型，让它自己现编一条流程出来（我给你地图和规则，你自己找路）

关键：既让Agent自己组织，又给它上护栏——依赖顺序、工具白名单、风险等级、模板安全都是被强制校验的。

### 卷Harness不卷模型

**智能路由**：简单任务→国产便宜模型，难任务→旗舰模型+深度思考，跨厂商组合
- 同一道简单HTTP题：火山方舟mini vs Opus差16倍价格
- 跨厂商路由：DeepSeek flash处理简单题，Claude Opus处理复杂推理题

**核心观点**：让用户为每次对话操心选哪个模型，产品没做好分内的事。自动路由把这负担免掉。

### 核心洞察

> 当技能多到一定程度，瓶颈早就不是模型聪不聪明，而是这一堆技能能不能被组织起来。

行业注意力几乎都在「模型又涨了几分」上。但工作流编排，正在变得比参数量更重要。

## Harness Engineering：给Agent套上确定性安全网

> 来源：InfoQ 整理 Thoughtworks Birgitta Böckeler QCon 演讲 | [13]
> 核心命题：把确定性约束系统（lint/结构测试/ArchUnit）包裹非确定性模型产出

### Context Engineering 演进

| 阶段 | 内容 |
|------|------|
| 一年前 | rules files（AGENTS.md/CLAUDE.md）|
| 今天 | skills + subagents + MCP + plugins + specs |

**核心**：精心筛选模型能看的信息，获得更好结果。

### Skills：模块化 + 按需加载

1. **模块化**：rules 拆成小 skill 文件夹，按功能分类
2. **按需加载**：LLM 先看 skill description，需要时才加载完整内容
3. **不止 Markdown**：skill 文件夹里可以放脚本让 agent 直接执行

### Subagents：上下文隔离

- 主 agent 派生子 agent 做专项任务，只把结果汇报回主 session
- 常见用法：code review agent（无历史上下文污染的独立窗口）

### 风险评估三维度

| 维度 | 问题 |
|------|------|
| 概率 | 它出错的概率有多大？ |
| 影响 | 如果出错，后果有多严重？ |
| 可检测性 | 你能不能发现它出了错？ |

### Harness Engineering：确定性约束包裹非确定性模型

| 机制 | 说明 |
|------|------|
| 结构性测试 | ArchUnit、Spring Modulith、dependency-cruiser |
| Lint 规则 | 可「增强语义」——错误信息里附加解释 |
| 自动化反馈 | 测试失败 → agent 收到反馈 → 自动修复 |

### 成本现实

- 有人日均消耗 **$380** → 年化 **$91,200**（已超过资深开发者薪资）
- 模型能力进步是「背景变量」，surrounding system 才是关键

### 未来展望

> 也许未来我们不再靠传统服务模板起步，而是一个 **Harness 模板**，实例化之后就能支撑代码库。

### Hook：Agent Runtime 的治理中间层

> 来源：下班后的AI局 | [14]
> 核心：Tool=能力层，Hook=治理层。Agent Runtime 在关键节点上长出可被外部规则介入的接口。

#### Tool vs Hook 的本质区别

| 维度 | Tool（能力层）| Hook（治理层）|
|------|-------------|-------------|
| 回答 | Agent 能做什么？ | Agent 按什么规则、受谁监督地做？|
| 示例 | BashTool、FileEditTool、BrowserTool | PreToolUse、PermissionRequest、PostToolUse |
| 地位 | Agent 动起来 | Agent 的行动进入严肃场景 |

#### Agent Runtime 的四层定位

| 层 | 说明 |
|-----|------|
| MCP | 扩工具 — Agent 多几只手 |
| Skill | 扩方法 — Agent 多一些做事方法 |
| Plugin | 扩分发 — 能力易安装和分发 |
| **Hook** | **扩治理 — Runtime 在关键节点被外部规则接管** |

#### Claude Code 已开放的 Hook 节点

| 节点 | 时机 |
|------|------|
| SessionStart | session 启动时 |
| UserPromptSubmit | 用户提交 prompt 前 |
| PreToolUse | 工具调用前（可拦截）|
| PostToolUse | 工具调用后（可审计/补充上下文）|
| PermissionRequest | 权限请求时 |
| PreCompact | 上下文压缩前（可预判风险）|
| PostCompact | 压缩后 |
| SubagentStart | 子 agent 启动时 |
| TaskCompleted | 任务完成时 |
| SessionEnd | session 结束时 |

#### Hook 能做四类事情

| 功能 | 说明 |
|------|------|
| 🚫 拦截 | 在模型准备执行前阻止危险操作 |
| 📝 补充 | 在工具执行后注入上下文或审计信息 |
| 📋 记录 | 记录耗时、成本、失败原因 |
| 🔄 接管 | 把控制权交给用户或外部系统 |
| 🔀 改变后续流程 | 影响模型下一步的行为选择 |

#### 企业绕不开 Hook

> 这些规则不可能全部写死在一个通用产品里。
> **Prompt 负责提醒模型，Hook 负责在运行时把规则落地。**

| 治理能力 | 解决什么问题 |
|----------|-------------|
| 权限检查 | 危险操作审批、工具调用拦截 |
| 审计记录 | 记录谁让 Agent 做了什么 |
| 成本控制 | token 预算、模型路由、超额提醒 |
| 上下文注入 | session start 时补充项目规则 |
| 工作流联动 | 任务完成后同步 Jira、Slack、飞书 |
| 评测采样 | 任务结束后沉淀评测样本 |
| 记忆沉淀 | 从完成任务里抽取长期记忆 |
| 安全检查 | 敏感文件、外发数据、危险命令检查 |

#### Hook 也是采样点

- `PostToolUseFailure` → 收集工具失败样本
- `PermissionDenied` → 收集模型越界尝试
- 用户接管位置 → 标注模型什么时候需要人帮忙
- `TaskCompleted` → 沉淀成功任务轨迹

> 一个好的 Agent Harness，不只是把模型包起来调用工具，还应该把真实任务轨迹收集起来，变成评测和训练的原料。

#### 与本体系关联

| 维度 | 关联 |
|------|------|
| **Harness Engineering** | Hook 是 Harness 在 Runtime 层的具体实现入口 |
| **Context Engineering** | SessionStart Hook 注入项目规则，PreCompact 预判风险 |
| **企业实践** | 企业级 Agent 治理的核心机制 |

## Agent Harness 综述：ETCLOVG 七层框架

> 来源：Datawhale 整理 CMU/Yale/JHU/Virginia Tech/Amazon 联合综述 | [15]
> 论文：Agent Harness Engineering: A Survey
> 核心：Agent 的下一场竞争，不只是模型能力，而是模型外面的工程外壳。

### 三次范式迁移

| 阶段 | 核心问题 | 工程对象 |
|------|----------|----------|
| Prompt Engineering (2022-2023) | 怎么跟模型说话？ | 提示词、few-shot、思维链 |
| Context Engineering (2024-2025) | 模型该看见什么？ | 上下文管理、记忆检索、窗口控制 |
| **Harness Engineering** (2025-) | 模型在真实世界怎么可靠干活？ | 执行环境、工具接口、编排、可观测、验证、治理 |

### ETCLOVG 七层框架

| 层 | 英文 | 解决什么问题 |
|----|------|-------------|
| **E**xecution | 执行环境 | Agent 在哪里跑？本地/容器/沙箱/桌面？边界在哪？|
| **T**ooling | 工具接口 | 工具怎么描述、发现、调用？如何防止模型乱选工具？|
| **C**ontext | 上下文和记忆 | 短期上下文、会话状态、长期记忆怎么管理？|
| **L**ifecycle | 生命周期和编排 | 单轮还是多轮？单Agent还是 planner+executor+reviewer？|
| **O**bservability | 可观测性 | 每次调用、工具执行、token成本、延迟都能追踪？|
| **V**erification | 验证和评估 | 结果对不对？失败原因在模型/工具/上下文还是环境？|
| **G**overnance | 治理和安全 | 权限、审批、审计是否闭环？|

### 关键发现

**不改模型，只改 Harness 最高提升 10 倍**
- 修改编辑工具格式和周边 harness，coding benchmark 上最高带来 10 倍提升
- 固定 GPT-5.2-Codex，通过重构 system prompt + 中间件注入 + 自验证 hooks，Terminal-Bench 2.0 从 52.8%→66.5%
- Meta-Harness 自动优化 harness，Terminal-Bench-2 达 76.4%，超手工方案

**可观测性和治理是独立层，不是附属功能**
- 没有可观测性 → Agent 失败了你不知道原因
- 没有治理 → Agent 成功了你也不敢用

**评估要 trace-native**
- 记录完整执行轨迹：模型输出、工具调用、环境状态、上下文快照、错误、重试、恢复动作、token 用量、延迟
- 判断三件事：结果是否正确、路径是否合理、评估器本身是否可信

### 三大矛盾

| 矛盾 | 表现 |
|------|------|
| **成本-质量-速度三角** | 越安全→更强沙箱/更细权限/更完整 trace→成本和延迟上升 |
| **能力-控制矛盾** | 工具越多→越容易选错+prompt injection面越大；执行环境越开放→失控半径越大 |
| **Harness 耦合** | 改 prompt/工具/memory/sandbox/verifier → 可能改变整个系统行为 |

### 从 Agent Framework 到 Agent Platform

| Framework | Platform |
|-----------|----------|
| Agent、tool、memory、loop | Durable workspace、managed sandbox、identity、billing |
| 局部抽象 | Observability、evaluation、governance、human handoff |
| 最快搭 loop | 长期可靠运行 |

### 少加脚手架

> 好 Harness 不只是会加控制，还要知道什么时候删控制。

- 每一个 wrapper/reset/verifier/planner/permission gate 都代表一个假设：模型自己做不好
- 模型变强后，某些控制可能不再必要，甚至会拖后腿
- Anthropic 案例：某些 context reset 对旧模型有用，但对强模型可去掉 → 成本下降，质量不降

### 与本体系关联

| 维度 | 关联 |
|------|------|
| **Harness Engineering** | ETCLOVG 为 Harness 提供完整架构参考，七层全覆盖 |
| **Hook** | Governance 层的具体实现入口，Runtime 可编程治理 |
| **AI-SDC 方案** | 可直接基于 ETCLOVG 框架评估和设计我行的 Agent 工程体系 |

## Anthropic Skill 方法论：Context Engineering 的最佳实践

> 来源：AI产品阿颖（整理自 Claude Code 团队博客 + Perplexity 实践）| [16]
> 核心：Skill 本质上是在做 Context Engineering——决定模型该看见什么、什么时候看见。

### Skill 文件夹 = 渐进暴露架构

| 层级 | 内容 | 何时加载 |
|------|------|----------|
| **SKILL.md** | 导航页 + Gotchas（踩坑经验）| 始终加载 |
| **references/** | 详细说明、API参考、边界条件 | 需要时才加载 |
| **scripts/** | 可执行脚本（查询、排查、分析）| 需要执行时 |
| **examples/** | 案例参考 | 需要类比时 |
| **assets/** | 模板、图片、固定素材 | 渲染输出时 |

> 如果所有信息都塞进 SKILL.md → 上下文爆炸。每次调用都重读一遍无用信息。
> SKILL.md 是导航页，不是数据库。它告诉模型该去哪找什么。

### Scripts > Instructions

| 维度 | Instructions | Scripts |
|------|-------------|--------|
| 解决什么问题 | 经验与判断（为什么要查）| 能力与执行（怎么查）|
| Token 成本 | 每次推理消耗 | 一次执行，zero token |
| 准确性 | 模型可能漏步或跳步 | 确定性执行 |
| 复用 | 每次重新推理 | 固化最佳实践 |

**原则**：不要让模型把有限的上下文和推理能力浪费在重复劳动上。把这些事情交给脚本。

### Gotchas > 常识

Skill 真正有价值的是**老师傅踩过的坑**，不是模型早已掌握的通识：

```
✓ 这个表不能按 created_at 排序
✓ staging 返回 200 不代表成功
✓ request_id 和 trace_id 是同一个东西
✗ HTTP 协议是什么
```

### Description = 路由规则，不是功能介绍

大多数人写 Description 的方式天然是错的：

| ❌ 错误写法 | ✅ 正确写法 |
|------------|------------|
| "PR Management Skill 帮助用户监控 PR 状态" | "用户说「帮我盯一下这个 PR」时加载" |

**检查方法**：写完 Description 后把整个 Skill 删掉，只保留这一行。问自己：模型看到用户的问题后，能不能知道什么时候该加载这个 Skill。

### Skill 管理轻量化

当 Skill 从几个变成几十个甚至几百个时，管理是关键：

| 阶段 | 方式 |
|------|------|
| 小团队 | `.claude/skills/` 项目目录共享 |
| 增长期 | 小范围试用 → 同事自行安装试用 |
| 成熟期 | 自然选择后提交 Marketplace |

**核心**：不先讨论 Skill 有没有价值，而是先让它在真实场景中接受检验。用的人多了，自然进入正式体系。

### 与本体系关联

| 维度 | 关联 |
|------|------|
| **女娲Skill** | 蒸馏机制 + 渐进暴露，本质一致 |
| **达尔文.skill** | Gotchas + Scripts = 达尔文优化的对象 |
| **Skill训练心法** | Scripts > Instructions 与「四翻车教训」高度吻合 |
| **5层Skill架构** | Anthropic 3层（md/references/scripts）vs 我们5层（多了examples/assets）|

## NLAH：自然语言智能体驾驭层——Harness 的新范式

> 来源：阿里云云原生（解读论文 Natural-Language Agent Harnesses）| [17]
> 核心：把驾驭策略从代码中外置为可执行的自然语言对象，让 Harness 本身成为可研究、可移植、可消融的工程对象。

### Agent 控制的三种范式

| 范式 | 控制方式 | 特点 |
|------|----------|------|
| **代码驾驭层**（Code Harness）| 程序逻辑对模型施加硬性外部控制 | 最强确定性，但策略与实现细节交织 |
| **NLAH+IHR** ⭐ | 自然语言文档描述策略 + 共享运行时执行 | 可读、可移植、可做消融实验 |
| **自驾驭**（Self-harnessing）| 控制器模型直接驾驭其他模型 | 无外部驾驭层，未来设计 |

### NLAH + IHR 四层架构

| 层 | 比喻 | 说明 |
|----|------|------|
| **基础智能体** | 马 | 最小化 agent 循环，LLM+唯一工具=终端，是一块白板 |
| **运行时策略** | 骑手 | 固定NL指令，把基础智能体变成IHR（对所有任务通用） |
| **NLAH 文档** | 路线图 | 可编辑、可替换的NL文档，描述阶段/角色/状态/验收/恢复/停止规则 |
| **脚本与适配器** | 信号灯 | 不需要LLM判断的硬逻辑（JSON校验、diff计算等） |

### 三个研究发现

**RQ1（驾驭实现）**：NLAH 与代码驾驭层取得相当成绩
- Live-SWE：6.01万 token 代码 → 2.9k token NLAH（压缩 20 倍）
- MHTBA：1.05万 → 0.8k
- 代价：当前原型运行时 token/调用开销更高

**RQ2（驾驭机制遵循）**：契约式/即时验证型机制遵循度高，跨阶段/跨子agent协调型遵循度低

**RQ3（模块消融）**：首次可在策略层而非代码层做消融实验

| 模块 | 价值 |
|------|------|
| ✅ **文件持久化状态** | 最稳的正贡献（agent把中间状态外化到文件比依赖上下文记忆可靠） |
| ✅ 自我进化 | 最高绝对性能，但 token 消耗大幅上升 |
| ❌ **多候选搜索** | 分散模型注意力且昂贵，反而损害性能 |
| ❌ **上下文压缩** | 省了 token 却丢了关键信息，在两个任务上都有害 |
| ⚠️ Markdown memory | 表现不稳定，收益依任务而定 |

### 与本体系关联

| 维度 | 关联 |
|------|------|
| **ETCLOVG 七层框架** | NLAH 是 Governance 层和 Lifecycle 层的具体实现路径之一 |
| **Hook** | Hook 是代码层硬性控制，NLAH 是 NL 层柔性控制，两者互补 |
| **Anthropic Skill** | Skill 文件夹+渐进暴露 vs NLAH 四层架构，思路一致但粒度不同 |
| **AI-SDC** | 我行的 Agent 治理可在代码钩子(Hook)和 NL 策略(NLAH)之间选择 |
