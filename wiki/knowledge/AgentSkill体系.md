# Agent Skill 体系

> 整合Skill宇宙、女娲Skill、同事Skill等Skill构建方法论

---

## 核心共识

### 1. Skill = 能力封装单元
Skill是将AI能力固化为可复用模块的核心机制：[1]

| 维度 | 说明 |
|------|------|
| 定义 | 可复用的工作流、知识、经验的封装 |
| 价值 | 不需要每次从零开始推理 |
| 特点 | 跨项目、跨场景、可持续迭代 |

### 2. Skill构建三层体系
| 层级 | 内容 | 说明 |
|------|------|------|
| 方法论 | 构建原则、设计模式 | 如何设计好的Skill |
| 技术实现 | SKILL.md格式、参数定义 | 如何让AI理解Skill |
| 生态运营 | 分享、发现、评价机制 | 如何让Skill流通 |

### 3. Skill vs Prompt 的本质区别
| 维度 | Prompt | Skill |
|------|--------|-------|
| 作用域 | 单次对话 | 跨会话复用 |
| 可进化 | 否 | 是（可迭代更新） |
| 携带上下文 | 无 | 可携带关联文件、示例 |
| 触发方式 | 手动输入 | 可按路径/关键词自动激活 |

## 观点分歧

### 关于Skill是否需要标准化
| 观点 | 来源 | 立场 |
|------|------|------|
| 开放技能标准（agentskills.io） | [1] Skill Universe | 标准化派 |
| 各自定义、百花齐放 | [2][3] 女娲Skill | 灵活派 |

> **整合理解**：标准化有利于生态流通，个性化有利于场景适配。两者不矛盾，可以标准化框架+个性化内容。

## 核心Skill类型

### 1. Persona Skill（人格技能）
| 类型 | 用途 | 特点 |
|------|------|------|
| 同事.skill | 模拟特定同事的工作方式 | 保留个人风格 |
| 前任.skill | 保留离职员工的经验 | 知识不流失 |
| 自己.skill | 提炼个人最佳实践 | 自进化 |

**构建方法**：
1. 收集该人格的工作样本、文档、决策记录
2. 提炼核心方法论、偏好、禁忌
3. 封装为Skill格式

### 2. Workflow Skill（流程技能）
| 类型 | 用途 | 示例 |
|------|------|------|
| 代码审查.skill | 标准化审查流程 | 步骤+检查清单 |
| 架构评审.skill | 评审设计方案 | 维度+打分表 |
| 发布检查.skill | 规范发布流程 | 检查项+审批流 |

### 3. Knowledge Skill（知识技能）
将特定领域的知识封装为可查询的Skill：
- 领域规则（银行合规、监管要求）
- 技术栈文档（内部框架、中间件）
- API规范（接口定义、使用示例）

## 女娲Skill技术解析

### 核心特性
| 特性 | 说明 |
|------|------|
| 三重验证 | 格式、语法、语义检查 |
| 蒸馏机制 | 从对话中提炼Skill |
| 自动分类 | 按领域/场景归类 |
| 版本管理 | 支持迭代更新 |

### 技术架构
```
对话记录 → 模式识别 → Skill模板 → 质量验证 → 发布
```

### 使用场景
1. **新人入职**：加载"资深工程师.skill"，快速上手
2. **项目交接**：加载"前任.skill"，保留知识
3. **跨团队协作**：加载"XX团队.skill"，理解对方工作方式

## 构建指南

### Skill设计原则
1. **单一职责**：一个Skill做一件事
2. **清晰边界**：明确输入、输出、适用场景
3. **可观测**：记录使用效果，持续优化
4. **可组合**：Skill之间可以组合使用

### SKILL.md基本结构
```markdown
---
name: skill-name
description: 一句话描述
trigger: 触发条件
---

# Skill名称

## 概述
一句话描述这个Skill做什么

## 触发条件
什么情况下应该使用这个Skill

## 执行步骤
1. 步骤一
2. 步骤二

## 注意事项
- 注意点1
- 注意点2

## 示例
\`\`\`示例输入
...
\`\`\`
```
