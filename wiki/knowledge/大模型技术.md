## DeepSeek-V4 发布：百万上下文+Agent能力跃升（2026-04-26）

### 核心发布

DeepSeek-V4 预览版正式上线并开源，分为两个版本：

| 版本 | 定位 | 核心特点 |
|------|------|----------|
| **V4-Pro** | 性能旗舰 | 比肩顶级闭源模型，Agent能力大幅提升 |
| **V4-Flash** | 经济之选 | 接近Pro的推理能力，更快捷经济的API服务 |

### 关键能力

**1. 百万字超长上下文（1M）**
- 开创全新注意力机制，token维度压缩 + DSA稀疏注意力（DeepSeek Sparse Attention）
- 1M上下文成为DeepSeek所有官方服务标配
- 大幅降低计算和显存需求

**2. Agent能力大幅提高**
- Agentic Coding评测达到当前开源模型最佳水平
- 内部员工反馈：使用体验优于 Sonnet 4.5，交付质量接近 Opus 4.6 非思考模式
- 针对 Claude Code、OpenClaw、OpenCode、CodeBuddy 等主流Agent产品适配优化

**3. 世界知识**
- V4-Pro大幅领先其他开源模型，仅稍逊于Gemini-Pro-3.1

**4. 推理性能**
- 数学、STEM、竞赛型代码测评超越所有已公开评测开源模型
- 比肩世界顶级闭源模型

### API变更

- 新模型名：`deepseek-v4-pro`、`deepseek-v4-flash`
- 旧模型名 `deepseek-chat` / `deepseek-reasoner` 将于 **2026-07-24** 停止使用
- 支持非思考模式与思考模式（reasoning_effort参数：high/max）
- 复杂Agent场景建议使用思考模式并设置max

### 开源资源

- HuggingFace: https://huggingface.co/collections/deepseek-ai/deepseek-v4
- ModelScope: https://modelscope.cn/collections/deepseek-ai/DeepSeek-V4
- 技术报告: https://huggingface.co/deepseek-ai/DeepSeek-V4-Pro/blob/main/DeepSeek_V4.pdf

### 对我行的影响

1. **Agent Coding工具选型参考**: V4-Pro在Agentic Coding上达到开源最佳，可作为内部Agent框架的底层模型评估对象
2. **长上下文场景**: 1M上下文适合代码库级分析、大型文档处理等银行研发场景
3. **API迁移提醒**: 旧模型名3个月后停用，需关注现有对接服务的兼容性
4. **私有化部署**: 开源权重支持本地部署，符合银行数据安全要求

## Pinecone "做空" RAG：知识编译范式崛起（2026-05-12）

> 来源：InfoQ | [2]
> 链接：https://mp.weixin.qq.com/s/7nz7Zj0YPmEr7TFWcX2jXA

### 核心事件

Pinecone（向量数据库开创者）发布面向 Agent 的知识引擎 **Nexus**，几乎等于亲口宣布 RAG 时代结束。过去4年 Pinecone 一直在推动 RAG 范式，现在却告诉开发者：retrieve-read-retrieve 循环成了瓶颈。

### RAG 的三大痛点

| 痛点 | 数据 | 原因 |
|------|------|------|
| 任务完成率低 | 仅50%-60% | Agent 陷在 retrieve-read-retrieve 循环 |
| 效率极低 | 85%精力消耗在"找上下文" | 每次临时抓取20段chunk，再消耗token理解 |
| 输出不稳定 | 缺乏字段级grounding | Agent无法区分事实vs猜测，无法引用来源 |

> Pinecone 把 retrieve-at-inference 形容为"Agent 检索的十条蓝色链接时代"——和早期搜索引擎一样，只负责甩链接，用户自己拼信息。

### Knowledge Compilation（知识编译）

**核心理念**：把"推理"提前——提前预编译源数据为带类型、可引用、面向具体任务的知识产物（artifacts），Agent 查询的不再是原始语料库，而是整理好的结果。

**类比 SQL 的诞生**：SQL 出现前，每个应用都自己实现数据访问层；SQL 给关系型数据库提供统一接口后，应用生态才真正建立。现在 Agent 也走到了类似的时刻。

### KnowQL：知识请求语言

6个原语封装进一次声明式调用：

| 原语 | 说明 |
|------|------|
| intent | 意图——Agent想得到什么 |
| filter | 过滤——限制范围 |
| provenance | 来源——答案来自哪里 |
| output shape | 输出格式——结构化而非原始文本 |
| confidence | 置信度——可信度标注 |
| latency budget | 延迟预算——如500ms内返回 |

> Pinecone 声称：任务完成率→90%+，token开销↓90%（待生产环境验证）

### 行业趋势：推理往上游移动

| 产品/方案 | 做的事 | 本质 |
|-----------|--------|------|
| Anthropic Skills | 预编译可复用的上下文包 | 推理提前 |
| Cursor Rules | 编辑器层预编译 | 推理提前 |
| Claude Code subagents | 为不同任务提前打包上下文+工具 | 推理提前 |
| LangChain context engineering | Harrison Chase强调的上下文工程 | 推理提前 |
| Pinecone Nexus | retrieval层预编译 | 推理提前 |

> "编译一次，多次复用（compile once, read many times）"比每次推理时临时抓取更适合 Agent 工作负载。

### 对我行的影响

1. **RAG不是终点**：我行若正在构建知识检索系统，应关注 Knowledge Compilation 趋势，避免过度投入传统 RAG pipeline
2. **Skill体系契合趋势**：我们正在建设的 AgentSkill 体系本质也是"预编译上下文"，与 Pinecone 方向一致
3. **检索基础设施化**：向量搜索将变成"水电煤"，真正的产品层在知识编译

## Dreaming V3：ChatGPT长期记忆架构（OpenAI 2026）

### 记忆三阶段演进

| 版本 | 时间 | 机制 | 时效正确率 |
|------|------|------|-----------|
| 保存记忆 | 2024.4 | 你告诉它记什么，没说的就忘 | 9.4% |
| Dreaming V0 | 2025.4 | 自动从聊天历史提取上下文（给旧架构打补丁） | 52.2% |
| **Dreaming V3** | **2026.6** | 后台异步聚类+分层摘要归档，时间感知修正 | **75.1%** |

### Dreaming V3核心能力

**时效感知**：后台感知时间流逝，自动识别状态切换（出差"即将去"→"已去过"），推荐起点自动更新。

**偏好跟随**：分散在一年对话里的零散偏好自动整合，旅行规划不再给通用攻略（成功率31.4%→71.3%）。

**算力降本**：维持记忆的计算量砍5倍（降幅80%），长期记忆从Plus/Pro付费专属→免费用户标配。

### 核心观点

> 无记忆AI = 随时跑路的临时工  
> 有状态记忆 = 数字员工的入场券

只有具备长期、可伸缩、能自我修正的记忆基础，Agent才有资格被真正委以重任。
