# 知识管理方法论

> 整合Karpathy的LLM知识库方法论与llm.wiki实战

---

## 核心共识

### 1. 双层分离存储
所有方法论都强调两层分离：[1][2]

| 层级 | 说明 | 特点 |
|------|------|------|
| **raw/** | 原始资料缓存 | 只追加不修改 |
| **wiki/** | 结构化知识库 | 持续更新整理 |

**Karpathy核心观点**：[1]
- 知识来源（raw）= 原始文档、网页、笔记
- 知识沉淀（wiki）= AI提炼、关联、更新
- 两层分离的好处：来源可追溯，沉淀可持续

### 2. 知识编译 vs RAG
| 模式 | 特点 | 局限 |
|------|------|------|
| RAG | 每次检索碎片化，无积累 | 无法跨文档关联 |
| 知识编译 | 编译一次，持续维护 | 需要好的解析工具 |

**llm.wiki的价值**：[2]
- 不是索引文档等着检索
- 而是立刻读、提炼、编入已有wiki
- 更新实体页、修订概念页、标注矛盾

### 3. 六步流程
Karpathy方法论核心：[1]

```
原始资料库 → AI编译wiki → 对wiki提问 → 多格式输出 → 定期健康检查 → 定制工具
```

## 观点分歧

### 关于解析工具的选择
| 观点 | 来源 | 建议 |
|------|------|------|
| pdfplumber | [2] | 简单文档可用 |
| TextIn API | [2] | 复杂格式推荐，信息密度更高 |

> **整合理解**：Garbage in, garbage out，解析质量决定知识库质量。优先选择能保留格式的解析工具。

### 关于是否自动更新raw层
| 观点 | 来源 | 立场 |
|------|------|------|
| raw层只追加不更新 | [2] 用户确认 | 保守派：保留原始记录 |
| 支持动态更新 | [2] | 演进派：保持新鲜度 |

> **整合理解**：raw层保持追加原则，wiki层按需更新，两者职责分离。

## llm.wiki 三层架构详解

### 架构图
```
raw/          → 原始材料（markdown/pdf/网页）
     ↓
wiki/         → AI维护的知识中间层
     ↓
AGENTS.md     → 告诉AI wiki怎么组织
```

### wiki/ 内部结构
| 目录 | 内容 | 说明 |
|------|------|------|
| entities/ | 实体页 | 具体概念、工具、人 |
| concepts/ | 概念页 | 方法论、原理 |
| comparisons/ | 对比页 | 不同方案的异同 |

### 关键机制
1. **实体关联**：每个实体都有明确的跨文档链接
2. **矛盾标注**：自动发现并标注不同文章的观点冲突
3. **持续维护**：不是一次性整理，而是增量更新

## 跨文档关联的价值

### RAG做不到的事
- 发现"两个人用不同方式解决同一个问题"
- 自动构建正反观点对比
- 回答"XX跟YY的商业化策略有什么不同"

### llm.wiki能做到
- Harness概念页同时容纳正反双方观点
- 表格对比Ralph Wiggum vs OpenAI vs Anthropic的流派差异
- 跨文档综合问题回答

---

## 实践要点

### 解析优化
1. **避免模型偷懒**：一份份处理，超长文档分段 [2]
2. **选择好的解析工具**：TextIn > pdfplumber
3. **保留格式信息**：表格用HTML表示，图片标注位置

### 质量保障
1. **信息密度 > 文件数量**：wiki页面应有实质内容，不是目录
2. **定期健康检查**：清理过时内容，更新过期信息
3. **人工审核关键决策**：AI提炼可能有偏差

## 知识分层架构扩展：五层存储模型

### 核心框架

腾讯AI Team实践提出的五层存储架构，与Karpathy双层分离思想形成互补：[3]

| 层级 | 存储位置 | 共享范围 | 生命周期 | 示例 |
|------|---------|---------|---------|------|
| Layer 0-P | 个人本地 ~/.ai-team/ | 纯个人 | 永久 | 缩进偏好、函数风格 |
| Layer 0-T | team-conventions/ | 团队级，Git共享 | 稳定 | 代码规范、Commit标准 |
| Layer 1 | tech-wiki/ | 跨项目通用 | 较长 | Spring Boot多租户模式 |
| Layer 2 | biz-wiki/[业务域]/ | 按业务领域 | 业务周期 | 广告审核流程规则 |
| Layer 3 | docs/knowledge/ | 仅当前项目 | 项目周期 | 本项目用TencentDB |

### 知识提升机制

知识可以"向上提升"，实现跨项目复用：

Layer 3 → 是否项目特有？→ 否 → 是否通用技术？→ 是 → 提升到 Layer 1 (tech-wiki)
Layer 3 → 是否项目特有？→ 否 → 是否通用业务？→ 是 → 提升到 Layer 2 (biz-wiki)

---

## 五种知识类型

定义知识的"形状"，便于结构化管理：[3]

| 类型 | 回答的问题 | 定义 | 示例 |
|------|-----------|------|------|
| model | 是什么？ | 实体定义、数据结构、关系图 | 广告计划包含预算/出价/投放时段三字段 |
| decision | 为什么选这个？ | 技术选型、架构决策及理由 | 选事件驱动而非RPC，因需解耦状态变更 |
| guideline | 怎么做才对？ | 推荐或禁止做法 | 公共模块变更后需做兼容性检查 |
| pitfall | 有什么坑？ | 已知风险、故障模式、排查步骤 | 广告预算扣减高并发下会超扣 |
| process | 怎么走的？ | 业务流程、状态机、操作步骤 | 广告审核：提交→机审→人审→上线 |

## 三级成熟度 + 自动衰减

知识需要生命周期管理：[3]

- draft：新提取，单一来源，置信度0.5-0.6
- verified：1个工作流成功引用，单项目验证
- proven：≥2项目+≥2人验证，可信赖知识

自动衰减机制：
- proven条目12个月未引用 → 降级为verified
- verified条目6个月未引用 → 降级为draft
- draft长期未引用 → 归档，移出活跃索引

借鉴Karpathy LLM Wiki的Lint机制：定期识别矛盾、孤儿页、缺失交叉引用。

## 团队知识库：独立Git仓库模式

与llm.wiki"知识中间层"思路一致，但更强调独立仓库的三个价值：[3]

1. 跨项目共享：同一团队多项目连接同一知识仓库，项目A沉淀的知识项目B自动受益
2. 生命周期独立：业务项目归档后，知识不随之消失
3. 权限独立：知识贡献和代码仓库权限可分开管理

贡献闭环（借鉴区块链思想）：
- 不可篡改追加：log.md只追加不修改，evidence记录贡献者+时间+会话哈希
- 成熟度共识：draft→verified需1人验证；verified→proven需≥2人+≥2项目

## 知识按需消费：三级渐进式索引

解决"上下文膨胀 vs 知识利用"的平衡问题：[3]

- Layer A 全景目录 (~50行)：知识库有什么，按阶段推荐查阅路径
- Layer B 分类清单 (~100-300行)：每条知识一行摘要+成熟度+标签
- Layer C 完整条目 (~50-200行)：背景+适用场景+来源追溯

渐进查询流程：读50行全景目录 → 读300行分类清单定位相关条目 → 按需读完整条目
对比"一次性推送50条完整知识"（5000-10000行），上下文效率提升一个数量级。

## 工作流知识闭环

知识沉淀不是额外工作，而是工作流的自然产物：[3]

- INIT（知识注入）：工作流启动时 git pull 知识仓库，注入查询入口
- 各阶段按需查询：Agent在决策点按预算查询知识库
- ARCHIVE（知识提取）：自动提取decision/pitfall/guideline，执行提升判定

引用追踪闭环：Agent查询知识后记录knowledgeReferences，ARCHIVE阶段批量更新evidence.last_referenced，形成自动化maturity提升。

---

## 知识工程 vs 大模型

核心观点：技能/Agent/工具链会迭代，领域知识是永恒的。[3]

- 更大更强的模型：概率性回报，下一代模型未必在特定场景更好
- 知识工程：确定性回报，每条proven知识所有后续工作流都受益

## LLM-WIKI + 领域专家SKILL体系（阿里实践）

阿里内部团队实践，提出了"LLM-WIKI知识库"与"领域专家SKILL包"双轨并行的体系，与腾讯五层存储模型形成互补。[4]

### 核心定位

| 组件 | 定位 | 核心能力 |
|------|------|----------|
| **LLM-WIKI 知识库** | 持续生长型 | 研发提效/自动化、知识沉淀 |
| **领域专家SKILL包** | 全自动研发流程 | 写技术方案\|开发代码\|技术评审\|自动化测试\|专业答疑\|问题排查 |
| **团队视角** | 成员共享 | 多agent平台可用 |

**最终目标**：用户提供PRD，剩下工作全部交给AI完成（全自动研发流程）。

### LLM-WIKI核心思想

> 把LLM从"每次查询时重新检索的RAG引擎"变成"持续维护个人Wiki的全职编辑"

- 知识不再每次重新发现，而是被一次次摄入、合并、交叉引用，沉淀为一份"不断变厚的、活的、可演化的"知识库
- 人类放弃wiki是因为维护成本随规模超线性增长；但LLM不会累、不会忘、一次能改多个文件，维护成本被压到接近零

### LLM-WIKI三层架构

| 层级 | 内容 | 特点 |
|------|------|------|
| **L1: Sources（原始层）** | 文档/图片/代码，LLM只读不写 | 是真相之源 |
| **L2: Wiki（知识层）** | LLM全权拥有的markdown文件集合 | 含实体页、概念页、综述、对比页 |
| **L3: Schema（灵魂层）** | 写给LLM的"工作规范"——目录约定、摄入流程、查询/巡检流程 | 把LLM从通用chatbot变成有纪律的wiki维护者 |

### 核心操作：Ingest / Query / Lint

| 操作 | 解释 |
|------|------|
| **Ingest（摄入）** | 丢一份新数据源→LLM读→与你讨论要点→写摘要页→逐一更新被影响的实体页/概念页→更新index→追加log。一次摄入触达多个页面。 |
| **Query（查询）** | 基于wiki答题。好答案要写回wiki——让你的探索也变成沉淀，不消失在聊天历史里。 |
| **Lint（巡检）** | 定期让LLM自检——找矛盾、找过时、找孤页、找该独立成页但还没有的概念、补交叉引用、提建议性的下一步研究方向。 |

### LLM-WIKI vs RAG 关键对比

| 维度 | 传统RAG | LLM-WIKI |
|------|---------|----------|
| 知识形态 | 原始文档片段（chunks）+向量索引 | 结构化、互链的人类可读markdown页面 |
| 核心动作 | Retrieve（检索）→Generate（生成） | Ingest（摄入并融合）→Read（直接读结论） |
| 知识载体 | 向量数据库（不可读、不可改） | Git仓库+Obsidian（可读、可改、可版本） |
| 合成时机 | 查询时（query-time） | 摄入时（write-time） |
| 知识增长 | 线性（多一份源=多一些chunk） | **复利式**（多一份源=整张网被重写一次） |
| 答案质量随用增长 | 不变（每次都重新拼） | **持续变好**（每次问答都能沉淀回wiki） |
| 典型代表 | NotebookLM, ChatGPT file upload | Obsidian+Claude/Codex+schema |

**关键洞察**：Wiki是一个"持续编译、持续保鲜"的产物（compounding artifact），不是查询时才生成的临时答案。

### SKILL包的迭代方法论

**核心观点**：好SKILL不是靠大量"补丁和情况枚举"，而是需要给房子一个"经久耐用、万变不离其宗"的"钢筋结构"。

**SKILL迭代闭环**：拿着真实PRD使用SKILL实战→总结遇到的问题→一遍遍迭代优化→形成方法论沉淀。

| 问题 | 优化 |
|------|------|
| 找不到默认代码仓或知识库需要向用户询问 | 新增"框架代码原则"章节 |
| 会把大量详细代码写在技术方案里 | 新增"Interaction Simplification Principle"表 |
| 非必要询问可跳过 | 加入"非必要询问跳过"判定矩阵 |
| 提示的代码仓存在硬编码路径问题 | 将代码仓路径改成相对路径，并放到知识库中 |

### 知识库质量保障：Evaluation-Driven四层闭环

| 层级 | 内容 |
|------|------|
| **结构层** | lint+schema_gate必须PASSED，PR阻塞门 |
| **实用层** | 15 task gold集，每次KB变更跑r-N evaluation |
| **信号层** | 任务提分需过3σ₀才算"真改进" |
| **治理层** | ≥10页变更必更log.md+PR写动机 |

### 工具选型

| 组件 | 选择 | 说明 |
|------|------|------|
| 知识库管理 | Obsidian | 可视化界面、图谱视图、双向链接 |
| 搜索引擎 | qmd | BM25+向量检索+LLM重排序 |
| 知识化方式 | 静态分析+LLM摘要 | 代码结构用静态分析提取，语义描述用LLM生成 |
| 更新策略 | 混合方式 | 代码定期更新，文档按需/实时更新 |
