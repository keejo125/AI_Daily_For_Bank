## Harness Engineering 全景总纲

> 来源：TRAE.ai（咸鱼）| [3]
> 核心：Harness = 除了 LLM 本身之外，让 Agent 真正能干活的一切基础设施。

### 为什么需要 Harness：REST 模型

| 维度 | 关键要求 |
|------|----------|
| **可靠性** (Reliability) | 失败可恢复、操作幂等性、行为一致性 |
| **效率** (Efficiency) | 资源可控（Token预算）、低延迟响应、高吞吐量 |
| **安全性** (Security) | 最小权限、沙盒执行、输入/输出过滤 |
| **可追溯性** (Traceability) | 全链路追踪、决策可解释、状态可审计 |

> 模型只解决了「智能」问题，Harness 解决的是「可靠」问题。

### Agent 核心闭环：PPAF

Perception（感知）→ Planning（规划）→ Action（行动）→ Feedback（反思）

| 环节 | Harness 职责 |
|------|-------------|
| **感知** | 上下文管理器——将外部状态翻译为结构化 Prompt |
| **规划** | 行为规划器——任务分解、工具选择、路径规划 |
| **行动** | 调用拦截器→工具执行器→沙盒隔离 |
| **反思** | 反馈汇编器——结果注入下一轮上下文 |

### Harness = 带边界控制的 REPL 容器

| 步骤 | 动作 | Harness 组件 |
|------|------|-------------|
| **Read** | 读取外部状态和记忆 | 上下文管理器（Context Manager）|
| **Eval** | 解析模型输出，路由到工具 | 调用拦截器（Call Interceptor）|
| **Print** | 捕获执行结果，重新注入 | 反馈汇编器（Feedback Assembler）|
| **Loop** | 循环直到目标达成或触发终止 | 状态机/工作流引擎 |

### 三维分析矩阵

横轴：认知循环（被动→主动规划反思）
纵轴：上下文效率（人工投喂→沙盒自动注入）

> Harness 的成熟度直接决定 Agent 能否从「低效被动」跃迁至「高效主动」。

### 控制平面 vs 数据平面

| 维度 | 控制平面（决定做什么）| 数据平面（如何去做）|
|------|---------------------|-------------------|
| 调度 | 任务调度、资源配额 | 实际 Agent 运行实例 |
| 规划 | 行为规划、策略制定 | 状态存储、记忆存储 |
| 治理 | 权限、成本配额、策略 | 沙盒执行环境 |

### 四大功能层级

| 层 | 核心职责 |
|----|----------|
| **模型接入层** | API Gateway、模型路由、多模型切换 |
| **基座能力层** | 技能（Skill）、记忆（Memory）、知识（Knowledge）| 
| **编排调度层** | 工作流引擎、规划器、任务管理器 |
| **治理与观测层** | 权限、成本配额、审计、沙盒 |

### 三大核心约束 + 六大设计原则

| 约束 | 说明 |
|------|------|
| **有限上下文** | Transformer 处理有限 Token 序列 |
| **非确定性输出** | 模型输出随机，Function Calling 每一步可能失败 |
| **状态不在模型中** | LLM 是无状态 CPU，状态必须在 Harness 外部管理 |

| 原则 | 含义 |
|------|------|
| **为失败而设计** | 异常是常态，所有组件需重试和降级能力 |
| **契约优先** | 交互必须由明确 Schema 定义 |
| **默认安全** | 最小权限、零信任、纵深防御 |
| **决策执行分离** | 规划与执行解耦 |
| **万物可度量** | 没有度量就没有优化 |
| **数据驱动进化** | 每次运行都是一次学习机会 |

### 沙盒四级隔离

| 级别 | 方案 | 适用 |
|------|------|------|
| L1 进程级 | chroot/namespace/seccomp | 可信内部工具 |
| L2 **容器级** | Docker/containerd | **默认选择** |
| L3 轻量虚拟机 | Firecracker | 不可信代码/高敏感数据 |
| L4 完整虚拟机 | KVM/QEMU | 极少数特殊任务 |

### 核心架构决策

- **状态分离**：LLM 视为无状态 CPU，跨轮次状态存在 Harness 外部
- **Token 转化流水线**：多源信息收集 → 相关性排序 → 压缩摘要 → 预算分配 → 模板组装
- **Function Calling 失败面管理**：反序列化失败→重试/回退文本；执行失败→交互补充/反思重规划

| 维度 | 关联 |

| **AgentSkill体系** | PPAF+REPL 是 Hook/ETCLOVG/NLAH 的底层运行模型 |

## 核心理念

Harness Engineering 不是某个工具，也不是某条提示词技巧，而是**一整套让 AI 在工程里稳定产出正确结果的工程系统**。

三个关键词：
- **稳定**：不是这次运气好做对了，而是换需求、换维护人仍能稳定工作
- **产出**：不只是写代码，还包括需求、方案、验证、交付等完整过程产物
- **正确结果**：不是"做完了就算"，而是有办法判断做得对不对

> 核心问题：如何让 AI 从一个会临场发挥的模型，变成一个**可约束、可协作、可校验、可持续维护的执行系统**。

## 六大核心组件

| 组件 | 回答的问题 | 工程角色 | 软/硬约束 |
|------|-----------|---------|---------|
| **Rule** | 什么事绝对不能乱来？ | 基础规矩、红线、底线 | 软（AI可能忽略/绕过） |
| **Skill** | 这件事具体应该怎么做？ | 固定动作的标准操作手册 | 硬（按剧本执行） |
| **Sub Agent** | 复杂任务由谁分工处理？ | 不同阶段的专业角色 | - |
| **Workflow** | 这些角色按什么顺序接力？ | 接力赛规则 | - |
| **Scripts** | 最后谁来判断做没做好？ | 总门禁脚本，事后验证闭环 | 最硬（直接检查） |
| **MCP** | AI怎么安全接外部工程系统？ | 外接能力与工具链标准插座 | - |

### 组件关系（逐层叠加）

```
Rule 负责告诉 AI 什么是底线
Skill 负责把高频动作标准化
Scripts 负责判断结果到底对不对
Sub Agent 负责把复杂任务拆成多个专业角色
Workflow 负责让这些角色能按顺序接力
MCP 负责把这套系统继续往外部工程系统延伸
```

## 组件详解

### Rule（工程规矩）

**定义**：给 AI 写的一套"工程制度"，告诉它什么能做、什么不能做、什么做完必须验证。

**核心作用**：帮助 AI **少犯本来不该反复犯的低级错误**，而非让 AI 变聪明。

**典型Rule示例**：
- 改完代码以后必须编译
- 编译通过后必须跑测试
- 测试通过后必须跑事后验证
- 三步不全过，开发不算完成

**Rule 的天花板**（重要）：
- AI 会"遗忘"：需求很长、上下文很重时，Rule 会被稀释
- AI 会"绕过"：给规则找理由，"这次是特殊情况"

> **结论**：Rule 只能做"原则约束"，不能做"流程执行"。Rule 非常重要，但不能解决所有问题。

---

### Skill（标准操作手册）

**定义**：告诉 AI 有些事情**严格按照固定步骤执行**，不要临场发挥。

**典型场景**：编译、测试、事后验证

以编译为例，实际工程里可能涉及：
- 要找对 MSBuild
- 要用固定的配置
- 要先还原
- 要把日志输出到指定文件
- 编译完以后还要看错误模式

**与 Rule 的分工**：
- Rule："这件事**必须做**"
- Skill："这件事**具体怎么做**"

**好处**：
1. Rule 变轻了，只保留真正的红线
2. AI 执行稳定性变高，不是理解而是调用
3. 维护成本更低，修改编译流程只改 Skill 本身

---

### Sub Agent（多角色分工）

**问题**：单 Agent 干到底会导致：
- 自己给自己做需求解释
- 自己给自己的方案打分
- 自己写代码、自己说自己没问题
- 天然更倾向于推进任务继续进行，而不是停下来承认当前有问题

**解决方案**：多个 Sub Agent，把不同阶段拆开，每个 Agent 只负责自己那一段，把产出写进文档，然后交给下一个 Agent。

**类比真实研发**：
- 需求分析的人只管把需求说清楚
- 方案设计的人只管把技术方案做完整
- 闸门的人专门负责判断"这东西能不能进开发"
- 开发的人只管实现
- 代码审查和测试的人负责收口
- PM 只负责流程

---

### Workflow（接力赛规则）

**核心**：不是"有一张图"，而是**让每一次前进、暂停、打回、重跑都有明确依据**。

**Workflow 三层**：
1. **写给人看**：整体研发链路怎么走
2. **写给系统看**：把阶段和迁移边界固定下来
3. **写给具体角色看**：明确每个角色接棒时该读什么、交棒时该写什么

**上下文纪律**：每一棒只给当前该看的那一份材料。不是要瞒信息，而是避免上下文越长、重点越散。

---

### Scripts（总门禁脚本）

**地位**：整个 Harness 里最"硬"的东西。

**Rule vs Scripts**：
- Rule：告诉 AI 应该怎么做
- Scripts：直接检查做没做到，AI 很难用"我觉得没问题"糊弄过去

**典型"总门禁脚本"检查项**：
- XAML 里有没有中文 / Emoji
- 有没有硬编码 UI 文案
- 日志格式对不对
- 主工程编译是否通过
- 测试是否通过
- 测试数量有没有异常减少
- .cs 文件有没有漏进 .csproj

---

### MCP（外接能力接口）

**定位**：不是 Harness 的主体，而是**外接能力接口**——把 AI 接进更大工程系统的标准插座。

**核心能力**：
- 调用 CI 平台发起构建
- 读取构建日志和构建结果
- 调用签名服务给安装包签名
- 上传制品到制品库或发布平台
- 调用发布系统做灰度、提审、发布
- 回写发布状态、版本号、交付记录

**比喻**：
- Rule 像制度
- Skill 像操作手册
- Scripts 像闸机
- MCP 像把 AI 接进更大工程系统的标准插座

## 多 Agent 技术选型对比

| 路线 | 优点 | 缺点 | 结果 |
|------|------|------|------|
| **强化单 Agent** | 成本最低、改造最少 | 角色冲突，长链路容易失稳 | 早期有效，不能成为最终形态 |
| **去中心化协作** | 灵活、像 AI 团队自由讨论 | 路径不稳定、责任边界不清、难维护 | 被放弃 |
| **结构化调度** | 流程清晰、可审计、可维护 | 前期设计成本高、产物多 | **最终选择** |

**结构化调度内部 PK**：

| 做法 | 特点 | 结果 |
|------|------|------|
| 固定角色 + 固定流程 | 稳定、易维护、文档化强 | **最终选择** |
| 固定 PM + 动态生成 Agent | 灵活性强 | 角色边界漂、维护成本高 |

> 真正贵的不是 token，真正贵的是失控。

## 七角色结构化调度体系

JK Launcher 项目实践中，角色是一步一步被问题逼出来的：

### 骨架三层（最早）
1. **需求分析** — 把模糊想法变成结构化需求
2. **方案设计** — 把需求翻译成能落地技术方案
3. **开发实现** — 按约束写代码，不边写边定义需求

### 第四层（闸门总控）
4. **闸门总控** — 站在"开发之前的最后一道门"，判断：
   - 需求够不够清楚
   - 方案有没有明显漏项
   - 在现有工程里能不能安全落地
   - 现在进开发会不会把风险带进编码阶段

> 没有闸门，很多问题都会在开发阶段才爆炸。而开发阶段是最贵的阶段之一。

### 收口两层
5. **代码审查** — 独立于开发，判断实现质量
6. **测试验证** — 独立验证功能正确性

### 流程管理层
7. **PM / 流程管理** — 负责流程调度

## 设计规格文档（SPEC）先行

**核心理念**：很多人一上来就想写 Rule、拆 Agent、接 MCP。但如果没有先把"这个工程到底想怎么开发、最终想交付什么"说清楚，后面所有约束都会变成无源之水。

**SPEC 的意义**：
- 不是为了文档看起来专业
- 而是先把"这个版本到底要解决什么问题、边界是什么、怎样才算做完"谈清楚

**与 AI 反复磨透 SPEC 的价值**：
- 不只是改文档措辞
- 是和 AI **共同加深对需求本身的理解**的过程
- 真正的需求常常在来回里被挖出来，而不是第一版就写死

**SPEC 必须做到**：
- 清晰写明**所有**需求，指明**必要**的边界条件
- **没有**任何不确定词语：建议、可以、推荐、可选等

## 从规格到交付的九步体系

```
1. 设计规格文档（SPEC） → 作战目标
       ↓
2. Rule                → 纪律/底线（绝对不能乱来）
       ↓
3. Skill               → 标准动作（关键流程不临场发挥）
       ↓
4. Sub Agent           → 兵种分工（不同阶段专业角色）
       ↓
5. Workflow            → 指挥链（前进/暂停/打回/重跑规则）
       ↓
6. Scripts/事后验证    → 验收闭环（做得对不对直接检查）
       ↓
7. dev-map（开发导航地图） → 项目结构理解（不重复造轮子）
       ↓
8. 任务看板            → 全局视角（项目历史+当前进展）
       ↓
9. MCP                 → 外部工程系统连接
```

## 工程案例效果

JK Launcher V3 "工程修复工具箱"功能，使用完整版 Harness **一次就直接完成，完成度相当高**。

项目特征：
- 功能持续迭代
- 既有 UI，又有服务和状态逻辑
- 既要工程正确性，也要用户体验
- 既要能改代码，也要能长期维护

**"人搭 Harness，AI 写代码"模式**：
- 人做：搭 Harness（SPEC → Rule → Skill → Sub Agent → Scripts → dev-map → 任务看板）
- AI 做：代码实现（人没有亲手写过一行代码）

## 落地建议

### 搭建顺序（推荐）

1. **先磨透 SPEC**：需求和设计说透之前，不要急着写 Rule、拆 Agent
2. **Rule 只做原则约束**：不要把流程细节写进 Rule，让 Rule 变轻
3. **高频固定动作 → Skill**：编译、测试、验证做成 Skill，减少临场发挥
4. **复杂任务 → 多 Agent**：从三角色骨架开始，被问题逼着拆更多角色
5. **Scripts 做硬门禁**：把验收标准做成可执行检查，不要只靠 Rule
6. **dev-map + 任务看板**：让 AI 快速理解项目上下文

### 避坑指南

- ❌ 不要一上来就写几十条 Rule → Rule 会稀释，AI 会绕过
- ❌ 不要迷信"灵活协商式多 Agent" → 好看但难维护
- ❌ 不要让单个 Agent 扮演所有角色 → 角色冲突不可避免
- ✅ 要追求**可控、可追踪、可复用** > 理论上的灵活性
- ✅ 要让 AI 做代码实现，人做 Harness 搭建
