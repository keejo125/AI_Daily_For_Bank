# Claude Code 源码泄露深度解析

> 来源：微信公众号  
> 收录：2026-04-06

## 泄露事件概述

**时间**：2026年3月31日  
**原因**：Anthropic 打包失误，59.8MB source map 文件意外暴露  
**规模**：约 2000 个文件，51.2 万行 TypeScript 代码

## 架构总览

**六层分层架构**：

| 层级 | 职责 |
|------|------|
| 入口层 | CLI 命令解析 |
| 展示层 | React + Ink 终端 UI |
| 核心引擎 | Agent 编排、上下文管理 |
| 执行层 | 工具调用、命令执行 |
| 协作层 | SubAgent、AgentTeams |
| 管理层 | 记忆、配置、安全 |

**工具定义层**：2.9 万行 TypeScript，每个工具的 prompt 描述精细到"什么时候该用、怎么用、用完之后期望什么结果"。

## 记忆管理系统

### 四种记忆类型

| 类型 | 说明 |
|------|------|
| **user** | 用户偏好 |
| **feedback** | 反馈修正（正反馈 + 负反馈） |
| **project** | 项目信息 |
| **reference** | 外部引用 |

### 正反馈记忆

> Claude Code 不仅记"负反馈"，也记"正反馈"——如果只记批评不记表扬，模型会趋向保守，不敢做决定。

### 三层记忆架构

| 层级 | 说明 |
|------|------|
| **底层 MEMORY.md** | 轻量级索引，始终加载在上下文中 |
| **中间层主题文件** | 具体项目知识，按需加载 |
| **上层原始对话** | 不整体回读，通过 grep 检索特定标识符 |

## KAIROS 与 autoDream：后台记忆整合

**核心设计**：当用户空闲时，后台启动守护进程执行"记忆整合"。

**类比**：像人类睡眠中大脑自动编排记忆——保留重要的，丢掉可以忘记的，整合重复的。

**运行方式**：
- 在名为 Undercover 的独立进程中运行
- 不依赖用户打开终端
- 作为操作系统守护进程运行

**更新维度**：
1. 用户偏好
2. 项目知识
3. 反馈历史
4. 外部引用

**价值**：用户第二天打开终端时，Claude Code 已经准备好整洁的起点，不需要重新阅读历史对话。

## 上下文工程

### 核心原则

> Context Engineering > Prompt Engineering

不是写固定 prompt 告诉模型"你是谁"，而是在每轮对话中精心组装完整的上下文环境。

### 自动压缩

- **上下文窗口**：200K token
- **性能下降点**：147K-152K token
- **自动压缩阈值**：64%-75% 容量

### 压缩策略

| 触发方式 | 说明 |
|----------|------|
| **自动压缩** | 接近容量上限时触发，总结对话历史，清除旧的工具输出 |
| **手动压缩** | `/compact` 命令，用户可自定义压缩提示词 |

**autoDream 的价值**：不是被动压缩，而是后台持续知识蒸馏，把短期记忆沉淀为长期记忆。

## SubAgent 与 AgentTeams

### SubAgent

**特点**：
- 独立于其他 Agent 和主 Agent 运行
- 拥有自己的上下文窗口、工具集和提示词
- 降低上下文溢出风险

**配置**：
- 在 Markdown 文件中定义
- 存储于项目特定目录或全局用户目录
- 明确列出允许访问的工具集

### AgentTeams

**两个角色**：

| 角色 | 职责 |
|------|------|
| **Team Lead** | 架构师 + 项目经理，生成队友、拆解任务、分配工作、综合结果 |
| **Teammates** | 独立的 Claude Code 实例，完成具体子任务 |

**核心机制**：

1. **共享上下文**：所有 Member 加载相同的项目相关信息，共享 CLAUDE.md"员工手册"
2. **任务看板**：任务状态（pending/in progress/completed），可设置依赖关系，文件锁防竞争
3. **IPC 通信**：可发送给指定队友，也可广播给全员

### SubAgent vs AgentTeams

| 维度 | SubAgent | AgentTeams |
|------|----------|------------|
| 自主性 | 低，结果汇报给主智能体 | 高，可主动认领任务 |
| 协作方式 | 主智能体统一协调 | 队员间直接通信 |
| 隔离 | 独立上下文窗口 | 独立 Claude Code 实例 |

## 成本控制：Prompt Cache Fork

**问题**：多 Agent 协同工作时，token 消耗指数级增长。

**解决方案**：
- SubAgent 采用 fork 模式
- 创建与父 Agent 字节相同的上下文副本
- 所有并行 Agent 命中同一份缓存
- 5 个并行 Agent 的 token 成本 ≈ 1 个 Agent 顺序工作

**监控**：对缓存命中率设有监控告警，命中率过低时触发紧急事件。

## 安全设计

- **Fail-closed**：安全优先设计
- **Undercover 模式**：后台进程隔离运行

## 对 AI-SDC 的启示

1. **记忆分层**：底层索引 + 中间层主题 + 上层原始对话
2. **后台整合**：KAIROS 模式——空闲时自动整理知识
3. **多 Agent 协作**：SubAgent + AgentTeams 参考架构
4. **成本控制**：Prompt Cache Fork 降低多 Agent 成本
