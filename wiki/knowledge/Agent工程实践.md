## Harness Engineering（驾驭工程）

### 来源

Mitchell Hashimoto（HashiCorp 联合创始人）提出，继提示词工程、上下文工程之后的第三个 AI 工程支柱。

**三次范式跃迁**（阿里妹，2026）：
1. **Prompt Engineering（2022-2024）**：单次交互优化，Few-shot、CoT、角色设定
2. **Context Engineering（2025）**：动态构建上下文窗口，决定"给 Agent 看什么"
3. **Harness Engineering（2026）**：跨多会话、多Agent、多阶段的系统架构设计

### 核心隐喻

> AI 智能体 = SOTA 模型（野马）+ Harness（缰绳）= 卓越执行者

**Harness** = LLM 之外的一切基础设施，用于约束、引导、纠偏智能体行为。

> "Agents aren't hard; the Harness is hard." — Ryan Lopopolo（OpenAI）

### 四种失败模式（Anthropic）

| 失败模式 | 表现 | 根因 |
|----------|------|------|
| **One-shot Syndrome** | 试图在单次上下文窗口完成全部工作 | 上下文填充率超40%后质量衰退 |
| **过早宣布胜利** | 部分完成就宣布结束 | 缺乏外部验收机制 |
| **过早功能完成** | 功能实现但未端到端验证 | 缺乏自动化测试 |
| **冷启动问题** | 新会话需大量Token重新理解项目 | 缺乏持久化记忆 |

**核心约束**：Agent 无法准确评估自身产出质量 → 必须外部化约束和反馈机制。

### 四根支柱

| 支柱 | 定义 | 关键实践 |
|------|------|----------|
| **上下文架构** | Agent 恰好获得当前任务所需上下文 | 分层加载、按需获取；AGENTS.md 控制在~100行作为索引 |
| **Agent专业化** | 受限工具集的专业Agent优于通用Agent | Planner/Generator/Evaluator 分离 |
| **持久化记忆** | 进度持久化在文件系统而非上下文窗口 | 标准启动序列：检查工作目录→读Git Log→定位未完成任务 |
| **结构化执行** | 未经审查批准的计划前禁止写代码 | 理解→规划→执行→验证，每阶段有质量门禁 |

### R.E.S.T 框架

| 维度 | 定义 | 关键要求 |
|------|------|----------|
| **可靠性** | 面对异常仍能稳定服务 | 故障恢复、操作幂等、行为一致 |
| **效率** | 有效使用计算资源 | Token 控制、低延迟、高吞吐 |
| **安全性** | 保护系统和数据 | 最小权限、沙箱执行、I/O 过滤 |
| **可追踪性** | 提供可理解的数据 | 端到端追踪、可解释决策、可审计状态 |

### PPAF 循环

```
感知(Observe) → 思考(Think) → 行动(Act) → 反馈(Reflect)
      ↑                                        ↓
      ←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←←
```

### 架构本质：REPL 容器

Harness 本质是一个 **REPL 容器**：
- **Read**：上下文管理器将外部世界翻译成结构化提示
- **Eval**：调用拦截器捕获意图，路由到工具执行器
- **Print**：反馈组装器将结果包装成"观察"
- **Loop**：循环直到目标达成或触发终止条件

### 核心转换机制

**无限状态 → 有限 Token**：
- 上下文管理：缩减规则 + 注入边界
- 函数调用：Schema 序列化 → 触发生成 → 反序列化 → 观察注入
- 状态分离：LLM 是无状态 CPU，状态下沉到外部存储

### 六条设计原则

1. **为失败而设计**：异常是常态，必须支持容错、重试、优雅降级
2. **契约优先**：所有交互通过显式 Schema/API/Event 定义
3. **默认安全**：最小权限、零信任、纵深防御
4. **关注点分离**：规划与执行解耦
5. **一切皆可度量**：每个行为必须可量化
6. **数据驱动演进**：每次运行都是学习机会

### 沙箱隔离四层架构

| Level | 方案 | 特点 | 适用场景 |
|-------|------|------|----------|
| 1 | 进程级隔离（chroot/namespaces） | 快，共享内核 | 可信内部工具 |
| 2 | 容器隔离（Docker） | 行业标准 | **默认选择** |
| 3 | 微虚拟机（Firecracker） | 独立内核 | 多租户/不可信代码 |
| 4 | 完整虚拟机（KVM/QEMU） | 最高安全 | 最敏感任务 |

### 控制平面 vs 数据平面

- **控制平面（What）**：任务调度、资源配额、策略执行
- **数据平面（How）**：运行时实例、状态存储、沙箱执行

### 运行治理

- 资源管理：Token/API调用/CPU时间预算
- 韧性：超时控制、重试策略、熔断器、优雅降级
- 策略网关：RBAC/ABAC权限、PII过滤、注入防御、审计日志

### 指标体系

| 类别 | 指标 |
|------|------|
| 任务有效性 | 成功率、指令遵循率、工具使用效果 |
| 服务质量 | 端到端延迟、首次行动时间、错误率 |
| 资源效率 | 平均 Token 消耗、工具调用次数 |
| 安全合规 | 策略拒绝率、安全事件数 |

### 实战案例：阿里妹（10万+行 Java 应用）

**十阶段开发流程**：
```
需求分析 → 需求评审 → 编码实现 → 编码评审 → 单元测试编写
→ 单元测试评审 → 代码推送 → CI验证 → 部署验证 → 用户确认
```

**Human-in-the-Loop 确认点**（5个）：
- 需求待决议确认、计划评审后确认、编码评审后确认、部署环境参数确认、最终交付确认

**评审循环上限**：需求评审最多3轮，编码/测试评审最多2轮，超出升级人工决策。

**效果数据**：

| 维度 | Harness引入前（3月） | Harness成熟后（4月） | 提升 |
|------|---------------------|---------------------|------|
| AI代码率（项目） | 24.86% | 90.54% | +65.68pp |
| AI代码率（个人） | 14.24% | 87.85% | +73.61pp |

**关键收益**：
- 返工大幅减少（Agent-to-Agent评审闭环）
- 交付质量可预期
- 知识沉淀形成活的"项目开发手册"

### 核心洞察

> "当模型撞墙时，我们就实现一种工程机制，确保同一类失败不会再次发生。"

> "工程师的角色正在从代码创造者，转变为创造过程的守护者。"

> "每发现一个错误，就工程化地消除它再次发生的可能性。" — Mitchell Hashimoto

> "将做事的 Agent 和评判的 Agent 分开，是一个强有力的杠杆。" — Anthropic

## 终端沙箱：Agent Harness 的基础设施

### 核心问题

Agent 执行终端命令时面临两类风险：
1. **模型幻觉**：模型把项目目录误认为临时目录，发出危险删除命令
2. **环境变量差异**：第三方工具在异常环境下产生破坏性操作

**真实案例**：开发者让 AI 清理项目根目录临时文件，项目下恰好有一个名为 `$HOME` 的子文件夹。AI 生成了 `rm -rf "$HOME"`。开发者确认了弹窗——他以为删除的是项目里的 "$HOME" 文件夹。但 shell 执行时 `$HOME` 被展开为真实主目录路径，磁盘几乎被清空。

### 为什么黑名单和弹窗不够

**黑名单问题**：字符串级匹配（如拦截 `rm -rf`、`sudo`、`format`）可被绕过。
- `cmd /c "rmdir /s /q"` 和 `rm -rf` 功能相同但写法不同
- 反引号嵌套、管道拼接、变量展开都能绕过规则

**弹窗问题**：
- 真实开发环境中命令太多，很多人最终会直接点"允许"
- 弹窗把安全责任推给用户，但用户无法弥补语义不一致

### 终端安全需要多一步：进程隔离

**核心思路**：风险命令可以运行，但不能随便读写你的机器。

进程隔离 = 命令在受限环境中执行，工作区目录可写，其余目录只读，系统路径不可达。

### 三层防护机制

| 层级 | 机制 | 说明 |
|------|------|------|
| **第一层：LLM 风险判定** | 模型生成命令时判断意图 | "这条命令在删除系统路径""变量值不确定" |
| **第二层：AST 命令解析** | 三套 AST 解析器处理 PowerShell/Bash/CMD | 提取结构化命令名和参数，识别隐藏危险命令（如反引号注入） |
| **第三层：内置危险命令名单** | 解析出的命令名逐一比对名单 | 命中名单→弹窗确认；命中前两层→直接进沙箱 |

**AST 解析示例**：
```
Prompt 注入让模型生成：whoami `rm -rf /`
肉眼看是一条 whoami。
AST 解析结果：{ cmd: "whoami", args: ["rm -rf /"] }
提取到两个命令名：whoami 和 rm → 危险命令被识别
```

### 三平台实现方案

| 平台 | 技术方案 | 特点 |
|------|----------|------|
| **macOS** | sandbox-exec（Apple 系统自带 Seatbelt） | 内核级隔离，性能调优，无第三方依赖 |
| **Linux** | bubblewrap（基于内核 user namespace） | 轻量隔离，根目录只读挂载，网络隔离一行搞定 |
| **Windows** | 自研 sandbox.exe（Rust 实现） | CreateRestrictedToken + DACL 动态授权 + Job Object，兼容 Win7+ |

**Windows 自研难点**：调研了 15 种隔离技术（Hyper-V/WSL2/AppContainer/Sandboxie 等），最终选择用户态机制（受限令牌 + Capability SID），平衡了兼容性（Win7+）和启动速度（毫秒级）。

### 性能开销

| 平台 | 开销数据 | 说明 |
|------|----------|------|
| **macOS** | +43-72ms（启动）/ 循环命令 +1-3% | tar +3%、seq +0.3%、find +2% |
| **Windows** | +7.7%（综合） | 主要影响进程启动，PowerShell spawn +19%（约+4.8秒） |

### 与 Harness Engineering 的关系

终端沙箱是 **Harness 的执行层**：
- R.E.S.T 框架的"安全性"维度需要沙箱支撑
- PPAF 循环的"行动"阶段在沙箱中执行
- 沙箱把"安全执行"从人工弹窗变成基础设施能力

**核心价值**：把风险关进更小的边界里。对个人开发者，少一次误删；对团队和企业，这是权限治理、审计和合规的起点。

### 对我行的启示

1. **推进智能研发必须同步建设终端沙箱**：Agent 能执行命令的地方就是风险点
2. **三平台覆盖是标配**：我行开发环境 macOS/Linux/Windows 并存
3. **AST 级解析 > 字符串黑名单**：能识别嵌套变形攻击
4. **性能开销可控**：毫秒级启动对开发体验影响小
