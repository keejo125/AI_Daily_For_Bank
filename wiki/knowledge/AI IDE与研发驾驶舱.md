# AI IDE与研发驾驶舱

> 研发入口的重构：从工作台到驾驶舱

---

## 核心命题

**过去 IDE 是开发人员写代码的"工作台"，未来 AI IDE 会变成调度 Agent 的"研发驾驶舱"。**

- 普通 AI IDE 解决的是"能不能写代码"
- 金融 AI IDE 解决的是"写出来以后能不能进流程、过审计、担责任"

金融 AI Coding 的审计，不该只看代码改了什么，还要看人如何指挥 AI 改了代码。

---

## AI IDE 的四个特征

真正的 AI IDE，至少要具备四个特征：

1. **感知上下文** — 知道当前工程结构、技术栈、依赖关系、历史代码和正在处理的任务
2. **执行任务** — 不只是回答问题，而是能改文件、跑命令、生成单测、触发检查
3. **审核机制** — AI 生成的结果不能直接进入生产，必须经过自动化检查和人工确认
4. **过程留痕** — AI 调用了什么知识，依据了什么规约，改了什么文件，谁审核采纳，都要能追溯

没有这四点，所谓 AI IDE 只是"带聊天框的编辑器"。

---

## 为什么入口一定会回到 IDE

聊天框离研发现场太远：

| 对比维度 | 聊天框 | IDE |
|---------|--------|-----|
| 上下文获取 | 依赖人说清楚 | 主动读取当前工程 |
| 跨文件修改 | 很难处理 | 天然连接项目目录和代码索引 |
| 工具连接 | 需额外接入 | 原生连接终端、Git、测试、调试和插件 |
| 任务状态管理 | 很难管理 | 把任务、文件、测试、提交、审核放在同一个研发现场里 |

AI Coding 走到深处，入口一定会回到 IDE，不是因为 IDE 时髦，而是因为 IDE 离研发现场最近。

## 美国大行路径

### Citi

- 用 GitHub Copilot 作为辅助型工具
- 用 Devin 处理软件补丁、升级等任务（Agent 委派）
- **边界划得很清楚**：不允许 Agent 直接部署代码
- Agent 只产出 artifacts，交给开发者，再经过自动测试和人工 review 后由人决定是否采纳

### J.P. Morgan

- 内部 LLM Suite 用于代码审查、单元测试编写、头脑风暴等软件开发场景
- 把模型能力放进内部可控环境里，降低数据泄露、合规和安全风险

### Bank of America

- 软件开发人员使用 GenAI 工具辅助代码编写和优化
- 强调 human oversight、transparency 和 accountability（人工监督、过程透明、结果问责）

### Morgan Stanley Research 判断

随着 AI coding assistants 和 agents 成为标准开发工作流，传统软件工程师会更多转向复杂应用，成为 **curators、reviewers、integrators 和 problem-solvers**。

开发人员从"亲手写每一行代码的人"，变成"组织 AI 完成复杂工程的人"。

**美国路径关键词**：生态工具、Agent 委派、代码仓库上下文、人类审核、自动测试、内部知识约束。

## 中国金融路径：六条线

中国大型金融机构更强调：

1. **数据安全** — 代码、日志、配置、接口、客户相关数据不能随意外流
2. **权限控制** — 不同人员、不同岗位、不同系统，能看什么、能调什么、能改什么，必须按权限执行
3. **国产适配** — 模型、算力、操作系统、芯片环境都要考虑金融机构的实际部署约束
4. **审计留痕** — 所有 AI 交互、知识调用、任务执行和人工采纳都要能追溯
5. **知识内化** — AI 要接入机构自己的知识库、规约库、代码资产、历史缺陷和业务规则
6. **流程嵌入** — AI 不能游离在研发流程之外，必须与 DevOps、安全门禁、测试平台、投产系统打通

### 招商银行 DevAgent 案例

自研研发智能体 DevAgent，采用"感知—规划—执行—反馈—进化"的多轮交互 ReAct 模式：
- 结合编程现场环境感知、企业研发知识检索等工具
- 以开发者业务目标为驱动，提供任务级功能需求开发能力
- 具备跨文件、大片段代码生成能力
- **月均处理 4.8 万+ 开发任务**

关键关键词：**编程现场、企业知识、任务级开发、跨文件生成**。

## 研发入口重构

传统工作流里，开发人员的工作被切散在很多系统里：

- 需求在一个系统
- 设计在一个系统
- 接口文档在一个系统
- 表结构在一个系统
- 代码在 IDE 里
- 测试在测试平台
- 流水线在 DevOps 平台
- 安全扫描在安全工具里
- 投产又在另一套流程里

开发人员真正的低效，很多时候不是代码写得慢，而是在不同系统之间来回切换、查资料、找资产、问规范、等环境、跑流程。

**AI IDE 的价值**：把这些分散的上下文重新拉回研发现场。

## 中小银行采购路径

不是所有银行都要自己从零开发 AI IDE。

中小银行更适合走一条"**金融级采购 + 本地化集成 + 持续运营**"的路线：

- 可以买成熟 IDE 底座
- 可以买 AI 编程助手
- 可以买模型适配能力
- 可以买知识工程工具
- 可以买 DevOps 集成能力
- 可以买审计留痕和安全治理模块

**但必须把几件事掌握在自己手里**：
- 代码和数据不能失控
- 知识库和规约库要接入本行资产
- 权限、审计、门禁和审批要符合本行制度
- AI 生成结果要经过可追溯的审核和采纳流程

供应商提供能力，但最终责任链条必须落在银行自己的研发治理体系中。

## 五个装配维度

AI IDE 真正的价值，是把金融研发现场重新装配一遍：

### 1. 装配上下文

AI 必须知道项目结构、技术栈、接口依赖、命名规则、公共方法、历史代码和当前任务。IDE 是最接近代码现场的地方，也是最容易获取上下文的地方。

### 2. 装配知识

把 Wiki、接口平台、代码库、群聊、设计文档、历史版本里的知识变成可调用上下文，让开发人员不用在几十个系统之间来回切换。

### 3. 装配工具

AI 要能改文件、跑命令、跑测试、调用 Git、查日志、触发流水线、执行安全扫描。Agent 如果没有工具调用，就只是会聊天；Agent 如果有了工具调用，才真正进入研发任务。

### 4. 装配流程

后面还有单测、复核、安全扫描、流水线、版本交付、投产审批和运维反馈。AI IDE 如果不能接这些流程，就只能停留在个人效率工具。

### 5. 装配审计

金融机构需要知道 AI 改了什么文件、依据什么知识、调用什么工具、测试是否通过、谁审核、谁采纳、结果流向哪里。

---

## 过程即审计

传统审计更多是事后追溯（代码改了什么，谁提交的，谁审批的）。

AI 进入研发后，仅看 Git Diff 已经不够了。银行还要知道：
- 开发人员给 AI 下了什么原始任务
- AI 调用了哪些知识库条目
- 引用了哪些规约
- 修改了哪些文件
- 调用了哪些测试、扫描、流水线工具
- 谁审核了结果，最终是否采纳
- 责任由谁闭环

**AI IDE 的价值**：不只是记录代码变化，而是记录"人如何指挥 AI 完成研发任务"的全过程。

以前审计看的是结果痕迹，以后审计还要看协作痕迹。

## 研发知识主权

通用模型懂通用代码，但不懂一家银行几十年的系统演进：

- 不懂内部接口习惯
- 不懂历史技术债
- 不懂某个核心系统为什么不能轻易改一行字段
- 不懂某张表为什么曾经出过事故
- 不懂某个流程背后的监管要求

银行真正的护城河，不是某个外部模型，而是自己的：
- 业务规则
- 代码资产
- 工程规约
- 测试案例
- 故障经验
- 合规要求

**研发知识主权包括**：
1. 内部接口规���
2. 历史技术债
3. 故障案例
4. 监管合规要求
5. 业务规则

外部工具可以帮助你生成代码，但未必理解你的系统边界、监管约束、过去踩过的坑。

## 开发人员价值重新定价

AI IDE 之后，开发人员的价值会被重新定价：

| 过去 | 未来 |
|------|------|
| 写代码 | 定义任务 |
| 查资料 | 调用知识 |
| 亲自实现 | 审核结果 |
| 单人编码 | 调度 Agent |
| 执行者 | 责任承接者 |

AI 最怕任务模糊。任务越模糊，生成越漂。未来开发人员最重要的能力之一，是把模糊需求变成 AI 可执行任务。

开发人员不是被 AI 替代，而是被 AI 重新分层。

## 建设路径：分层推进

不能走两个极端：
- ❌ 完全买外部工具直接推广（很难解决数据安全、知识接入、流程贯通问题）
- ❌ 从零造一个 IDE（周期太长，投入太大）

**更合理的路径**：分层推进

### 底层
- 基于成熟 IDE、插件体系、内部已有 IDE 或研发平台
- 接内部模型、国产模型、模型路由和模型测评平台

### 知识能力
- 接内部知识库、代码库、规约库、RepoWiki、RAG 和项目记忆

### 工具能力
- 接 DevOps、Git、流水线、测试平台、安全扫描、制品库、投产系统

### 治理能力
- 接权限、审计、门禁、日志、审批和风险控制

### 上层
- Rules、Spec、Skills、MCP、Tools、SubAgent、异步任务、沙箱执行和任务状态管理

### 四阶段路线

1. **第一阶段** — 基于现有 IDE 或插件体系，把内部模型、代码库、知识库接进来，让开发人员在熟悉入口里用起来
2. **第二阶段** — 建设任务级 Agent 能力，支持跨文件修改、单测生成、工具调用和任务状态管理
3. **第三阶段** — 把规约库、Skills、RepoWiki、测试平台、安全扫描、流水线和审计留痕逐步接入
4. **第四阶段** — 形成统一研发触点，让 IDE、插件、CLI、Web IDE、DevOps 平台协同起来

---

## 终局：研发驾驶舱

AI IDE 的终局：

- 左边是代码和工程上下文
- 右边是知识库和规约库
- 上面是任务和 Agent 状态
- 下面是终端、测试、流水线和安全扫描
- 中间是人对结果的审查和决策

开发人员不再只是输入代码的人，他要像飞行员一样看仪表、定航线、发指令、看反馈、处理异常、决定是否接管。

**掌握这个入口，就能把 AI Coding 从个人技巧变成组织能力。**
