## 核心框架：三层指标金字塔

智能研发指标体系分三层，分别回答不同管理层级的问题：

| 层级 | 受众 | 核心问题 | 典型指标 |
|------|------|----------|----------|
| **战略层** | CIO | AI 产出了多少可控价值 | 研发周期缩短率、生产问题下降率、故障恢复时间、AI 合规入库率、单位任务成本 |
| **工程层** | 开发中心 | AI 代码有没有进入主干和版本 | 单测覆盖率、代码复核检出率、安全门禁阻断率、版本问题提前发现数 |
| **行动层** | 一线团队 | 今天该改什么 | 场景采纳率、人工修正率、上下文缺失率、工具调用失败率、知识命中不准率 |

> **关键升级**：指标从衡量「个人体感」升级为衡量「工程结果」

---

## 一、先看全景：智能研发指标金字塔

银行做 AI Coding，最容易陷入"指标很多、看不清重点"的问题。指标的本质要回答一个核心问题：**AI Coding 到底在制造热闹，还是在改善研发体系**。

头部银行的实践方向：AI Coding 不能只放在插件、代码补全和个人效率里看，它正在被放进智能研发平台、模型底座、知识资产、场景工程和组织运营里一起管理。

---

## 二、警惕表面繁荣：从流量思维转向价值思维

AI Coding 推广初期最关注调用量和生成代码行数。这些指标容易统计、汇报好看，但不能停在"有没有用 AI"。

**表面指标**（容易误导）：
- 多少人开通了账号
- 生成了多少代码
- 覆盖了多少项目组

**有价值的指标**（后半段）：
- 设计阶段提前检出了多少问题
- 单测覆盖率有没有提升
- 代码复核检出率有没有改善
- 安全门禁阻断了多少高危风险
- 版本问题有没有在投产窗口前暴露出来
- 故障定位时间是否缩短

---

## 三、CIO 真正关心的六张账

| # | 账目 | 核心指标 | 管理意义 |
|---|------|----------|----------|
| 1 | AI 代码合规入库 | AI 合规入库率 | 代码有没有穿过研发全流程 |
| 2 | 缺陷密度 | AI 生成代码缺陷密度（/千行） | 质量有没有同步改善 |
| 3 | 安全拦截 | 安全问题提前发现率、提交前拦截率 | 越早发现处理成本越低 |
| 4 | 版本问题 | 版本问题提前发现数 | 投产窗口才暴露风险极高 |
| 5 | 生产稳定性 | MTTR、生产问题下降率 | 最终要接受生产检验 |
| 6 | 单位成本 | 单位价值成本 | 可规模化才是真成功 |

### AI 合规入库率（关键定义）

$$
\text{AI 合规入库率} = \frac{\text{AI 参与生成、经过人工审核、通过测试和安全门禁、成功合并入库的代码量}}{\text{总入库代码量}}
$$

**漏斗分解**：建议数 → 采纳数 → 代码变更数 → PR 数 → 测试通过数 → 安全通过数 → review 通过数 → 合并数 → 版本发布数 → 上线后问题数

## 四、产品视角：五类关键指标

| 类别 | 核心指标 | 含义 |
|------|----------|------|
| **任务完成率** | AI 任务走到可 review 状态的比例 | 衡量端到端任务能力 |
| **证据完整率** | 附带测试结果、日志、扫描结果、变更说明的交付比例 | 可验证、可审计、可追责 |
| **工具调用成功率** | 文件读写、测试执行、扫描工具、MCP 调用成功率 | Agentic Coding 的核心价值 |
| **上下文与规则生效率** | 项目上下文覆盖率、规则生效率、企业知识库调用有效率 | 比"知识库有多少文档"更重要 |
| **企业后台可观测性** | 受控工具链覆盖率、权限/规则/知识库/审计配置能力 | 企业级 vs 个人工具 |

## 五、覆盖率与活跃度：只是入场券

**覆盖率**回答"AI Coding 有没有进入研发体系"：
- 研发人员覆盖率、部门覆盖率、应用覆盖率、语言栈覆盖率、项目组覆盖率
- **金融专属指标**：受控工具链覆盖率 = 在统一 AI Coding 平台内发生的研发任务数 / 全部 AI 相关研发任务数

**活跃度**回答"大家有没有真的在用"：
- 日活、周活、月活、人均调用次数、人均使用时长、连续使用天数

> 覆盖率和活跃度只能证明工具进入了现场，价值有没有进入流程还要看后面的工程结果。

---

## 六、代码采纳率：衡量体感 vs 有效采纳率

**粗口径采纳率的局限**：
- 直接采纳 ≠ 代码质量可靠
- 修改后采纳 ≠ 上下文匹配
- 采纳率高 ≠ 团队不是对 AI 过度信任

**有效采纳率**（更合理）：

$$
\text{有效采纳率} = \frac{\text{被采纳并通过测试、扫描、评审的 AI 代码量}}{\text{AI 生成代码总量}}
$$

**委派深度**：开发人员只是让 AI 解释代码，还是愿意让它生成计划、修改多文件、运行测试、提交 PR。委派深度越高，产品信任越强。

---

## 七、AI 合规入库率：从体感进入工程结果

代码采纳率衡量开发人员是否接受 AI，AI 合规入库率衡量代码是否真正进入代码库并完成工程校验。

**分层看入库**：
1. AI 参与生成并**提交**的代码 / 总提交代码
2. AI 参与生成并进入**主干**的代码 / 总入库代码
3. AI 参与生成并进入**版本**的代码 / 总版本代码
4. AI 参与生成并通过安全扫描、测试验证、人工 review 的代码 / AI 入库代码

**入库后漏斗**：生成多、PR 少 → 生成质量或信任不足；PR 多、测试失败多 → 质量不稳；测试通过多、安全失败多 → 规约和安全上下文不足。

---

## 八、质量指标：写得对比写得快更重要

| 类别 | 关键指标 |
|------|----------|
| **测试覆盖** | 单测覆盖率、接口测试覆盖率、AI 生成测试案例有效率 |
| **缺陷发现** | AI 生成代码缺陷密度（/千行）、评审检出率、AI 辅助测试发现缺陷数 |
| **返工** | AI 代码人工修改比例、需求/设计/测试返工率 |
| **可维护性** | 圈复杂度变化、技术债新增量、代码可读性评分 |
| **设计阶段** | 设计阶段问题检出数、接口影响分析命中率、计划通过率 |

> **计划质量指标**：AI 不宜直接进入代码修改，应先生成可 review 的实施计划，识别影响文件、接口依赖、测试范围、安全风险和回退方案。计划通过率越高，说明 AI 对代码库和工程上下文理解越稳。

---

## 九、安全指标：进入门禁和审计体系

| 类别 | 核心指标 |
|------|----------|
| **安全问题前移** | 编码阶段安全问题发现率、提交前风险拦截率、AI 生成代码安全问题占比 |
| **安全门禁** | 高危漏洞阻断数、敏感信息泄露拦截数、开源风险阻断数 |
| **AI 特有安全** | AI 生成代码安全问题密度、Agent 越权动作拦截率、模型上下文脱敏合规率 |
| **安全修复回路** | 漏洞平均修复时长、修复建议采纳率、同类问题复发率 |
| **生产安全** | 生产安全事件数量、漏洞引发生产问题数量 |

> CIO 最关心的：哪些风险被提前拦住，哪些安全经验回流成了规则，哪些 AI 行为被有效管住。

## 十、交付指标：投产窗口前暴露问题才是真改善

| 类别 | 核心指标 |
|------|----------|
| **周期** | 需求到开发完成周期、测试通过周期、缺陷修复到合并周期 |
| **风险前移** | 持续集成问题定位准确率、版本风险提前识别数、环境差异提前识别率 |
| **交付结果** | 上线成功率、投产问题数量、版本回滚率、灰度验证通过率 |

**综合指标**：AI 参与版本健康度 = AI 代码比例 × 测试通过率 × 安全通过率 × 投产成功率 × 上线后问题下降率

## 十一、运维指标：生产稳定性检验前面所有指标

| 类别 | 核心指标 |
|------|----------|
| **故障发现定位** | 告警降噪率、故障定位准确率、根因分析耗时、运维智能体建议采纳率 |
| **恢复效率** | MTTR 下降率、故障恢复时间、应急处置建议采纳率 |
| **变更关联** | 故障与近期变更自动关联准确率、AI 参与代码引发故障比例 |
| **复盘回流** | 故障复盘自动生成率、故障知识回流率、同类故障复发率 |

## 十二、知识指标：AI 是否带着本行经验写代码

| 类别 | 核心指标 |
|------|----------|
| **知识覆盖** | RepoWiki 覆盖率、核心应用知识覆盖率、历史故障案例覆盖率 |
| **知识质量** | 知识更新及时率、知识过期率、知识引用准确率 |
| **知识调用** | 知识命中率、有效引用率、AI 回答中知识引用占比 |
| **知识回流** | 代码评审意见回流率、测试缺陷回流率、规则更新回路完成率 |

**CIO 重点看**：
$$
\text{有效知识命中率} = \frac{\text{AI 在真实任务中引用正确知识并被人工确认有效的次数}}{\text{AI 触发知识检索的总次数}}
$$

## 十三、组织指标：人有没有升级

| 维度 | 关键指标 |
|------|----------|
| **使用分层** | 高频有效使用者 vs 偶尔使用 vs 使用多但产出低 |
| **岗位能力变化** | 开发/测试/交付/运维/安全人员是否从执行者升级为责任承接者 |
| **专家经验放大** | 专家经验是否通过规约、模板、Skills、知识库扩散到更多团队 |
| **人工接管率** | AI 执行任务时哪些环节必须人工接手（过高=Agent未融入流程） |
| **组织运营** | 种子用户数量、实战培训后有效采纳率提升、周活跃有效用户占比 |

## 十四、组织结构指标：自有+分级外包+AI 组合效率

中国金融机构研发组织由自有人员、不同级别外包、测试外包、驻场支持共同构成。AI Coding 进入后要重新计量组合效率。

**四类外包指标**：

| 外包级别 | 重点指标 |
|----------|----------|
| **初级外包** | AI 辅助任务完成率、返工率、规约违规率、人工 review 耗时 |
| **中级外包** | 模块交付一次通过率、AI 合规入库率、缺陷密度、单测补齐率 |
| **高级外包** | 复杂任务承接能力、方案质量、关键问题解决率、知识沉淀贡献 |
| **测试外包** | 有效用例率、缺陷检出密度、核心路径覆盖率、生产缺陷漏出率 |

**自有人员审查负担**：
$$
\text{自有人员审查负担率} = \frac{\text{自有人员用于复核 AI/外包产出的时间}}{\text{自有人员总研发管理时间}}
$$

**单位需求综合成本**：
$$
\text{综合成本} = \text{自有人员成本} + \text{外包人员成本} + \text{AI 资源成本} + \text{review 成本} + \text{返工成本} + \text{缺陷修复成本}
$$

## 十五、成本指标：单位价值成本有没有下降

| 维度 | 核心指标 |
|------|----------|
| **单任务成本** | 生成单测/修复漏洞/完成交付分析/定位故障的平均花费 |
| **模型路由效率** | 简单任务是否错用昂贵模型，复杂任务是否用能力不足的模型 |
| **Token 有效率** | 知识切片是否精准，无效上下文是否过多塞进模型 |
| **部署形态选择** | API 调用 vs 私有化部署的场景匹配 |
| **多模型路由准确率** | 任务是否被路由到合适的模型和环境 |

> 成本指标不能只看模型调用成本，还要看需求交付总成本（开发→测试→投产→稳定运行的全部人力+等待+返工+故障成本）。

## 十六、防止指标变成新的形式主义

**三个原则**：
1. **指标必须能驱动行动**：连续几个月没变化的指标应从核心仪表盘降级或移除
2. **指标必须能追溯来源**：口径统一，AI 代码如何标记、安全问题如何归因必须明确
3. **指标必须能防止行为扭曲**：只考核生成量会鼓励多生成，只考核采纳率会降低审查标准

**两个关键**：
- 数据最好从工具链自动产生，不要靠团队手工填
- 指标口径要跨 IDE、CLI、PR、流水线、知识库和 Agent 平台统一归集

## 十七、失败原因分布比失败率更有价值

| 失败原因 | 对应调整 |
|----------|----------|
| 上下文不足 | 补项目知识图谱和 RepoWiki |
| 知识命中错误 | 重新做知识切片和审核 |
| 工具调用失败 | 优化 MCP 工具、权限和执行环境 |
| 测试失败多 | 加强计划生成和测试策略 |
| 安全扫描失败多 | 把安全规约前置到生成阶段 |
| 人工 review 否决多 | 看任务拆解、代码风格、规约匹配和开发者信任 |

> "AI Coding 不能只放在插件、代码补全和个人效率里看，它正在被放进智能研发平台、模型底座、知识资产、场景工程和组织运营里一起管理。"

> "指标的本质，要从衡量个人体感升级为衡量工程结果。只看活跃度和调用次数，AI Coding 很容易变成一场昂贵的内部运动。"

> "从代码采纳率走到 AI 合规入库率，指标开始从个人体感进入工程结果。采纳率代表开发人员觉得好用，合规入库率代表研发体系接得住。"

## AI编程度量危机：Token时代的度量突围

> 来源：茹炳晟聊软件研发（茹炳晟）| [3]
> 核心：Token 是 AI 时代的「代码行数」——一个被行业淘汰的虚荣指标，换了个名字卷土重来。

### 核心问题

| 问题 | 说明 |
|------|------|
| **Tokenmaxxing** | 亚马逊和Meta员工故意让agent跑不必要任务来刷token消耗量，因leaderboard上这代表「AI使用活跃度」 |
| **生产力悖论** | GitHub Copilot引入后，核心开发者需多审查6.5%代码，自身产出下降19%；AI相关PR审查时间增加91% |
| **成本悖论** | 推理成本降280倍，但应用端反而涨10倍（前沿模型token价格未降 + 交互token量成倍增加） |
| **组织级抵消** | IDC调查318名决策者，92%表示AI Agent成本高于预期；AI采纳率与公司级KPI相关性在组织层面消失 |

### 三个演化中的度量框架

| 框架 | 思路 | 价值 |
|------|------|------|
| **DORA+SPACE+AI** | 部署频率/变更前置时间/变更失败率 + 满意度/绩效 + AI采纳/影响层 | 古典框架的AI扩展 |
| **LCOAI** | 借鉴能源行业平准化电力成本，将AI全生命周期成本平摊到每单位「有效推理产出」 | 解决API vs 自托管不可比问题 |
| **AWU（Agentic Work Unit）** | Salesforce 2026 Q4推出：agent完成的每一个可验证的离散任务 | 从按消耗计费→按结果计费 |

### 度量体系三层

| 层 | 问题 | 关键指标 |
|----|------|----------|
| 成本度量 | 花了多少钱？ | 每有效产出token成本、开发者人均AI成本、隐藏维护成本 |
| 效率度量 | 变快了吗？ | 交付吞吐量、周期时间、流动效率、任务成功率 |
| 价值度量 | 结果变好了吗？ | 代码质量与返工率、开发者效能感（信任度仅24%）、业务影响 |

### 对我行启示

- 效能量化必须从「代码采纳率」走向「多维度度量体系」
- 警惕 Tokenmaxxing 式虚荣指标，建立输出导向的考核机制
- 流动效率比局部编码速度更能反映AI的真实价值
- ROI 计算必须包含隐性维护成本和信任建设投入

## 十八、从个人到组织：AI Coding 的组织级度量落地实践

> **来源**：从个人生产力到组织能力：LoongSuite-Pilot×SLS 的 AI Coding 度量实践
> **原文**：https://mp.weixin.qq.com/s/SEOGc3KIGm7eFpyUK6nR4g（阿里云云原生，徐可甲）| [3]
> **核心**：从个人体感到组织能力的度量体系，需要事件级统一采集、SQL 驱动的灵活分析层、以及能驱动行动的空白发现机制。

### 1. 核心认知：DORA 的 J-Curve

Google Cloud DORA 团队的《ROI of AI-Assisted Software Development》报告明确了几个关键判断：

- **AI 是放大器，不是变压器**：放大优势和功能障碍，不会自动解决组织问题
- **J-Curve 效应**：团队采纳 AI 工具会先经历一段生产力下降期（工作流适配、习惯切换、prompt 调优），之后才反弹
- **场景差异悬殊**：Greenfield 项目效率增益 35-40%，Legacy 代码低于 10%——同一公司内部不同团队收益曲线可能完全不同
- 缺少事件级度量能力的组织，连"自己的 J-Curve 走到了哪里"都无从判断

### 2. 数据底座：事件级统一采集

当前多数研发组织只有两类数据：个体自报的满意度问卷 + CI/CD 的聚合 KPI。中间缺失**事件级的 AI 使用行为日志**。

LoongSuite-Pilot 的解决方案：
- **标准先行**：基于 OpenTelemetry GenAI semantic conventions，扩展为 LoongSuite GenAI 语义规范（已在 GitHub 开源）
- **统一采集**：Claude Code、Copilot、Cursor、Qoder 等异构 Agent 按同一字段结构入库，跨工具可比
- **事件级粒度**：每次 Agent 调用、每次 tool.call、每次 Token 消耗独立上报，不是会话汇总
- **事实表 + 维表分离**：事件事实表（用户/会话/Agent/模型/Token/Tool/仓库） + 人员维表（工号/部门路径），通过工号 JOIN

**三个工程价值**：异构 Agent 可比性、会话级溯源能力、Skill 与 Tool 可观测性（通过 file_path 关联到具体 SKILL.md）

### 3. 分析层：SLS SQL 驱动的 8 个 Section 递进看板

| Section | 内容 | 分析层次 |
|---------|------|----------|
| 核心概览 | 使用人数/总Token/事件数/人均/环比 | 整体水位感知 |
| 结构分布 | 按 Agent/Model/Provider 的 Token 占比 | 资源配置判断 |
| 趋势 | 覆盖规模/Token消耗/时段分布 | 走向判断 |
| 部门统计 | Token明细/覆盖率/人均/未上报列表 | 团队级对比 |
| 组织与人员 | 员工明细/Agent和模型画像 | 个人穿透 |
| Skill & 工具 | Skill调用次数/使用人数/Tool调用 | 知识资产复用 |
| 代码仓库 | Repo Token Top10/Git Domain占比 | 投入战略对齐 |
| Token 集中度 | Top10%/20% Token占比/每日趋势 | 均值校验 |

**工程关键**：公共 CTE（Common Table Expressions）保证 30+ 图表口径一致——口径约束只定义一次，改一处 CTE 所有图表自动生效。

### 4. 关键设计模式

- **LEFT JOIN 暴露空白**：维表 LEFT JOIN 事件表 + `WHERE user_id IS NULL`，找出"在册但未上报"的员工名单
- **集中度校验**：Top10% 人群 Token 占比是否过高——集中度持续下降，AI Coding 才从个人扩散为组织共同能力
- **退回到原始事件表**：通用 CTE 不是越宽越好，仓库/Skill/时段分布等需要更细粒度的场景退回到原始事件表单独查询

### 对度量体系建设的启示

- 度量体系需要三层：**采集层**（统一语义事件）→ **分析层**（灵活 SQL + CTE 工程基座）→ **行动层**（空白发现 + 集中度校验）
- 当前我们已有指标框架（三层金字塔），但缺少采集层规范和分析层工程实现——LoongSuite GenAI 规范可参考
- 人员维表的设计模式（LEFT JOIN 找空白）直接适用于组织推广阶段的"谁还没在用"的精准识别
- Token 集中度校验正好解决当前"均值指标是否被头部撑起"的疑问
