# Claude Code 深度解析与实战

> 整合 Claude Code 架构源码分析与实战教程，系统梳理设计理念、工程实现与实践要点

---

## 第一部分：架构设计与核心理念

### 1. 七层架构设计

Claude Code 采用分层架构，从源码分析得出的共识：

| 层级 | 职责 | 来源 |
|------|------|------|
| 入口层 | CLI 命令解析 | [1][2] |
| 展示层 | React + Ink 终端 UI | [1][3] |
| 核心引擎 | Agent 编排、上下文管理 | [1][2] |
| 执行层 | 工具调用、命令执行 | [1][2] |
| 协作层 | SubAgent、AgentTeams | [2][3] |
| 管理层 | 记忆、配置、安全 | [2][3] |
| 工具定义层 | 2.9万行TS精细定义 | [3] |

### 2. 命令 + 工具双轨设计

理解 Claude Code 整个系统最重要的概念：[01]

| 轨道 | 触发方 | 用途 | 示例 |
|------|--------|------|------|
| **命令（Command）** | 用户（"/"前缀） | 用户主动操作 | `/review`、`/commit`、`/config` |
| **工具（Tool）** | AI（自主决策） | AI与宿主环境交互 | `BashTool`、`FileReadTool` |

**设计哲学**：用户用命令"指挥"，AI用工具"执行"，两者通过 QueryEngine 协同。

### 3. QueryEngine 是调度中枢

QueryEngine 负责：接收用户输入 → 决策调用命令还是工具 → 与 Claude API 对话 → 返回结果。是整个系统的"大脑"。[01][08]

### 4. Context Engineering > Prompt Engineering

不是写固定 prompt 告诉模型"你是谁"，而是在每轮对话中精心组装完整的上下文环境。[3]

## 第二部分：记忆系统

### 1. 层级化记忆设计

| 层级 | 说明 | 来源 |
|------|------|------|
| L1 工作记忆 | 当前会话对话历史 | [1][3] |
| L2 项目上下文 | 项目文件结构、规范 | [1][3] |
| L3 任务历史 | 已完成的子任务记录 | [1] |
| L4 工具结果 | 执行的命令和结果 | [1] |
| L5 外部知识 | 检索到的相关代码 | [1] |
| L6 系统提示 | 系统级指令和约束 | [1] |
| L7 持久化记忆 | 跨会话的知识积累 | [1] |

### 2. 后台记忆整合（KAIROS/autoDream）

当用户空闲时，后台启动守护进程执行"记忆整合"。

**类比**：像人类睡眠中大脑自动编排记忆——保留重要的，丢掉可以忘记的，整合重复的。

**运行方式**：
- 在名为 Undercover 的独立进程中运行
- 不依赖用户打开终端
- 作为操作系统守护进程运行

**价值**：用户第二天打开终端时，Claude Code 已经准备好整洁的起点，不需要重新阅读历史对话。[3]

### 3. 正反馈记忆

Claude Code 不仅记"负反馈"，也记"正反馈"——如果只记批评不记表扬，模型会趋向保守，不敢做决定。[3]

### 4. 记忆压缩策略

| 细节 | 说明 | 来源 |
|------|------|------|
| 上下文窗口 | 200K token | [3] |
| 性能下降点 | 147K-152K token | [3] |
| 自动压缩阈值 | 64%-75% 容量 | [3] |
| 手动压缩命令 | `/compact` | [19] |

## 第三部分：工程实现要点

### 1. 初始化流程

1. 解析CLI参数 → 2. 加载配置 → 3. 初始化状态 → 4. 启动UI → 5. 等待用户输入。[02]

### 2. 权限系统

- 敏感操作需要用户确认（文件写入、网络请求）
- 不同命令有不同的权限级别 [03]

### 3. 工具系统架构

核心模块：`Tool.ts`（基类）、`tools.ts`（注册中心）、`tools/`（具体实现）。[05]

**核心工具**：
| 工具 | 用途 |
|------|------|
| `BashTool` | 执行Shell命令 |
| `FileReadTool` | 读取文件内容 |
| `FileWriteTool` | 写入文件 |
| `GrepTool` | 文本搜索 |
| `GlobTool` | 文件模式匹配 |

### 4. 技能系统是命令的超集

技能（Skill）是命令（Command）的进化版本：[10]

| 维度 | 命令 | 技能 |
|------|------|------|
| 结构 | 单个.md文件 | `技能名/SKILL.md`目录 |
| 关联文件 | 不支持 | 支持`files`字段 |
| 执行隔离 | 仅内联 | 支持`context:fork`子代理 |
| 条件激活 | 不支持 | 支持`paths`按路径激活 |

### 5. 状态管理机制

包括：全局配置（API Key、模型选择）、会话状态（当前任务、上下文）、任务队列（待办、完成、进行中）。[11]

## 第四部分：高级特性

### 1. 多 Agent 协作

| 模式 | 适用场景 | 说明 |
|------|----------|------|
| SubAgent | 隔离复杂任务 | 降低上下文溢出风险 |
| AgentTeams | 多角色协作 | Team Lead + Teammates 模式 |

**观点分歧**：
- 保守派（社区Ralph Wiggum）：单 Agent 足够，反对多 Agent 复杂度
- 演进派（[2][3]）：SubAgent + AgentTeams 是未来方向

> **整合理解**：Claude Code 实际上同时支持两种模式，工具选择取决于场景复杂度。

### 2. 成本控制：Prompt Cache Fork

多 Agent 协同工作时 token 消耗指数级增长。Claude Code 采用 fork 模式：
- 创建与父 Agent 字节相同的上下文副本
- 所有并行 Agent 命中同一份缓存
- 5个并行 Agent 的 token 成本 ≈ 1个 Agent 顺序工作

### 3. 安全优先（Fail-closed）

安全优先设计，后台进程隔离运行（Undercover 模式）。[3]

### 4. 可扩展架构

插件系统、支持多种入口（CLI/IDE/API/Web），核心引擎与功能解耦。[1][2]

## 第五部分：跨工具对比

### Claude Code ↔ Qoder

| 维度 | Claude Code | Qoder |
|------|-------------|-------|
| 上下文管理 | 层级压缩 | 向量检索 |
| 记忆机制 | 7层 + KAIROS后台整合 | 5大知识源 + ProcMEM |
| 多角色协作 | SubAgent + AgentTeams | 专家团 |
| 工程知识注入 | 无内置 | RepoWiki |
| 架构风格 | 单智能体 | 多智能体协作 |

**共性**：都意识到记忆/上下文管理是核心挑战，都在探索层级化方案。

### Claude Code ↔ Hermes Agent

| 维度 | Claude Code | Hermes Agent |
|------|-------------|--------------|
| 部署方式 | 客户端工具 | 自托管服务器 |
| 记忆系统 | KAIROS后台整合 | 内置MEMORY.md + FTS5 |
| 技能沉淀 | SubAgent定义文件 | 自动从任务中提炼skill |
| 训练数据闭环 | 无 | 有（Atropos） |

**共性**：都强调记忆的持续维护，都在探索"越用越强"的机制。

## 第六部分：Harness Engineering 综合

Harness Engineering = 上下文工程 + 记忆管理 + 工具编排 + 安全控制的综合体系。[25]

**核心原则**：
1. 精心组织的上下文 > 海量检索
2. 后台记忆整合 > 被动等待
3. 安全优先（Fail-closed）
4. 可观测性全程追踪

---

## 第七部分：对智能研发工具建设的启示

1. **记忆分层是基础**
   - 底层索引（始终加载）
   - 中间层主题（按需加载）
   - 上层原始对话（通过 grep 检索）

2. **后台整合提升体验**
   - KAIROS 模式——空闲时自动整理知识
   - 用户每次打开都有"整洁的起点"

3. **多 Agent 协作参考架构**
   - SubAgent：适合隔离复杂任务
   - AgentTeams：适合多角色协作

4. **成本控制不可忽视**
   - Prompt Cache Fork 是降低多 Agent 成本的关键
   - 监控缓存命中率

5. **上下文工程 > Prompt 工程**
   - 精心组装上下文 > 写固定 prompt
   - 压缩优于扩展
