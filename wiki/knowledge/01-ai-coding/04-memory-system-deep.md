# Memory 系统深度解析：从 Raw Ledger 到可治理闭环

> 来源：微信公众号 - 「纯干货」几万字都讲不明白的Memory架构与思考  
> 收录：2026-04-07

## 核心命题

> Memory 的本质不是一个组件，而是一个闭环系统：
> Raw Ledger（权威）→ Views（可用）→ Policy（控制）→ Commit（回写）→ Provenance（可回放）

## 三层抽象

### 1. Raw Ledger（原始账本）

**定义**：追加式记录每次写入/更新/删除发生了什么。

**核心要素**：
- 对记忆状态的变更（ADD/UPDATE/DELETE/NONE）
- 当时的输入、时间、scope 等

**特点**：
- 像"账本/黑匣子"
- 只追加，不修改
- 提供可审计性

### 2. Views（视图层）

**定义**：面向检索/推理的派生状态。

**类型**：
- 向量索引
- Keyword/Hybrid 检索
- KG/TKG（时序知识图谱）
- Timeline
- Skill Index

**特点**：
- 可以多、可以 lossy
- 必须可回指到 Raw Ledger
- 把历史变成能力

### 3. Policy（策略层）

**定义**：决定何时读、读多少、何时写、如何更新、如何遗忘。

**关键要求**：
- 决策必须显式化为可记录/可回放的 Action 序列
- 不是"靠 prompt 里一句话暗示"

## System 1 + System 2 架构

```
┌─────────────────────────────────────────┐
│           System 1: General Agent       │
│         (LLM + tools + planner)         │
└─────────────────────────────────────────┘
                    ↑↓
┌─────────────────────────────────────────┐
│           System 2: Memory System       │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐ │
│  │ Ledger  │  │ Views   │  │ Policy  │ │
│  └─────────┘  └─────────┘  └─────────┘ │
│                                         │
│  Retrieve → Evidence → Early Stop      │
│  Observability: trace/log/metrics      │
└─────────────────────────────────────────┘
```

### 为什么需要 System 2

1. **能力正交性**：记忆能力和 LLM 通用 Agent 能力相对正交
2. **可插拔**：记忆可迁移、可复用
3. **可归因**：决策显式化，便于诊断

## 程序性记忆（Procedural Memory）

### 从"知识"到"技能"

传统 RAG 的短板：
- 擅长检索"信息片段"
- 难以检索并复用"推理过程/操作流程/可执行技能"

### Skill-MDP 形式化

**Skill 三元结构**：
- 触发条件（Trigger）
- 执行步骤（Execution）
- 终止条件（Termination）

### ProcMEM 三阶段

| 阶段 | 动作 |
|------|------|
| **生成** | 语义梯度 → 总结成功/失败原因 → 批级更新 |
| **验证** | PPO Gate 信任域门控 → 反事实验证 |
| **维护** | 在线评分 → 淘汰低价值技能 → 剪枝冗余 |

**关键特点**：
- 可执行
- 可验证
- 可治理
- 非参数化优化（不更新 LLM 权重）

## Memory 整合层

### 问题：阻抗匹配

外部记忆通常是文本或结构化数据，LLM 的核心计算对象是 token 与注意力。

**传统链路**：
```
外部内容 → 文本化 → 拼接到上下文 → 模型再次编码为向量
```

**问题**：
- 额外的编解码开销
- 信息损失与语义偏差

### Latent Memory 方向

**LycheeMemory**：
- 训练 compressor，把输入压缩成潜层 token
- 直接注入到注意力计算
- 减少文本编解码链路

**MemAdapter**：
- 生成式子图检索
- 把结构化信息映射到 LLM 语义空间
- 零样本跨范式融合

## Memory OS 架构

### 五层抽象

| 层级 | 职责 |
|------|------|
| **内核（Kernel）** | System 2 的慢回路与调度器，决定何时检索/写入/更新/遗忘 |
| **文件系统（File System）** | Raw Ledger + Views，时序一致性与语义压缩 |
| **可执行文件（Executable）** | Skill/Macro/Workflow，程序性单元 |
| **总线接口（Interface）** | 记忆与推理之间的接口层，低开销注入 |
| **学习引擎（Learning Engine）** | 在线适应，持续改进信号 |

## 核心结论

1. **三件套必须**：Raw Ledger + Views + Policy，缺一不可
2. **相对正交**：记忆能力与 LLM 通用能力可拆分
3. **Policy 是瓶颈**：最被低估的瓶颈
4. **时间约束**：bi-temporal + time-sliced recall 是硬约束
5. **抽象不依赖实现**：五层架构描述"必须存在的模块"，不是"怎么拼装方案"
