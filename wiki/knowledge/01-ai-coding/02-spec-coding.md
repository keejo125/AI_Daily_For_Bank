# 2026 年 AI 编码的"渐进式 Spec"实战指南

> 来源：微信公众号 - 阿里云开发者  
> 收录：2026-04-02

## 核心观点

**模型是地基，方法论是上层建筑。地基不行，上面盖得再好也白搭。**

## 模型能力分层（T0-T2）

| 层级 | 能力 | 特点 |
|------|------|------|
| **T0** | 一次生成全链路，主动处理边界情况 | 顶级模型 |
| **T1** | 多提示一两轮可达接近效果 | 优秀模型 |
| **T1.5** | 基本可用但容易漏边界 | 中等模型 |
| **T2** | 能写骨架但需较多人工调整 | 入门模型 |

**差距不在"能不能写"，而在"一次成功率"和"边界处理"。**

## Agent 本质公式

```
Agent = while 循环 + Tool Use + 工具执行器
```

**执行循环**：
1. 思考：分析代码，决定修改方案
2. 行动：调用工具执行
3. 验证：运行编译检查
4. 自愈：读取错误，自动修复

**关键理解**：工具的边界就是 Agent 的能力边界。

## 复杂度理论

### 两种复杂度

| 类型 | 定义 | 工具能否消除 |
|------|------|-------------|
| **本质复杂度** | 业务决定（如交易状态机） | ❌ 不能 |
| **偶然复杂度** | 工具/流程引入 | ✅ 可以 |

**评判标准**：方案好不好，看它能多高效应对本质复杂度，同时自身引入的偶然复杂度有多低。

## Spec Coding 方法论

### 背景：Vibe Coding 的四个工程问题

1. 上下文丢失：对话历史爆炸
2. 需求漂移：越聊越偏
3. 一致性难保：前后矛盾
4. 知识不沉淀：下次从零开始

### 三条铁律

> Code is Cheap, Context is Expensive

1. **高质量输入**：把需求、约束、代码现状写进 Spec
2. **减少试错**：AI 不用反复试错，输出大幅减少
3. **轮次压缩**：对话从 20 轮降到 3-5 轮

### 渐进式原则

- **简单需求不承担复杂流程**：改个字段不需要先写 spec
- **流程是可选增强**：Rules 始终生效，Spec 按复杂度加载
- **分层设计**：prompt、模板、rules 都是代码库中的普通文件

## 完整框架

### 目录结构

```
rules/          # 编码规范、安全红线、异常处理
knowledge/      # 领域知识、踩坑记录
changes/        # 进行中的变更
  ├── spec.md   # 需求规格
  ├── tasks.md  # 任务拆分
  └── log.md    # 执行日志
archives/       # 已完成的变更
```

### 7 个命令

| 命令 | 功能 |
|------|------|
| `/init` | 初始化项目 |
| `/propose` | 提出方案 |
| `/apply` | 执行编码 |
| `/fix` | 增量修正 |
| `/review` | 代码审查 |
| `/archive` | 归档沉淀 |
| `/knowledge` | 知识提取 |

### 工作流

```
Propose → Apply → Review → Archive
   ↓        ↓        ↓        ↓
  设计     编码     审查     归档
```

## 关键实践

### Propose 阶段

1. **Research 先行**：分析代码现状，每个结论有代码出处
2. **渐进式提问**：一次只问一个问题，优先给 2-3 个选项
3. **分段输出**：不一口气生成完整 spec，每段等用户确认
4. **显式确认**：spec 确认后禁止任何编码动作

### Apply 阶段

1. **证据导向**：每个 task 完成后展示可验证的证据
2. **即时沉淀**：发现踩坑/新规则立即写入 log.md
3. **紧急停车**：遇到逻辑冲突或 spec 缺失立即停止

### Review 阶段

**两阶段审查**：

| 阶段 | 内容 |
|------|------|
| 阶段一 | 功能覆盖：逐条比对 spec 与实际代码 |
| 阶段二 | 质量审查：编码规范、安全红线、异常处理 |

**核心原则**：不信报告只信代码

### Archive 阶段

- 展示知识发现和踩坑记录
- 询问用户是否沉淀到 knowledge/
- 变更目录移到 archives/

## 两层 AI 架构

```
编排层 AI（决策）
    ↓ 分发任务
执行层 AI（编码）
```

**为什么分层**：
- 编排需要强模型做决策
- 编码可以用安全模型降低成本
- 混在一起要么太贵，要么质量不够

## 工具选择建议

| 场景 | 推荐工具 |
|------|----------|
| Claude 系列模型 | Claude Code 开箱即用 |
| 灵活切换模型 | opencode |
| IDE 工作流习惯 | Cursor / Windsurf |

**方法论适配**：rules/ + knowledge/ + changes/ 框架可适配任何编码工具

## 对银行智能研发的启示

1. **分层落地**：简单需求用 Vibe Coding，复杂需求用 Spec Coding
2. **知识沉淀**：每次变更自动沉淀到 knowledge/
3. **渐进式流程**：流程是可选增强，不是强制前提
4. **透明工具链**：模型版本可见、context 可查、token 透明
