## 核心共识

### 1. 多角色协同架构（专家团模式）
Qoder 核心设计：Team Lead + 多专家协同。[1]

| 角色 | 职责 | 特点 |
|------|------|------|
| Team Lead | 拆解任务、分配方向 | 只决策不写代码 |
| 调研专家 | 资料收集、分析 | 异步并行 |
| 编码专家 | 代码生成 | 分工协作 |
| QA专家 | 测试验证 | 质量门禁 |
| 评审专家 | 代码审查 | 架构把关 |
| 浏览器专家 | Web操作 | 扩展能力 |

### 2. 工程知识五大源
Qoder 具备领域知识注入能力：[1]

| 知识源 | 说明 |
|--------|------|
| 业务规则 | 领域逻辑、约束 |
| 代码规范 | 编码风格、最佳实践 |
| 项目结构 | 模块关系、依赖 |
| API文档 | 接口定义、调用方式 |
| 历史决策 | 为什么这么设计 |

### 3. 三大工程挑战及解法
| 挑战 | Qoder解法 |
|------|-----------|
| 上下文爆炸 | RepoWiki知识提炼 |
| 知识断层 | 自动分析项目结构 |
| 规范混乱 | Hooks质量门禁 |

## 观点分歧

### 关于多Agent必要性
| 观点 | 来源 | 立场 |
|------|------|------|
| 单Agent足够，反对多Agent复杂度 | Claude Code社区 | 保守派 |
| Team Lead+专家团是多角色协作最优解 | [1] Qoder专家团 | 演进派 |

> **整合理解**：两者并不矛盾，Qoder的专家团本质是"分工"而非"竞争"，复杂任务需要分工，简单任务单Agent更高效。

### 关于代码审查
| 观点 | 来源 |
|------|------|
| Code Reviewer的核心价值在于语义理解而非语法检查 | [2] |
| 追溯调用链、下游影响分析是关键能力 | [2] |

## 核心能力详解

### 1. Experts Mode 专家团模式
**执行流程**：
```
用户输入 → Team Lead拆解 → 并行分发 → 专家执行 → 结果汇总 → Team Lead整合
```

**关键特性**：
- 异步并行，多专家同时工作
- 中枢协调，统一调度
- 自进化，经验技能自动沉淀

### 2. Code Reviewer 评审专家
**核心能力**：[2]

| 能力 | 说明 |
|------|------|
| 语义理解 | 理解为什么改，而非只看不改 |
| 调用链追溯 | 下游影响分析 |
| 架构影响判断 | 跨文件、跨模块逻辑审查 |

**两种使用方式**：
1. **Experts Mode 自动审查**：完整评审流程
2. **CLI 命令审查**：快速验证

### 3. 老项目改造方案
**核心挑战**：[3]

| 挑战 | 解决方案 |
|------|----------|
| 上下文爆炸 | 先建RepoWiki，知识提炼 |
| 知识断层 | 自动分析项目结构，生成架构文档 |
| 规范混乱 | Hooks配置质量门禁 |
| 测试缺失 | 智能单测生成 |

**改造步骤**：
1. 自动分析项目结构
2. 生成RepoWiki（架构文档）
3. 配置Hooks（规范约束）
4. 智能单测补充

### 4. 智能单元测试生成
**快手实践案例**：覆盖率从3%→80% [4]

| 版本 | 问题 | 改进 |
|------|------|------|
| V1.0 | 编译错误、mock问题 | 沙箱执行、多轮修复 |
| V2.0 | 生成时间长 | 并发流程优化 |
| V3.0 | 逻辑理解不足 | 规则+知识双驱动 |

**核心公式**：
```
智能单测 = 规则引擎 + 领域知识 + 多轮修复
```

### 5. Hooks 事件驱动扩展
**解决三大痛点**：[5]

| 痛点 | Hooks解法 |
|------|-----------|
| 不可控 | 阻断危险操作（exit 2） |
| 不可观测 | 事件日志记录 |
| 不可进化 | 上下文注入，数据积累 |

**5种事件类型**：
| 事件 | 可阻断 | 说明 |
|------|--------|------|
| UserPromptSubmit | ✅ | 用户提交提示词 |
| PreToolUse | ✅ | 工具调用前 |
| Stop | ✅ | Agent停止 |
| AssistantEdit | ❌ | AI编辑后 |
| ToolResult | ❌ | 工具执行后 |

**8个实战场景**：
1. 敏感词拦截
2. Rule/Skill使用分析
3. 文件变更追踪
4. 全局数据分析
5. 危险命令拦截
6. 质量门禁
7. 资产自进化
8. Skill自动注入

## Claude Code vs Qoder 对比

| 维度 | Claude Code | Qoder |
|------|-------------|-------|
| 架构 | 单智能体 | Team Lead + 专家团 |
| 记忆 | KAIROS后台整合 | 5大知识源 |
| 知识注入 | 无内置 | RepoWiki |
| Code Review | 基础审查 | 语义理解+调用链追溯 |
| 扩展机制 | 插件系统 | Hooks |
| 目标场景 | 开发者个人 | 团队协作 |
