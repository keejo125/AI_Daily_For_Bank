| QA专家 | 测试验证 | 质量门禁 |
| 评审专家 | 代码审查 | 架构把关 |
| 浏览器专家 | Web操作 | 扩展能力 |

## 核心能力详解

### 1. Experts Mode 专家团模式
**执行流程**：
```
用户输入 → Team Lead拆解 → 并行分发 → 专家执行 → 结果汇总 → Team Lead整合
```

**关键特性**：
- 异步并行，多专家同时工作
- 中枢协调，统一调度
- 自进化，经验技能自动沉淀

### 2. Code Reviewer 评审专家
**核心能力**：[2]

| 能力 | 说明 |
|------|------|
| 语义理解 | 理解为什么改，而非只看不改 |
| 调用链追溯 | 下游影响分析 |
| 架构影响判断 | 跨文件、跨模块逻辑审查 |

**两种使用方式**：
1. **Experts Mode 自动审查**：完整评审流程
2. **CLI 命令审查**：快速验证

### 3. 老项目改造方案
**核心挑战**：[3]

| 挑战 | 解决方案 |
|------|----------|
| 上下文爆炸 | 先建RepoWiki，知识提炼 |
| 知识断层 | 自动分析项目结构，生成架构文档 |
| 规范混乱 | Hooks配置质量门禁 |
| 测试缺失 | 智能单测生成 |

**改造步骤**：
1. 自动分析项目结构
2. 生成RepoWiki（架构文档）
3. 配置Hooks（规范约束）
4. 智能单测补充

### 4. 智能单元测试生成
**快手实践案例**：覆盖率从3%→80% [4]

| 版本 | 问题 | 改进 |
|------|------|------|
| V1.0 | 编译错误、mock问题 | 沙箱执行、多轮修复 |
| V2.0 | 生成时间长 | 并发流程优化 |
| V3.0 | 逻辑理解不足 | 规则+知识双驱动 |

**核心公式**：
```
智能单测 = 规则引擎 + 领域知识 + 多轮修复
```

### 5. Hooks 事件驱动扩展
**解决三大痛点**：[5]

| 痛点 | Hooks解法 |
|------|-----------|
| 不可控 | 阻断危险操作（exit 2） |
| 不可观测 | 事件日志记录 |
| 不可进化 | 上下文注入，数据积累 |

**5种事件类型**：
| 事件 | 可阻断 | 说明 |
|------|--------|------|
| UserPromptSubmit | ✅ | 用户提交提示词 |
| PreToolUse | ✅ | 工具调用前 |
| Stop | ✅ | Agent停止 |
| AssistantEdit | ❌ | AI编辑后 |
| ToolResult | ❌ | 工具执行后 |

**8个实战场景**：
1. 敏感词拦截
2. Rule/Skill使用分析
3. 文件变更追踪
4. 全局数据分析
5. 危险命令拦截
6. 质量门禁
7. 资产自进化
8. Skill自动注入

## 知识引擎 新增能力（2026-06）

### 团队共享知识引擎

来源：[6] Qoder公众号，2026-06

**核心定位**：把代码、Commit、Plan、对话中的经验"编译"成 AI 可消费的结构化知识卡片，之后每次提交自动更新、每次对话自动学习和记忆，知识越用越厚。

**知识引擎 vs 传统方式**：

| 传统方式 | 知识引擎方式 |
|---------|------------|
| 人工上传文档 → 切片向量化 → 查询时检索 → 每次从头拼凑答案 | 多源知识自动采集 → AI理解并结构化 → 建立知识关联图谱 → 持续自迭代更新 → 全团队Agent直接调用，无需重复生成 |

**核心理念**：从"一次性编译产物"，变成一份会自己生长、人人可修、AI自己会学的团队资产。

---

### 新增：记忆智能体（Memory Agent）

**架构**：常驻后台运行的 Background Agent，持续监听对话，主动判断哪些信息值得长期沉淀、何时应当被调用。

**"监控–检索–反思"闭环**：
- 监控：对话全程持续监听
- 检索：主动判断何时调用已有知识
- 反思：主动判断哪些值得沉淀

**对比上一代任务流记忆系统**（92个标准对比用例）：

| 指标 | 任务流方案 | 记忆智能体 | 提升 |
|------|----------|----------|------|
| 记忆写入通过率 | 31% | 48% | +55% |
| 记忆整理通过率 | 35% | 65% | +86% |
| 记忆检索成功率 | 40% | 77% | +93% |

> 记忆检索成功率从约四成跃升至近八成——这恰恰是任务流方案最薄弱的环节。

**对Agent任务的实际效果**（典型软工场景）：

| 知识类型 | 端到端评分提升 | Token降低 |
|---------|-------------|---------|
| 架构知识 | +25% | -30% |
| 技术栈知识 | +25% | -15% |
| 编码规范知识 | 完整通过 | - |

### 知识卡片体系

**三大核心知识**（围绕Coding Agent落地效果）：

| 知识类型 | 回答的问题 | 支撑层面 |
|---------|-----------|---------|
| 架构知识 | 系统整体结构 | 系统理解 |
| 编码规范知识 | 代码可合入性 | 代码可合入性 |
| 技术栈知识 | 运行环境兼容性 | 运行环境兼容性 |

**知识来源**（自动采集）：
- 代码工程
- Commit Message（经验根因、解法）
- 开发者对话（隐性知识）
- Plan计划

**一人生成，全团队受用**：
- 新人入职→基于团队知识库精准协助
- 团队共享，Token消耗降低（超大仓库最高-27.8%）

### 知识引擎安全机制

| 机制 | 说明 |
|------|------|
| **代码不出域** | 知识在客户端本地生成，服务端只接收结构化知识卡，零外泄风险 |
| **版本锁裁决** | 服务端按 repo+branch 加锁，commit 版本裁决，旧版本不覆盖新版本 |
| **Wiki CLI共享** | `.qoder/repowiki` 目录 git 共享；CLI 无 IDE 批量生成接 CI/CD |
| **管理员管控** | 统一维护团队知识库——保留/剔除/自迭代规则设定 |

---

### 人机协同共建

**人的一侧（/knowledge）**：
- 用户随时可对 RepoWiki 和 Knowledge Card 修改、补充或重写
- 人工修订被系统识别为新的认知沉淀，反向同步到对应知识卡片

**AI的一侧（Memory Agent）**：
- 边对话边判断、边沉淀边复用
- 上一代"被动执行" → 升级为"主动回忆与学习"

---

### RepoWiki 自动生成

**能力**：指向代码仓库，知识引擎自动阅读代码、理解模块结构、生成结构化Wiki文档。

**效果**：
- 不用手动写文档，AI自己帮你把项目"读一遍"并记住
- 以后问"这个接口的设计逻辑是什么"、"这个模块和哪些其他模块有依赖"，直接回答而非重新读代码

**对话经验自动沉淀**：
- 对话中解决了一个Bug，过程中的隐性知识（根因、排查思路、解法）被自动提取并归入知识库
- 下次遇到类似问题，Agent直接调用已沉淀经验

**知识自迭代**：
- 代码提交了新版本，知识引擎自动检测变化并更新相关Wiki页面
- 接口签名改了、模块拆分了、依赖关系变了，知识库自己跟着进化，不需要人手动维护

## 亚信科技案例（知识沉淀标杆）

来源：[6]

**背景**：30年通信行业积累，专家经验靠"传帮带"，越来越带不动。Agent进入研发流程后，也得"看得懂"这套系统。

**做对了三件事**：
1. 从代码里自动抽取规范、API资产和架构特征，把对话沉淀为知识卡片——专家不用专门写文档
2. 长文档拆成小卡片配合混合检索，Agent像翻笔记一样快速命中
3. 库级共享让一个人贡献的经验，全团队和所有Agent都能受益
