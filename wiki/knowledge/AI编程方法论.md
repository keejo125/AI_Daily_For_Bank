# AI 编程方法论

> 整合AI编程核心方法论,从Vibe Coding到Memory系统

---

## 核心共识

### 1. Vibe Coding:新的编程范式
AI 编程演进路径:[1][2]

| 阶段 | 人类角色 | AI角色 | 特点 |
|------|----------|--------|------|
| 1.0 辅助编码 | 编码者+审核者 | 代码补全 | 效率有限 |
| 2.0 Agent模式 | 审批者 | 自主执行 | 效率大幅提升 |
| 3.0 Vibe Coding | 架构师 | 全程实现 | "用嘴巴编程" |

**Karpathy观点**:"AI执行+人类审核"成为标准流程,人类不再需要理解代码,只需要描述清晰意图。[1]

### 2. 渐进式 Spec 方法论
模型能力决定方法论选择:[2]

| 层级 | 能力 | 适用方法 |
|------|------|----------|
| T0 | 一次生成全链路 | 直接使用 |
| T1 | 多提示可达接近效果 | 渐进式提示 |
| T1.5 | 基本可用但漏边界 | Spec前置约束 |
| T2 | 只能写骨架 | 人工+AI协作 |

**核心公式**:
```
Agent = while循环 + Tool Use + 工具执行器
```

### 3. Memory 是核心能力差距
AI 的差距不在"理解",而在"记忆":[3][4]

| 场景 | 无记忆 | 有记忆 |
|------|--------|--------|
| 多轮对话 | 每次从零开始 | 上下文连贯 |
| 长期协作 | 无法积累 | 越用越懂你 |
| 知识沉淀 | 不复用 | 自动复用 |

### 4. Harness Engineering：Agent 编码的"规训"体系

> 来源：InfoQ《严禁手写代码、一天烧不完 10 亿 Token 就是失职》| OpenAI 工程师 Ryan Lopopolo 访谈 [10]

Harness Engineering 与 Vibe Coding 互补——Vibe Coding 告诉你怎么用 AI 写代码，Harness Engineering 告诉你**怎么让 AI 持续产出高质量代码**。

#### 核心思想

> 「你只有两个杠杆：你提供什么上下文，以及你提供什么工具。」

Harness Engineering 的核心：将好代码的**非功能性需求**写下来，让 Agent 在正确时间看到它们。通过工具调用对 Agent 进行"提示注入"，管理其上下文窗口。

#### 零人类手写代码

Ryan 团队自 2025 年 6 月起**禁止任何人手写代码**，也不对 Agent 生成的代码做人工审查。

| 阶段 | 做法 | 效果 |
|------|------|------|
| 初期（GPT-5.2） | 当 Agent 卡住时委托人类 | 极痛苦，人均 3.5 PR/周 |
| 中期（5.3-5.4） | 自动化重复模式，构建测试+lint 闭环 | PR 吞吐量起飞 |
| 成熟（5.5） | Agent 自主完成全流程 | 人均 70 PR/周（20倍增长） |

**关键机制**：
- **自动CI反垃圾循环**：定期的自动化任务扫描代码库违规行为，提出修复 PR
- **人类反馈蒸馏**：on-call 工程师对 Agent PR 的「点赞/点踩」→ Agent 下次运行自省改进
- **错误可学习**：允许低后果错误进入主分支，让 Agent 从中学习

#### 零人类代码审查

审查焦点从代码转向更高层次的文档/SPEC：

> 「如果你依赖文本文档驱动实现，那文档的措辞是否精确、是否充分描述了任务，变得至关重要。」

| 传统审查 | Harness 审查 |
|----------|-------------|
| 逐行审查代码 | 不审查代码 |
| 审查侧重点：实现 | 审查前置到：**文档/SPEC 质量** |
| 事后发现 | 事前保证+自动化程序化护栏 |

#### SPEC 三阶段 Pipeline

Ryan 团队独创的 SPEC 精炼流程，本质上是**用 Token 换确定性**：

```
Phase 1: Codex 读原始实现 → 生成 SPEC Markdown（可复现系统的描述）
Phase 2: 新 Agent（无原始代码访问）按 SPEC 重新实现系统
Phase 3: 裁判 Agent 对比新/旧系统 → 提出 SPEC 修改 → 下一轮迭代
```

结果：一套高度精炼、能可靠复现原始系统的 SPEC，同时保留歧义性空间让使用者适配自己的业务上下文。

#### 工程师能力转型

| 转型前 | 转型后 |
|--------|--------|
| 50-70% 时间写代码 | 30% 最难重构 / 30% 从零到一 / 30% 客户交流 |
| 核心技能：编码 | 核心技能：**系统思维、预见未来、扫清障碍** |
| 关注"怎么实现" | 关注"建不建"和"排什么优先级" |

> 「每个通过 Codex 作为唯一入口进入代码库的人，默认就能获得所有人的最佳实践。新人不再需要一到三个月去吸收团队的最佳实践——它们已经'铺'在代码库里了。」

#### 每天十亿 Token

> 「一天不用掉十亿 Token，几乎就是失职。」

核心论点：从模型能提取的智能量与 Token 消耗量**线性相关**。这不是显摆算力，而是逼团队找到规模化使用 Agent 的正确模式——并行化、异步循环、组织级 Agent，而非只影响个人的结对编程。

## 观点分歧

### 关于"是否需要理解代码"
| 观点 | 来源 | 立场 |
|------|------|------|
| 不用理解代码,相信AI | [1] Vibe Coding | 激进派 |
| 仍需保持批判思维,验证AI输出 | [1] Vibe Coding | 保守派 |

> **整合理解**:Vibe Coding 不代表完全放弃审核,而是将审核焦点从"语法"转向"逻辑"和"业务"。

### 关于"代码审查是否必要"
| 观点 | 来源 | 立场 |
|------|------|------|
| 零人类代码审查，审查前置到SPEC | [10] Harness Engineering | 激进派 |
| 仍需人工审查代码 | [1] Vibe Coding | 保守派 |

> **整合理解**:Harness 实践验证了——当测试、lint、自动化CI护栏足够强时，逐行代码审查可以被消除；但前提是团队建立了完善的**程序化质量护栏**和**端到端验证闭环**。

### 关于模型能力与效率
| 观点 | 来源 |
|------|------|
| 模型能力差距在"一次成功率"和"边界处理" | [2] Spec方法论 |
| AI价值不在能做多少,而在解决真实问题 | [5] AI产品断舍离 |
| Token消耗量与可提取智能量线性相关 | [10] Harness Engineering |

> **整合理解**:效率提升 ≠ 功能堆砌,"200美元测试法"--用户愿不愿意为此付高价,才是真正的价值判断标准。

## AI 产品设计理念

### 断舍离四原则
来自得到AI产品实践:[5]

| 功能类型 | 常见误区 | 正确做法 |
|----------|----------|----------|
| 模板堆砌 | 功能越多越好 | AI自动匹配,用户无感 |
| 伪需求 | 想象用户需要 | "愿不愿意付200美元?" |
| 信息压缩过度 | 一键生成完整结果 | 保留用户加工空间 |
| 门槛过高 | 追求技术领先 | "我奶奶能不能用?" |

### "200美元测试法"
问团队一个问题:**"如果这个功能标价200美元,你们愿不愿意掏钱?"**

如果摇头 → 不做。

## 团队知识沉淀体系

> 整合自腾讯技术工程《Harness不是目的，知识才是护城河》[8]

### 核心论点：知识是护城河

Harness Engineering 三支柱（上下文工程、架构约束、持续治理）中，**知识管理本身就是核心能力**，而非附属品。工作流只是管道，知识才是流过管道的活水。

**团队知识的永恒价值**：领域模型、架构决策、最佳实践、已知陷阱、业务流程——这些不因模型换代而失效。

### 知识分类：三层次级

| 类型 | 定义 | 示例 |
|------|------|------|
| 散点型 | 孤立的事实 | "广告计划包含预算/出价/投放时段三个核心字段" |
| 因果型 | A导致B的推理链 | "选择事件驱动而非RPC同步，因为广告状态变更需要解耦" |
| 时空型 | 特定场景/时间窗口才成立 | "高并发下广告预算扣减会超扣，需用Redis+Lua保证原子性" |

越高级的知识，越难从模型获得，越依赖团队实践积累。

### 五层存储架构

| 层级 | 内容 | 范围 | 特点 |
|------|------|------|------|
| Layer 0-P | 个人偏好 | 本地 | 不共享 |
| Layer 0-T | 团队约定（宪法） | 团队级，Git共享 | 相对稳定 |
| Layer 1 | 技术知识 | 团队级，跨项目 | tech-wiki/ |
| Layer 2 | 业务知识 | 团队级，按领域 | biz-wiki/{domain}/ |
| Layer 3 | 项目知识 | 项目级 | 随项目走 |

**提升规则**：Layer 3项目知识经判定可提升至Layer 1（通用技术）或Layer 2（通用业务）。

### 三级成熟度 + 自动衰减

| 成熟度 | 定义 | 升级条件 | 衰减触发 |
|--------|------|----------|----------|
| draft | 新提取，单一来源 | — | 6个月未引用 |
| verified | 单项目验证 | 1个工作流成功引用 | — |
| proven | 多项目验证/可信赖 | ≥2项目验证 | 12个月未引用降为verified |

**衰减意义**：知识会过时，定期降级/归档比保留过时知识更有价值。

### 三级渐进式索引（按需消费）

| 层级 | 文件 | 大小 | 作用 |
|------|------|------|------|
| Layer A | knowledge-catalog.md | ~50行 | 知识库有什么——分类统计+按阶段推荐查阅路径 |
| Layer B | 各目录catalog.md | ~100-300行 | 每个分类有哪些条目——ID+标题+成熟度+标签 |
| Layer C | TK-*.md / BK-*.md | ~50-200行 | 这条知识说了什么——完整内容+背景+适用场景 |

**查询流程**：A(50行全貌) → B(300行定位) → C(按需读完整条目)。Agent用~350行成本获取全貌，只在真正需要时才读完整内容。

### 团队知识库架构

独立Git仓库（team-knowledge.git），不寄生于业务项目：

```
team-knowledge.git
├── knowledge-catalog.md          ← Agent查询入口
├── .knowledge-config.yaml        ← 团队配置
├── team-conventions/             ← Layer 0-T: 团队约定
├── tech-wiki/                    ← Layer 1: 技术知识
├── biz-wiki/{domain}/            ← Layer 2: 业务知识
├── project-profiles/             ← 项目画像
└── contributions/
    ├── pending/
    └── conflicts/
```

**独立仓库价值**：跨项目共享、生命周期独立、权限独立。

### 三种团队角色

| 角色 | 权限 | 适用人群 |
|------|------|----------|
| maintainer | 裁决冲突、审批proven提升、管理成员 | 负责人、资深工程师 |
| contributor | 通过工作流自动贡献（创建/验证/标记矛盾） | 正式成员 |
| reader | 只消费知识，不贡献 | 新成员试用期 |

### 工作流即知识沉淀闭环

```
/flow-run（每次需求）
  INIT（知识注入）: git pull团队知识仓库，注入查询入口
    ↓
  各阶段执行（知识消费）: Agent按需查询（预算控制）
    ↓
  ARCHIVE（知识提取）: 自动从产物中提取知识条目 + 提升判定
    ↓
  下一人/fow-run 自动受益
```

**三级渐进式索引**+查询预算控制，避免上下文膨胀。

### Lint治理机制

借鉴Karpathy的LLM Wiki，定期检测：

| 检查项 | 处理方式 |
|--------|----------|
| 孤儿条目（无引用、无验证） | 降级为draft |
| 矛盾（同主题相反结论） | 标记冲突，等待maintainer裁决 |
| 12个月未引用 | proven→verified |
| 6个月未引用 | verified→draft |
| 重复/相似条目 | 标记合并候选 |

触发：每10个工作流自动触发 / `/knowledge lint`手动触发 / 连续30天未执行时启动提醒。

### 与Memory体系互补

本文的五层存储与Memory系统形成互补：

- **Layer 0-P**：对应个人Memory（MEMORY.md、日记录）
- **Layer 0-T**：对应AGENTS.md团队约定
- **Layer 1/2**：对应团队知识库（tech-wiki/biz-wiki）
- **三级成熟度**：对应Memory的半衰期机制（draft→verified→proven ≈ 活跃→衰减→归档）
- **Lint治理**：对应Memory的consolidation（日记录→原子记忆→MEMORY.md）

## AGENTS.md 实践体系

### 核心理念：地图而非手册
AGENTS.md 是给 AI Coding Agent 看的项目指令，与 README.md（给人看）互补。[^1]

| 文件 | 读者 | 核心内容 |
|------|------|----------|
| README.md | 人 | 项目介绍、快速开始 |
| AGENTS.md | AI为主 | 构建命令、编码规范、验证流程 |

**200行原则**：AGENTS.md 应是 ~200 行的导航地图，告诉 Agent "去哪里找什么"，详细内容通过链接指向 docs/ 专题文档。[^1]

### 五大核心实践

**① 仓库聚合（解决上下文割裂）**
- 前后端分离 → monorepo：AI 同一上下文全栈编码
- 存量项目用脚本聚合（setup-repos.sh）作为折中[^1]

**② 统一环境配置（让 AI 能启动项目）**
- 环境变量统一配置在 `~/.<project>_env`（避免 Git 提交）
- 一键启动脚本封装：构建 + 启动 + 健康检查
- AI 只需调用 `./scripts/start-server.sh`，不需理解细节[^1]

**③ 验证闭环（改完代码不算完）**
- curl 验证规范：每个 curl 独立执行，用临时文件传递数据
- Agent Browser 验证页面渲染（前端）
- "写完代码不算完，自测过功能才算完"[^1]

**④ 自动化检查（规则的执行力）**
- 分层依赖检查：`make lint-arch` 自动扫描 import 禁止跨层依赖
- 格式检查：`make lint-format / format`
- **规则优先级**：能自动化检查的 > 写在 AGENTS.md 的 > 口头约定的[^1]

**⑤ 参考项目引入（给 AI 喂够上下文）**
- 私域组件 / 开源内核 / 其他产品 → 放入 `reference-projects/`（git submodule）
- 源码是最准确的文档，永远不会过时
- 配套架构说明文档（ref-*.md）帮 AI 快速定位[^1]

### AGENTS.md 标准模板（9章节）

```
# AGENTS.md
## 1. 项目概述（必读，前10行建立心智模型）
## 2. 快速命令（构建/启动/格式/检查）
## 3. 后端架构（包结构树 + 核心子系统）
## 4. 前端架构（技术栈/路由/组件库）
## 5. 关键约定（5-10条硬性编码规则）
## 6. 本地开发及验证流程（curl模板/Token/日志）
## 7. 质量检查（make lint-arch/format/build/test）
## 8. 参考项目约定（列表 + 优先级规则）
## 9. 文档导航（所有详细文档索引）
```

### 实施建议：Bad Case 驱动迭代
不要一次写完，从 AI 犯的错误出发：
1. AI 犯了错误（如跨层依赖）
2. 思考："如果 AGENTS.md 里多写一条 XX 规则，AI 会不会不犯？"
3. 判断改哪里：全局规则 → AGENTS.md，模块细节 → docs/


**工具支持**：Claude Code `/init` 或 `harness-creator` Skill 可一键生成初始版本 + lint 脚本 + Makefile。[^1]

## Claude Skill 编写方法论

Claude 官方 Skill 编写指南，核心解决"Skill 写了但 AI 不用"的问题。[^7]

### 渐进式披露机制

Claude 启动时**只读 Skill 的 name + description**（30-60 token），匹配后才加载正文。[^7]

这意味着：
- **description 决定触发率**：弱描述~20%，强描述（具体动词+触发场景+边界定义）可达72-90%
- **正文只写 Claude 不知道的信息**：偏好标准、工作流步骤、踩过的坑、输出规范、边界定义

### 强 description 公式

```
[触发场景] + [具体动词] + [边界定义]

示例：
弱："帮助用户写邮件"（20%触发率）
强："当用户说'起草邮件'、'写一封邮件'、'拟邮件'时，调用本 Skill 生成专业商务邮件草稿"（72%+）
```

### 正文内容组织原则

| 写什么 | 不写什么 |
|--------|----------|
| 用户的偏好标准（如"微信消息≤50字"） | Claude 已知的通用知识 |
| 工作流步骤（先A后B再C） | 基础概念解释 |
| 踩过的坑和修正策略 | 显而易见的操作 |
| 输出格式规范 | 通用最佳实践 |
| 边界定义（什么不归本 Skill 管） | 宽泛的能力描述 |

### 内容过多时的分文件策略

```
skill-name/
├── SKILL.md          # 精简框架（name + description + 核心规则）
├── references/
│   ├── workflow.md   # 详细工作流
│   └── examples.md   # 示例库
├── assets/
│   └── template.json # 模板文件
└── examples/
    └── case1.md      # 典型案例
```

**按需加载**：Claude 只在需要时读取 references/ 下的文件，避免一次性加载过多内容。[^7]

### 与 AGENTS.md 的互补关系

| 维度 | AGENTS.md | Skill |
|------|-----------|-------|
| 作用域 | 项目级（一个项目一个） | 任务级（多个 Skill 可组合） |
| 读者 | AI Coding Agent | Claude 主 Agent |
| 内容 | 项目结构、构建命令、编码规范 | 任务流程、偏好标准、边界定义 |
| 更新频率 | 随项目迭代 | 随任务模式固化 |

> **实践建议**：AGENTS.md 解决"AI 懂项目"，Skill 解决"AI 懂任务"。两者结合 = 项目上下文 + 任务 expertise。[^7]

### 与现有方法论互补
- **Spec 七步法**：AGENTS.md 是 Spec 的落地载体（验证方案写进 Design 文档）
- **Memory 体系**：AGENTS.md 是项目级 Memory，Agent 打开项目即读懂上下文

## DESIGN.md：设计系统的"双向沟通"方案

Google Labs 开源的设计规范格式，核心解决"AI懂数值、不懂意图"痛点。

### 核心理念：机器层 + 人类层

**机器可读YAML层（Front Matter）**
- 定义所有设计token：colors、typography、rounded、spacing、components
- 确保AI精确执行，不乱猜品牌色

**人类可读Markdown层（Body）**
- 8个固定章节：Overview → Colors → Typography → Layout → Elevation → Shapes → Components → Do's and Don'ts
- 用自然语言解释"为什么这么设计"

### CLI工具链

| 命令 | 用途 |
|------|------|
| lint | 规范校验，规避错误 |
| diff | 版本对比，追溯变化 |
| export | 多格式导出（tailwind/dtcg） |
| spec | 生成规范文档 |

### 核心价值

- 设计系统可随代码仓库版本化管理
- AI能精确理解设计规范和需求
- 设计师无需反复向AI和开发者解释意图

## 核心原则总结

### Vibe Coding 四原则
1. 不要试图理解代码—相信AI
2. 专注于需求描述—清晰的意图 > 精确的指令
3. 接受不完美—快速迭代比完美重要
4. 保持批判思维—验证AI的输出

### Harness 实战原则
1. **杠杆是上下文+工具** — 只用两个杠杆，让 Agent 在正确时间看到正确信息
2. **零手写代码** — 将自己的时间全部投入 Agent 编排和系统思维
3. **前置审查** — 不审查代码，审查 prompt/SPEC/设计文档的质量
4. **自动化护栏** — 用测试+lint+CI 循环替代人工代码审查
5. **Token 不省** — 十亿 Token/天 不是浪费，是规模化效应的前提

### Spec 编码七步法
1. 需求澄清
2. 技术约束确认
3. 设计评审
4. AI生成代码
5. 自测验证
6. 代码审查
7. 持续集成

### Memory 最佳实践
- **上下文控制**:20%-30%利用率最优
- **存储策略**:精选 vs 全存,质量 > 数量
- **更新机制**:增量更新,避免覆盖重要旧信息

## Spec 设计方法论：Spec 该长什么样才审得动

> 来源：dolphin07 公众号（SSD驱动开发系列） | [9]
> 核心：Spec 应是「架构设计的极限压缩版」——一屏摘要＋关键图＋契约/不变量＋验收。

### 核心命题：Spec 审不动 = 病根

> 「需要人 review 才能保证正确的东西，恰恰说明它没有自证能力。」
> AI 把写代码打到地板价，**稀缺的不再是写，是审**。

### 20年前的答案：离岸外包的结构

```
intent → 架构设计 → 外包写 code       （2000年代）
intent → spec → AI 写 code         （AI时代）
```

架构师交给外包的从来不是详细需求文档，是 **HLD/LLD、ICD、SOW、UAT**。Spec 的参照物应是**架构设计**，不是 PRD。

### Spec 四要素（可校验的交付物）

| 要素 | 说明 | 为什么传不走样 |
|------|------|---------------|
| **设计视图（图）** | 上下文/时序/状态/ER 图 | 一段文字十人读十版；一张图谁看都一样 |
| **接口契约** | 字段/状态码/调用时序 | 唯一零歧义、还能自动验 |
| **参考实现** | 样例 + 规范 | 规则会被误读，样例难误读 |
| **验收+UAT** | 怎么算做对 | 只有可执行的完成定义挡得住不可信的执行者 |

### 图正在被低估，它该回归

> 这个判断经过时间验证——20年前架构师交付的结构和形式，今天依然成立。

- 人审文字会 skim，「看着都对」
- 人审图，结构不对一眼就露
- **图是为「人审」量身定的形式**

**更高阶用法**：让 AI 多画图，人只审图。图有损——只审得出结构级错（漏状态、越界、链路断），审不出阈值/并发/精度。所以图不替代契约和验收，是分工。

### Harness Engineering 视角的 Spec 审查

OpenAI Ryan 的实践为 Spec 审查提供了**更激进的注脚**：当 AI 能可靠生成代码，人类审查应**完全剥离逐行代码**，聚焦在：

1. **Prompt/SPEC 质量** —— 措辞是否精确，任务描述是否充分
2. **高层设计图** —— 架构图、状态机、时序图是否完整
3. **验收标准** —— 能否自动执行

> Ryan：「不审查代码」≠ 不审查质量。我们审查的是**能让 Agent 正确编码的文档**。

这与 dolphin07 的「审图不审代码」不谋而合，且更进一步——连审图也可以自动化，人类只需守住**业务判断和红线**。

### AI 三点变更 Spec 形态

| AI 属性 | Spec 对策 |
|---------|----------|
| 机械强、判断弱 | 业务判断和红线凸显到最前，人专审这个 |
| 没有记忆（每次从零开始）| Spec 当外挂记忆——一份写下来、能版本化、下次直接喂 |
| 不担责、跑得快 | Spec 必须薄到**三分钟能审完**；验收必须能自动跑 |

> **Spec 该薄，不是图省事，是你根本审不过来。**
> **它写得多全不重要，你审得多快才重要。**

### Spec = prompt 的蒸馏

Vibe Coding 每次都重新 prompt，用完即弃。Spec 不一样：

| 好处 | 为什么 |
|------|--------|
| 复用 | 框架统一，新人照着填，跨人跨需求都能接 |
| 省 token | 稳定结构 + 复用 spec，不必每次长篇 prompt 重新喂 |
| 提效 | reviewer 看的永远是同一套结构，审得更快更稳 |

### 模板骨架

```
标题：〈功能名〉

## Part I — 背景与目标
- 一句话：做什么、为什么做
- 命名约定（API / DB / 错误码）

## Part II — 系统设计
- 2.1 架构图 + 核心时序图（mermaid）← 关键图
- 2.2 协议设计：统一响应 / 错误码表 / 接口协议  ← 契约
- 2.3 数据模型：ER 图 + DDL + 索引说明        ← 关键图 + 契约
- 2.4 方案选型：只保留有真实取舍的决策
- 2.5 资金安全：状态机 + 流转规则（涉资产才写）← 不变量/红线

## Part III — 质量保障
- 3.1 验收标准：Given-When-Then 表格         ← 验收
- 3.2 风险与审查自查表：每项填「设计依据」    ← 给人审的入口
- 3.3 测试策略：单测 / 集成 / 并发           ← 对照 AC

## Part IV — 上线发布
- 4.1 上线 Checklist + Seed SQL
- 4.2 灰度与回滚

## 决策确认表                               ← 该人拍板的决策全拎到一处
| # | 决策点 | 影响范围 | 默认决策 | 状态 |
```

### 建模：程序员要重新捡起的手艺

> AI 把「将模型翻译成代码」打到接近免费，没被打掉的是「先想清楚系统该长什么样」。

| 建模图 | 让 reviewer 一眼审出 |
|--------|---------------------|
| 上下文图 | 越界、漏依赖 |
| 组件/模块图 | 改动落点、职责错位 |
| 时序图 | 链路断点、事务/锁边界 |
| 状态图 | 非法迁移、漏掉的状态 |
| ER 图 | 外键/基数、金额/库存一致性 |
| 部署图 | 上线、回滚、容量风险 |

> **画不出来的图，就是你还没想清楚的地方。**
