# 金融智能研发

> 银行业智能研发体系建设的实践、方法论与行业动态

---

## 核心命题

**金融行业智能研发 = AI能力 × 金融级约束（安全、合规、审计、国产化）**

- 普通 AI IDE 解决的是"能不能写代码"
- 金融 AI IDE 解决的是"写出来以后能不能进流程、过审计、担责任"

金融 AI Coding 的审计，不该只看代码改了什么，还要看人如何指挥 AI 改了代码。

---

## AI IDE 与研发驾驶舱

### 四个特征

真正的 AI IDE，至少要具备四个特征：

1. **感知上下文** — 知道当前工程结构、技术栈、依赖关系、历史代码和正在处理的任务
2. **执行任务** — 不只是回答问题，而是能改文件、跑命令、生成单测、触发检查
3. **审核机制** — AI 生成的结果不能直接进入生产，必须经过自动化检查和人工确认
4. **过程留痕** — AI 调用了什么知识，依据了什么规约，改了什么文件，谁审核采纳，都要能追溯

没有这四点，所谓 AI IDE 只是"带聊天框的编辑器"。

### 为什么入口一定会回到 IDE

聊天框离研发现场太远：

| 对比维度 | 聊天框 | IDE |
|---------|--------|-----|
| 上下文获取 | 依赖人说清楚 | 主动读取当前工程 |
| 跨文件修改 | 很难处理 | 天然连接项目目录和代码索引 |
| 工具连接 | 需额外接入 | 原生连接终端、Git、测试、调试和插件 |
| 任务状态管理 | 很难管理 | 把任务、文件、测试、提交、审核放在同一个研发现场里 |

AI Coding 走到深处，入口一定会回到 IDE，不是因为 IDE 时髦，而是因为 IDE 离研发现场最近。

## 美国大行路径

### Citi
- 用 GitHub Copilot 作为辅助型工具
- 用 Devin 处理软件补丁、升级等任务（Agent 委派）
- **边界划得很清楚**：不允许 Agent 直接部署代码
- Agent 只产出 artifacts，交给开发者，再经过自动测试和人工 review 后由人决定是否采纳

### J.P. Morgan
- 内部 LLM Suite 用于代码审查、单元测试编写、头脑风暴等软件开发场景
- 把模型能力放进内部可控环境里，降低数据泄露、合规和安全风险

### Bank of America
- 软件开发人员使用 GenAI 工具辅助代码编写和优化
- 强调 human oversight、transparency 和 accountability

### Morgan Stanley Research 判断
随着 AI coding assistants 和 agents 成为标准开发工作流，传统软件工程师会更多转向复杂应用，成为 **curators、reviewers、integrators 和 problem-solvers**。

**美国路径关键词**：生态工具、Agent 委派、代码仓库上下文、人类审核、自动测试、内部知识约束。

## 中国金融路径：六条线

1. **数据安全** — 代码、日志、配置、接口、客户相关数据不能随意外流
2. **权限控制** — 不同人员、不同岗位、不同系统，能看什么、能调什么、能改什么，必须按权限执行
3. **国产适配** — 模型、算力、操作系统、芯片环境都要考虑金融机构的实际部署约束
4. **审计留痕** — 所有 AI 交互、知识调用、任务执行和人工采纳都要能追溯
5. **知识内化** — AI 要接入机构自己的知识库、规约库、代码资产、历史缺陷和业务规则
6. **流程嵌入** — AI 不能游离在研发流程之外，必须与 DevOps、安全门禁、测试平台、投产系统打通

### 招商银行 DevAgent 案例
- 自研研发智能体 DevAgent，采用 ReAct 模式
- 结合编程现场环境感知、企业研发知识检索等工具
- **月均处理 4.8 万+ 开发任务**

## 研发入口重构

传统工作流里，开发人员的工作被切散在很多系统里。AI IDE 的价值：把这些分散的上下文重新拉回研发现场。

### 交互渠道重构（兴业银行吕家进）

未来会有越来越多的个人和企业通过智能体筛选金融服务。金融机构面对的客户很可能是硅基人而不是碳基人——APP 将从一个个独立的系统，隐身为被智能体调用的功能模块，需要未雨绸缪、有序收敛。

## 五个装配维度

1. **装配上下文** — 项目结构、技术栈、接口依赖、命名规则、公共方法、历史代码和当前任务
2. **装配知识** — Wiki、接口平台、代码库、群聊、设计文档、历史版本里的知识变成可调用上下文
3. **装配工具** — 改文件、跑命令、跑测试、调用 Git、查日志、触发流水线、执行安全扫描
4. **装配流程** — 单测、复核、安全扫描、流水线、版本交付、投产审批和运维反馈
5. **装配审计** — AI 改了什么文件、依据什么知识、调用什么工具、测试是否通过、谁审核、谁采纳

## 过程即审计

传统审计是事后追溯。AI 进入研发后，仅看 Git Diff 已经不够：
- 开发人员给 AI 下了什么原始任务
- AI 调用了哪些知识库条目
- 引用了哪些规约
- 修改了哪些文件
- 谁审核了结果，最终是否采纳
- 责任由谁闭环

以前审计看的是结果痕迹，以后审计还要看协作痕迹。

## 研发知识主权

通用模型懂通用代码，但不懂一家银行几十年的系统演进。银行真正的护城河：
- 业务规则、代码资产、工程规约、测试案例、故障经验、合规要求

外部工具可以帮助你生成代码，但未必理解你的系统边界、监管约束、过去踩过的坑。

## 知识工程体系

> 模型决定 AI 会不会写代码，知识工程决定 AI 会不会写银行能用的代码

### 核心判断

- **模型不是护城河，组织记忆才是护城河**
- **私有数据才是真正的"护城河"**——互联网公开数据逐渐耗尽，有自身"味道"的数据才能建出有自身特点的模型（兴业银行吕家进）
- **知识是数据变为智慧的必经环节**——金融业对大模型的精准性、可解释性要求高，更需知识增加模型性能，弥补逻辑性不强、可解释性不足等缺陷（兴业银行吕家进）
- 没有知识工程的 AI，像一个聪明但不熟路的新员工——不知道系统边界、不知道哪些字段不能改、不知道哪些公共方法有坑
- 三类检索分工：RAG 解决"查得到"，代码图谱解决"关系看得清"，规约资产解决"边界守得住"

### 八类研发知识

| 类别 | 内容 | 对 AI 的价值 |
|------|------|-------------|
| 代码知识 | 源码、目录结构、公共方法、组件、SDK、框架脚手架、历史兼容逻辑 | 告诉 AI 这家机构过去怎么写、怎么复用、怎么处理异常 |
| 文档知识 | 需求、设计、接口文档、表结构、技术方案、评审意见、复盘报告 | 变成有版本、有责任人、有适用范围的资产 |
| 规约知识 | 编码/安全/架构/接口/测试/投产/日志/异常处理/开源规范 | AI Coding 的护栏，没有规约越快风险越大 |
| 测试知识 | 测试案例、测试数据、历史缺陷、覆盖率报告 | 决定 AI 生成代码能不能被验证 |
| 交付知识 | 流水线配置、门禁结果、版本说明、环境差异、投产交接 | 让交付阶段风险提前发现 |
| 运维知识 | 日志、指标、链路、告警、工单、故障案例、处置手册 | 让 AI 按专家排障路径缩短定位时间 |
| 合规安全知识 | 数据安全、反洗钱、消保、审计要求、CVE 情报 | 监管和合规边界 |
| 专家经验知识 | 老架构师、老开发、老测试、老运维的隐性知识 | 最难沉淀、最值钱，从个人经验变组织记忆 |

### 主动知识捕获

不从"让人写文档"开始，而从研发过程自然发生处捕获：

| 来源 | 捕获方式 |
|------|----------|
| IDE 中修改 AI 生成代码 | 影子学习——自动提取"为什么专家这样改" |
| 代码评审 | 驳回意见、修改建议、风险提醒 → Rules 和反例 |
| Git 提交 | 通过 Hook 关联需求、测试、缺陷 → 变更知识 |
| 故障复盘 | 告警→日志→根因→处置→验证，结构化沉淀 |
| IM 讨论/会议纪要 | 轻量模型提取知识候选，责任人确认入库 |

### 知识加工流水线

进入 AI 之前必须加工：去噪→去重→去过期→去敏感→切分→打标签→向量化→图谱化

**切分原则**：不能简单按字数切。需求按业务流程切，代码按函数/类/模块/调用链切，故障按现象→根因→处置切，规约按规则→触发条件→反例切。

**标签体系**：系统名、应用名、业务域、知识类型、适用版本、权限级别、责任人、更新时间、引用来源、有效期。

### 代码图谱与 RepoWiki

**RepoWiki**：自动生成每个应用的工程说明书——做什么、有哪些模块、核心流程、公共方法、接口调用、表结构、依赖组件、历史易出问题处。

**代码图谱**：函数关系、模块依赖、接口调用、公共组件复用、字段血缘、测试覆盖、缺陷关联、版本变更。

**研发全生命周期图谱**：代码→需求→设计→测试→缺陷→运维→反哺代码和规约。这张图一旦建起来，AI 才能回答"这次修改会影响哪里"。

### 规约变成硬约束

概率性生成 vs 确定性约束——AI 可以建议，红线必须硬拦：

| 形态 | 作用 |
|------|------|
| Rules | 告诉 AI 哪些规则必须遵守 |
| Spec | 告诉 AI 任务如何表达、拆解、验收 |
| SOP | 告诉 AI 流程如何走 |
| Template | 告诉 AI 输出长什么样 |
| Policy-as-Code | 告诉系统哪些红线必须拦截 |

安全漏洞、敏感信息、开源风险、权限边界、生产变更——不能靠模型自己判断，必须 AST 扫描、规则引擎、门禁系统强制校验。

### 知识进入研发流程

知识工程终局不是让人去找知识，而是让知识在正确时间来到正确任务现场：

| 阶段 | AI 调用的知识 |
|------|-------------|
| 需求 | 业务规则、历史需求、同类案例、需求模板 |
| 设计 | 应用资产视图、RepoWiki、接口文档、架构规约 |
| 编码 | Rules、Spec、代码上下文、公共方法、工程规范、安全规范 |
| 测试 | 需求、设计、代码变更、历史缺陷、测试数据 |
| 交付 | 流水线记录、门禁结果、环境差异、历史投产问题 |
| 运维 | 日志、指标、链路、告警、变更记录、历史故障、处置手册 |

### Agent OS 分层记忆

| 层级 | 范围 | 适用场景 |
|------|------|----------|
| L1 即时上下文 | 当前文件、当前错误、直接依赖、当前测试结果 | 代码补全、简单解释、实时纠错 |
| L2 项目记忆 | RepoWiki、项目历史、接口图谱、近期缺陷、项目规约 | 模块开发、Bug 修复、局部重构 |
| L3 组织记忆 | 全行规约、公共组件、安全红线、合规要求、历史故障库 | 架构设计、安全审查、关键变更 |

简单任务走 L1，复杂任务走 L1+L2，重大变更才调用 L3 和图谱推理。还要有 Router 动态判断任务应该调用什么知识、什么模型、什么工具。

### 知识主权与防火墙

- **多租户知识底座**：物理可隔离，逻辑可统一目录
- **LLM Firewall**：知识输出前做敏感信息识别、脱敏、权限校验、合规检查
- **知识主权**：外部工具可以采购，模型可以切换，但组织记忆必须掌握在本行或可信私有化环境中
- **银行可以买能力，但不能把组织记忆交出去**

### 激励机制

- **Knowledge Credit**：知识贡献积分——规约被 AI 高频调用并减少代码问题，应被记录和认可
- **知识合伙人**：每个重点应用/领域/系统都有人对知识质量负责
- **新角色**：规约工程师、数据资产工程师、RepoWiki 工程师、研发知识图谱工程师、AI 测评工程师、Skills 运营工程师

### 90 天 MVP 路径

| 阶段 | 内容 |
|------|------|
| 第一个 30 天 | 选种子场景：底层框架组（Rules/Spec/RepoWiki）+ 运维 SRE（日志/告警/故障案例） |
| 第二个 30 天 | 最小知识加工：1-2 个应用接入代码库/接口/测试/故障；生成第一版 RepoWiki；沉淀 10-20 条高频规约；基础 RAG + 初版代码关系 |
| 第三个 30 天 | 真实任务验证：一个开发任务+一个测试任务+一个运维任务，观察 AI 知识命中率、引用准确率、生成可用性 |

### 北极星验收指标

| 指标 | 含义 |
|------|------|
| TTM 缩短率 | 需求到上线的周期缩短 |
| MTTR 下降率 | 生产故障平均修复时间下降 |
| 生产事故/高风险缺陷下降率 | 规约更可执行、安全门禁更前置 |

辅助指标：知识命中率、引用准确率、AI 代码入库率、单测覆盖率、人工修正率、反馈闭环率、Token 成本、缓存命中率

⚠️ **代码采纳率陷阱**：采纳率高不一定代表质量好，可能审核变松或开发懒得改。必须同时看长期缺陷率、复核检出率、回滚率。

### 组织革新四转型（兴业银行吕家进）

金融智能化不仅是技术升级，更需要体制机制系统性改革：
1. **科技研发从瀑布式转向敏捷化**

3. **组织架构从专业分工转向业技融合**

## 增程器与飞控系统（2026-05-06 新增）

> 核心命题：金融 AI 要进入生产，既要补足能力（增程器），也要守住控制权（飞控系统）

### 增程器：让 AI 先跑起来

**类比**：新能源汽车的增程器 — 纯电是方向，但增程器让车先进入真实生活。

金融 AI 今天面临的现实：大量老系统、复杂流程、孤岛数据、非标准接口、人为经验。模型生成的高维智能，不能天然变成旧系统可以执行的低维指令。

**增程器具体形态**：规则、知识库、流程系统、人工复核、多模型路由、专用工具、RPA、API 封装

**应用场景**：
- 模型可以先写授信报告，关键风险由审批人确认
- 模型可以先生成客户回复，高敏投诉由客服主管把关
- 模型可以先写代码和单测，架构、安全、性能仍由研发负责人复核
- 模型可以先做合规比对，最终风险点由合规专家确认

### A2H：增程器也是数据漏斗

A2H = Agent-to-Human。真正有价值的增程器，把人的介入变成数据。

**Because 数据**：审批人为什么改了 AI 的风险判断？合规人员为什么标注这段条款？客服主管为什么删掉一句客户回复？研发负责人为什么重写某个接口？

> 金融系统过去记录了很多结果：批了还是拒了，通过还是驳回。但真正决定组织能力的，往往是中间那句"为什么"。

- 没有 Because 数据的 A2H = 只是在人工补漏
- 有 Because 数据的 A2H = 在训练未来的数字员工

增程器不仅补动力，也在捕捉**组织记忆**——把资深业务专家的判断、修改、标注和驳回沉淀下来。

### 飞控系统：不能让 AI 失控

**类比**：飞机有飞控系统，限制危险动作、防止失速、修正姿态，在关键时刻保护飞机不进入不可控状态。

金融 AI 的飞控系统 = 定义"飞行包络线"。在这条包络线内 Agent 可以工作；越界时系统必须干预。

**越界必须停的情况**：
- 逻辑前后矛盾，要停
- 权限越界，要停
- 风险超阈值，要停
- 监管口径不匹配，要停
- 异常成本飙升，要停
- 高敏客户投诉，要转人工
- 核心系统变更，要进入审批链

### 飞控系统的五个组成部分

1. **权限控制**：Agent 能看哪些数据、调用哪些接口、代表谁执行、推进到哪一步，必须有明确边界
2. **合规和风险校验**：模型生成内容需经过监管口径、内部制度、风险规则、敏感表达、客户保护检查
3. **流程控制**：AI 可以生成建议，但流程是否推进、是否放行、是否升级，必须由流程层决定
4. **审计留痕和责任闭环**：事后能还原 — 谁发起指令，模型调用哪些数据，中间谁修改，最终谁确认
5. **异常熔断和安全降级**：模型不确定时停，结果冲突时停，权限异常时停，成本异常时停

### 业务风控同步编码（兴业银行吕家进）

传统架构中，业务和风控呈串联、博弈的线性关系——"业务冲在前、风控踩刹车"。智能时代，业务和风控的边界将被重新定义：业务和风控团队必须围绕共同的数据和模型开展工作，风险参数和业务逻辑从开始就被同步编码。实时感知风险、即时切断风险将成为金融机构在智能时代的核心竞争力。

### 增程器和飞控系统的关系

| 维度 | 增程器 | 飞控系统 |
|------|--------|----------|
| 本质 | 战术性桥梁 | 战略性基座 |
| 解决的问题 | 模型现在还不够强 | 模型再强也不能失控 |
| 未来变化 | 随着模型能力提升，部分弱化 | 永远不会消失 |

> 金融行业永远需要权限控制、合规校验、审计留痕、责任闭环和异常熔断。

### Agent OS 应该同时承载两者

**增程器层**：知识增强、多模型路由、人工协同、规则兜底、流程补偿、旧系统适配

**飞控系统层**：权限控制、合规校验、动作边界、异常熔断、审计留痕、责任闭环

这两者结合，金融 Agent 才能从"能回答问题"走向"能参与生产"。

### 后链路智能体：测试·交付·运维

> 没有智能测试、智能交付和智能运维的 AI Coding，就像没有刹车和仪表盘的高速车：越快，越危险

**核心判断**：成熟的 AI Coding Agent，交付的不是代码片段，而是一条可测试、可审计、可追责的工程证据链。

#### 测试智能体：重做质量出口

真正可用的测试智能体，至少做五件事：

1. **基于需求和设计生成测试案例**：结合结构化需求、详细设计、接口说明、表结构、业务规则和验收标准，覆盖主流程、异常路径、边界条件、权限控制和数据状态
2. **基于代码变更做影响分析**：改了什么模块，影响哪些接口，涉及哪些表，关联哪些历史缺陷，哪些地方需要补测
3. **辅助构造测试数据**：账户状态、交易状态、授信状态、风控标签等，按脱敏和权限要求生成构造建议
4. **生成和修复测试脚本**：接口测试、自动化测试、批量测试、安全测试生成初稿，判断失败原因是环境问题、数据问题还是脚本错误
5. **生成测试报告和质量分析**：总结风险点、缺陷分布、覆盖缺口、回归建议和上线风险

**测试智能体解决的不只是"测没测到"，更是在重构银行的质量信用体系。** 过去测试质量高度依赖个人经验和团队成熟度；未来测试知识要逐步变成组织资产。

#### 交付智能体：把风险前移到研发中途

交付智能体的核心价值：**把投产风险从"上线前夜"前移到"研发中途"的雷达**。

分支是否合并正确、依赖版本是否一致、环境配置是否有差异、数据库脚本是否完整、接口变更是否通知下游、批量任务是否受影响、制品是否匹配版本——这些问题如果都堆到上线前夜，任何一个小错误都可能变成投产风险。交付智能体要在研发中途持续监测风险指标，让问题有充足时间被识别和修复。

#### 运维智能体：复杂架构下的故障推理系统

运维智能体的本质：**不是日志翻译器，而是复杂架构下的故障推理系统**。

在信创、老旧系统和分布式架构并存的银行环境里，智能运维的本质是复杂性治理。AI 参与生成的代码，生产出现问题后，运维人员需要理解：当时 AI 基于什么上下文改了什么代码、为什么这样改、谁审核了结果、测试是否覆盖、交付是否识别过风险。运维智能体要把这个推理链固化下来。

#### 后链路不是兜底部门，而是组织记忆的固化剂

> 站在金融 CIO 视角看，AI Coding 可能带来一个新的治理问题：它会加速数字熵的上升。代码写得越快，系统复杂度的膨胀就越快；变更越频繁，测试、交付、运维和审计的压力就越大。

**真正的智能研发不是单纯提速，而是对冲复杂度。** 让更多变更、更快迭代、更复杂架构，仍然处在可测试、可交付、可运维、可审计的秩序之中。

### AI Leader 最该抓什么

1. Agent 遇到未定义异常时，是继续生成还是停下来/降级/转人工？
2. 专家每一次修改、驳回、补充说明，是消失在页面里还是进入 Because 数据变成组织记忆？
3. 简单任务/低风险任务/高频任务，是否都在调用最贵的模型？
4. 模型输出能不能被校验，动作边界能不能被控制，审计链路能不能还原，异常场景能不能熔断？

### 核心观点

> 未来金融机构之间的 AI 差距，表面看是模型能力，往下看，会越来越变成工程体系成熟度的差距。

> 真正的生产级 AI，是让模型在增程器的补足下进入业务，在飞控系统的约束下进入生产。

## 安全体系深度：从飞控到证据链（2026-05-09 新增）

> 本节提炼自《AI Coding十讲 08：AI决定研发的上限，安全决定生产的底线》。与「飞控系统」章节互补——飞控系统定义了控制框架，本节定义具体的工程落地路径和量化体系。

### 核心判断

**安全不是 AI Coding 的刹车，而是 AI Coding 进入金融生产流水线的通行证。**

金融 AI Coding 的审计，不只看代码改了什么，还要看：
- 开发人员给 AI 下了什么原始任务
- AI 调用了哪些知识库条目、引用了哪些规约
- 修改了哪些文件、谁审核了结果、最终是否采纳
- 责任由谁闭环

### 安全信任鸿沟

AI Coding 正在引发研发范式变化，但效率与安全之间存在一道「信任鸿沟」——这是大模型进入银行核心研发流程的最后一公里。

**这道鸿沟的三层含义：**
1. 速度提升后，风险有没有同步收住
2. 代码生成后，安全有没有前置校验
3. Agent 参与研发后，权限有没有被管住

没有安全底座，AI Coding 生成得越快，风险也可能堆得越快。

### 安全放行公式

金融级生产代码 = AI 生成逻辑 + 确定性扫描报告 + 自动化测试存证 + 人工复核签名。

**缺一不可。**
- 少了扫描报告 → 无法证明风险被检查过
- 少了测试存证 → 无法证明逻辑被验证过
- 少了人工复核签名 → 无法证明责任被承接过

### AI Coding 放大的五类风险

1. **代码安全**：输入校验不足、异常处理不完整、敏感信息泄露、权限判断遗漏、SQL注入风险
2. **开源和依赖**：引入 CVE 风险、许可证冲突、不可控组件
3. **数据安全**：客户信息泄露、测试数据违规使用、跨域访问
4. **权限安全**：Agent 越权调用工具、执行命令、读写文件
5. **审计和责任**：无法还原「谁发起→AI调用什么→生成什么→谁审核→谁放行」的责任链

### 安全规约工程化（三层）

| 层级 | 内容 | 形态 |
|------|------|------|
| 硬约束 | 硬编码密钥、敏感信息泄露、禁止依赖、高危漏洞、越权访问、生产数据违规 | 规则引擎、扫描工具、门禁平台、权限系统（确定性执行） |
| 软约束 | 异常处理风格、日志规范、接口返回规范、错误码规范、历史风险案例 | RAG、Prompt 模板、示例库、代码片段库（提高生成质量） |
| 反馈约束 | 每一次安全扫描结果、人工复核意见、修复建议采纳、豁免理由、生产安全事件 | 安全规则库、漏洞案例库、规约资产（回流到上下文） |

**AI 与门禁冲突时**：硬规则 > 模型建议，门禁结果 > 生成结果，安全责任人 > 自动修复。

### 代码安全智能体（边界定义）

代码安全智能体**不是**安全审批人，不能替安全团队盖章。它的真正价值：
1. **解释扫描结果** — 结合代码上下文、历史规则、调用链，区分真风险/误报
2. **给出修复建议** — 符合本行规范的修复方案（而非通用建议）
3. **识别相似风险** — 基于代码图谱和模式匹配，横向排查同类问题
4. **沉淀安全知识** — 每次风险识别、修复、复核、豁免、复盘回流为安全知识

### 安全成熟度路线

| 阶段 | 核心目标 | 关键动作 |
|------|----------|----------|
| **基础级** | 入口可控 | 统一入口、基础 SAST/SCA/Secrets 检测、人工 review、提交门禁 |
| **增强级** | 流程可控 | 安全规则进 AI IDE/DevOps/CI/CD：编码现场实时提示、提交前检查、流水线门禁、开源白名单、CVE 情报 |
| **领先级** | 体系可控 | Agent OS 底层治理：统一身份/权限/工具调用/数据边界/Skill 上架/安全证据链/责任链 |

### 金融研发 Agent 安全准入清单（8问）

上线前至少回答：
1. 这个 Agent 的责任人是谁？
2. 它能访问哪些代码库和知识库？
3. 它能调用哪些工具，是否有越权阻断机制？
4. 它能否读取日志、数据和配置，是否做了脱敏？
5. 它生成的代码是否经过扫描、测试和人工 review？
6. 它的每一次工具调用是否留痕？
7. 它的输出是否可以被审计和追责？
8. 它下架、回滚、禁用和版本更新机制是否明确？

### 关键安全指标体系

**前移指标**（安全提前发现）：
- 编码阶段安全问题发现率
- 提交前风险拦截率
- AI 生成代码安全问题占比

**门禁指标**（阻断高危）：
- 高危漏洞阻断数
- 敏感信息泄露拦截数
- 开源风险阻断数
- 投产前未关闭安全问题数量

**AI Coding 特有指标**：
- AI 生成代码的安全问题密度（每千行高危/中危数）
- AI 修复建议复核通过率
- Agent 越权动作拦截率
- 模型上下文脱敏合规率
- AI 生成依赖的开源合规通过率

**生产指标**：
- 生产安全事件数量
- 高风险变更拦截率
- 漏洞引发生产问题数量

> 金融 CIO 关心的不是 AI 多能写，而是 AI 写出来以后，系统还能不能稳，风险还能不能控，责任还能不能落。

## 政策与监管（2026-05-10 新增）

> 提炼自兴业银行董事长吕家进在第九届数字中国建设峰会数字金融主题交流活动的演讲

### 三大政策建议

1. **推动联合创新** — 美国AI发展出现科技巨头加强合作、组建联盟、交叉持股的趋势，而国内头部科技企业普遍全栈发展、相互进入对方领域。建议推动金融机构之间、金融机构和产学研之间组建大的创新联合体，集中力量办大事，打造金融智能硬科技成果
2. **促进数据共享** — 金融机构普遍感到市场上获取的数据质量参差不齐，合规成本不断上升。借鉴欧盟《开放数据指令》《数据治理法案》，建议加快公共数据管理机制改革，推动更多公共数据走向市场
3. **鼓励"沙盒"实验** — 目前我国金融业主要在内部使用AI，还不能用于对外面客服务。海外头部银行（美洲银行、摩根大通等）已在推动基于生成式AI的面客服务。建议加快构建金融智能化"沙盒"监管机制，明确准入、运行、退出规则，允许在限定规模、限定场景内实验新技术、新产品、新服务

### 关键判断

- 大模型本质是概率模型，寻求"相关关系"而非"因果关系"，是"概率合理"而非"事实正确"
- 未来基础大模型可能趋于同质化，成为类似水电煤的基础设施
- 决定金融机构最终高度的不是基础大模型的引进部署情况，而是根据自身资源整合的利用情况
- 智能体将一人分饰多角——训练企业金融、零售金融、同业金融知识，不再有岗位界限

## 银行智能研发十讲 07：后链路智能体（2026-05-21 新增）

> 提炼自《银行智能研发十讲 07：AI Coding 写得更快之后，谁来兜住生产？》

### 核心命题

**AI Coding 提升的是开发速度，后链路智能化守住的才是生产秩序。**

只做编码智能体而不做测试/交付/运维智能体，AI Coding 会从"提效工具"变成"后链路压力放大器"。

### 为什么后链路不能慢

| 现象 | 后果 |
|------|------|
| 代码生成多了 | 测试要验证的东西更多 |
| 变更频率高了 | 交付要判断的风险更多 |
| 系统调用复杂了 | 运维要解释的链路更多 |
| AI参与研发后 | 代码来源、修改逻辑、过程依据更复杂，复核/审计/追责要求更高 |

> 站在金融 CIO 视角，AI Coding 可能带来一个新的治理问题：**它会加速数字熵的上升。**
> 代码写得越快，系统复杂度的膨胀就越快；变更越频繁，测试、交付、运维和审计的压力就越大。

**真正的智能研发不是单纯提速，而是对冲复杂度。**

### 测试智能体：不是多生成几条案例，而是重做质量出口

测试智能体至少做五件事：
1. **基于需求和设计生成测试案例**：结合结构化需求、详细设计、接口说明、表结构、业务规则和验收标准
2. **基于代码变更做影响分析**：改了什么模块、影响哪些接口、涉及哪些表、关联哪些历史缺陷
3. **辅助构造测试数据**：账户状态、交易状态、授信状态、风控标签等，按脱敏和权限要求生成构造建议
4. **生成和修复测试脚本**：判断失败原因是环境问题、数据问题、接口变更还是脚本本身错误
5. **生成测试报告和质量分析**：总结风险点、缺陷分布、覆盖缺口、回归建议和上线风险

**测试智能体解决的不只是"测没测到"，更是在重构银行的质量信用体系。**

### 交付智能体：把风险从上线前夜前移到研发中途

交付智能体的核心价值：把投产风险从"上线前夜"前移到"研发中途"的雷达。

成熟交付智能体做四类事情：
1. **持续集成问题分析**：流水线失败后判断是代码编译/依赖版本/单测失败/配置缺失/制品权限/环境差异中的哪一种
2. **版本风险检查**：改了哪些应用/涉及哪些接口/影响哪些下游/是否含数据库变更/是否触发安全门禁
3. **投产材料生成和校验**：版本说明/变更说明/投产影响/回退方案/验证清单，与实际代码变更一致性校验
4. **投产后验证**：关联日志/指标/链路/业务监控，辅助判断上线是否稳定

**真正有价值的交付智能体**：能够提醒团队——这个接口、这个表、这个批量任务、这个配置项，在历史上是否曾经导致过严重问题。

### 运维智能体：不是日志翻译器，而是故障推理系统

运维智能体的本质：站在金融CIO视角，AI Coding可能带来新治理问题——**它会加速数字熵的上升**。代码写得越快，系统复杂度膨胀越快；变更越频繁，测试/交付/运维/审计压力越大。真正的智能研发不是单纯提速，而是对冲复杂度。

运维智能体至少做六类事情：
1. **告警降噪**：识别告警关联关系，压缩重复告警，识别主告警和派生告警
2. **日志语义分析**：结合时间/服务/接口/调用链/错误码/版本变更/配置变更判断可能原因
3. **链路分析**：识别故障在哪一段——入口/服务/数据库/中间件/下游依赖
4. **变更关联**：自动关联最近上线版本/配置变更/数据库脚本/接口调整/批量任务变化
5. **历史案例匹配**：调用历史故障案例，判断是否出现过类似问题、根因是什么、怎么处置
6. **处置建议和复盘沉淀**：生成处置建议/检查清单/回退判断/影响说明/复盘初稿，故障结束后沉淀回知识工程

> **在信创、老旧系统和分布式架构并存的银行环境里，智能运维的本质是复杂性治理。**

### 后链路的共同底座：知识工程

测试/交付/运维三个智能体，背后依赖同一个底座：**知识工程**。

| 智能体 | 依赖的知识 |
|--------|-----------|
| 测试智能体 | 需求知识、设计知识、历史缺陷、测试案例、测试数据、覆盖规则、环境知识 |
| 交付智能体 | 版本知识、流水线知识、环境知识、门禁规则、投产流程、历史交付问题 |
| 运维智能体 | 日志知识、链路知识、告警知识、变更记录、故障案例、处置手册 |

**没有知识工程，智能体只能做浅层问答**——测试智能体会生成模板化案例，交付智能体会生成普通版本说明，运维智能体会复述日志文本。

### 成熟的 AI Coding Agent 必须交付证据链

银行金融研发需要的不是"神秘地生成一段代码"，而是"可验证、可审计、可追责地完成一项工程任务"。

一个成熟的 AI Coding Agent，至少应该交付六类东西：

1. **代码变更**：改了哪些文件，新增/删除了哪些逻辑，影响了哪些模块
2. **测试结果**：跑了哪些单测/接口测试/回归测试，通过/失败情况，失败原因
3. **扫描结果**：是否触发安全漏洞/开源风险/代码规范/敏感信息/性能风险检查
4. **PR或变更说明**：为什么这么改，解决什么问题，影响哪些系统，有哪些已知风险
5. **审查依据**：引用了哪些需求/设计/规约/历史缺陷/故障案例/知识库内容
6. **人工确认链**：谁review、谁采纳、谁否决、谁放行、谁承担最终责任

**AI生成代码，必须自带审计基因。**

### 数字考古：后链路智能体的特殊价值

很多银行系统不是没有知识，而是知识散落在十几年前的文档、历史代码、接口习惯、运维脚本、批量任务、故障复盘和老员工经验里。

后链路智能体的特殊价值：**它要参与数字考古**——从旧需求/旧缺陷/历史测试案例里找边界条件，从历史上踩过坑的变更路径里识别风险，从凌乱日志/历史处置记录/批量运行轨迹里拼出系统真实运行逻辑。

### 三个关键指标（AI Leader应该看什么）

| 维度 | 关键指标 |
|------|----------|
| 测试侧 | 案例采纳率、单测覆盖率、复核检出率、回归缺陷下降率 |
| 交付侧 | 问题定位准确率、版本风险发现数、环境差异提前识别率、上线后回滚率 |
| 运维侧 | 告警降噪率、故障定位准确率、MTTR下降率、历史案例命中率 |

**更高层要看三个指标**：生产事故有没有减少、故障恢复时间有没有缩短、后链路经验有没有反哺前链路。

### AI Leader问自己的三个问题

1. **我到底是在买一堆临时助手，还是在建设一套可沉淀的数字研发资产？**
2. **当AI生成的代码引发生产问题时，我能不能完成合规定责？**
3. **我的团队是在从"写代码"升级到"审逻辑、控风险、管闭环"，还是退化成"给AI擦屁股"？**

### 真正的闭环：把研发风险往前移

**从管理视角看，测试、交付、运维智能化最重要的价值，是把风险往前移。**

| 过去 | 未来 |
|------|------|
| 测试阶段发现需求边界不清 | 设计阶段就暴露 |
| 交付阶段发现环境差异 | 开发阶段就识别 |
| 投产阶段发现版本依赖问题 | 集成阶段就拦截 |
| 生产阶段发现容量风险 | 容量评估阶段就前置 |

**金融研发的核心不是追求快，而是追求在可控前提下更快。**
