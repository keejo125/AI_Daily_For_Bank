# Claude Code 实战教程精华

> 整合25篇 Claude Code 源码分析教程，提炼核心知识点与工程实践

---

## 核心共识

### 1. 命令 + 工具双轨设计
这是理解 Claude Code 整个系统最重要的概念：[01]

| 轨道 | 触发方 | 用途 | 示例 |
|------|--------|------|------|
| **命令（Command）** | 用户（"/"前缀） | 用户主动操作 | `/review`、`/commit`、`/config` |
| **工具（Tool）** | AI（自主决策） | AI与宿主环境交互 | `BashTool`、`FileReadTool` |

**设计哲学**：用户用命令"指挥"，AI用工具"执行"，两者通过 QueryEngine 协同。

### 2. 七层模块化架构
Claude Code 由职责明确的模块构成：[01][05]

| 层级 | 核心问题 | 代表模块 |
|------|----------|---------|
| 入口层 | 谁来启动？ | `main.tsx` |
| 初始化层 | 如何准备运行环境？ | `init.ts`、`setup.ts` |
| 命令层 | 用户能做什么？ | `commands/` |
| 工具层 | AI能调用什么？ | `tools/`、`Tool.ts` |
| 查询层 | 如何与Claude对话？ | `QueryEngine` |
| 状态层 | 运行时状态如何维护？ | `state/` |
| UI层 | 终端界面如何呈现？ | `components/`、`ink/` |

### 3. QueryEngine 是调度中枢
QueryEngine 负责：接收用户输入 → 决策调用命令还是工具 → 与 Claude API 对话 → 返回结果。是整个系统的"大脑"。[01][08]

### 4. 技能系统是命令的超集
技能（Skill）是命令（Command）的进化版本：[10]

| 维度 | 命令 | 技能 |
|------|------|------|
| 结构 | 单个.md文件 | `技能名/SKILL.md`目录 |
| 关联文件 | 不支持 | 支持`files`字段 |
| 执行隔离 | 仅内联 | 支持`context:fork`子代理 |
| 条件激活 | 不支持 | 支持`paths`按路径激活 |

### 5. 状态管理机制
Claude Code 有完善的状态管理系统，包括：[11]
- 全局配置（API Key、模型选择）
- 会话状态（当前任务、上下文）
- 任务队列（待办、完成、进行中）

## 工程实践要点

### 初始化流程
1. 解析CLI参数 → 2. 加载配置 → 3. 初始化状态 → 4. 启动UI → 5. 等待用户输入。[02]

### 权限系统
Claude Code 有严格的权限控制：[03]
- 敏感操作需要用户确认（文件写入、网络请求）
- 不同命令有不同的权限级别

### 工具系统架构
核心模块：`Tool.ts`（基类）、`tools.ts`（注册中心）、`tools/`（具体实现）。[05]

**核心工具**：
| 工具 | 用途 |
|------|------|
| `BashTool` | 执行Shell命令 |
| `FileReadTool` | 读取文件内容 |
| `FileWriteTool` | 写入文件 |
| `GrepTool` | 文本搜索 |
| `GlobTool` | 文件模式匹配 |

### Shell执行工具
- 支持流式输出
- 超时控制
- 工作目录管理 [06]

### 文件操作工具
- 原子写入（避免部分写入）
- 自动创建目录
- 符号链接处理 [07]

### QueryEngine 调度中枢
- 意图识别：判断用户是想执行命令还是进入对话
- 上下文组装：按层级组装上下文（见深度解析）
- 模型调用：与 Claude API 交互 [08]

### 智能代理工具
- 负责任务分解
- 调用子工具链
- 错误重试机制 [09]

- 终端优先体验
- 管道与重定向支持
- 脚本化集成 [24]

## Harness Engineering 综合
Harness Engineering = 上下文工程 + 记忆管理 + 工具编排 + 安全控制的综合体系。[25]

**核心原则**：
1. 精心组织的上下文 > 海量检索
2. 后台记忆整合 > 被动等待
3. 安全优先（Fail-closed）
4. 可观测性全程追踪
