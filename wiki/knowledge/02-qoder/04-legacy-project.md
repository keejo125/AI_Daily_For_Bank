# Qoder 老项目实战指南

> 来源：微信公众号  
> 收录：2026-04-03

## 问题背景

老项目接入 AI Coding 工具的常见问题：
- 代码量大、历史包袱重
- 规范不统一、风格混乱
- 文档缺失、知识断层

## 核心挑战

| 挑战 | 说明 |
|------|------|
| **上下文爆炸** | 老项目代码量大，塞不进上下文 |
| **知识断层** | 原开发者离职，没人懂为什么这么写 |
| **规范混乱** | 不同时期代码风格不一致 |
| **测试缺失** | 改了不知道会不会出问题 |

## 推荐策略

### 1. 先建 RepoWiki

让 Qoder 自动分析项目结构，生成架构文档。

**产出**：
- 项目结构说明
- 模块关系图
- 关键文件说明

### 2. 建立 Rules 文件

在项目根目录创建 `.qoder/rules/`：

```
rules/
├── coding-style.md      # 代码风格
├── security.md          # 安全红线
├── naming-convention.md # 命名规范
└── architecture.md      # 架构约束
```

### 3. 渐进式改造

**不要一次性改造整个项目**：

| 阶段 | 做法 |
|------|------|
| 第一阶段 | 新功能用 AI 写，老代码不动 |
| 第二阶段 | 改 Bug 时顺便让 AI 理解相关模块 |
| 第三阶段 | 重构时用 AI 补测试 |

### 4. 建立 Knowledge 沉淀

每次踩坑后记录：
- 项目的特殊约定
- 历史包袱的原因
- 改动的高风险区域

## 关键经验

1. **不要试图让 AI 理解整个项目**
   - 分模块、分文件逐步理解
   - 用 RepoWiki 做概览，深入时再展开

2. **先建约束，再让 AI 写代码**
   - Rules 文件比文档更重要
   - AI 写代码前先检查是否符合规范

3. **改老代码必须补测试**
   - AI 生成的测试先跑通
   - 确保改动不破坏现有功能

4. **知识要持续沉淀**
   - 每次改动后更新 knowledge/
   - 形成项目专属的知识库
