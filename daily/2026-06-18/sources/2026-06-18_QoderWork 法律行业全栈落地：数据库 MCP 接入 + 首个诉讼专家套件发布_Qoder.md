# QoderWork 法律行业全栈落地：数据库 MCP 接入 + 首个诉讼专家套件发布

> 原文链接：https://mp.weixin.qq.com/s/f6TsLaWYRYichJd-Hi0FiQ
> 公众号：Qoder

法律行业是 AI 落地最难啃的骨头之一。原因很简单：这个行业对"准确"的要求高到近乎偏执。法条错一个字、案号差一位数、引用一部已废止的旧法，就可能构成执业事故。而恰恰在"准确"这件事上，大模型有天然软肋：它的法律知识来自训练数据的"记忆"，不会主动查库，不会校验时效，不会区分现行有效和已废止。它很聪明，但记性不总是靠谱。

过去一年，法律科技行业的共识正在从"AI 能不能写法律文书"转向一个更务实的问题：

AI 怎样才能成为法律人真正信赖的工作伙伴？

答案不是更大的模型，而是两件事同时到位：可靠的数据底座让 AI 说话有据可查，有纪律的工作流框架让 AI 做事有章可循。

QoderWork 在法律行业的落地正是沿着这条路径展开。先接通数据管道，再跑通工作流，让法律 AI 从"能聊"升级为"能用"，再从"能用"升级为"好用"。今天，两块拼图同时就位：北大法宝与华宇元典两大法律数据库 MCP 正式接入，首个诉讼领域专家套件“中国民商事诉讼工具箱（qoder-for-cn-litigation）”同步已上架专家套件市场。

法律数据库 MCP：让每一条引用都经得起核查

北大法宝 MCP 与华宇元典 MCP 是中国法律数据库领域的两家重量级基础设施，现已上架 QoderWork MCP 市场，一键安装即可调用。

法宝MCP新接入阿里QoderWork：让AI拥有法律专家的“手和眼”

两家 MCP 提供的核心能力包括法律法规语义检索与关键词检索、司法案例语义检索与关键词检索，以及专门做"引用防幻觉"的校验接口等等。

在 QoderWork 上，法律 MCP 能做什么？

以下是几个典型的应用场景：

场景一：写文书时实时引用校验。

AI 在生成法律文书时，每引用一条法规都从数据库里实时取回最新条文原文；写完之后整篇草稿再通过MCP来一遍幻觉校验，自动抽取所有引用并输出"一致/不一致/未命中"的逐条判定。

数据底座的价值不只是"能查"，而是让 AI 的每一次输出都可溯源、可核验。

场景二：法律问题研究。

假设你想知道“债权质押通知的适格要件有哪些？”，可以先让QoderWork用语义检索MCP基于口语化的问题转成有精确法条命中的指令去获取法条原文，再基于案例检索MCP拉取类案裁判规则，最后按找你希望的分析框架输出研究报告——每一条结论都有可溯源的法条和判例支撑，而不是一个“看起来很自信但实际上无法验证”的回答。

场景三：撰写新法解读文章时的新旧对比。

新司法解释一出，法律人往往需要第一时间跟进解读，对外发文或者内部沉淀。此时可以让QoderWork从法律数据库里分别取出新法全文和旧法版本，生成逐条对比表——每一行都是从权威库取回的原文引用。

场景四：结合法律法规和工商信息制作财产线索调查报告。

诉前或诉中财产保全是诉讼实务中至关重要的一环，而且执行规范浩如烟海，想要高效无遗漏地检索并非易事。在 QoderWork 上，可以同时调度企查查 MCP 拉取目标公司的工商信息、股东出资、对外投资、失信被执行、股权冻结等几十个维度的结构化数据，同时通过法律数据库 MCP 实时检索《民事诉讼法》及相关司法解释的执行与保全相关条款等规范的最新生效版本——确保每一条保全建议都附带准确的法律依据，最终将这两路数据编排为一份完整的财产线索调查报告。

法律数据库 MCP 的接入不是孤立事件。它与 QoderWork 已有的企查查 MCP、钉钉文档与 AI 表格通过统一协议形成生态：法律数据库提供权威条文，企查查提供商业数据，协作工具承载团队流程，三者可以在同一个工作流中自由编排，适配从法律研究到财产保全的各种实务场景。

诉讼专家套件：从"接上数据库"到"跑通整条工作流"

数据底座就位后，一个自然的问题是：然后呢？能干啥？

中国民商事诉讼工具箱专家套件就是对这个问题的一次完整回答。这是 QoderWork 平台上首个诉讼领域专家套件，包含 20 项专业技能，覆盖从收案建档到结案归档的完整生命周期。它不是"一个能帮你写法律文书的 AI"，而是一套有纪律的工作流系统：20 项技能共享同一份案件上下文、走同一条质量管线、经过同一道核验闸门。

背景值得一提。2026 年 5 月 Anthropic 开源了 Claude for Legal，面向美国律师的全套工作流插件，全球法律科技界都评价很高。但它直接搬到中国基本用不了：法律体系不同、数据源不通、执业习惯差异巨大。中国民商事诉讼工具箱专家套件由阿里云法务合规部游初同学开发完成，是基于中国诉讼实务从零构建的本土工作流框架，且与qoder产品生态充分融合。

一个案件跑起来是这样的：把起诉状丢进去，AI 自动提取案号、当事人、法院等信息完成建档；事实梳理输出时间线大事记、法律关系图和时间轴；攻防分析输出结构化的"分析骨架"：法律依据、争议焦点、举证责任分配；此后答辩状、证据目录、庭审提纲、代理词逐一生成。

每个技能都站在上游产出的肩膀上往前走一步，引用同一份分析成果，不会出现"答辩状的论点跟庭审提纲对不上"的情况。

除诉状、答辩状等核心文书外，套件还覆盖了保全申请、管辖异议、追加当事人等几十种程序性文书，一个入口解决全部需求，此外还有诉讼可视化、法律研究报告、新法解读、财产线索调查等辅助能力以及贯穿整个套件的案件管家和诉讼知识库，前面图中展示的MCP运行效果均来自该套件。

四个核心设计：让工作流有纪律

统一质量管线。所有核心文书走同一条生产线：加载用户画像、读取案件配置、解析材料、法律分析、检索现行法条、起草文书、核验闸门、交付。差异只在"起草"过程各自套用不同规范，其余环节完全统一。

法律核验闸门。每份文书交付前强制执行"出厂检验"：自动扫描全文法条引用，调用法律数据库 MCP 逐条比对现行有效版本，不通过就不放行。这正是数据底座与工作流框架产生化学反应的地方：法宝和元典提供的MCP校验能力，被固化为流程中不可跳过的一环。

能力槽抽象。套件不绑定任何一家数据供应商，声明的是"我需要法条检索能力"，至于背后谁在提供服务，运行时自动识别。换供应商不改代码，多家供应商自动灾备，没有 MCP 也能用。

轻量骨架加留白。完整的管线和规范，但刻意不预填知识库、不预设模板。

框架提供纪律，用户注入灵魂。

用一个月后你的工具箱和同事的会完全不同，但质量标准是统一的，如果再搭配了丰富的私域知识库、充分融合了自己的办公流程，会更加贴合用户的个性化需求。
