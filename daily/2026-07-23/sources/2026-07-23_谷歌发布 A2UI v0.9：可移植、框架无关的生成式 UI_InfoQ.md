# 谷歌发布 A2UI v0.9：可移植、框架无关的生成式 UI

> 原文链接：https://mp.weixin.qq.com/s/Gqf0y9uEwAlH2AA4KSLrGA
> 公众号：InfoQ

作者 | Daniel Curtis

译者 | 张卫滨

谷歌发布了 A2UI v0.9，这是一种与框架无关的标准，允许 AI 智能体声明用户界面意图并在 Web、移动与桌面上原生渲染，而无需分发任意可执行的代码。该发布在 谷歌开发者博客 上公布，其核心思想是：智能体应使用应用已有设计系统的表达方式，而不是自行发明新的组件。

最显著的变化是理念上的转变。根据 CopilotKit 的说法，v0.9 不是小幅更新：核心哲学、JSON 结构与 schema 均发生了改变，协议也变为双向的。此前的可选组件集“Standard”已重命名为“Basic”，以表明前端开发者应将智能体接入已有组件。客户端方面，有一个共享的 web-core 库现在为浏览器渲染器提供基础，该版本同时包含官方的 React 渲染器，以及版本更新的 Flutter、Lit 与 Angular 渲染器，并为社区渲染器设立了专门位置。

智能体端也有了自己的工具包。新的 A2UI Agent SDK 为更低的延迟引入了缓存层，向 Python 智能体添加 SDK 只需一条安装命令：

pip

install a2ui-agent-sdk

该 SDK 负责处理版本协商、运行时切换 schema 的动态目录，并具备增量解析与修复不完整 LLM 输出的健壮流式处理，使组件能随着内容到达而逐步渲染。新特性包括客户端定义的验证函数、用于协作编辑的客户端到服务器数据同步、改进的错误处理以及更简化的模块化 schema。传输层也被精简，A2UI 现在可在 MCP、WebSockets、REST、AG-UI 与新推出的 A2A 1.0 上运行。

迁移指南也已经发布。v0.8 到 v0.9 的演进指南 记录了模型提示词方面向“提示词优先”的直接嵌入式模式的演进、模块化文件的重构以及属性重命名的速查表。团队需要注意，当前稳定系列的生产发布为 v0.9.1，并且网站上已提供 v1.0 候选规范 及相应的 v0.9.1 到 v1.0 的演进指南。

CopilotKit 的 Atai Barkai 将 v0.9 称为 “一次重大改进，内置 AG-UI 支持”，并认为其已具备大规模采用的条件，但也有怀疑的声音。Hacker News 上的一位评论者质疑为何要信任 LLM 输出的 UI，并担心安全漏洞、UI 冒充攻击与可用性问题。

为什么会有人想要这个东西？为什么要相信 LLM 能输出 UI 呢？你是在自找安全漏洞、UI 模拟攻击、糟糕的可用性等等。

有一篇 Reddit 帖子 认为目录模型的限制性太强，警告“每个界面都会变得雷同”。架构师 Brian Love 提出了一个折衷的方案（即 固定目录和动态覆盖），从而在验证失败时提供确定性的回退策略。

A2UI 并不是该领域孤立存在的项目。它与 AG-UI、MCP Apps、Vercel 的 json-renderer 及甲骨文的 Agent Spec 等项目是同一个领域的竞争者，面向 Web 的项目（比如，syntux）认为基于聊天机器人生成式 UI 所产出的“可弃用（disposable）”界面，难以满足真实 Web 应用对一致性与缓存性的要求。目前谷歌将 A2UI 定位为位于这些方案之下的可移植契约。

A2UI 是由谷歌制定的开放、与框架无关的规范，用于在 Web、移动与桌面声明生成式的用户界面。完整的 v0.9 规范、演进指南以及官方 React、Flutter、Lit 与 Angular 渲染器均发布在 a2ui.org 站点上，A2UI Agent SDK 的 Python 版本可通过 PyPI 获取。
