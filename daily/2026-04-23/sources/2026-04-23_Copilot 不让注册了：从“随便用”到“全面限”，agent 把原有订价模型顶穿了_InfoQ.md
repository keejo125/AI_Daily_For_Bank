# Copilot 不让注册了：从“随便用”到“全面限”，agent 把原有订价模型顶穿了

> 公众号：InfoQ

编译 | Tina

GitHub 暂停 Copilot 新用户注册并收紧限制：“长时间运行、并行执行的工作流……已经对我们的基础设施和定价结构构成挑战”。

GitHub 宣布将暂停 Copilot 个人订阅计划的新注册，同时对现有用户施加更严格的使用限制。这一调整意味着，这款全球使用最广泛的 AI 编程工具之一，正在承受越来越大的系统压力。

这家隶属于 Microsoft 的公司，已经从原本基于请求次数的限制体系，转向对“会话”和“每周 token 使用量”分别进行独立限制。GitHub 同时建议用户，在处理简单任务时选择“倍率更低”的模型，并尽量分散请求时间，以减轻系统压力。

在本周发布的一篇博客中，产品副总裁 Joe Binder 表示：“随着开发者逐渐意识到 agent 和 sub-agent 在解决复杂编程问题中的价值，我们看到所有用户的使用强度都在上升。”

他同时承认：“这些长时间运行、并行执行的工作流确实能带来巨大价值，但也对我们的基础设施和定价结构提出了挑战。”

从“无限用”到“全面限”

GitHub 此次面临的问题，并不只是“用户变多”，更关键的是“使用方式发生了变化”。GitHub 在 2021 年推出 Copilot 时，它还只是编辑器里的代码补全工具。如今，它早已不只是“补全”，而是逐步变成 agent 化的编程助手。随着 Copilot CLI 等能力上线，开发者开始把调试、重构甚至完整功能开发直接交给它执行。

而且新一代模型支持更大的上下文窗口，这意味着开发者可以提交更长的提示，触发更复杂的处理流程，对算力的消耗也随之显著增加。例如，Anthropic 的 Opus 4.6 和 4.7 支持高达 100 万 token 的上下文窗口，而 Opus 4.5 仅为 20 万。

针对 GitHub 的新政策，安全解决方案架构师 Nathan McNulty 在 X 上评论道：“大家都在要更大的上下文窗口，但现实是，一次高级请求就可能耗掉一整周的 Opus 4.6 配额，这种模式根本不可持续。

而他转发的，则是 Ed Zitron 的一条爆料。后者称：微软正在收紧 GitHub Copilot 的使用限额，将 Opus 模型从每月 10 美元的订阅中移除，并计划在 2026 年晚些时候转向基于 token/API 的计费模式。这也说明，微软已经在寻找降低 AI 服务成本的办法。

甚至还有开发者直接感叹：“Copilot 的时代结束了。我试了下用 Sonnet 按 API 计费，2 分钟就花了 49 美元。”

Copilot 开始限人、限量、限模型

随着 Copilot 的 agentic 能力快速扩展，智能体承担的工作越来越多，也有越来越多用户开始触及原本用于保障服务稳定性的使用上限。 Binder 表示，如果不做调整，整体服务质量就会下降。

接下来，GitHub 的这一决策将影响多个付费层级，包括 Copilot Pro、Pro+ 以及学生计划，这些套餐的新订阅将暂时关闭。现有用户仍可在不同套餐之间升级，而 Copilot Free 仍然对新用户开放，这也意味着 GitHub 仍在保留未来付费用户的入口。

与此同时，平台还引入了更严格的使用上限，对不同模型和功能在各个套餐中的分配方式进行了调整。Pro+ 套餐现在提供的使用额度超过标准 Pro 的五倍，但 GitHub 并未披露具体数值，也没有明确说明与此前相比收紧了多少。用户如果需要更高额度，可以升级到 Pro+。

过去，付费套餐往往意味着接近“无限制”的使用；现在，GitHub 正在更依赖使用控制机制，通过追踪请求量，在达到一定阈值后进行限制。换句话说，Copilot 正在向“按使用量约束”的服务形态靠拢，尽管计费方式本身尚未改变。

VS Code 和 Copilot CLI 也从本周开始，都会直接显示剩余可用额度。GitHub 的说法是，这样可以帮助用户避免“意外触达资源限制”。

CLI 的使用限制

在模型访问方面也发生了变化。Opus 模型已不再提供给 Pro 用户，新发布的 Opus 4.7 仅限 Pro+ 使用。同时，GitHub 也确认，旧版本的 Opus 4.5 和 4.6 也将从 Pro+ 中移除。

规则一改再改，门槛一提再提

事实上，GitHub 在 2026 年 4 月 10 日就已经引入了一系列“服务可靠性限制”和“模型级使用限制”，用于应对“高并发和高强度使用”。那次调整，GitHub 暂停了所有 Copilot Pro 的免费试用，理由是存在滥用行为。

这期间，不少用户还遇到了意外的限流。

用户 John Clary 表示：“我只是个业余开发者，但每个月在额外额度上也要花几百英镑。最近我经常遇到短时间的限流，中断请求，而且没有明确倒计时，只提示‘X 秒后重试’，通常不到 30 秒。但如果你真的这么做，很容易在 AI 正在处理关键任务时再次被打断，所以我一般会等 15 分钟再继续。”

Clary 表示，这种情况还能勉强应对。但随后他很快就触发了一个长达 44 小时的“每周限流”。起初他以为是 bug，等了一阵，但一个小时后发现限流并不会自动解除，于是他提交了工单，并无奈切换到了 Auto 模式，让 Copilot 自动选择模型。这种模式很可能会优先选择推理成本更低的模型，因为 Clary 表示，性能明显变差了。

“从那之后，我的项目几乎没什么进展。Auto 模式选的模型质量很差，经常偷偷走捷径又不说明，我还得花时间让它改回来。”

类似的抱怨，也开始在 GitHub Copilot 社区中大量出现，一些关于限流的讨论帖，在这半个月里就新增了数十条投诉。不少用户形容这些限制“离谱地长”，有人甚至被告知需要等待几天才能再次使用服务。

当限流变成常态，这就不再是简单的容量问题了。越来越多开发者开始意识到，问题可能出在更底层：Copilot 的定价模式，本身已经跟不上现在的使用方式。

开发者 Roman Kir 指出，问题的转折点出现在 2026 年 3 月。彼时，GitHub 发现其限流系统存在一个 bug：一直在低估新一代模型（如 Claude Opus 4.6 和 GPT-5.4）的 token 使用量。这些模型单次请求消耗的基础设施资源远高于上一代，但这个 bug 让它们看起来成本很低。

当 GitHub 修复该问题后，限额立即恢复到原本配置的水平。对于正常使用的用户来说，这些限制瞬间变得过于严苛——几十个小时的封锁，coding agent 在任务中途被终止，也没有恢复路径，除了 Pro+ 之外，没有其他升级方案。

Roman Kir 进一步指出：“这一点之所以重要，是因为这个执行层的问题只在 agent 使用场景下才会出现。单次 API 请求在限流时还能‘降级运行’，但多步骤的 agent 任务不行。一旦在执行过程中触发限流，任务就会被打断，进入一个无法恢复的状态。这意味着，同样是算力受限，不同使用方式会导致完全不同的失败结果，而当前的订阅体系并没有区分这种差异。”

这背后更深层的原因是：GitHub 的定价架构，是基于一个旧模型成本结构设计的。过去，“一次高级请求”大致意味着相近的基础设施成本；但在前沿模型时代，这一假设已经不成立。GitHub 一方面以订阅形式出售前沿模型能力，另一方面要承担这些模型对应的真实基础设施成本，这两者已经不再对等了。

其他网友在限速讨论中也做出了类似的推测：“GitHub 的补贴水平似乎已经达到了难以为继的地步。因此，这与其说是一个漏洞，不如说是一项商业决策……”

GitHub 的举措并非孤例

GitHub 的这一调整，也延续了整个行业近期的一系列“紧急刹车”动作。例如，Anthropic 在 3 月收紧了 Claude 的会话限制，以“应对不断增长的需求”；而 OpenAI 也对 Codex 引入了新的使用上限。

开源项目 OpenClaw 等第三方智能体编排工具的兴起，也在进一步放大对 AI 工具的需求。这类工具可以让智能体几乎 7×24 小时自动调用模型，持续消耗 token。因此，Anthropic 还在本月早些时候限制了其订阅在第三方工具（如 OpenClaw）中的使用，这类调用不再包含在订阅内，而是单独计费。

Anthropic 的 Claude Code 负责人 Boris Cherny 在 X 上表示，这一变化源于订阅设计与实际使用方式之间的错配。“我们一直在努力应对 Claude 使用需求的增长，但当前的订阅模式并不是为这些第三方工具的使用方式设计的，”Cherny 写道，“算力是一种需要精细管理的资源，我们优先保障直接使用我们产品和 API 的用户。”

另一边，Google 也在 2026 年 1 月为其 Antigravity 服务引入了每周使用上限，并据称开始封禁通过 OpenClaw 使用该平台的账户，原因是“恶意使用大幅增加”。

整体来看，一个共识正在形成：当 AI 编程从“辅助工具”变成“长时间运行的智能体系统”，原有的订阅模式与算力供给，正在被彻底拉到极限。
