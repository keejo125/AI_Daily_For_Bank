# 你的项目你做主——灵码 Project Rules 功能详解

> 原文链接：https://mp.weixin.qq.com/s/qdtUa2fnOVVJR9m5MfYjQA
> 公众号：灵码Lingma

用AI写代码的开发者，大概都经历过这样的场景：

你让灵码生成一段业务逻辑，代码跑通了，但变量命名用的是 camelCase——而你的项目规范是 snake_case。你让它写个单元测试，框架用的是 Mocha，但团队统一用 Jest。你让它加注释，格式又和项目文档规范对不上。

改一次两次还好，可当AI的输出始终不符合你的项目规范，这种"纠正"就变成了额外的负担。

问题的根源很简单：AI不了解你的项目。它不知道你用什么框架，不清楚团队的命名约定，也看不到你写在 Wiki 里的编码规范。

灵码的 Project Rules（项目规则）功能，就是为了解决这个问题——让你把项目的规范、偏好、约定告诉AI，让它按你的规矩来。

01

什么是 Project Rules

Project Rules 是灵码支持的项目专属规则机制。这些规则以文件形式存储在项目目录下的

.lingma/rules

文件夹中，仅对当前工程生效。

你可以把它理解为给AI的"项目手册"。在这份手册里，你可以定义编码风格（比如变量命名规范、缩进方式）、框架约束（比如"本项目使用 React Hooks，不要用 Class Component"）、甚至模型回答的默认角色（比如"回答时偏重实操，少说原理"）。

规则设定后，灵码的智能会话和行间会话都会遵循这些规则生成代码和回答。一次设定，全程生效。

02

四种规则类型，覆盖不同场景

为了灵活适配各种开发场景，Project Rules 提供了四种触发方式不同的规则类型。你可以根据实际需要组合使用。

始终生效（Always）

顾名思义，这类规则在智能会话和行间会话的所有请求中都会生效，适合设定项目级的通用规范。

典型场景：

你的团队有统一的编码风格要求——如变量用camelCase、常量用 UPPER_SNAKE_CASE、组件文件用 PascalCase 命名。把这些写进一条"始终生效"规则，灵码在任何对话中都会自动遵循，不再需要每次提醒。

模型决策（Model Decision）

这类规则不是每次都生效，而是由模型根据你定义的场景描述自主决策是否应用。你只需描述"什么时候该用这条规则"，模型会在合适的时机自行判断。

典型场景：

你希望AI在生成单元测试时遵循团队的测试规范（比如使用 pytest + fixture 模式），但在日常问答中不需要这条规则。写一条描述为"当生成单元测试代码时生效"的模型决策规则，灵码会在检测到你在写测试时自动调用，其他场景不干扰。

指定文件（Specific Files）

通过通配符模式匹配文件路径，让规则只在特定类型的文件上生效。如果你的项目同时使用多种语言或有不同目录的不同规范，这个类型非常实用。

典型场景：

前端 TypeScript 要求 strict 模式和特定的类型规范，后端 Python 代码要求加 type hints 和 docstring。你可以分别创建两条规则——一条匹配

*.ts, *.tsx

，另一条匹配

*.py

，各自定义对应的编码要求。再比如，

docs/**/*.md

下的文档文件要求使用中文撰写并保持口语化风格。

手动引入（Manual）

这类规则不会自动生效，需要在对话中通过

#rule

（VS Code）或

@rule

（JetBrains）手动引入。适合按需执行的一次性任务或特定工作流。

典型场景：

你有一套标准的代码审查清单——检查空指针处理、并发安全、日志规范、异常处理等。把这套清单写成一条手动规则，需要审查代码时引入即可，不会干扰日常对话。又比如，偶尔需要让灵码按特定的重构模式改写代码，用手动规则一次性完成，用完即走。

04

团队协作：规则也能 Git 同步

规则文件存储在项目目录中，天然支持通过 Git 在团队间共享。把团队的编码规范作为规则提交到仓库，所有成员拉取后即刻生效——不需要每个人单独配置。

如果某些规则只想在本地生效（比如个人的回答风格偏好），在

.gitignore

中加入

.lingma/rules

即可。

版本要求：

使用 Project Rules 需要确保灵码插件版本不低于 VS Code v2.5.16、JetBrains v2.1.5、Visual Studio v0.4.4，或直接使用 Lingma IDE。

05

规则怎么写才有效

规则的效果很大程度上取决于你怎么写。分享几条实践中总结的建议。

保持简洁明确。

一条规则解决一个问题，避免在一条规则里塞太多内容。"写好代码"这种模糊描述不会有什么效果，"变量命名使用 camelCase，函数命名使用动词+名词结构，如 getUserInfo"这样的具体规则才有用。

结构化表达。

用项目符号、编号列表和 Markdown 格式来组织规则。相比长段落文字，结构化格式更便于模型理解和执行。

提供代码示例。

在规则中给出具体的代码示例（Do / Don&#x27;t）比只描述抽象规范更有效。模型能从示例中准确理解你的意图，而不是靠猜。

迭代优化。

规则不是写一次就定型的。建议在实际使用中观察AI的输出，根据效果不断调整和完善规则描述。

灵码也提供了开源的 规则模板库，里面有各类场景的规则最佳实践，你可以直接参考或复用。

提示：单个规则文件最大限制为 10000 字符，规则请使用自然语言描述，暂不支持图片或链接的解析。当规则和记忆存在冲突时，灵码会优先遵循规则执行。

06

让 AI 写"你的项目"的代码

要让 AI 真正融入你的开发工作流，光靠每次在对话里描述需求是不够的。把你的项目规范变成 Project Rules，让灵码从"能写代码"变成"能写你的项目的代码"——这才是 AI 编码助手该有的样子。
