# QoderWork 发布国内首个 AI Native 设计工作台：一句话搞定专业设计

> 原文链接：https://mp.weixin.qq.com/s/MV390RVbHp7mx1a6CG9-2g
> 公众号：Qoder

数字产品设计行业有一个根本矛盾，从 Sketch 时代到 Figma 时代，十年来从未被真正解决：设计产物是图片，研发产物是代码。格式不同，每次交接必然有损。还原度普遍不足 70%，单页面往往经历 3–5 轮标注与返工。

QoderWork 设计工作台提出了一条完全不同的路径：设计产物本身就是代码。设计师在画布上操作的、产品经理在预览中查看的、研发最终接手的，是同一份可运行文件。

这不是效率优化，而是交付形态的根本性变革——从 Design to Code，走向 Design as Code：

设计师的产出不再止步于「被看的图」，而是直接成为「被运行的工程」。

五个核心模块

设计工作台不是一个「输入 prompt → 输出网页」的生成器。它是一个完整的设计工作环境，由五个模块构成，每个模块解决设计流程中一个具体问题。

Canvas 无限画布

AI 生成的界面不是作为附件出现在聊天记录里，而是直接呈现在画布上。用户可以看到页面全貌、选中具体元素、查看结构层级、进行局部操作。设计是空间性的工作，画布让设计产物从「对话中的文件」变成「可以被直接操作的对象」。专业用户可以深入文件和工程结构，普通用户可以停留在 Canvas 和自然语言——两者操作的是同一个产物。

Preview 真实预览

Preview 让用户像浏览网页一样查看设计，而非只看一张静态截图。静态截图里「看起来还行」的布局，跑起来可能立刻暴露问题。Preview 让用户更早发现这些问题，而不是等到研发还原后再返工。

Style References 风格参考

我们内置了百余个风格参考。它们不是简单的颜色模板或主题皮肤，而是面向不同产品类型、行业气质、视觉语言和界面密度的设计资产。

背后的逻辑是：大多数人有审美判断力，但缺少设计表达能力。他们知道自己「想要什么感觉」，但很难从零开始构建一个视觉系统。风格参考解决的就是这个问题——用户选定方向，AI 在这个方向上生成、迭代、精修。不是从白纸开始，而是从一个有品质基准的起点开始。

AI 也可以根据用户描述的产品类型和目标受众，主动推荐更合适的风格方向。

Design Files 工程文件

Design Files 让用户看到设计背后的文件结构。无论是 HTML、CSS 还是 React 工程文件，它们都不是隐藏在系统内部的黑箱产物，而是作为设计的一部分被呈现出来。设计师可以不看文件，但研发需要看——产物必须是一个可以被理解、被检查、被交接的工程环境。

画笔标注

用户可以直接在画布上圈出来、划出来、标出来。比如把某个区域圈住，说「这里重排一下」；把某个按钮划出来，说「这个弱一点」；把一块内容标出来，说「这里像卡片但不要这么重」。

这些画笔信息不是简单的涂鸦。它们作为结构化上下文，连同当前 frame、命中区域、截图和用户文字一起交给 Agent。AI 能精确知道用户指的是哪里、当前状态是什么、要改成什么方向。

这是设计协作中最自然的交互方式——圈出来，说一句，改好了。不需要写 prompt 描述「第三行第二个卡片的右上角阴影减弱」，直接圈那个位置就行。

三个流程机制

除了五个功能模块，设计工作台还通过三个流程机制重构了「输入 → 生成 → 不满意 → 重来」的传统 AI 交互循环。

Questions：先对齐，再执行

传统 AI 工具的问题是：用户说一句模糊的需求，AI 就开始猜测执行。猜对了皆大欢喜，猜错了用户要重新描述、重新生成，三四轮下来时间全耗在无效迭代上。

设计工作台的做法是：当输入信息不足时，AI 先结构化追问。它会明确问你：这个页面服务什么用户？信息层级怎么组织？有没有已有的品牌规范要遵循？问完对齐后再执行，一次命中率远高于反复重来。

信息损耗是无效迭代的最大来源。Questions 把损耗堵在流程最前端。

Design Plan：先计划，再动手

大模型有一个固有特性：同一个 prompt 每次产出都可能不同。这对设计任务来说是风险——用户还没确认方向，AI 就已经生成了一个完整页面，方向不对就全部作废。

Design Plan 在生成前插入一个确认点：AI 先输出结构化的设计计划（布局方案、风格方向、内容层级、信息架构），经用户确认后才执行。方向对齐在前，执行在后。这让用户对产出有预期，也让 AI 的执行更聚焦。

Nudge：参数化微调

生成之后的精修同样重要。传统方式是重新用语言描述修改意图——「配色再暖一点」「间距再大一点」「圆角再小一点」。每次修改都是一轮完整的 prompt → 生成循环。

Nudge 把关键设计决策以可调参数的形式暴露出来。配色、间距、圆角、字号——这些高频微调项直接可拖拽调整，无需重新描述，也不会引发整体重新生成。保持设计意图的同时，让精修变得直觉化。

两条交付路径：HTML 与 React

设计工作台提供两种产物形态，服务于设计流程中的不同阶段。

HTML workspace 适用于快速探索。当用户需要快速看到一个设计方向时——比如一个首页的三种风格尝试、一个 Dashboard 的布局方案、一个落地页的信息架构——HTML 的优势是轻量、直接、速度快。适合快速探索视觉方向、页面结构、信息层级和交互雏形。

React + Vite 工程 适用于明确要交付的场景。当用户确认了方向，需要将设计推进到可被研发接手的状态时，设计工作台可以生成一个准工程级别的 React 项目。这个项目里有真实文件、有入口、有运行环境、有组件结构，研发接手后可以直接进入业务接入、组件抽象和工程治理。

谁在用，怎么用

设计师：画笔迭代，产出即工程

选定风格参考作为起点，快速生成首版；通过画笔标注连续定向迭代——圈选区域、标注意图，AI 基于画面上下文即时调整；用 Nudge 实时微调配色与间距，反复精修直到满意。最终导出 React + Vite 工程，研发直接接入业务逻辑。

产品经理：无需设计技能，产出高保真原型

选定风格参考，描述页面结构与核心模块，AI 先通过 Questions 对齐意图，再经 Design Plan 确认方向，即可在画布上获得具备设计品质的可交互原型。直接用于需求评审或团队汇报，不再需要等设计排期，也不再需要解释「这只是线框图」。

市场运营：多方向并行，覆盖高频视觉需求

输入活动主题与调性，通过 Design Plan 确认方向后即可生成；切换不同风格参考即获得不同方向的方案，覆盖海报、Banner、落地页等高频需求。设计资源紧张时，运营也能自主产出有品质的视觉稿。

为什么是「工作台」而不是「功能」

设计工作台不是 QoderWork 里新增的一个按钮或一组对话指令。它是一个完整的、独立的工作环境——有自己的画布、预览、文件系统、风格库和交互方式。

这背后是 QoderWork 的一个架构决策：扩展机制。

自 2023 年以来，几乎所有 AI 办公产品都被锁定在同一种交互形态里——聊天框。无论任务多复杂、场景多专业，用户面对的永远是一个输入框和一串对话记录。这个形态足够通用，但也已经触及天花板：设计需要画布，数据需要表格，写作需要文档结构，它们无法被一个对话窗口承载。

QoderWork 通过扩展机制解决这个问题：解耦 UI 组件、业务逻辑和工作区区块，让 AI 办公助手能够围绕不同工作场景生长出专属模式。每个模式有自己的视图、工具和产物标准，但共享同一个 Agent 底座和任务系统。

设计工作台是这个机制下的第一个领域模式。它验证了一件事：AI 办公产品的形态不必是固定的，它可以根据用户的工作场景动态适配。你的工作是什么形态，它就变成什么形态。

还有一件事

设计工作台同期，QoderWork 已支持语音输入。直接用语音描述需求、下达指令、讨论方案，QoderWork 实时理解并执行。不用打字，不用切换工具，动动嘴就能推进工作。

设计产物正在从图片走向代码，设计协作正在从翻译走向共建，AI 产品正在从聊天框走向领域工作台。这是第一步。
