# ICML 2026 | 多智能体系统也能搭积木？Agent Primitives让MAS走向模块化复用

> 公众号：机器之心

本文工作由伊利诺伊大学厄巴纳 - 香槟分校（UIUC）DREAM Lab 团队完成。通讯作者 Haohan Wang 教授为 UIUC 信息科学学院教授、DREAM Lab 负责人，研究方向包括可信人工智能、大语言模型与多智能体系统。

论文标题：Agent Primitives: Reusable Latent Building Blocks for Multi-Agent Systems

论文链接：https://arxiv.org/pdf/2602.03695

实验室网址：https://dream.ischool.illinois.edu/

背景与动机

多智能体系统（Multi-Agent Systems，MAS）展示了令人印象深刻的能力：一个模型负责提出方案，另一个模型进行批评，还有模型承担投票、规划或执行。通过角色分工和多轮协作，系统能够解决单个模型难以稳定完成的数学推理、代码生成和知识问答任务。

一个 MAS 的例子。出自我们 team 的 TMLR survey paper：Chen, K., Wang, P., Yu, Y., Zhan, X., & Wang, H. (2025). Large language model-based data science agent: A survey. arXiv preprint arXiv:2508.02744.

但这种能力背后，存在一个容易被忽略的工程问题：今天的大多数 MAS 都是围绕具体任务手工搭建的。面对一个新问题，研究者往往需要重新定义 Agent 角色、重新设计提示词、重新规定消息的传递顺序，并不断调试协作协议。换一个数据集、一个模型，甚至一个任务表述，原来的架构就可能需要大幅修改。

这与现代深度学习的发展路径形成了鲜明对比。构建卷积神经网络时，我们不会为每一个视觉任务重新发明卷积；搭建 Transformer 时，也不需要从头设计注意力机制。成熟的神经网络之所以能够快速演进，部分原因正是研究社区找到了卷积、注意力、残差连接等可复用的基础构件。

那么，多智能体系统是否也能拥有类似的抽象形式呢？是否可以把复杂的 Agent 工作流拆解成少量稳定、通用的计算单元，再像搭乐高一样，将它们按需组合起来呢？

Agent Primitives 给出的答案是：多智能体系统也可以拥有自己的「基础模块」。

从「复用 Agent」到「复用协作模式」

Agent Primitives 的出发点，是重新审视 MAS 中什么才是真正应该被复用的对象。传统方法通常把一个 Agent 看成原子单元：它拥有固定身份、固定提示词，并在系统中承担一个预先规定的角色。但「数学专家」、「代码审查员」或「规划者」这些角色，本身往往与任务高度绑定，很难直接迁移。

论文观察到，尽管不同 MAS 的角色名称和流程看起来五花八门，它们内部反复执行的计算模式却高度相似：有的系统让模型生成答案后进行自我审查；有的系统并行生成多个候选，再通过比较达成共识；还有的系统先把复杂目标拆成计划，再逐步执行。

因此，论文不再把 Agent 当作最小构件，而是进一步向下拆解，把这些反复出现的内部计算模式抽象为 Agent Primitives，即面向大语言模型多智能体系统的、可复用的潜在构件。

基础的 Primitives

论文实例化了三类基础 primitive。它们并非试图穷尽所有协作方式，而是从现有 MAS 中提炼出三种高频、互补的计算结构。

Review Primitive：

把「生成 — 批评 — 修正」变成通用循环

Review Primitive 对应的是多智能体系统中最常见的迭代改进模式。模型先形成一个候选解答，随后对其中的推理漏洞、事实错误或实现缺陷进行审查，再依据反馈更新答案。这个过程可以重复进行，直到输出趋于稳定。

它的价值不只是增加一次「请检查答案」的提示。通过把审查与修正封装成独立 primitive，系统可以在数学证明、代码调试、开放域问答等任务中复用同一种计算结构，而无需为每个任务重新设计一套 critic 和 refiner。

Voting & Selection Primitive：

在多个候选中形成共识

当一个问题存在多条合理推理路径时，单次生成容易受到采样波动影响。Voting & Selection Primitive 让系统并行探索多个候选，再对这些潜在表示进行比较、聚合和选择。

传统投票通常依赖最终文本答案：先把每条推理完整解码，再通过字符串匹配或额外提示进行选择。相比之下，该 primitive 直接作用于内部潜在表示，使系统能够保留候选方案中更丰富的语义和推理信息，而不是只依据已经压缩成文本的结果做决定。

Planning & Execution Primitive：

将高层规划与底层执行分离

复杂任务往往不是「想得更多」就能解决，而是需要先确定做什么，再决定如何做。Planning & Execution Primitive 把高层任务分解和低层步骤执行拆开：规划阶段建立整体路线，执行阶段围绕子目标逐项完成，并把中间状态继续传递。

这种分离可以降低模型在长任务中同时维护全局目标与局部细节的负担，也让同一个规划结构能够适配数学推导、程序编写与多阶段问答等不同问题。

绕过自然语言瓶颈

仅仅把工作流命名为 primitive，还不足以解决现有 MAS 的根本局限。大多数多智能体系统通过自然语言交换信息：一个 Agent 把内部思考解码成文本，另一个 Agent 再读取文本并重新编码。每经过一轮，信息都要经历一次「潜在表示 — 文本 — 潜在表示」的转换。

这种方式直观、可读，却也代价高昂。文本解码会丢失内部表示中的细粒度信息；长链路交互会不断积累噪声和误差；重复生成与读取大量文本，则带来可观的 token 开销和推理延迟。系统中的 Agent 越多、协作轮数越长，这种通信瓶颈越明显。

Agent Primitives 让 primitive 内部直接通过 Key-Value Cache（KV Cache）进行潜在通信。KV Cache 原本用于保存 Transformer 已处理 token 的注意力状态，避免自回归生成时重复计算。论文进一步把它作为 primitive 之间传递内部信息的载体，使后续模块能够直接继承前序计算形成的潜在状态，而不必先将其完整解码为自然语言。

在连接不同阶段的 KV Cache 时，系统还需要处理位置编码的一致性。论文通过旋转位置编码（RoPE）的重新编码与对齐，使来自不同计算阶段的缓存能够被后续模块正确接续。由此，多个 primitive 不只是概念上可以组合，在模型内部的计算状态层面也能够真正「插接」起来。

论文用实验证明了利用自然语言处理时，长上下文中的任务遗忘：在长上下文的开头、中间或结尾注入一条辅助指令，比较两种通信方式能否在完成原始任务的同时保留新增约束。结果显示，当指令位于上下文中间时，自然语言通信的指令遵循率骤降至 15.6%，而 KV Cache 通信仍达到 73.3%。

论文进一步考察了通信噪声对多智能体协作的影响。具体而言，研究者在数学推理历史中逐步插入来自其他问题的无关推理句子，以模拟 Agent 交互过程中不断累积的噪声。实验结果显示，当插入 3 条无关句子时，自然语言通信的准确率已下降至 73%；当噪声增加到 10 条时，准确率进一步降至 47%。相比之下，KV Cache 通信在相同设置下仍能分别保持 100% 和 93% 的准确率。

上述结果表明，相较于自然语言通信，基于 KV Cache 的潜在通信能够更完整地保留前序推理状态，并显著降低无关信息在多阶段交互中的累积影响，从而提升多智能体系统在噪声环境下的通信稳定性与推理鲁棒性。

谁来决定如何搭这些「积木」？

有了可复用构件，下一个问题是：面对一个新查询，系统应该选择哪种 primitive，又应该以什么顺序组合？如果仍然由人手工指定，架构设计的负担并没有真正消失。

为此，论文引入了一个自动化的 LLM Organizer。Organizer 首先分析当前查询的性质与难点，再从 primitive 集合中选择合适的模块并组织执行流程。例如，需要验证和修正的任务可以调用 Review；存在多种候选路径的问题可以加入 Voting & Selection；长程复杂任务则更适合先使用 Planning & Execution，再对局部结果进行审查。多个 primitive 也可以串联或嵌套，形成更完整的 MAS。

Organizer 的决策由一个轻量级 Knowledge Pool 提供参考。Knowledge Pool 记录过去任务中成功的 primitive 配置，让系统能够利用已有经验，而不是针对每个新查询从零开始搜索。随着成功配置不断积累，构建 MAS 逐渐从「手工设计一条固定工作流」，转变为「根据问题动态检索并组合基础构件」。

方法框图

完整框架如下图所示。Organizer 根据用户查询和 Knowledge Pool 中的历史经验选择并组合 primitives；Review、Voting and Selection、Planning and Execution 分别封装三类可复用协作模式；所有 primitive 的内部协调均通过 KV Cache 完成，并对外保留与普通 LLM Agent 一致的接口。

实验结果

论文在数学、代码和知识问答等八个基准、五种大语言模型骨干上进行了系统评估。实验关注的不只是最终准确率，也同时比较 token 使用、推理延迟、相对单智能体的额外开销，以及跨模型骨干的稳定性。

从具体任务来看，Primitives-based MAS 在数学推理、代码生成和知识问答上均取得了稳定提升，并且没有出现传统 MAS 方法在部分任务上提升、在另一些任务上明显退化的情况。更重要的是，这种性能增益并不是依靠无限增加 Agent 数量或生成更长的交互文本获得的。综合准确率、token 消耗、推理延迟和跨模型表现，可以将 Agent Primitives 的主要实验结果概括如下：

这些数字揭示了一个重要事实：多智能体协作并不必然意味着数倍乃至数十倍的推理成本。传统 MAS 的高开销，很大一部分来自重复文本生成和冗长通信，而不完全来自解决问题本身所需的有效计算。当协作被重新组织为潜在空间中的模块化计算后，系统可以在保留性能增益的同时，把额外成本控制在更接近单智能体推理的范围内。

走向可组合的智能系统

今天构建 MAS，仍然很像早期编写神经网络：大量能力被写死在具体架构中，每个新任务都需要重新搭建。Agent Primitives 提出了一条不同的路线 —— 把角色背后的协作规律抽离出来，封装为潜在空间中的标准构件，再由 Organizer 根据任务动态组合。

如果这种思路进一步成熟，未来开发者面对复杂任务时，或许不必先问「我需要设计几个 Agent、分别写什么 prompt」，而可以先问：「这个问题需要哪些基础计算模式，它们应该怎样连接？」

从重复造轮子，到复用基础模块；从文本消息驱动的固定团队，到潜在状态连接的动态系统，这正是 Agent Primitives 希望推动的范式转变。
