# 数字员工的第 1 天和第 30 天：QoderWake 自进化能力升级

> 原文链接：https://mp.weixin.qq.com/s/Xj3X8uJznV0eCwpmJ1u24A
> 公众号：Qoder

QoderWake 是 Qoder 家族的数字员工产品，已开启全球公测。Waker 是你在 QoderWake 平台上创建的数字员工，有身份、有记忆、有技能，和团队里其他成员一样长期在岗，能自主完成软件工程、数据分析、内容运营等各类工作任务。

v0.0.18 对 QoderWake 核心能力「自进化」做了全面升级：

Skill 会自己长出来：

Waker 干完复杂任务后会自动复盘，把做法沉淀成可复用的 Skill。你也可以上传自己的 Skill，选择是否让他在入职后持续改进。Skills 市场里的 Skill 为只读，不会被改写。

记忆和技能有迹可查：

首页新增鱼骨时间线，Waker 什么时候记住了什么、学会了什么、哪个 Skill 被降噪或恢复，一条线全部看得见，点开就能查版本。

记忆在后台自动沉淀：

不用你主动说“记住这个”。对话间隙，Waker 会自己把你的身份、项目目标、长期偏好整理成记忆，不打断你手头的事情。

这三个能力背后是同一套机：一次好的对话，通常不只是解决眼前的问题。它还会隐式带出用户的长期偏好、项目的稳定约定、某类任务更高效的做法，以及 Waker 未来应该避免重复踩坑的经验。

QoderWake 的自进化能力，就是把这些“对话里的经验”沉淀下来，让 Waker 在新的对话、新的任务、新的项目里更快进入状态。QoderWake 不是简单地把聊天记录塞进上下文，而是把经验分成两类资产：

Memory

和

Skill

。

Memory 存事实，Skill 存方法。两类资产各司其职。

Waker 的 Memory 和 Skill

到底有什么区别？

Memory 负责记住"长期为真"的信息

它像 Waker 的长期记忆，记录的是"这个人是谁、这个项目是什么、长期目标是什么、用户偏好什么、哪些约定未来还会反复用到"。例如：

用户是某个账号的 TL，长期负责小红书内容运营。

当前项目的目标是 30 天涨粉 1w。

用户偏好先给结构化方案，再进入执行。

某个项目里汇报对象、协作语言、禁用方案、固定口径等长期约定。

Skill 负责沉淀"下次怎么做"的方法

它像 Waker 的工具书，记录的是可复用的流程、检查清单、模板、验证步骤和工作策略。例如：

遇到小红书选题任务时，先判断账号定位，再拆内容方向、发布节奏和素材需求。

遇到代码评审任务时，优先检查安全、正确性、兼容性和测试覆盖。

遇到某类运营复盘时，固定按数据、洞察、行动项输出。

所以，对于 QoderWake 而言，

Memory 更像是"我应该记住什么"，而 Skill 更像是"遇到类似任务我应该怎么做"。

这个边界很重要，Memory 不应该变成一堆可执行流程，Skill 也不应该塞满用户画像和一次性事实。只有各司其职，Waker 才不会记得很多却不会用，或者会做很多却不懂你。

自进化的整体架构：

从对话经验到下一次复用

自进化不是"模型想写就写"。它是一条带治理的闭环：

QoderWake 自进化架构：从对话到下一次复用，每一步都有判断，每一步都留记录。

当用户在对话中给出稳定信息、修正 Waker 行为、补充项目定义，或者明确要求"以后都这样做"时，系统会先把这些内容抽取出来，再判断它应该进入 Memory、Skill，还是应该被拒绝。

进入 Memory 的信息，会经过作用域判断、去重合并、容量治理和版本快照，最终成为可追溯的长期记忆。进入 Skill 的信息，会走一条受控的技能更新流程，写入 Waker 级技能版本，并保留差异、贡献来源、冲突处理和回滚能力。

整条流程的目标不是"尽可能多地记录"，而是"只沉淀未来真的有复用价值的经验"。这也是为什么自进化只接收高价值的经验：长期身份、稳定职责、项目定义、关键偏好、明确纠错、可复用流程，这些才值得进入 Waker 的长期资产。

Memory 自进化：让 Waker 逐渐理解你和你的项目

Memory 自进化主要解决一个问题：

新的对话不应该每次都从零开始。

一些细节、事实性的沟通，例如用户说明自己负责什么、偏好什么、项目目标是什么，Waker 就应该逐渐形成稳定理解，而不是下一轮又问一遍。Memory 会把这些长期信息整理成可读的记忆文件，并在需要时注入到新的对话里。

当 Memory 沉淀或更新后，用户可以在对话中实时点击记忆标题，查看具体的变更内容，也能快速进行手动编辑。

它有几个关键特点：

分清个人与项目：

个人记忆记录跨项目长期有效的信息，项目记忆记录某个项目内的稳定约定。

不会无限堆积：

Memory 有容量治理，会合并相似信息、替换低价值信息，并保留版本快照。

可以人工查看和编辑：

用户可以在记忆管理页查看时间线、版本差异，也可以手动调整。

后台异步运行：

Memory 的总结和写入不阻塞主对话，失败时不会打断当前回答。

更重要的是，Memory 不是把原始聊天直接塞进去，而是把对话中的稳定事实重新组织为结构化、长期可用的内容。尤其是项目粒度的 Memory，Waker 会基于搜索能力，找到最相关的记忆条目进行注入，保证拿到的是"清晰结论"，不是一坨上下文噪音。

Skill 自进化：让 Waker 越做越会做

Skill 自进化解决的是另一个问题：

同类任务做多了，Waker 应该把方法沉淀下来。

当用户反复纠正某个工作方式，或者完成了一个复杂的任务，亦或是某次任务中形成了明确可复用的流程，系统会尝试更新相关 Skill。下次再遇到类似任务时，Waker 就能直接使用新的方法，而不是重复探索。

比如用户说"帮我阅读某个代码仓库，总结成一份报告保存到桌面"，这更像 Skill，因为它描述的是一种未来可复用的工作流程。

复杂任务完成后，Waker 自行总结沉淀做法，连同踩过的坑和试过的方案一起。

Skill 管理页：沉淀下来的「GitHub 仓库报告」技能，带着「自进化」标签。

Skill 自进化有几个核心设计：

所有自进化 Skill 都是 Waker 级资产：

不再维护项目粒度的技能分支，任何项目贡献的经验都会沉淀到 Waker 级技能中。

项目仍然有贡献记录：

如果某个项目触发了 Skill 更新，管理页会展示它对该技能的贡献路径。

版本可追溯：

每次更新都会生成新版本，用户可以查看差异，也可以回滚。

冲突需要处理：

多个会话同时修改同一个 Skill 时，会通过版本对照判断是否冲突；冲突时用户可以忽略、保留本次进化，或手动编辑后提交。

更新过程受控：

对话中的 Skill 更新通过受控的管理能力完成，不直接改临时同步目录，也不绕过版本流程。

这让 Skill 既能自动成长，又不会变成不可控的文件修改。

用户上传 Skill：默认安全，允许主动开启自进化

用户上传的 Skill 有一个特殊点：它既可能只是"我安装来用"，也可能是"我希望 Waker 后续帮我一起改进"。

因此上传时会有明确确认：是否允许这个 Skill 参与对话中的自进化。

如果用户同意，它会带上"自进化"标记，后续可以在对话中被更新，也可以在详情页关闭。

如果用户不同意，它就像市场下载的 Skill 一样，只读使用，不参与自动或对话内改写。

不论是否开启自进化，用户手工管理自己上传的 Skill 和 Waker 自动改写权限是分开的。

这个设计的重点是：

用户拥有明确控制权。

生命周期管理：会成长，也会被治理

一个会自进化的系统，不能只会增加内容。它还需要会审查、会降噪、会回滚、会停用。

Memory 侧更像一套"长期记忆的 LRU 治理"。它不是无限追加，而是保持小而准：Personal Memory 和每个 Project Memory 都有着严格的条数控制，避免造成上下文爆炸。新信息进来时，系统会先判断它是否真的长期有价值，再看能不能合并到已有记忆里；只有不能合并、且容量还没满时，才会新增一条。

如果已经达到最大条数限制，Memory 不会粗暴写入新增内容，而是进入替换判断。这个判断可以理解成"带重要度的 LRU"：优先考虑置信度低、很久没有被实际使用、使用次数少、更新时间更早的内容。

被替换也不是静默消失。Memory 会先生成版本快照，再在 Markdown 和 metadata 中完成合并、替换或标记，用户仍然可以在版本管理里看到变化、查看 diff，必要时回滚。

Skill 侧则是一套"活跃度 + 价值"的生命周期。自进化 Skill 会记录引用、使用、更新、查看、Review、回滚等活动记录。后台判断一个 Skill 是否长期未使用时，不只看页面上的数字，而是看最近一次有效活动时间：优先看最近更新，其次是最近使用、最近引用、最近查看、最近 Review、最近回滚，最后才回退到 Skill 自身更新时间。过去单纯"被注入过"不再算有效活动，因为注入不代表真的帮到了用户。

当一个可自进化 Skill 未被 pin，且最近 30 天没有有效活动时，它会从后续会话的默认注入候选中移除。这就是自动失效。这个动作会同步到运行时，并记录到首页"记忆与积累"的时间线里，用户能看见 Waker 为什么少用了某个 Skill。

但失效不等于删除。这些 Skill 仍然保留版本、历史和内容，只是先不主动参与后续对话。它有两条重新激活的路径：用户可以在管理页手动打开；如果当前激活的 Skill 都不适合承载新经验，而某个已失效的 Skill 又和当前场景强相关时，会自动恢复为激活状态，并在同一条 Waker 级版本链上继续更新。

以上生命周期的管理，让 Memory 和 Skill 都保持"会呼吸"的状态：常用的升温，过时的降噪，重要的可保护，误改的可回滚，需要时还能重新回来。

自进化的 Guardrail：自进化 ≠ 随便改

自进化能力最容易被误解成"模型可以自己改一切"。实际设计恰好相反：它是一套受控写入和用户可追溯的治理系统。

几个关键护栏包括：

内置 Skill 和市场 Skill 默认只读：

不会被对话随意改写。

用户上传 Skill 需要显式授权：

只有用户同意后，才允许参与自进化。

冲突不静默覆盖：

同一技能的并发修改会通过版本对照判断，冲突时让用户选择。

版本可回滚：

每次有效更新都有版本记录，用户可以后悔。

主对话不被阻塞：

后台自进化失败不会影响当前对话继续进行。

这套机制的目标是让 Waker 有成长能力，同时让用户始终知道发生了什么、能控制什么、能恢复什么。

快速上手

第一，

告诉 Waker 哪些信息以后还要记住。

例如：

"我是这个项目的 TL，后续方案都默认面向小红书风光摄影账号。"

"以后给我方案时，先给结论，再给执行拆解。"

"这个项目的月度目标是涨粉 1w，之后所有内容建议都围绕这个目标。"

这些更容易进入 Memory。

第二，

告诉 Waker 哪些做法以后要复用。

例如：

"以后做选题，先判断账号定位，再列内容方向。"

"下次代码评审时，先看安全和回归风险，不要先改格式。"

"这个发布复盘模板以后都按数据、问题、行动项三段输出。"

这些更容易进入 Skill。

第三，

在管理页查看、确认和回滚。

用户可以在首页"记忆与积累"中看到 Waker 最近学到了什么，也可以进入 Memory 或 Skill 管理页查看版本、差异、贡献来源和生命周期状态。对于 Skill 冲突，输入框上方会出现提醒，用户可以忽略、立即处理，或手动编辑后提交。

员工后台首页，可以看到 Waker 记忆积累的全景。

首页「记忆与积累」时间线：什么时候学到了什么技能，积累了什么记忆，一目了然。

版本管理页：每一次记忆更新的改动都能确认，历史版本可以直接回滚。

自进化所带来的体验升级

自进化能力真正想改变的不是某一次对话，而是长期使用 Waker 的感觉。

过去，用户需要不断重复上下文：我是谁、项目是什么、我想怎么做、上次哪里不对。现在，Waker 会逐渐把这些经验组织起来，变成自己的长期理解和工作方法。

它会让 Waker 更像一个稳定协作的数字员工：

知道你的长期偏好。

理解项目的稳定目标。

记得上次被纠正过的地方。

把做得好的流程变成下次可复用的 Skill。

在内容变旧、低频或不适用时自动治理。

也就是说，QoderWake 不只是"帮你完成一次任务"，而是在每一次任务之后，变成一个更懂你、更懂项目、更会做事的 Waker。
