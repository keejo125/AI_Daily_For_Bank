# 传统研发的 AI 化转型：从 AI 原生工程到 AI 原生团队

> 原文链接：https://mp.weixin.qq.com/s/DAq6qW6aTzm2ZWNB6wTQjA

大家好，我是郑云龙，来自阿里云块存储（EBS）业务管控团队，现在担任系统架构师，同时也是EBS整个核心架构组的成员。在加入阿里之前曾在ThoughtWorks担任技术咨询师，参与过多家大型企业的敏捷和DevOps的技术转型。

一、AI Native愿景

24年开始随着AI大模型以及Agent技术的持续快速发展，我们在EBS内部发起了AI兴趣小组，希望通过兴趣小组的形式持续探索和总结AI应用实践。

同时也提出了在我们组织内部在AI应用上的愿景。我们希望通过将AI技术内置在软件的端到端交付过程中，提升组织的软件研发效率，降低成本，加速业务交付速度，同时改进产品质量。

人与AI协作范式的快速演进

而到目前为止，在块存储业务管控团队在研发活动中有超过80%有AI的直接参与，覆盖有开发、Code Review 测试运维。过程中，从LangChain到Dify， 从MCP到Skill， 从Vibe Coding到SPEC驱动开发，一直到最近QoderWork产品以及小龙虾的爆火，大模型和Agent技术每天都在发生变化。

我们在不断地尝试新事物，同时也对未来变革不确定性感到困扰和焦虑。而所有的焦虑从我自己的角度来看，是因为我们没有理解AI这个新事物对我从事的这个职业带来影响的本质是什么，所以我们只能盲目地去追新。

同时对我个人而言，希望找到一条“第一性原理”，搞清楚问题的本质，才能在这场AI引发的变革中找到一个可能正确的方向。

一、AI工程化应用的基本原则

首先第一件事情就是：我们到底在生产系统里面用AI要解决什么问题？我们首先要了解AI在生产系统中应用的关键原则和一些挑战。

理解AI工程应用的第一性原理

首先相比于人，AI作为生产力工具极大地提升了代码的生成效率，几乎是即时生成，同时从大模型技术本身我们可以看到几个基本事实：

AI的无状态性：

AI的推理过程本身是无状态，缺少长期记忆

有限的上下文窗口：

信息丢失是必然发生的

幻觉永远无法消除：

LLM结构的固有特性，无法通过技术手段消除

因此如果要总结一条在软件研发中使用AI的核心目标，我的理解就是要：

去基于大模型的不确定性去构建确定性的系统支撑持续变化的业务需求。

破局之道：对抗不确定性

基于不确定性的基本原理，如果我们希望让AI更好的服务软件交付，我们要做的事情，我们就是要做好这两个事情：

1. 通过必要的手段对抗AI不确定性

2. 同时释放AI生产力以支持业务价值更快的交付

这两部分必不可少且相互制约。

二、用Harness控制不确定性

如何约束AI的不确定性，做好AI Coding前的第一公里问题。

迭代0：做好AI Coding第一公里

对于熟悉敏捷研发的同学，一定都听过一个词迭代0：迭代0是指我们在正式进行软件业务开发之前的准备环节，包括项目的脚手架、开发环境、版本控制，CI/CD。在做好这些准备工作之后，从第一行业务代码开始完成系统的迭代交付。那过去当我们代码的开发对象主要是人时，我们会通过各种文档去约定代码的开发规范和共识，并通过持续集成的方式对这些内容进行校验。

当开发对象变成AI AGENT后，基于前面我们提到的AI的几个基本事实我们需要：

为AI提供明确的上下文知识去提升AI首次输出代码的可用性

同时为AI构建反馈通道，让AI能够自我纠正，确保代码的最终质量

在前馈和反馈的相互制约下为AI提供持续输出可用代码的高速通道。

什么是Harness Engineering?

这种方式本质上就是最近非常火爆的一个词：Harness工程。

什么是Harness工程，这个本身没有一个准确的定义。如果大家去看大多数的技术文章，会有一个公式：

Agent = Model + Harness。

本质上Qoder就是一个Harness的实现，而回到我们正式业务的场景，我们不是要去重新做一个Agent，我们是要去交付正式软件功能。所以我这里对这个公式进行了扩展：

Mode + Harness + Harness. = Agent的运行时环境。

基于这个高速通道，让Agent可以持续地输出可用的代码，支撑真实业务的持续交付。

迭代0：Harness 构建约束和护栏

如何构建一个Coding Agent运行时环境？这里我们给了一个参考的实现方式：

首先就是如何高效的管理知识，基于大模型有限上下文窗口的事实，我们不能一股脑的将所有的内容全部丢给大模型。这抢占你珍贵的上下文空间，因此我们需要理解Qoder的上下文加载机制。默认加载和按需加载。

其次构建开发阶段护栏，这里我们采用了git hook机制，通过pre-commit框架，注册了一组用于校验Agent输出的本地护栏，让Agent可以自助修正，减少人类参与。

最后我们也需要在代码提交后，定义明确门禁，100%的测试覆盖率，以及业务的E2E回归。

架构师Harness工作流

对于不同的软件项目，大家都会有一些根据内在需求产生的架构约束。以EBS管控而言，我们最大的挑战是多云交付。多云交付的问题是什么？比如在公有云上有不同的售卖模式，在专有云下售卖模式可能不一样。一套代码在不同的云下基础设施可能有差异，SDK甚至可能不兼容。公有云上要横向接入的服务，在专有云上可能不存在。外部依赖存在差异，同一个特性在不同环境下交付时也会有特性差异。

所以我们整个系统在设计时，会天然形成一些设计上的约束。包括基于领域驱动设计和领域边界的模式，将核心业务的云上资源管控和售卖模式进行边界解耦。另外基于六边形架构和依赖倒置原则，把核心模块和外部接口以及基础设施之间的依赖进行解耦。

基于Harness的逻辑，在我们的项目中架构师会完成以下几个工作：

准备前馈：

与Qoder做Vibe Coding，让它基于我们的架构规范形成架构文档，并且索引到AGENTS.MD里面。

构建反馈：

我们采用了ArchUnit一个架构测试框架，可以让我们像写单元测试一样去验证架构的分层逻辑、模块之间的边界、以及相互调用的关系。

设置护栏：

在有了前馈和反馈渠道后，我们基于Pre-Commit的设置强制护栏，让Agent在提交代码的时候能够触发ArchUnit的架构测试用例。除了本地护栏以外，在CI上也要持续验证。因为一旦涉及到团队协作时，最终集成后的结果，我们也希望一定是能够符合架构测试规范的。

架构的长期守护：

你的架构测试用例一定是会随着系统的演进而持续发生变化的。因为它不是一次性的事情，架构设计需要长期的、基于整个项目演进的过程去维护好这些架构用例。

传统的架构师就从过去的"我写文档然后告诉同学去做事情"，变成了一个团队的实际"立法者"。而且这些一定是强制约束的，不管是对Agent还是对于人，都遵循同一套规范。这样对于团队而言，我们可以逐步完成从人到Agent协作的转型。

三、AI原生工程

在基于Harness工程为Agent构建了一套软件开发的高速通道后，接下来就需要进一步从软件交付的端到端的视角去看我们的软件交付过程。

将AI内嵌到端到端交付

以EBS管控团队为例，从代码生成、Code Review、测试用例生成、线上故障的快速诊断、定期对线上系统的异常日志和Trace异常数据进行分析。这些所有的东西我们都可以基于Agent的方式去实现端到端能力的自动化。

AI Coding协作模式

从AI Coding协作模式上，我们采纳了Vibe Coding和Spec驱动开发的。 在需求分析和创意阶段，通过Vibe Coding与 AI协同产生Spec，在开发阶段由AI基于Spec自主完成研发，人类审核最终结果。

Qoder协同开发4步工作法

如果大家关注过Spec驱动开发的话，在开源社区上也有各种各样的开源实现。Spec驱动开发本身非常好：基于这种模式可以尽可能减少幻觉、减少AI的返工，提升首次代码生成的有效性。

但大家也可以反过来思考一点：如果连过去非常严谨的代码，维护都会有问题的话，当你的系统变成了Spec之后，由于Spec本身缺少严格规范，当量多了之后，它本身的维护就是一件非常痛苦的事情。因为它不像编程语言有确定的输入输出，Spec完全是根据个人经验产生的没有标准格式的东西。

在Qoder里面，可以非常好地帮助我们消除这样的问题。

首先我们去跟Qoder Vibe完成Spec的设计，定义好需求以及验收规范。然后具体实现，让Plan基于Qoder的Plan模式或专家团模式，基于输入的需求规格去产生自己的实现计划。

对于人而言就是做对于它结果的验收。

最后一个点就是知识显性化：

Qoder提供了Repo Wiki的能力，可以让我们基于最终输出的代码产生一个知识库。

这个知识库跟代码本身是一一对应的，这样我们就不需要去维护Spec内容：对于系统而言，最后的"真理"的部分还是在代码以及最终的Wiki结果上。

AI辅助故障诊断：现有工作流的自动化

在运维阶段我们也基于Qoder提供的Slash Command以及自定义skill扩展能力，将现有的工作流进行自动化，以AI辅助故障诊断为例：

过去当线上出现异常或告警之后，你需要去看监控指标、应用日志、Trace链路。几个不同系统之间切换的过程非常繁琐。而且需要非常熟悉可观测性基础设施的同学才能很快完成故障定位。

但基于Agent的自动化能力，比如Qoder提供了Slash Command或Skill这样的扩展能力，我们可以让Agent帮助自己去完成故障诊断。

给它一个请求ID之后，可以让它自动拿到这个请求ID相关的所有日志、链路的整个上下文以及监控指标，然后基于这些指标产生结果：告诉我们故障到底是因为什么，完成整个故障的定位定界。

它最终形成的东西就是你的Slash Command、自定义工作流，以及一些基础的Skill。

对于所有基础设施，我们也希望能做一些AI友好的改造：过去基于UI的东西可能对AI并不友好。我们可能会在Skill里面做一些CLI工具，或者如果企业已经对系统做了MCP化的改造，可以把基础能力通过MCP暴露给大模型。当然本身基于API也没问题，因为你可以让Qoder基于你系统的API去生成一个新的脚本或Skill。基于这个模式，我们可以完成现有工作流的自动化。

AI原生工程Everything As Code

如果把这些所有东西放在一起，就是我们团队在上个阶段做的事情：

AI原生工程：基于AI实现软件端到端交付过程的自动化，将上下文、Slash Command、Skills等内置在代码库当中。工程本身就是一个所有的Agent.

四、AI原生团队

但这是不是我们在软件工程中应用AI技术的重点？ 通过AI的方式完成自动化，去不断地补齐短板，提升局部效率？ 当然不是，本质上是我们依然是在过去经典敏捷和DevOps框架下的Automate Everthing只是方式从过去的脚本语言，换成了自然语言。那未来的软件研发方式应该是什么样的呢？

软件研发范式转型（项目管理铁三角）

要理解未来软件研发模式的发展方向，我们首先还是需要现回归一些软件工程的基本原理。

熟悉软件工程项目管理的同学，一定清楚

项目管理铁三角

。 那项目管理铁三角讲诉了一个基本事实：

项目质量受时间，范围，成本的相互制约，在质量不变的情况下，任一要素的变更都将引起至少一个其他要素的调整；比如当你想缩短交付时间，要不你就加人加成本，要不就是减少范围。

瀑布模式通常假定软件的

范围应该是固定的，但是资源和时间是可以协调的

，因此我们需要在软件项目一开始，进行详细的设计以给出一个确定的范围，并在这个基础上协调相关的资源和时间。 从公司组织结构上也主要是以大型的职能部门为主，由不同的部门负责软件研发的各自阶段。

随着互联网时代的发展，我们发现软件的需求和范围总在持续变化，过去的这套瀑布玩法在高强度的竞争下不生效了。因此敏捷研发模式中，我们认为人是稀缺资源，业务响应力是核心竞争力。因此我们采用全功能团队，通过全功能团队，通过小周期迭代，快速交付一个客户满意的范围；

而对于大模型时代而言，大模型可以在极短的时间内输出大量的代码，代码本身变得“廉价”，只要可能“资源”（算力）几乎是无限的。 那对于软件研发而言。因此在

极其理想情况下

，我们几乎可以假设未来的软件研发将会以通过AI在

无限算力（资源）

，在

极短周期（时间）

内，快速去交付一个

客户满意的产品（范围）

为目的进行转变。

从Kanban方法说起

前面我们学习了通过Harness和SPEC驱动开发，通过人类必要的审核我们可以在控制AI不确定的同时，释放AI的生产效率。那如果把这个模式扩展到软件的端到端交付过程，会产生一个什么样的变化？

那这里我们以Kanban方法为例，Kanban是经典敏捷的一种项目管理方式，通过可视化的方式管理软件交付进度，通过拉动信号驱动项目需求的持续交付。在以人为核心的协作过程中，我们还能通过限制在制品数量的方式，最大化项目交付的吞吐。

基于Kanban的AI原生团队（异步协同）

而如果我们将看板方法中需要人去观察拉动信号（工作项状态）变成让Agent去观察拉动信号，我们就可以得到一个全新的人-Agent的工作协同方式。Agent可以主动地完成软件的设计，开发，测试以及发布，而人在这个过程中持续地审核Agent产出的结果。

Harness持续重构AI原生工作流

同时通过Harness护栏的持续左移，我们可以进一步减少人类介入，提高Agent的自主性。最终实现更短的流程，更少的协同，更高的并发。

研发范式演进到组织和个人转型

同时这个过程也会催生出超级个体的诞生，研发人员不再局限于过去基于人与人协作而产生的职能分工。人能不能降业务需求转变为严谨的提示词，定义清晰的目标和验收标准，同时能够对AI产出的最终质量负责？能不能将隐形的知识转变为明确的规则和约束，确保系统和业务的长期可持续迭代和演进？能不能不断的优化人与AI的协作流程，实现更高的并发需求交付？这些才是在AI时代下人的核心竞争力。

五、持续学习、提升能力，更好地驾驭Agent

最后，我们经常会在社交媒体上听到类似于技术平权，程序员即将淘汰。 但是事实是AI缺失降低了技术的门槛，但是同时AI也进一步加大了人与人之间的差距。AI本质是个人能力的放大器。你的能力决定了AI的上限，同时AI也在无限放大你个人的上限；

所以对个人而言我们要做的依然是持续学习、提升个人能力，这样才能让你更好的驾驭AI。
