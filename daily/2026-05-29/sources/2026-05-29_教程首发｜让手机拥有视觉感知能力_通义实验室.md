大家在玩  Agent 时，有没有遇到这个痛点，Agent 对物理世界完全是“盲”的，大模型看不到开发板的状态、听不到设备的运行异响，也无法直接获取手机摄像头捕捉的画面。

本篇是

「手机上的创意AI」

挑战赛首期教程，看完本期内容，你能够：

在 Android 手机上部署 MCP 感知服务器；

用 Claude Code 远程一键调用 phone_look / phone_listen；

核心架构：为什么选择端侧 MCP？

想让 AI 感知物理世界，传统做法是把手机摄像头当成网络摄像头，通过 RTSP 协议将原始视频流实时推送到云端分析。但这种方案的痛点是：

成本高、算力重、隐私风险

。

我们可以让手机当“本地大脑”。

在手机端部署多模态大模型，将看到的画面和听到的声音，在本地实时转化为几 KB 的结构化文本（如场景描述、异常状态列表），只把“结果”汇报给云端 AI。这就是所谓的

端侧语义提取

。

为了让云端 AI 能轻松、标准化地调用手机能力，我们在 App 中内置了 MCP 服务。你可以把它理解为 AI 世界的“标准 USB-C 接口”：无需复杂抓包或自建 WebSocket 桥接，云端只需一行命令，就能安全、无缝地接入手机感知能力。

我们需要做什么？怎么做？

我们的实现路径也非常简单：

构建 Android App 并内置 MCP Server；

接入

将预转换的 Qwen3-VL 模型文件部署至手机本地；

生成 MCP 鉴权信息，通过命令行在 Claude Code 中完成接入。

踩坑经验

一些备用踩坑经验, 避免大家重蹈覆辙：

输出截断

模型经常输出类似 {"summary": "

" 后就停止，tokens 计数仅 2 就直接结束。排查发现，预编译版 libMNN 在步进推理（step decode）时，一旦遇到 Markdown 代码块前缀（如

），会错误地触发假  结束符，导致 JNI 层误以为生成完成而提前终止。

解决方法：

在 C++ 层打了一个简单补丁——如果生成的内容长度不足 48 个字符，则忽略该  信号，强制继续生成。

模型卡在“verifying...”

模型加载和检测过程建议使用后台协程处理，避免在主线程中执行，否则界面会长时间无响应，显示卡死在“verifying...”状态。

：

居家隐私监控

：

将旧 Android 手机置于家中角落，所有图像和声音数据仅在本地处理，云端大模型只接收结构化的文字总结。无需购买昂贵的云端摄像头，同时完全保护家庭隐私。
