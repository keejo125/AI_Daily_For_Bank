# Agent 永不下班，我用 QCA 搭了个工作台

> 原文链接：https://mp.weixin.qq.com/s/UATtb53cGpm5gCEVsBN0pQ
> 公众号：Qoder

作者｜屈立威（安陈），Qoder Cloud Agents 产品经理

过去一年，一个问题变得越来越明显：

AI 已经很会完成一次任务了，但还不太会长期上班。

让 AI 写一篇文章、分析一份数据、做一个网页，第一次对话往往很惊艳。可当工作进入第二天、第二周，问题就来了：上次确认了什么？哪些方案已经否决？文件放在哪里？为什么当时这样决定？这一次又应该从哪里继续？

如果这些信息只存在于聊天记录里，每次新会话都像重新来了一位能力很强、但刚入职的同事。你重新介绍一遍背景，它完成一件事；换个窗口，又从自我介绍开始。

在我参与 Qoder Cloud Agents（下文简称 QCA）产品工作的过程中，越来越多持续执行的真实任务进入视野。QCA 已经把云端 Session、隔离沙箱和 GitHub 协作做得很好用：Agent 可以真正进入项目并持续完成工作。也正因为执行能力足够强，另一个问题才更值得被单独讨论——当一次 Session 结束以后，项目上下文怎样留给下一次执行？

为了把这个问题放进真实工作里验证，一个基于 QCA 的个人试验在工作之外逐渐成形，它就是

Operra

。它不是为了再造一个更聪明的聊天框，而是想回答一个更朴素的问题：

当 AI 长期进入日常工作，我们该怎样给它一间不会失忆、能够交接，而且始终属于用户自己的办公室？

Operra 最后选择的底座，是一个看起来甚至有点“复古”的东西：

文件系统。

一、长期工作的上下文，不应该只活在对话里

今天谈 Agent，我们很容易先聊模型、Prompt、工具调用和编排。但真正要把事情做完的人，通常只关心几个问题：

我要的结果是什么？

现在做到哪里了？

有什么需要我做决定？

最终成果在哪里？

下一次能不能接着做，而不是从头再来？

现实中的公司不会要求老板每天重新向员工解释“公司是谁”。它有项目资料、工作规范、历史方案、会议纪要和交付文档。员工会变，工具会换，但组织记忆会留下来。

可今天使用 AI 的主流方式，仍然以“会话”为中心：临时塞入一批上下文，希望模型在有限时间内理解整个项目。

这很适合问答，却不完全适合长期工作。因为办公不是一连串互不相关的问题，而是一段持续积累的历史：今天的决定会成为明天的约束，这次的修改会影响下次的判断，一个项目也会逐渐形成自己的语言、偏好和方法。

所以在 Operra 里，会话没有被当作项目本身。会话只是一次协作，

项目才是持续存在的工作空间。

这并不是一个从零开始的新概念。在 Coding Agent 领域，把项目规则、执行计划和可复用技能放进仓库，已经越来越常见。Operra 真正想验证的是另一件事：

这套在代码项目中有效的方法，能不能走出 IDE，进入更一般的办公任务；又能不能被包装得足够简单，让不懂 Git 的用户也能使用？

二、文件系统怎样成为“可恢复的上下文”

“File-System-Based”听起来很技术，其实它非常生活化。

想象一家公司搬进了一间办公室：文件夹是项目和部门，文档是共识与规则，任务摘要是交接单，版本历史是账本，入口处的索引则告诉新同事先看什么、当前做到哪里、下一步去哪找。

这就是本文所说的 File-System-Based 办公：

工作的目标、上下文、经验和成果，不只存在某个 AI 的脑子或某次对话里，而是沉淀成用户真正拥有的文件。

在 Operra 中，每个项目对应一个用户自己的 Git 仓库。里面不只有最终产物，也包括项目为什么存在、有哪些约束、最近发生了什么、过去学到了什么，以及每个任务做过什么。

这些文件不是把同一件事反复抄写，而是一套分工明确的上下文恢复机制。

新的 Agent 进入项目，会先从

AGENTS.md

找到阅读顺序；从

INDEX.md

知道资料在哪里；从

SPEC.md

理解目标、边界和执行规范；从

JOURNAL.md

了解近期变化；从

MEMORY.md

继承长期经验；最后带着当前 Task 的 brief，按需进入相关文件工作。

完成后，它再更新成果、任务交接和必要的日志。稳定且经过验证的经验才进入 MEMORY，项目方向或规范的变化则回到 SPEC，Git 继续记录每次变化的版本证据。

所以这里所说的“上下文可恢复”，并不是某个模型能记住无限长的聊天，而是：

在这些文件得到持续维护的前提下，即使会话结束、Agent 更换、电脑重启，下一位执行者仍有清楚的路径重建工作所需的上下文。

严格来说，文件系统并不等于模型此刻看到的 Context。模型的上下文窗口依然有限，也不应该在每次任务开始时把整个仓库全部塞进去。文件系统更像是

上下文的持久事实层：

它负责让信息留下来；入口索引、检索、摘要和任务范围负责决定这一次应该读取什么。

文件写下来，只完成了一半

文件也会过时、冲突和膨胀。仓库里多几个 Markdown，并不会自动得到高质量的长期上下文。至少还需要三条纪律：

同一事实有明确归属。

目标和规则回到 SPEC，近期变化进入 JOURNAL，经过验证的经验才进入 MEMORY，避免同一个结论在多处各写一版；

信息有生命周期。

临时过程需要归档，错误经验需要纠正，长期记忆也需要定期审查和遗忘；

读取和写回可以验证。

Agent 通过短入口找到相关资料，而不是把所有文件一次性读完；重要变化再由测试、版本历史或人工审查确认。

文件和 Memory 也不是对立面。文件适合承载显式、可审阅、需要长期归属的事实与成果；Memory 系统更擅长从执行中提炼经验、在合适的时刻检索，并清理已经过时的内容。两者解决的是上下文的不同层次。

2025 年初，我在设计另一个产品时，团队曾把一个“文件目录动态挂载”功能命名为

Context

，还引起过不少用户误会——大家理解的 Context 通常是一段 Prompt 或会话历史，而这里说的却是一个真实目录。

现在回头看，那个名字似乎并没有叫错，只是当时这件事还没有完全发生。

Prompt 是这一次要说的话，文件系统保存的是这份工作一路走来的历史。

三、为什么是 Git，而不是共享云盘？

说到这里，一个很自然的问题是：既然核心是文件，为什么要放进 Git，而不是一个大家都能访问的共享云盘？

共享云盘当然很有价值。它擅长保存和同步文件，回答的是：

最新版文件在哪里？

但长期工作的 Agent 还要回答另外几个问题：这个版本是怎样形成的？哪些修改共同属于同一个 Task？两个 Agent 同时工作时怎样避免互相覆盖？一项修改应该直接生效，还是先交给人检查？出了问题，又该回到哪个确定的状态？

这正是 Git 比“共享存储”多出来的部分。它不只保存文件，还保存项目演进的秩序。

第一，是

连续而准确的历史

。一个 Task 可能同时更新执行规范、工作日志、长期经验和最终报告。Git 可以把这些变化收在同一个 commit 里，成为一次完整的工作检查点。

第二，是

人和多个 Agent 之间清楚的协作边界

。每项工作可以先进入独立分支或隔离工作区。Git 不会消灭冲突，但能让冲突被明确暴露，而不是安静地覆盖；Pull Request 则让人的审查继续留在关键位置。

第三，是

一条天然的自动化接口

。QCA、其他 Agent、IDE 和人都能理解同一组基本动作：clone、修改、commit、push。一次 push 又可以触发 GitHub Actions，让测试、检查和后续流程自动发生。

选择 Git，并不是因为“Git 比共享云盘更高级”，而是二者解决的问题不同：

共享云盘管理文件的当前位置，Git 管理工作的演进。

当然，也不是所有东西都应该进入 Git。报告、规范、日志、任务摘要、代码和小型成果适合放在仓库里；视频、模型等大文件更适合对象存储或 Release，仓库只保存说明、地址和校验信息。

同样，File-System-Based 也不意味着把所有办公数据复制进仓库。日历、审批、CRM、在线表格等持续变化的数据，原系统仍然应该是事实来源；仓库更适合保存项目目标、决策、任务交接、稳定知识、成果，以及指向这些业务系统的引用。

Git 的可追溯性也是一把双刃剑：一旦把 Token、密码或敏感数据提交进历史，事后删除并不简单。因此凭证不应该进入项目仓库，敏感资料也需要单独的权限、存储和保留策略。在 Operra 里，凭证留在本机安全存储中，仓库只保存不含密钥的项目事实与成果。

四、文件是地基，用户不应该每天参观地下室

File-System-Based 很容易让人产生另一个误会：用户以后是不是要天天维护目录、写配置、手动整理上下文？

答案恰恰相反：

底层应该开放，表层应该简单。

文件系统是地基，不是要用户每天参观的地下室。

在 Operra 里，用户面对的核心单位是

Task（任务）

，不是文件、Agent 或工作流。

你可以直接对 AI 运营官 Oopz 说：

帮我比较三个客服方案，给出推荐，最后整理成一份可以发给团队的报告。

Operra 把这句话组织成一个有目标、有上下文、有成果归属的 Task；QCA 在云端执行；报告和过程中形成的有效经验，再回到项目仓库。

用户日常只需要说清楚结果、在真正需要判断时介入，然后拿走成果或继续修改。默认界面先展示结果；只有想了解原因、纠偏或排查问题时，才继续展开计划、过程和原始执行记录。

这背后有一个产品判断：

用户想管理工作，不想管理 Agent。

Agent 可以替换，Task 会持续；会话可以结束，项目仍然保留。

Operra 里其实有两层 QCA

Operra 里不是只有一个 QCA，而是让两层 QCA 分别承担两种工作。

第一层是

Oopz，也就是管家 QCA

。它离用户最近，负责理解需求、查看项目、创建 Task、确认关键操作，再把工作安排到合适的位置。它掌握整个工作台的概要，但不亲自生产最终成果。

第二层是

任务 QCA Agent

。当一个 Task 开始执行时，Operra 会为它启动独立的 Session 和执行环境。这个 Agent 进入对应的 GitHub 项目仓库，读取入口、SPEC 和任务 brief，完成修改、测试与交付，再把结果写回项目。

一个更像接需求、做协调的运营管家，另一个像进入具体项目执行的同事。两者都使用 QCA，但角色、Session 和环境彼此分开。这样，管家不会被某个漫长任务淹没，任务 Agent 也不需要知道整个工作台的所有信息。

云端的 QCA，为什么能在本地“动手”？

QCA 的 Environment 既可以是托管的云端沙箱，也可以是用户自己运行的 Self-hosted Environment。

可以把这种架构理解成一种

脑手分离

：云端负责保持 Session、理解意图、规划和调度；真正执行动作的“手”可以位于云端沙箱，也可以位于用户本地。本地 worker 主动从 QCA 的 work queue 领取任务，在本地沙箱和权限边界内执行，再把结果送回 Session。

因此，更准确的说法不是“云端接管了用户的电脑”，而是“用户在本地提供了一双受控的手，让云端大脑把工作交给它”。数据在哪里、能调用哪些工具、哪些动作必须由人确认，仍然由本地环境决定。

还需要强调一点：写在 SPEC 里的规则可以指导 Agent，但不能代替真正的安全边界。删除数据、对外发布、花钱等高风险动作，仍然要由工具权限、运行时策略或人的确认来约束。

Operra 的 Oopz 能查看本地工作台、创建项目和 Task，靠的就是这条链路。Operra 只开放少量明确的工作台工具，而不是整台电脑的任意权限；真正的任务交付则交给第二层任务 Agent，在独立沙箱和 GitHub 仓库中完成。

五、QCA 如何让这个项目持续迭代

Operra 本身也是一个 File-System-Based 项目。仓库里的

AGENTS.md

规定 Agent 的阅读入口，

STATE.md

记录当前进度和下一步，SPEC 与设计文档保存产品边界和执行规范，Git 历史留下每次迭代的证据。

在此基础上，开发流程被接成了一个循环：

把一次修改写成明确任务，或从反馈中整理出一个低风险工作项；

QCA 创建云端 Session，在隔离沙箱中取得 Operra 仓库，先按入口读取当前状态和相关 SPEC；

Agent 修改代码与文档，并在沙箱里运行测试；

变化提交后，GitHub Actions 独立运行完整 CI，不把 Agent 自己声称“测过了”直接当成结论；

失败证据进入下一轮修改；通过后形成可审查的变化，最终合并和发布仍由人决定；

结果、决策和下一步重新写回仓库，供下一次 Session 接手。

这里最关键的，不是让一个 Session 永远不结束，而是每一轮结束后，项目都留下足够清楚的文件和版本记录。下一轮即使换了新的 Session，也可以从仓库恢复上下文，再接着往前走。

QCA 的 Session 和沙箱运行在云端，GitHub Actions 也在云端执行，周期工作还可以通过 QCA 的云端能力定时触发。因此即使 Operra 没有打开、本地电脑正在休眠，已经安排好的工作仍然可以继续。

这里的“7×24 小时”并不是让某一个 Session 永远运行，也不是让一台个人电脑永远亮着，而是把执行、存储、验证和下一次交接都放进了可以持续接力的系统里。

六、仍然只是一个个人试验

Operra 没有证明某种未来办公方式。它只是一个还很粗糙的个人试验，许多代码、文档和界面也是 Agent 按着仓库规范一点点做出来的。

但它已经能被频繁用于建立项目、派发任务、修改成果和运行周期工作。这里需要说清楚：Operra 之所以已经可用，主要不是这层封装有多成熟，而是

QCA 本来就很好用

。持续 Session、隔离沙箱、长任务执行、Self-hosted Environment 和 GitHub 项目协作，这些真正让任务得以执行和接力的能力都来自 QCA。Operra 只是享受了这些能力加成，再用项目、Task、成果展示和人工确认把它们组织成一层个人工作台。

这个试验只想验证一个很小的问题：

当 Agent 已经足够能干以后，用文件系统和 Git 保存项目上下文，能不能让它更自然地参与长期工作？

目前的使用体验偏向“可以”，但这只是一个正在使用中的样本，不是结论。
