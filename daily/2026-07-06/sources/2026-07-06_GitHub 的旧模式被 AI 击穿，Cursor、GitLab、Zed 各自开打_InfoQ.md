# GitHub 的旧模式被 AI 击穿，Cursor、GitLab、Zed 各自开打

> 原文链接：https://mp.weixin.qq.com/s/RwSxpir9-dVJB3ssjs2mQg
> 公众号：InfoQ

作者 | Paul Sawers

译者 | 平川

策划 | Tina

本文最初发布于 TheNewStack 博客。

图片来自 Unsplash，由 Egor Komarov 提供

2026 年 6 月 16 日，人工智能界最大的新闻无疑是，埃隆·马斯克的 SpaceX 同意以价值 600 亿美元的全股票交易收购 AI 编码初创公司 Cursor。

但就在同一天，Cursor 在旧金山举办了一场 仅限受邀请的开发人员参加的会议。Tomas Reimers宣布 了一个可能对开发工具行业同样具有重大影响的新项目。

这个项目的名称是 Origin，是 一个与 Git 兼容的代码托管平台。 该项目从一开始就是为由智能代理而非人类完成大部分工作的世界而设计的。

值得一提的是，Tomas 是 Graphite （一家代码审查初创公司）的联合创始人。在 12 月份的时候，Cursor 透露他们正在收购这家公司（显然，这笔交易在 1 月完成了）。当时，有一些评论员注意到了这笔交易对 GitHub 的影响，其中包括 Gergely Orosz。

Orosz 是《实用工程师通讯》的作者，也是 Graphite 的投资者。他在 LinkedIn 上写道：“我告诉你们：GitHub 的最大竞争对手可能很快就会是 Cursor 。在我看来，Graphite 是最好的 AI 代码审查 + 堆叠差异 + PR 工作流产品。GitHub 已经在追赶 Cursor/Graphite。”

简而言之，Graphite 已经构建好了 GitHub 急于复制的工作流工具。而且，在 Cursor 的资源支持下，它们之间的差距只会越来越大。现在，有了 SpaceX 这个价值 2.5 万亿美元的公司的支持，事情可能即将变得非常有趣。

Origin 的源起

在旧金山的舞台上，在宣布 Origin 之前，Reimers 指出，Graphite 的客户群包括 Shopify、Snowflake、Notion 和 Figma。他想借此证据，问题在 Origin 存在之前就已经出现。

Reimers 说，“在过去的几年里，我们注意到了这些公司采用 AI 工具的趋势。他们依赖的工具开始变得不可靠。那是因为在过去的几年里，AI 工具彻底改变了我们的行业。它使每个开发人员的生产力都提升了 10 到 100 倍，但这种变化需要完全不同的工具。那就是为什么，当我们被 Cursor 收购时，我们加快了我们最有雄心的项目，从头开始重新构建了那个工具。”

在 SpaceX 登陆公开市场、一夜之间跻身全球最有价值公司之列，以及斥资 600 亿美元收购一家成立仅四年的初创公司等种种喧嚣之中，人们很容易理解为什么 Origin 会悄然无声地被忽视。但它致力于解决的基础设施问题却是切实存在的。

GitHub，这个在世界上占据主导地位的代码托管平台，无疑正在经历一段艰难的时期。正如 The New Stack 在 6 月份报道的那样，该平台在过去的 12 个月里记录了数百起事件，他们在 努力跟上 AI 代理生成代码的节奏。该公司表示，他们现在每月处理大约 14 亿次提交（2025 年全年为 10 亿次），仅 AI 代理每月就生成超过 1700 万次拉取请求。

讽刺的是，GitHub 在 2021 年推出 Copilot 时帮助启动了 AI 编码时代，现在他们正在承受它的重压。对有些人而言，这种影响在他们的日常习惯中已经初现端倪。

最近，GitHub 前开发者关系总监 Brian Douglas成立 了自己的 AI 基础设施初创公司 Paper Compute。他告诉 The New Stack 说，转变已经在进行当中。

Douglas 说，“代理正在迅速扼杀人们参与开源的意愿。我很想知道， GitHub 的[月活跃用户] 数如今是个什么样子，因为我确信，有很多人选择在其他地方进行代码审查，或者仅与代理合作完成最后的工作，而且这种情况目前正处于历史最高水平。”

值得一提的是，Douglas 说，他自己也是他们中的一员，他现在大部分的审查和 PR 工作都是直接在 AI 编码工具中完成的。

他还说，“作为一个 GitHub 重度用户，我发现自己使用它的频率越来越低，我现在更多地依赖 Claude 和 Codex 进行代码审查和 PR 互动”。

后 GitHub 世界？

在按照计划于秋季正式发布之前，Origin 目前仍处于仅限候补名单的模式，而根据出席 Compile 大会的人士所透露的细节，已经足以勾勒出该项目的雄心壮志。开发大使兼独立评论员 Shawn Wang Yuexian（网名 swyx）将其 描述 为“一个期待已久的 Git 竞争对手，不仅能够适应代理工作负载的可扩展性，还支持通过 API 和 MCP 进行扩展，并内置了合并冲突和 CI 失败的代理解决机制。”

无论 Origin 在正式发布时呈现何种面貌，可以肯定的是，人们对于打破现状、寻求替代方案的渴望正在日益增强。自 2008 年 GitHub 将“拉取请求”模式推广开来之后，软件开发领域已经发生了巨大的变化——Douglas 将这一功能称为“有史以来最棒的功能”。但“拉取请求”的设计初衷，是针对人类有意识地编写和审查代码、一次只进行一项修改的开发环境。而这样的时代正在迅速远去。

Douglas 说，“目前，项目创建的速度已经让 GitHub 不堪重负，工程师们根本无暇查看代码。所以如果目标是将其放在云端，让代理管理代码，我认为这绝对是一个开启颠覆性变革的机会。”

因此，随着 AI 代理以人类审查者无法跟上的速度推送代码，拉取请求有沦为走过场之虞——成为一个需要打勾的选项，而非有实质意义的质量关卡。这引发了一个更深层次的问题：整个行业究竟应该如何衡量软件工作的价值。

对于 Douglas 来说，答案存在于另一种完全不同的计量单位中。在当前这个代理能在几秒内生成数千行代码的世界里，提交次数和代码行数，这些传统上用来衡量开发者产出的指标几乎已经无法提供有意义的信息。相比之下，Token 是与计算成本直接挂钩的，因此它能直接反映出实际付出的努力和创造的价值。这种重新定义工作量计量的方式非常适合 Cursor。

Douglas 说，“Token 是比提交更好的度量标准。它们与花费的美元对齐，与工作的努力相关。以前，我们假装代码行数是度量标准，事实证明，这是不正确的。但 Token 加上代理会话就等于客户价值。而 Cursor 处于有利地位，能够占据协作栈中更深层的部分。”

不过，持有这种观点的并非只有 Cursor 一家公司。目前，一系列旨在为“代理时代”重建该基础设施的尝试正涌现出。

在 6 月 10 日于伦敦举行的 Transcend 大会 上，GitLab 宣布 了一个名为 Next Generation Source Code Management 的工具的私有测试版本，他们内部称之为 Switch 项目。 GitLab 首席产品和营销官 Manav Khurana 在舞台上 宣布 了这个新的后端。它保持了 Git 协议的完整性，但完全重新设计了底层架构，允许代理在服务器端查询存储库而不是完整地克隆它们。

GitLab 表示，它使得每个代理的任务执行速度提高了达 50 倍，同时消耗的 Token 量减少了 3 倍。值得注意的是，Anthropic 是该项目的一个设计合作伙伴。

Khurana 说，“全球最受欢迎的 Git 平台正因不堪重负而陷入瘫痪，这不仅是因为你的团队在克隆、分支和合并代码，还因为数十个——在某些情况下甚至是数百个——代理在同时运行，给这些系统带来了巨大的压力”。

在 GitLab 的 Transcend 公告发布后第二天，Zed 联合创始人 Nathan Sobo 公布 了 DeltaDB 的细节，这个项目 首次被该公司提及 是在去年秋天。与 Origin 或 Switch 项目相比，DeltaDB 是一个更激进的方案，它完全用连续的细粒度增量流替换了 Git 基于提交的模型——每个代理执行的操作都会直接链接到产生它的对话。Sobo 确认，测试版几周之后就可以推出。

与此同时，HashiCorp 联合创始人 Mitchell Hashimoto 已经看到了这一点。早在去年 12 月，他就在 X 上写道，“AI 公司成为 GitHub 的速度有望比 GitHub 成为 AI 公司的速度更快。”

当 Origin 于本周宣布时，他转发了自己的推文，并写道，“Cursor 今天宣布了 Origin。还会有更多。”

Hashimoto 是另一个面向代理的代码托管初创公司 East River Source Control（ERSC）的 投资者。 该公司正在 构建 一个兼容 Git 且每秒可以处理数千次提交的平台。

对于 Douglas 来说，各方竞相从头开始重建版本控制系统的努力最终趋于相同，这并不令人感到特别意外。他指出，在过去的一年中，开发者沙箱（即编写和测试代码的环境）领域也曾出现过类似的动态—— Docker、Cloudflare 和 Vercel 等公司纷纷进军该领域，因为开发者们正将时间投入到那里。

现在，同样的吸引力正在影响着版本控制。开发者的工作方式已经发生了根本性的变化。以前，他们直接在编辑器中编写代码；现在，许多人花时间指导 AI 代理为他们编写代码。IDE 不再主要是一个打字的地方；它越来越多地成为一个观看、审查和进行指导的地方。

Douglas 说，“现在，IDE 正面临着这样一种困境：开发者已经进化到使用基础模型框架来编写代码，因此它们需要重新定位，变成一种打开即可观察代理编写代码的工具。我认为，所有参与这个故事的人都有机会，我们需要重新思考我们的基础设施，为这一点做好准备。”

然而，支撑这一切的是商业现实。Cursor 早已经为此布局多时：该公司于 2025 年推出了自有的 第一方编码模型 Composer，并于今年 5 月 对 Composer 2.5 进行了迭代升级，实现了更经济的内部推理，而不是完全依赖成本高昂的 Anthropic 和 OpenAI API 调用。在执行同等任务时，Composer 2.5 的成本仅为 Claude Opus 的零头，在输出 Token 的数量上，两者差距甚至高达十倍。换言之，拥有模型才是让拥有整个技术栈其余部分成为可能的关键。

Douglas 说，“很明显，你不能仅仅插入一个 OpenAI 密钥，就期望在这个市场上实现超速增长或长期发展。相反，要想胜出，你需要拥有自己的模型。”

SpaceX 的实力究竟会加速这一目标的实现，还是会使其变得更加复杂，还有待观察。但那些押注于下一代软件开发的公司，并不打算坐等 GitHub 迎头赶上。
