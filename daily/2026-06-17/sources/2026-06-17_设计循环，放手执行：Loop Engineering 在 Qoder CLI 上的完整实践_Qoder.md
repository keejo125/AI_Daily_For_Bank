# 设计循环，放手执行：Loop Engineering 在 Qoder CLI 上的完整实践

> 原文链接：https://mp.weixin.qq.com/s/RAbsjNSg7jpmd4HJ3CRw2g
> 公众号：Qoder

从 Prompt Engineering 到 Loop Engineering

AI 工程的新概念来得很快。

半年前大家还在聊 Prompt Engineering，接着 Context Engineering 火了一阵，Harness Engineering 刚刚进入视野，2026 年 6 月又出现了 Loop Engineering。

为什么这个词突然被广泛讨论？因为 AI 编程走到了一个新的阶段。早期我们关心的是"一次调用怎么出好结果"，现在的问题变成了——Agent 需要读代码、调工具、看报错、改文件、跑测试、再根据结果决定下一步。这整个过程本质上是一个循环：

感知环境 → 判断策略 → 采取行动 → 接收反馈 → 调整方向 → 继续执行

问题在于，今天大多数人还是手动推动这个循环的每一步。改完代码你得说"跑下测试"，看完结果你得说"这个没过，再试试"，一来一回几轮下来，半小时过去了，你全程的作用就是盯着屏幕按回车。

Loop Engineering 的核心主张很简单：这个循环应该由 agent 自己跑完，人只负责定义规则和验收最终结果。

过去几年，围绕 AI 编程的工程方法论一路演进。每一代都在解决上一代留下的瓶颈，视野逐步从"单次调用"拉高到"系统持续运行"：

每一代的主题都是同一件事：

把人从执行层抽出来，放到设计层。

Loop Engineering 是目前这条路上走得最远的一步。核心就一个转变：

人从 loop 内部退到 loop 外部。

以前你在循环里面，每一轮都得你推；现在你站在循环外面，只在两个点介入——

入口（定义规则）和出口（验收结果）

，中间全部无人值守。

下面这张图展示了这个空间关系——人在外圈定义规则，内圈是 agent 自动运转的闭环：

再看两种模式的对比——左侧人在 loop 内部，右侧人在 loop 外部：

左边的循环里人是每一轮的推动者，人离开就停。右边的循环人只触碰两次：入口定义规则，出口验收结果。中间可以跨小时、跨天持续运行。

Loop Engineering 的构成

显然人要退到 loop 外面，但要让循环自己能跑起来，它内部得具备哪些能力？根据社区实践的归纳，一个可用的 Loop Engineering 系统由五块拼图构成——调度负责"什么时候开始、什么时候停"，知识负责"怎么干活"，连接负责"能触达哪些系统"，协作负责"谁执行谁验证"，记忆负责"下次从哪接着"：

五块拼图协同工作，构成完整的 Loop Engineering。按职责可以分为两侧：

触发侧

（调度）负责让 loop 不依赖人工就能启动和推进，执行侧（知识 + 连接 + 协作 + 记忆）负责让 loop 跑得稳、跑得久。

和 CronCreate 负责触发，

/goal

负责退出条件，Skills 负责知识，MCP 负责连接，Subagent 负责协作（配合

/verify

验证输出、Worktree 与 Remote Agent 支撑多任务并行），记忆系统负责状态。不需要自己拼脚本搭架子，直接用就行。

接下来看看每块拼图在 Qoder CLI 里是怎么落地的

① 调度：/loop 与 /goal

调度解决的是循环的两个端点问题：

什么时候开始和什么时候停

。没有触发就没有 loop，没有退出条件 loop 就会永远跑下去。

Qoder CLI 里触发 agent 有两种方式：

方式一：

/loop

指令

最简单的用法。给一句 prompt 挂上定时：

/loop 5m /babysit-prs

/loop 30m check the deploy

/loop 1h /standup 1

/loop check the deploy every 20m

格式是

/loop

[间隔] <prompt>

，不写间隔默认 10 分钟。注册后立即执行一次，然后按间隔循环。

方式二

：CronCreate 工具

需要更精细的调度就直接告诉 agent：

帮我设个定时任务，工作日每天早上 9 点检查 PR 状态

agent 会调用 CronCreate，写入标准五字段 cron 表达式。搭配 CronDelete 取消、CronList 查看。

定时任务默认 7 天过期。需要长期跑的任务，CronCreate 支持

durable: true

参数让任务重启后依然存在。

什么时候停：

/goal

有些任务的关键不在于"什么时候跑"，而在于"做到什么程度算完"。

/goal

就是干这个的：

/goal

set

确保所有测试通过且 lint 零错误

给一个目标，agent 自己往前走，做一轮检查一轮，没达标就继续，达标了就停。全程不需要人介入。可以随时

/goal pause

暂停、

/goal resume

继续、

/goal clear

放弃，也可以加

--max-turns 15

设硬上限兜底。

触发和退出条件是同一个硬币的两面：一个决定 loop 什么时候转起来，一个决定它什么时候可以停。

② 知识：Skills

Loop 里 agent 每一轮都在做决策：该构建还是该测试？用哪个命令？哪些文件不能碰？如果这些信息每次都靠 agent 自己猜，loop 的质量就不可控。

Skills 的定位是 loop 的「执行手册」。每个 skill 是一份

.qoder/skills/<name>/SKILL.md

文件，内容告诉 agent：这个项目怎么构建、测试怎么跑、什么场景用什么工具、哪些路径有特殊约定。agent 在执行时自动加载匹配的 skill，不需要你每轮重复交代。

写 skill 并不难。Qoder CLI 提供了

/create-skill

命令帮你快速沉淀——你可以把一次成功的对话过程直接转化成可复用的 skill，也可以手动写。关键的设计思路是：

skill 定义的是 loop 的执行路径

，包括什么时候构建、什么时候测试、什么条件下调用哪个 subagent。

内置 skill 举几个例子：

/run

— 启动应用并在运行状态下验证代码改动

/verify

— 通过应用的真实接口检查改动是否生效

/create-skill

— 帮你把对话经验快速固化为新的 skill

一个设计得好的 skill 体系，能让 loop 的每一轮都走在正确的路上，而不是每次启动都在重新摸索。

③ 连接：MCP Server

Loop Engineering 的目标是让 agent 跑完「整个工作流」，而非只完成工作流中的「写代码」环节。要做到这一点，agent 的行动半径必须超出代码仓库。

在 Qoder CLI 里，MCP Server 负责把外部系统接入 agent 的决策范围。接上之后，loop 的一轮可以覆盖从发现问题到解决问题的全链路：

通过 GitHub MCP 查 CI 状态、创建 PR、回复 review comment

通过 Slack MCP 在修复完成后通知团队

通过 Linear MCP 自动更新工单状态、关联 commit

通过数据库 MCP 查生产数据验证修复效果

举个具体的例子：一个 CI 守夜人 loop 的一轮可能是——GitHub MCP 发现 CI 红了 → agent 读代码定位原因 → 修复 → 跑测试 → GitHub MCP 提 PR → Slack MCP 通知 channel「CI 已修复，PR

#43

待 review」。整条链路在一轮内闭环，中间没有人需要切窗口做任何操作。

MCP 在 Loop Engineering 里的角色是 agent 的「手」——loop 的大脑做决策，MCP 负责把决策变成真实动作。

④ 协作：Subagent

一个 agent 自己干活然后自己判断"做得行不行"，很容易陷入“自己批自己作业”的困境。Loop Engineering 里解决这个问题的方式是“maker/checker 分离”——写代码的和审代码的必须是不同的 agent，有不同的 prompt、不同的工具集、甚至不同的模型。

Qoder CLI 通过自定义 subagent 实现这一点。你在

.qoder/agents/

目录下定义一个

.md

文件，声明它的职责、可用工具和模型：

---

name: code-reviewer

description: 审查代码变更，找逻辑错误和安全漏洞

tools:

- Read

- Grep

- Glob

model: gemini-2.5-pro

---

你是代码审查者。检查最近的变更，重点关注逻辑错误和安全风险。

背后的设计哲学是：

在一个自主循环里，执行者和验证者必须分离。

同一个 agent 写了代码再自己说"没问题"，和一个人自己出题自己答没有区别。拆成两个 agent 之后，reviewer 带着完全不同的上下文和标准审视同一份代码——它不知道"写的时候在想什么"，只看结果是否符合规范。

更进一步，你可以定义多个不同职责的 subagent 构成一个团队：一个负责实现、一个负责测试、一个负责安全审查。每个 agent 有自己的 prompt、工具集和模型配置，各司其职。这种设计让 loop 的每一步都有独立的质量把关，而不是全靠一个 agent 的自觉。

⑤ 记忆：状态持久化

Loop 的一个核心挑战是：agent 每次启动都是"失忆"的。上一轮做到哪了、试过什么方案、哪些路走不通——如果这些信息不能跨轮次保留，每一轮都等于从零开始，loop 就无法真正"累积进展"。

解决方案是把状态存在 agent 外部。Qoder CLI 的记忆系统会自动持久化 agent 在工作中形成的关键信息——项目规范、历史决策、任务进度——下次启动时自动加载。

另一种更直观的方式是利用文件本身作为状态载体。比如长线特性开发里的 spec.md：每完成一个 TODO 就打勾，agent 下一轮启动时读到这个文件就知道还剩哪些没做。git log、TODO 文件、进度文档都可以充当 loop 的跨 session 状态。

你也可以通过 MCP 接入 Linear 或 Notion 之类的项目管理工具来管理 loop 的状态——任务的完成度、阻塞点、待决策事项都可以写入外部看板，人在任何时候打开看板就能知道 loop 进展到哪了。这比翻聊天记录或 git log 更直观。

组合起来：两个完整场景

上面的五块拼图单看都是独立能力，Loop Engineering 的价值在于把它们组合成一条自动运转的流水线。

场景一：日常运维自动化（CI / PR / 巡检）

这类场景的共同特点：有固定节奏、退出条件明确、人是瓶颈。

/loop 30m 检查 CI，如果有失败的 job 就分析原因并尝试修复

/goal

set

确保 PR

#42 CI 全绿、所有 review comment 都 addressed

帮我设个定时任务，工作日早上 9 点扫描所有项目的 lint 错误和过期依赖

涉及的构件：调度（

/loop

或

/goal

）→ 知识（项目构建 skill）→ 连接（MCP 查 CI、回复 review、发 Slack）→ 协作（subagent 审查修复）→ 记忆（下次启动知道上轮修了什么）。

你做了什么？写了一次 skill、配了一次 subagent、敲了一行命令。之后它自己盯着——半夜 CI 红了自己修，review comment 来了自己回，巡检报告每天 9 点自动生成。

场景二：长线特性开发

一个跨好几天的大特性，拆好 spec，让 agent 自主推进：

/goal

set

按照 spec.md 完成所有 TODO 项，每完成一个打勾 --max-turns 50

涉及的构件：调度（

/goal

定义终点）→ 知识（spec.md 作为执行路径）→ 协作（实现和验证由不同 subagent）→ 记忆（spec.md 里的打勾就是跨 session 状态）。

这里没有定时触发——不需要。关键是退出条件（所有 TODO 打完勾）和硬上限（50 turns 兜底）。agent 跨天自主推进，你定期看 spec.md 的打勾进度就行。

放手的前提

前面聊的都是怎么让 loop 跑起来。但跑起来只是第一步，真正的挑战在于：怎么让它持续跑得可控。

Loop Engineering 的收益很诱人——但任何把执行权交出去的行为都伴随代价。认清代价的人才能真正用好收益。这一节把两者摆在一起看。

收益：人退到循环外部后发生了什么

最直接的变化是自动化程度的跳跃。传统模式下人是循环的一部分，每轮都得在场；Loop Engineering 下 agent 可以 7×24 跑，凌晨 CI 红了自己修，周末长线任务继续推。人的精力从「盯盘」释放出来，投入到规则设计和结果验收上。

产出效率也是结构性的提升。一个 loop 一天可以自主迭代几十轮，同样的工作等人推每一步可能拖一整周。每一轮执行标准一致，不会因为今天精力不好就跳过某一步。而且你可以同时跑多个 loop——CI 监控、日报巡检、长线 feature 并行推进，人同时只能盯一件事。

代价：必须提前想清楚的三件事

第一是安全边界。Loop 可以自主写代码、调工具、改文件，但有些操作绝对不能让它自主决定——生产环境发布、支付逻辑变更、不可逆的数据操作。这些场景必须在设计 loop 时就划出禁区，loop 跑到这里应该停下来等人确认。没有边界的自主执行就是失控。

第二是 token 成本。每一轮循环都是 API 调用，30 分钟间隔的 loop 一天跑 48 轮，复杂任务每轮可能几万 token。挂 loop 前先做一次算术：单轮 token × 预期轮次 × 天数。

第三是理解债。agent 跑得越快你越容易跟不上它改了什么。loop 帮你写了 500 行代码，如果你不读，出了问题也定位不了。定期抽查产出、读 diff、验证结果，是使用 Loop Engineering 的必要纪律。

护栏：Qoder CLI 的安全设计

loop 的自主程度越高，保护机制越重要。好的 loop 设计应该覆盖以下几个维度：

什么时候用，什么时候不用

三个条件都满足再上 Loop Engineering：任务有重复节奏或明确终点、退出条件可以自动验证、人在中间是瓶颈。不满足的时候，直接 prompt 就行，别勉强挂 loop。

Loop Engineering 代表的是 AI 编程工具的一次范式跳跃：从「你和 agent 对话」到「你设计系统让 agent 自己跑」。人的精力从每一轮的乒乓对话中释放出来，投入到更高层的规则设计和结果验收上。
